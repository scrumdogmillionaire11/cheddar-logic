# WI-0620: Decompose db.js into domain-scoped modules

**ID**: WI-0620
**Goal**: Break `packages/data/src/db.js` (4,254 lines) into domain-scoped modules that share a single connection singleton, reducing diff conflicts and enabling isolated testing.

**Background**:
`db.js` is a monolithic module combining connection lifecycle, migrations, health checks, backup utilities, lock management, and 40+ data-access functions spanning odds, games, players, telemetry, model outputs, job runs, and settlements. Any change to any domain must touch this file. Diff conflicts are frequent. No domain is testable in isolation.

**Target structure**:
```
packages/data/src/
  db/
    connection.js       — SQLite open, WAL mode, pragma setup
    migrations.js       — runMigrations(), migration file discovery
    health.js           — health check, backup utilities, lock management
    odds.js             — insertOddsSnapshot, getOddsSnapshot, etc.
    games.js            — insertGame, getGame, updateGameState, etc.
    players.js          — insertPlayer, getPlayerPropLines, etc.
    telemetry.js        — insertTelemetry, getCalibration, etc.
    models.js           — insertModelOutput, getModelOutput, etc.
    settlements.js      — insertCardResult, getCardResults, etc.
    index.js            — re-exports all public functions (backward compat shim)
```

**Out of scope**:
- Changing any function signatures or query logic
- Migrating to an ORM
- Changing the public API surface (index.js re-exports everything)

**Acceptance**:
- All existing callers continue to work via `require('packages/data/src/db')` without modification.
- Each domain module can be required independently in tests.
- `npm --prefix apps/worker test` passes.
- `npm --prefix packages/data test` passes.
