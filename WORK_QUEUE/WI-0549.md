# WI-0549: Segment Results by PLAY vs SLIGHT EDGE

**ID**: WI-0549
**Goal**: Segment `/results` outputs into explicit decision-tier buckets (`PLAY` and `SLIGHT EDGE`) so performance is visible by actionable tier.

**Scope**:

- `WORK_QUEUE/WI-0549.md`
- `web/src/app/api/results/route.ts`
- `web/src/app/results/page.tsx`
- `web/src/__tests__/api-results-flags.test.js`
- `web/src/__tests__/ui-results-smoke.test.js`
- `web/src/__tests__/api-results-decision-segmentation.test.js`

**Out of scope**:

- Worker settlement logic or grading math
- DB schema/migration changes
- Changes to non-results pages/routes

**Acceptance**:

- `/api/results` returns decision-tier segmentation metadata that cleanly separates `PLAY` and `LEAN` rows (displayed as `SLIGHT EDGE`) from the same filtered/deduped result set.
- `/results` presents separate decision-tier sections for `PLAY` and `SLIGHT EDGE` with tier-level settled count, win/loss/push, win rate, and ROI.
- Totals across decision tiers reconcile to the overall summary for the same response payload (no double-counting, no dropped rows).
- Existing filter behavior (`sport`, `card_category`, `market`, `min_confidence`, `dedupe`) applies consistently to both decision tiers.
- Regression tests cover API decision-tier segmentation shape and UI payload assumptions.
