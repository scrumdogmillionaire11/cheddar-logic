# WI-0786: Increase CF convergence polling from 12×5s to 18×5s per route and extend post-purge wait

| Field              | Value |
|--------------------|-------|
| **ID**             | WI-0786 |
| **Goal**           | Increase the `retry_public_route_refs` attempt count from 12 to 18 (90s per route) and extend the post-CF-purge wait from 45s to 60s in `deploy-production.yml`, to reduce false-positive rollbacks caused by CF propagation exceeding the current 60s polling window |
| **Priority**       | Medium |
| **Owner agent**    | Unassigned |
| **Time window**    | TBD |
| **Coordination flag** | `needs-sync` — modifies `.github/workflows/deploy-production.yml` |

---

## Problem

From the deploy log, `retry_public_route_refs()` polls public routes to verify they serve the same asset hashes as origin. Current parameters:

```
attempts=12
delay=5
```

This gives **60 seconds per route** across 4 routes (/, /cards, /fpl, /results) before triggering `fail_with_rollback`. Total worst-case polling time: ~4 minutes.

Cloudflare cache propagation time is typically 30–60s but can reach 90–120s under high cache volume, zone-wide invalidation contention, or PoP-specific lag. When CF takes longer than 60s to converge, the deploy triggers a rollback of a valid, correctly-deployed build.

The post-purge wait is currently 45s (`sleep 45`) before polling begins. This leaves only ~15s of buffer if propagation takes the typical 45–60s.

---

## Scope

| File | Change |
|------|--------|
| `.github/workflows/deploy-production.yml` | In `retry_public_route_refs()`: change `attempts=12` → `attempts=18` (90s per route at 5s delay); change post-purge `sleep 45` → `sleep 60` |

## Out of Scope

- Changing the `retry_url` attempt/delay parameters (used for individual asset fetches, not route convergence)
- Changing the turbopack guard logic (WI-0636, already in place)
- Adding additional routes to the convergence check
- Setting CF propagation target to a specific SLA (no change to architecture)

---

## Rationale

At 18 × 5s = 90s per route, the total worst-case polling window across 4 routes is ~6 minutes — still well within the acceptable deploy window. The 60s post-purge wait ensures CF has had a full minute to begin propagating before the first probe fires.

This is a defensive tuning change. If CF propagation consistently exceeds 90s, a follow-on WI should investigate CF cache analytics.
