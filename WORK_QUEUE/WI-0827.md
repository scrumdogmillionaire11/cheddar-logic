# WI-0827: Feature Timestamp Audit — Enforce Strict Event-Time Data in All Model Inputs

| Field | Value |
|---|---|
| **ID** | WI-0827 |
| **Goal** | Audit all raw_data fields fed to models and assert that every pre-game feature was available before bet placement time. Add `available_at` metadata to the odds snapshot ingestion pipeline for the highest-leakage-risk fields. Block any feature whose `available_at` is after `bet_placed_at`. |
| **Priority** | High |
| **Depends on** | WI-0820 (model status gate; leakage check is an additional gate not a replacement) |
| **Coordination flag** | `needs-sync` — touches odds snapshot ingestion, possibly ESPN metrics enrichment |

---

## Background

The audit identified several features with potential forward-looking leakage risk:

1. **`umpire_factor`** — only clean if sourced from pre-game umpire assignment, not post-game box score
2. **`homeGoalieCertainty`** — only clean if sourced from pre-game lineup reports; post-game backfill contaminates backtests
3. **`homeGoalsForL5`** — should be L5 games *prior* to the current game; if the enrichment pipeline runs after game completion and includes today's game, it leaks
4. **`rolling_14d_wrc_plus_vs_hand`** — same issue; must be computed from games that closed before snapshot_time

Without `available_at` timestamps on these fields, there is no way to enforce or verify event-time ordering in backtests.

## Scope

**Files to change:**
- `apps/worker/src/jobs/run_pull_odds.js` (or equivalent ingestion job) — record `fetched_at` timestamp on each odds snapshot (already exists per AGENTS.md; confirm it's populated)
- ESPN metrics enrichment (wherever `homeGoalsForL5`, `wrc_plus_vs_hand`, goalie certainty are written into `raw_data`) — add `feature_timestamps` sub-object to `raw_data` with `available_at` per field

**Files to create:**
- `apps/worker/src/models/feature-time-guard.js` — `assertFeatureTimeliness(rawData, betPlacedAt)` — checks known high-risk fields against their `available_at` timestamps; returns violations list

**Out of scope:**
- Rebuilding historical snapshots with timestamps (future backtest work)
- Adding timestamps to every single raw_data field (Phase 1 covers high-risk fields only)
- Changing how odds are fetched

## High-Risk Fields to Timestamp in Phase 1

| Field | Source | Leakage Risk |
|---|---|---|
| `umpire_factor` | Umpire assignment API | Post-game backfill would include actual zone data |
| `homeGoalieCertainty` / `awayGoalieCertainty` | Lineup reports | Confirmed starter known only post-game warmup |
| `homeGoalsForL5` / `awayGoalsForL5` / `homeGoalsAgainstL5` / `awayGoalsAgainstL5` | ESPN rolling stats | If enrichment runs post-game, includes same-day game |
| `rolling_14d_wrc_plus_vs_hand` | FanGraphs team splits | Same risk as L5 stats |

## Guard Function

```js
'use strict';

/**
 * High-risk fields that must have available_at <= betPlacedAt.
 * Each entry: { field, path } where path is the dot-path into raw_data.
 */
const HIGH_RISK_FIELDS = [
  'umpire_factor',
  'homeGoalieCertainty',
  'awayGoalieCertainty',
  'homeGoalsForL5',
  'awayGoalsForL5',
  'homeGoalsAgainstL5',
  'awayGoalsAgainstL5',
  'rolling_14d_wrc_plus_vs_hand',
];

/**
 * @param {object} rawData       - parsed raw_data from odds snapshot
 * @param {string} betPlacedAt   - ISO8601 timestamp of bet decision
 * @returns {{ ok: boolean, violations: Array<{field: string, available_at: string, bet_placed_at: string}> }}
 */
function assertFeatureTimeliness(rawData, betPlacedAt) {
  const timestamps = rawData?.feature_timestamps ?? {};
  const betTime = new Date(betPlacedAt).getTime();
  const violations = [];

  for (const field of HIGH_RISK_FIELDS) {
    const availableAt = timestamps[field];
    if (availableAt == null) continue; // not tracked yet — skip (warn only in Phase 1)
    const availableTime = new Date(availableAt).getTime();
    if (availableTime > betTime) {
      violations.push({ field, available_at: availableAt, bet_placed_at: betPlacedAt });
    }
  }

  return { ok: violations.length === 0, violations };
}

module.exports = { assertFeatureTimeliness, HIGH_RISK_FIELDS };
```

In Phase 1: violations produce a warning log and are recorded in card payload metadata. They do NOT hard-block (we need baseline data to know prevalence). Phase 2 (after two weeks of data): violations on a confirmed-leaky field hard-block to `NO_BET`.
