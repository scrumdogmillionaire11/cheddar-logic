# WI-0920: Sunset Data-Layer Legacy Aliases and DB Path Compatibility

**ID**: WI-0920
**Goal**: Reduce production risk and maintenance burden by deprecating legacy DB path env handling and card-payload alias acceptance, keeping only explicitly approved compatibility behavior.

**Depends on**: none

**Scope**:
- `packages/data/src/db-path.js`
- `packages/data/src/validators/card-payload.js`
- `packages/data/src/db/cards.js`
- `packages/data/src/db/connection.js`
- `packages/data/README.md`
- `packages/data/__tests__/db-path-production.test.js`
- `packages/data/__tests__/validators/card-payload.news.test.js`
- `packages/data/__tests__/db-modules-smoke.test.js`

**Out of scope**:
- Web route/transform contract updates
- Worker model-evaluation algorithm changes
- Scheduler cadence and alert policy tuning

**Acceptance**:
- Production DB path resolution enforces `CHEDDAR_DB_PATH` as primary source and clearly fences legacy vars behind explicit compatibility behavior.
- Deprecated card-type aliases have a strict policy: either removed from write-path acceptance or kept read-only with hard comments/tests proving no new writes target them.
- Legacy official-status fallback from plain payload status is isolated and documented for historical rows only.
- Database connection proxy/shim behavior is either removed or explicitly justified with tests for remaining dependent call sites.
- Data package tests demonstrate deterministic behavior for path resolution and alias policy.
