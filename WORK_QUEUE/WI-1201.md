# WI-1201: Wire Discord to DecisionOutcome

**ID**: WI-1201
**Goal**: Convert Discord posting flow to a pure consumer of `DecisionOutcome`, ensuring Discord output counts and status match web exactly while preserving current user-facing Discord message format.

**Scope**:
- `apps/worker/src/jobs/post_discord_cards.js` (import canonical builder from @cheddar-logic/data at function entry; apply to decision_v2 inputs; consume outcome only for status/count classification; remove all locally-defined decision_v2 → status mappers/classifiers that are no longer called)
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js` (add parity assertions and snapshots; add blocker rendering test; add zero-reinterpretation test; add cross-mapper audit; verify no dead mapper code remains)
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js` (remove test cases for old mapper functions if they exist)
- Audit `apps/worker/src/utils/` for Discord-specific decision helpers that are no longer called after builder switch; delete easy hanging dead code (functions with no external callers, not exported publicly)
- Import shared fixture corpus from `packages/data/__tests__/fixtures/decision-outcome-parity-shared-corpus.json` (created in WI-1200; same 50+ decision_v2 objects used by web parity tests)
- `apps/worker/src/jobs/__tests__/fixtures/discord-decision-outcome-parity.expected.json` (generated from shared corpus; mirrored from web parity baseline)
- Post-Discord blocker handling template in `post_discord_cards.js` (append blockers to message when `DecisionOutcome.reasons.blockers` present)

**Out of scope**:
- Implementing Discord-specific decision mappers (use canonical builder from WI-1199 only)
- Changing worker decision_v2 production
- Comprehensive refactoring or reorganization of Discord jobs directory (covered in WI-1203)
- Webhook payload contract changes
- **In scope (not deferred to WI-1203):** Removing "easy hanging" dead code — functions that become unused after builder switch, have no external callers, and are not exported. This reduces maintenance burden immediately.

**Acceptance**:
- **Fixture corpus sourcing (FIXES blockers #1, #2, #8):**
  - Shared fixture corpus: `packages/data/__tests__/fixtures/decision-outcome-parity-shared-corpus.json` (50+ decision_v2 objects)
  - Created by WI-1200; imported by Discord parity test via `import corpus from '@cheddar-logic/data/fixtures/decision-outcome-parity-shared-corpus.json'`
  - Discord parity test (`post_discord_cards.test.js -t "DecisionOutcome parity"`) processes entire shared corpus; expected counts locked in `discord-decision-outcome-parity.expected.json`
  - Expected counts are byte-identical to web baseline (WI-1200) when processing same corpus; no subset blind spots

- **Canonical builder usage (FIXES blocker #7):**
  - At entry to `post_discord_cards.js` function, decision_v2 list is immediately transformed: `const outcomes = decisionV2List.map(d => buildDecisionOutcomeFromDecisionV2(d, metadata))`
  - No intermediate mapping or recomputation; all PLAY/PASS/SLIGHT_EDGE routing uses only `outcome.status`
  - Builder imported: `import { buildDecisionOutcomeFromDecisionV2 } from '@cheddar-logic/data'`

- **Zero status reinterpretation (FIXES blocker #9, warning #3):**
  - Unit test: `post_discord_cards.test.js -t "status is never reinterpreted"` — creates DecisionOutcome fixtures with status='PASS', status='SLIGHT_EDGE', status='PLAY'; confirms Discord message classification matches outcome.status exactly (no conversion)
  - Test verifies: if outcome.status='PASS', Discord message shows PASS count; if outcome.status='PLAY', Discord message shows PLAY count (no fallback or consensus logic)

- **Blocker rendering (FIXES blocker #3):**
  - When `DecisionOutcome.reasons.blockers` is non-empty, Discord appends blocker list to message text
  - Template: Discord message text includes line: `"Blockers: [blocker1, blocker2, ...]"` (exact formatting TBD by existing post_discord_cards style; must not change emoji/section labels)
  - Unit test: `post_discord_cards.test.js -t "blockers are included without status reclassification"` — DecisionOutcome with status='PLAY' + blockers=['NO_MARKET_LINE'] renders PLAY classification + appended blocker context; status remains PLAY
  - Blocker rendering does NOT reclassify status (e.g., PLAY + blockers still = PLAY count, not PASS)

- **Mapper code audit + dead code removal (FIXES blocker #6 + NEW):**
  - Task: Audit `post_discord_cards.js` for local decision_v2 → status mapping; confirm all such logic removed and replaced with builder call
  - Task: Identify functions in `post_discord_cards.js` and `apps/worker/src/utils/` that were called by old mapper logic and are now unreachable (e.g., old `classifyDecision()`, `mapDecisionToStatus()`, helper functions used only by dead mappers)
  - **Delete easy hanging dead code:** Remove identified functions that meet ALL criteria: (a) no longer called after builder switch, (b) not exported publicly to other modules, (c) no test cases specifically testing them
  - Test: Add assertion in test suite that `post_discord_cards.js` does not export or call functions matching pattern `*mapper*`, `*classify*`, `*statusRecompute`; confirm only `buildDecisionOutcomeFromDecisionV2` is used
  - Snapshot test verifies: for same decision_v2 input, builder-backed output is stable across runs
  - Git diff shows removed lines from dead mappers (proof of cleanup)

- **Manual cross-surface validation (FIXES blocker #5):**
  - Query `/api/cards` using shared fixture corpus (50+ decisions); extract status enum counts: X plays, Y slight_edges, Z passes
  - Run `post_discord_cards.job()` locally using same corpus
  - Verify Discord message counts exactly match /api/cards counts (same X, Y, Z)
  - Spot-check 5 decisions: Discord status classification for each decision matches /api/cards classification for same decision_v2 input

- `post_discord_cards.js` calls `buildDecisionOutcomeFromDecisionV2()` from canonical builder (imported from @cheddar-logic/data); does NOT implement its own mapper
