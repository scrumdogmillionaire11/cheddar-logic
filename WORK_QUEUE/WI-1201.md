# WI-1201: Wire Discord to DecisionOutcome

**ID**: WI-1201
**Goal**: Convert Discord posting flow to a pure consumer of `DecisionOutcome`, ensuring Discord output counts and status match web exactly while preserving current user-facing Discord message format.

**Scope**:
- `apps/worker/src/jobs/post_discord_cards.js` (import canonical builder from @cheddar-logic/data, apply to decision_v2 inputs at boundary, consume outcome only for status/count classification)
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js` (add parity assertions and snapshots using DecisionOutcome-driven fixtures)
- `apps/worker/src/jobs/__tests__/fixtures/discord-decision-outcome-parity.fixture.json` (new fixed 20-card fixture corpus for cross-surface parity)
- `apps/worker/src/jobs/__tests__/fixtures/discord-decision-outcome-parity.expected.json` (new locked expected counts/status summary mirrored from web golden baselines)

**Out of scope**:
- Implementing Discord-specific decision mappers (use canonical builder from WI-1199 only)
- Changing worker decision_v2 production
- Deleting old formatters/mappers (WI-1203)
- Webhook payload contract changes

**Acceptance**:
- `post_discord_cards.js` calls `buildDecisionOutcomeFromDecisionV2()` from canonical builder (imported from @cheddar-logic/data); does NOT implement its own mapper
- Discord status/count classification is zero-reinterpretation: PLAY/PASS/SLIGHT_EDGE routing comes only from DecisionOutcome.status
- Discord message text, section labels, and emoji remain unchanged from existing `post_discord_cards` snapshots unless explicitly required by DecisionOutcome parity assertions
- Parity test uses fixed fixture corpus in `apps/worker/src/jobs/__tests__/fixtures/discord-decision-outcome-parity.fixture.json`
- Expected parity outputs are locked in `apps/worker/src/jobs/__tests__/fixtures/discord-decision-outcome-parity.expected.json` and must mirror web golden baselines declared in `web/src/__tests__/cards-decision-outcome-parity.test.ts`, `web/src/__tests__/games-decision-outcome-parity.test.ts`, and `web/src/__tests__/results-decision-outcome-parity.test.ts`
- Snapshot/assertion coverage proves same-card corpus yields same PLAY/SLIGHT_EDGE/PASS counts in Discord and web
- If DecisionOutcome has `status='PLAY'` and `reasons.blockers`, Discord includes blocker context without reclassifying status
- All existing Discord tests pass
- No Discord-specific mapper code exists (dead code removal only; do not delete files yet)
