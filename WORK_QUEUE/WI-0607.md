# WI-0607: Persist + backfill results market period token

**ID**: WI-0607
**Goal**: Persist `market_period_token` onto `card_results.metadata` at settlement time and backfill historical settled rows through the worker so `/api/results` can read a stored full-game vs 1P classification without resettling cards.

**Scope**:
- `WORK_QUEUE/WI-0607.md`
- `apps/worker/src/jobs/settle_pending_cards.js`
- `apps/worker/src/jobs/backfill_card_results.js`
- `apps/worker/src/jobs/__tests__/settle_pending_cards.phase2.test.js`
- `apps/worker/src/__tests__/settlement-pipeline-integration.test.js`
- `web/src/app/api/results/route.ts`
- `web/src/__tests__/api-results-decision-segmentation.test.js`
- `web/src/__tests__/ui-results-smoke.test.js`
- `docs/DATA_CONTRACTS.md`
- `docs/ops-runbook.md`

**Out of scope**:
- Re-settling historical cards or changing any existing `result`, `status`, `settled_at`, or `pnl_units` values
- New `/results` filters, tabs, or UI redesign
- Persisting broader analytics segment families beyond the period token
- Web-side writes or maintenance paths that violate the single-writer DB contract

**Background**:
`WI-0567` surfaces 1P vs full-game labels on `/results`, but the distinction is still derived at read time from `market_key`, payload period fields, and `card_type`. The database already supports additive metadata on `card_results`, and the worker already owns settlement + backfill flows. This follow-up should make period segmentation durable for newly settled cards and historical settled cards while preserving current settlement outcomes and keeping `/api/results` backward-compatible during rollout.

**Acceptance**:
- Newly settled cards write `metadata.market_period_token` with `1P` or `FULL_GAME` during the worker settlement path, using the same classification rules currently used by `/api/results`.
- Historical settled rows can be backfilled through a worker-owned job path that updates only `card_results.metadata`; no rows are re-graded or re-settled.
- The backfill path supports a dry-run mode that reports candidate/update counts before applying changes.
- `/api/results` prefers persisted `metadata.market_period_token` when present and falls back to the existing derived logic when absent, so mixed old/new data remains safe during rollout.
- Existing `/results` 1P badge behavior remains correct for both newly settled and backfilled rows.
- The implementation preserves the single-writer architecture: all persistence/backfill writes happen in worker code, not web code.
