# WI-1173: MLB Pitcher Command-Context Integration and Dead-Field Prune

**ID**: WI-1173
**Goal**: Use MLB pitcher game-log command/context fields as explicit pitcher-K model inputs while removing truly dead run-allowed fields from the active fetch/write and model-input paths.

**Scope**:
- apps/worker/src/jobs/pull_mlb_pitcher_stats.js
- apps/worker/src/jobs/__tests__/pull_mlb_pitcher_stats.test.js
- apps/worker/src/jobs/run_mlb_model.js
- apps/worker/src/jobs/__tests__/run_mlb_model.test.js
- apps/worker/src/models/mlb-model.js
- apps/worker/src/models/__tests__/*.test.js (nearest existing pitcher-K model test file only)
- docs/models__mlb_pitcher_k_inputs.md (or equivalent existing MLB pitcher-K input doc)

**Out of scope**:
- Full MLB threshold retuning beyond the explicit command-context adjustments in this WI
- New external data sources
- F5 ML/F5 totals settlement behavior changes
- Schema-drop migrations for historical columns (separate serialized DB WI if desired)
- UI changes

**Implementation notes**:
- Preserve and consume active signal fields: `walks`, `batters_faced`, `home_away`, `number_of_pitches`, `innings_pitched`, `strikeouts`.
- Remove active write-path and model-input dependency on truly dead fields: `hits`, `earned_runs`.
- Before removing write-path usage, perform repo-wide verification that `hits` and `earned_runs` are not consumed by active model execution, recurring analytics pipelines, or runtime diagnostics in this repo.
- Add derived command-context contract for pitcher-K path:
  - `recent_walks`, `recent_batters_faced`, `recent_bb_pct`
  - `recent_bb_pct_status` in `OK | MISSING | SMALL_SAMPLE`
  - `home_away_context` in `HOME | AWAY | MIXED | UNKNOWN`
  - `command_risk_flag`
- Derivation rules:
  - `recent_bb_pct = sum(walks) / sum(batters_faced)`
  - Lookback window is explicit: last `N=10` starts by default, aligned with pitcher-K lookback.
  - `SMALL_SAMPLE` if `recent_batters_faced < 40`
  - `MISSING` if walks/batters-faced context is unavailable
  - `command_risk_flag = true` when `recent_bb_pct >= 0.095`
- Threshold handling:
  - `0.095` is provisional and must be documented as calibratable (not a permanent constant).
  - Keep threshold definition centralized so follow-up calibration can tune it without refactoring model flow.
- Apply small, bounded effect in pitcher-K logic:
  - projection adjustment: `-0.15 Ks` when command risk is true
  - confidence adjustment: `-5` for command risk, `-2` for small sample
  - missing context adds degradation reason without crash/auto-pass
  - guard against compounding with existing leash/fatigue penalties using an explicit cap: max combined projection penalty from overlapping risk controls is `-0.30 Ks`.
- Home/away handling remains confidence-only and should only apply when context is attributable; no heavy projection swing from home/away alone.
- Ensure pitcher-K output contract carries command context fields for explainability and downstream audit:
  - `recent_bb_pct`
  - `recent_bb_pct_status`
  - `command_risk_flag`
- Add/retain reason codes used by tests/docs:
  - `COMMAND_RISK_RECENT_BB_RATE`
  - `COMMAND_CONTEXT_SMALL_SAMPLE`
  - `COMMAND_CONTEXT_MISSING`
  - `HOME_AWAY_CONTEXT_SHIFT` (confidence-only)

**Acceptance**:
- `pull_mlb_pitcher_stats.js` no longer persists `hits` and `earned_runs` in active write path.
- Repo-wide search in this repository confirms `hits` and `earned_runs` are not required by active model runtime paths, recurring analytics pipelines, or runtime diagnostic code after this WI.
- Pitcher-K path explicitly consumes `walks`, `batters_faced`, and `home_away` via derived command context.
- `recent_bb_pct` is computed from `sum(walks) / sum(batters_faced)` over the explicit 10-start lookback and influences model output with bounded adjustments.
- Pitcher-K output payload includes `recent_bb_pct`, `recent_bb_pct_status`, and `command_risk_flag` for traceability.
- Missing/small-sample contexts are handled deterministically with reason codes and no crash.
- Command-risk adjustments cannot over-stack with existing leash/fatigue controls beyond the defined projection cap.
- Active MLB consumers for `number_of_pitches`, `innings_pitched`, and `strikeouts` remain intact.
- Tests cover command-context behavior and dead-field removal assumptions.
- Docs reflect retained vs deprecated fields, formulas, threshold status (provisional/calibratable), and penalty caps.
