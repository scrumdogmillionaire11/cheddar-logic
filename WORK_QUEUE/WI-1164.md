# WI-1164: Send Discord Hook Call Results To Operators

**ID**: WI-1164
**Goal**: Surface concrete success/failure results of Discord webhook calls so operators can see what was sent, what failed, and why, without reading raw runtime logs.

**Requirements**: [WI-1164-DISCORD-RESULTS-01, WI-1164-DISCORD-RESULTS-02, WI-1164-DISCORD-RESULTS-03]

**Roadmap traceability**:

- Executes phase plan: `.planning/phases/discord-hook-results/discord-hook-results-01-PLAN.md`
- Follows `discord-hook-eligibility` after WI-1163 closes and canonical posting behavior is stable.

**Scope**:

- `apps/worker/src/jobs/post_discord_cards.js`
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js`
- `.planning/ROADMAP.md`
- `.planning/phases/discord-hook-results/discord-hook-results-01-PLAN.md`
- `WORK_QUEUE/WI-1164.md`

**Out of scope**:

- Rewriting Discord content formatting
- Adding new external notification providers (Slack/email/SMS)
- Changing card scoring or bucket classification logic

**Acceptance**:

- [WI-1164-DISCORD-RESULTS-01] Each webhook attempt records a structured result object including endpoint label, attempt count, status (`success` or `failed`), HTTP status/error, elapsed time, retry count, and posted card count when successful.
- [WI-1164-DISCORD-RESULTS-02] Job summary output includes an operator-readable transport results block with counts for successes, failures, and retries plus one compact per-target line showing target label, final status, attempts, and posted card count or failure reason.
- [WI-1164-DISCORD-RESULTS-03] When at least one webhook call fails, the final job result clearly reports partial failure, exposes failed target details programmatically, and includes failed target label plus failure reason in the operator-facing result block.
- Tests cover all-success, partial-failure, and retry-then-success cases and assert final result-block contents rather than only helper state.
