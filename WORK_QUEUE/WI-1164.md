# WI-1164: Send Discord Hook Call Results To Operators

**ID**: WI-1164
**Goal**: Surface concrete success/failure results of Discord webhook calls so operators can see what was sent, what failed, and why, without reading raw runtime logs.

**Requirements**: [WI-1164-DISCORD-RESULTS-01, WI-1164-DISCORD-RESULTS-02, WI-1164-DISCORD-RESULTS-03]

**Roadmap traceability**:

- Extends execution after `discord-hook-eligibility` phase by adding delivery-result visibility for webhook transport calls.
- Depends on canonical posting behavior from WI-1163.

**Scope**:

- `apps/worker/src/jobs/post_discord_cards.js`
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js`
- `WORK_QUEUE/WI-1164.md`

**Out of scope**:

- Rewriting Discord content formatting
- Adding new external notification providers (Slack/email/SMS)
- Changing card scoring or bucket classification logic

**Acceptance**:

- [WI-1164-DISCORD-RESULTS-01] Each webhook attempt records a structured result object including endpoint label, attempt count, status (`success` or `failed`), HTTP status/error, and elapsed time.
- [WI-1164-DISCORD-RESULTS-02] Job summary output includes an operator-readable transport results block with counts for successes, failures, and retries.
- [WI-1164-DISCORD-RESULTS-03] When at least one webhook call fails, the final job result clearly reports partial failure and includes failed target details.
- Tests cover all-success, partial-failure, and retry-then-success cases.
