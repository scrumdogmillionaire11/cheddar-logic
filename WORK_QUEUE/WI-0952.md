# WI-0952: Stale-Only Moneyline Auto-Recovery and Single-Retry Flow

**ID**: WI-0952
**Goal**: Convert stale-only execution-gate failures into deterministic targeted refresh + single retry so recoverable moneyline plays are not permanently dropped.

**Depends on**: WI-0950, WI-0951

**Scope**:

- `apps/worker/src/jobs/run_mlb_model.js`
- `apps/worker/src/jobs/run_nhl_model.js`
- `apps/worker/src/jobs/__tests__/run_mlb_model.test.js`
- `apps/worker/src/jobs/__tests__/run_nhl_model.test.js`
- `WORK_QUEUE/WI-0952.md`

**Out of scope**:

- Broad multi-retry orchestration
- New queueing infrastructure
- Changes to market-eval ranking logic
- Web presentation changes

**Acceptance**:

- When execution gate fails only on stale snapshot, runner triggers one targeted odds refresh attempt and one deterministic retry.
- Retry path is idempotent and guarded to prevent infinite loops or duplicate publishes.
- Non-stale blockers (`NO_EDGE`, `CONFIDENCE`, integrity gates) do not trigger retry flow.
- Payload metadata records stale recovery attempt and final outcome for audit.
- Tests cover stale-only success recovery, stale-only failed recovery, and mixed-blockers no-retry behavior.

**Tech debt and legacy expectation**:

- Retire duplicate stale-handling branches in runner logic where new shared stale-recovery flow applies.
- Remove legacy ad-hoc stale bypass patterns in touched code; keep one explicit recovery contract.
- Any retained legacy conditional in-scope must be justified with comment plus follow-up WI reference.
