# WI-0962: Reduce non-production DB lock warning spam during manual worker runs

**ID**: WI-0962
**Goal**: Preserve the single-writer safety contract while making non-production/manual job runs stop emitting repeated lock warnings that do not correspond to a real failure.

**Depends on**: none  <!-- WI-0899 is a read-only audit; its table-classification output does not gate this work. Lock deduplication does not require knowing which tables are canonical vs. derived. Dependency removed. -->

**Scope**:
- `WORK_QUEUE/WI-0962.md`
- `packages/data/src/db/connection.js` — add non-production lock-warning suppression metadata that is bounded, cross-process, and safety-neutral
- `packages/data/src/db/__tests__/connection.test.js` — targeted public API coverage for lock-warning suppression and production refusal behavior
- `docs/QUICKSTART.md` — clarify expected dev behavior vs. actionable lock failures
- `docs/ops-runbook.md` — same guidance update

**Out of scope**:

- Changing production single-writer behavior
- Enabling `CHEDDAR_DB_ALLOW_MULTI_PROCESS=true` by default
- Replacing sql.js or the DB locking approach
- Table-access-aware conditional warning suppression (WI-0899 domain)

**Root cause**:
`acquireDbFileLock()` is called once per `loadDatabase()` invocation. `loadDatabase()` is called from `getDatabase()` only when `dbInstance === null`. Since `dbInstance` is module-level singleton, the lock contention warning fires **once per process** in production-safe code. However, in manual runs (e.g., direct script invocations with fresh module context), each script re-initializes the module and triggers the lock attempt. There is no per-process dedupe for the `console.warn` in the non-production path.

**Acceptance**:
- The exact warning `[DB] Refusing to open ... Set CHEDDAR_DB_ALLOW_MULTI_PROCESS=true to bypass.` is emitted once for the same `(dbFile, lockPath, ownerPid)` key within a 10-minute window in non-production environments.
- Warning suppression survives fresh Node module and process lifecycles by storing best-effort metadata under `os.tmpdir()` (for example `cheddar-logic-db-lock-warnings/<sha256(dbFile)>.json`).
- Warning re-emits when owner PID, DB path, lock path changes, metadata is missing/corrupt/unreadable, or the suppression window expires.
- Metadata read/write failure fails open by emitting the warning; warning suppression never changes lock acquisition, stale lock handling, retry behavior, or DB safety.
- `closeDatabase()` does not reset warning suppression state.
- Production behavior is unchanged: `NODE_ENV=production` still throws the existing refusal Error and does not downgrade to warn-only.
- Operator docs distinguish expected informational warnings (local multi-process dev) from actionable production lock failures.
- Tests cover: (a) first non-production contention warns once, (b) second contention after `closeDatabase()` + module reload suppresses for the same owner, (c) changed owner PID re-warns, (d) production path throws and does not warn, (e) corrupt/unreadable metadata fails open by warning.

**Execution plan**:

## Phase 1: Implement bounded cross-process warning suppression

1. Add non-production-only suppression around the existing lock-contention warning path in `connection.js`.
2. Key suppression by `(dbFile, lockPath, ownerPid)`.
3. Store metadata as best-effort JSON under `os.tmpdir()/cheddar-logic-db-lock-warnings/<sha256(dbFile)>.json`.
4. Suppress duplicate warnings only when the stored key matches and the timestamp is less than 10 minutes old.
5. Fail open on metadata read/write errors by emitting the warning.
6. Preserve existing public API, production throw behavior, stale-lock recovery, retry behavior, and `CHEDDAR_DB_ALLOW_MULTI_PROCESS` default behavior.
7. Do not reset suppression from `closeDatabase()`.

## Phase 2: Update operator docs

1. `docs/QUICKSTART.md`: Add section "DB lock warnings in dev"
   - Expected: informational, rate-limited warning when running manual jobs against a worker-owned local DB
   - Actionable: production throw, or a warning with changed DB path / lock path / owner PID
   - Resolution: stop worker, run maintenance, restart

2. `docs/ops-runbook.md`: Same section under "Lock management"
   - Distinguish informational non-production warning suppression from production lock conflicts
   - Reference lock file location: `$CHEDDAR_DB_PATH.lock`
   - Emphasize never using `CHEDDAR_DB_ALLOW_MULTI_PROCESS=true` in production

## Phase 3: Add targeted tests

File: `packages/data/src/db/__tests__/connection.test.js`

Use only public `getDatabase()` / `closeDatabase()` behavior. Use temp DB paths, pre-created `.lock` files with live PID data, `jest.resetModules()` or `jest.isolateModules()`, `console.warn` spies, and a `better-sqlite3` mock if needed.

Coverage:
1. First non-production contention warns once.
2. Second contention after `closeDatabase()` + module reload suppresses for the same owner.
3. Changed owner PID re-warns.
4. Production lock contention throws and does not warn.
5. Corrupt/unreadable metadata fails open by warning.
