# WI-0962: Reduce non-production DB lock warning spam during manual worker runs

**ID**: WI-0962
**Goal**: Preserve the single-writer safety contract while making non-production/manual job runs stop emitting repeated lock warnings that do not correspond to a real failure.

**Depends on**: none  <!-- WI-0899 is a read-only audit; its table-classification output does not gate this work. Lock deduplication does not require knowing which tables are canonical vs. derived. Dependency removed. -->

**Scope**:
- `WORK_QUEUE/WI-0962.md`
- `packages/data/src/db/connection.js` — add process-scoped lock-warning dedup flag (lines ~276-282 in `acquireDbFileLock`)
- `docs/QUICKSTART.md` — clarify expected dev behavior vs. actionable lock failures
- `docs/ops-runbook.md` — same guidance update

**Out of scope**:
- `apps/worker/src/jobs/pull_odds_hourly.js` — the warning originates in connection.js; no changes needed in the job file itself. Removed from scope.
- Changing production single-writer behavior

- Replacing sql.js or the DB locking approach
- Table-access-aware conditional warning suppression (WI-0899 domain)

**Root cause** (pre-diagnosis):
`acquireDbFileLock()` is called once per `loadDatabase()` invocation. `loadDatabase()` is called from `getDatabase()` only when `dbInstance === null`. Since `dbInstance` is module-level singleton, the lock contention warning fires **once per process** in production-safe code. However, in manual runs (e.g., direct script invocations with fresh module context), each script re-initializes the module and triggers the lock attempt. There is no per-process dedupe for the `console.warn` in the non-production path.

**Acceptance**:
- The exact warning `[DB] Refusing to open ... Set CHEDDAR_DB_ALLOW_MULTI_PROCESS=true to bypass.` is emitted **at most once per process lifetime** in non-production environments.
- Production behavior is unchanged: NODE_ENV=production still throws an Error (no change to throw path).
- A second call to `acquireDbFileLock()` from the same process (e.g., after closeDatabase() + re-open) re-emits the warning — this is expected and correct.
- Operator docs distinguish expected informational warnings (local multi-process dev) from actionable production lock failures.
- Tests cover: (a) warning emitted once when lock held, (b) no second warning within same module lifecycle, (c) production path still throws.

## Phase 2: Implement per-process deduplication

Simple fix — one new module-level flag in connection.js:

```javascript
// Add alongside existing module-level dedup flags (line ~9-14)
let warnedDbLockContention = false;

// In acquireDbFileLock(), replace the console.warn call:
if (!warnedDbLockContention) {
  warnedDbLockContention = true;
  console.warn(
    `[DB] Refusing to open ${dbFile}: another process holds the lock (${lockPath}, pid=${ownerPid}). ` +
    'Set CHEDDAR_DB_ALLOW_MULTI_PROCESS=true to bypass. [This warning will not repeat for this process.]',
  );
}
```

**Constraints:**
- Flag is `false` by default, set to `true` on first emit
- Flag resets to `false` on `closeDatabase()` call (so a new open attempt in the same process gets one warning)
- Look at existing pattern `warnedDbPathContract` (line 9) — use identical pattern for consistency
