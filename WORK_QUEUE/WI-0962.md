# WI-0962: Reduce non-production DB lock warning spam during manual worker runs

**ID**: WI-0962
**Goal**: Preserve the single-writer safety contract while making non-production/manual job runs stop emitting repeated lock warnings that do not correspond to a real failure.

**Depends on**: WI-0899

**Scope**:
- `WORK_QUEUE/WI-0962.md`
- `packages/data/src/db/connection.js`
- `apps/worker/src/jobs/pull_odds_hourly.js`
- `docs/QUICKSTART.md`
- `docs/ops-runbook.md`

**Out of scope**:
- Changing production single-writer behavior
- Enabling `CHEDDAR_DB_ALLOW_MULTI_PROCESS=true` by default
- Replacing sql.js or the DB locking approach

**Acceptance**:
- In non-production, a manual run against a DB already owned by the worker no longer emits the same `Refusing to open ... Set CHEDDAR_DB_ALLOW_MULTI_PROCESS=true to bypass.` warning repeatedly when the run can still proceed safely.
- Production behavior is unchanged: a live conflicting writer still hard-fails, and stale-lock semantics remain intact.
- Operator docs distinguish clearly between actionable lock failures and expected informational behavior in local/dev workflows.
- Any warning that remains is emitted once per process or per attempted conflicting open, with explicit context about whether the path is informational or blocking.

**Execution plan (deferred):**
- Phase 1: Reproduce lock-warning spam during manual non-production runs while worker holds the writer lock.
- Phase 2: Classify lock outcomes into informational vs blocking without changing production single-writer enforcement.
- Phase 3: Implement once-per-process (or once-per-open-attempt) warning dedupe for informational paths.
- Phase 4: Update quickstart/ops docs with explicit guidance on expected dev behavior vs actionable production lock failures.
- Phase 5: Run integration tests and validate manual run output is concise, classified, and non-repetitive.
