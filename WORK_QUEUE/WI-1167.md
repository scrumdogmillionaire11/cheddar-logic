# WI-1167: Canonical Cards Eligibility Gate Unification

**ID**: WI-1167
**Goal**: Remove duplicated card eligibility logic across SQL and post-query filtering by defining one canonical eligibility path for both `/api/cards` endpoints.

**Scope**:

- `web/src/app/api/cards/route.ts`
- `web/src/app/api/cards/[gameId]/route.ts`
- `web/src/lib/cards/query.ts`
- `web/src/lib/cards/payload-classifier.ts`
- `web/src/lib/games/projection-surface.ts`
- `web/src/__tests__/cards-projection-exclusion.test.js`
- `web/src/__tests__/api-cards-lifecycle-regression.test.js`
- `web/src/__tests__/api-cards-eligibility-contract.test.js`
- `web/src/__tests__/api-cards-diagnostics-contract.test.js`
- `WORK_QUEUE/WI-1167.md`

**Out of scope**:

- Changes to worker card production payloads
- New product surface types unrelated to existing cards endpoints
- Large UI refactors

**Acceptance**:

- `/api/cards` and `/api/cards/[gameId]` both delegate projection-only/synthetic eligibility decisions to one canonical shared layer instead of reimplementing the same gate in SQL and response serialization.
- The retained canonical layer returns explicit drop reason codes that match the diagnostics semantics introduced by WI-1166, and route diagnostics aggregate those canonical codes by reason and `card_type`.
- Eligible visible prop families including `nhl-player-shots` are preserved, while projection-only/synthetic rows still suppress where policy requires suppression.
- Regression coverage proves both endpoints preserve lifecycle and projection exclusion behavior after the unification.
