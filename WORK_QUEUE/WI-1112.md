## Context

### What exists today

**Settlement pipeline (worker)**
- `settle_mlb_f5.js` — settles `mlb-f5` and `mlb-f5-ml` card_results against live F5 game state
- `settle_projections.js` — stamps `actual_result` on `card_payloads` and writes rows to `projection_proxy_evals` for NHL 1P totals, MLB F5 (total and ML), pitcher K, player shots
- `settle_pending_cards.js` — settles odds-backed card_results; now guards `nhl-pace-1p` and F5 rows from MISSING_MARKET_KEY auto-close (WI-1107 fix)
- `projection_proxy_evals` table — stores graded proxy rows per card (proj_value, actual_value, recommended_side, tier, graded_result, hit_flag, etc.)

**Projection accuracy telemetry (worker)**
- `projection_accuracy_evals` + `projection_accuracy_line_evals` tables — per-card accuracy capture with synthetic line evaluation, confidence bands, MAE, bias
- `getProjectionAccuracyEvalSummary()` — feeds `/api/results/projection-accuracy`

**Web /results presentation**
- `/api/results/projection-settled` — merges `projection_proxy_evals` rows (NHL 1P) with settled `mlb-f5-ml` card_results rows into a unified `ProjectionProxyRow` feed
- `/api/results/projection-accuracy` — global `projection_accuracy_evals` summary and market health
- `ProjectionResultsTable` — groups settled rows by card family, renders direction as team name for F5 ML (WI-recent fix), stable keys
- Model stats summary table — synthesizes F5 ML rows from settled feed when `projection_proxy_evals` has no rows for that family

### Known gaps as of 2026-04-21

1. **NHL 1P projection_proxy_evals is empty in dev DB** — all 4,441 `nhl-pace-1p` card_results were in `error` state due to MISSING_MARKET_KEY auto-close (now fixed); `projection_proxy_evals` has never been populated for this family. A backfill run is needed after the guard fix is deployed.

2. **MLB F5 ML projection_proxy_evals not populated** — settled `mlb-f5-ml` rows exist (10 settled) but `settle_projections.js` doesn't write `projection_proxy_evals` rows for the ML variant — it only stamps `actual_result` on `card_payloads`. The web API works around this by doing a secondary card_results query, but it means MAE/bias/confidence tracking is absent for F5 ML.

3. **Projection summary model stats table misses F5 ML from the accuracy engine** — the `projection_accuracy_evals` table has no `MLB_F5_ML` rows, so the model stats table synthesizes them on the fly from settled rows (thin proxy, no MAE/bias/confidence).

4. **F5 ML backfill coverage is ~10%** — 85 of 95 pending rows remain unresolved due to missing gamePk map entries for historical matchups.

5. **Projection Confidence Engine shows anomalous numbers** — 22813 weak directions, 22-0 record, MAE 1.02 — because `projection_accuracy_line_evals` has ~22k rows across multiple families, almost none graded (underlying card_results were errored). Once the settlement fix flows through, real graded rows will replace this noise.

6. **No per-family date-range filter on `/results`** — all projection families show all-time data. A lookback filter would let users focus on recent accuracy.

7. **No "Projected" vs "Actual" numeric display for F5 ML** — projected shows HOME/AWAY (team side), actual shows 0 or 1 (binary). For a moneyline, this is semantically correct but visually confusing — it should display win probability or simply WIN/LOSS in the Actual column instead of a numeric.

8. **Projection settlement panel description is generic** — copy says "NHL 1P totals and MLB F5 ML" but will need to cover more families as they're added.

## Questions to resolve in this WI

### Settlement pipeline
- Q1: Should `settle_projections.js` write `projection_proxy_evals` rows for `mlb-f5-ml`? If yes, how do we map ML (HOME/AWAY win) to the proxy eval schema (proj_value, actual_value, proxy_line)?
- Q2: What is the right backfill strategy for the 85 still-pending F5 ML rows with missing gamePk entries? Options: (a) one-time map alias insert, (b) add fallback to fetch live game data by date+teams without gamePk, (c) mark as no_contest after a staleness window.
- Q3: After the MISSING_MARKET_KEY guard fix, what is the right backfill strategy for the 4,431 errored `nhl-pace-1p` card_results? Can we reset their status to pending, or do we need a separate backfill job?
- Q4: Should `projection_accuracy_evals` capture be extended to `mlb-f5-ml`? If yes, what are the `projectionKeys` and `actualKeys` for this market family?

### /results presentation
- Q5: For F5 ML rows, should the Projected column show win probability (e.g. `0.72`) or the selected team name (already showing in Direction)? The current dual display is redundant.
- Q6: Should the Actual column for F5 ML show `WIN` / `LOSS` / `PUSH` as text badges instead of `0.000` / `1.000`?
- Q7: Is the model stats table the right place to show F5 ML accuracy, or should ML results be surfaced in a separate "Moneyline Model" section?
- Q8: What lookback window should be the default for projection-settled data? All-time, 30 days, 90 days?
- Q9: Should the Projection Settlement section be gated behind a minimum settled count (e.g. don't show the section at all until N ≥ 5)?
