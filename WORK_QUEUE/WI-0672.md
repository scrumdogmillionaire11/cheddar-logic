# WI-0672: Draft Lab State Visibility and Reset to Baseline

**ID**: WI-0672
**Goal**: Users must be able to see what constraints are currently active, and reset to the recommended baseline at any point. Without this the system becomes a black box — users don't know what they've applied, can't verify their state, and the product feels opaque. State visibility and reset are trust surfaces, not convenience features.

**Depends on**:

- WI-0668 (intent translation and undo endpoint must exist)
- WI-0660 (Draft Lab must exist)

**Scope**:

- `WORK_QUEUE/WI-0672.md`
- `cheddar-fpl-sage/backend/routers/draft_lab.py` (add `GET /draft-lab/{session_id}/constraints` and `POST /draft-lab/{session_id}/reset`)
- `cheddar-fpl-sage/backend/services/constraint_state_service.py` (new)
- `cheddar-fpl-sage/backend/models/constraint_state_models.py` (new: `ConstraintState`, `AppliedIntent`, `BaselineSnapshot`)
- `cheddar-fpl-sage/tests/test_constraint_state_service.py`
- `web/src/components/fpl-constraint-panel.tsx` (new — always-visible panel strip showing active constraints)
- `web/src/__tests__/fpl-constraint-panel-contract.test.js`

**Out of scope**:

- Full intent history beyond the current session (only last 3-5 shown in panel; complete log deferred)
- Undo beyond the most recent intent (WI-0668 covers last-intent undo)
- State persistence across sessions (resets on new session)

## Required behavior

### 1. Active constraint state panel (always visible in Draft Lab)

`GET /api/v1/draft-lab/{session_id}/constraints` returns the full current constraint state:

    {
      "session_id": "<uuid>",
      "applied_intents": [
        { "type": "lock_player",       "params": { "player": "Salah" },  "applied_at": "<ISO8601>", "human_label": "Locked Salah" },
        { "type": "risk_down",         "params": {},                      "applied_at": "<ISO8601>", "human_label": "Made build safer" },
        { "type": "bench_priority_up", "params": {},                      "applied_at": "<ISO8601>", "human_label": "Improved bench depth" }
      ],
      "locked_players": ["Salah"],
      "banned_players": [],
      "risk_posture": "conservative",
      "bench_priority": "high"
    }

`fpl-constraint-panel.tsx` renders this as a persistent strip inside the Draft Lab — always visible, never collapsed by default. Shows:

- Locked players (pill with lock icon)
- Banned players (pill with ban icon)
- Risk posture (label: Conservative / Balanced / Aggressive)
- Bench priority (label: Light / Standard / Heavy)

Panels update after every intent application without full page reload.

### 2. Reset to baseline

`POST /api/v1/draft-lab/{session_id}/reset` clears all applied intents and restores the session to the profile-recommended baseline state. Returns the clean baseline `ConstraintState`.

UI: a "Reset to Sage Recommendation" button in the constraint panel. Only visible when `applied_intents` is non-empty. Requires a single confirmation tap (not a modal — an inline confirm state on the button itself).

### 3. Speed requirement

Both endpoints must respond in under 200ms. Constraint state is read from ProductStore — no computation on read. Reset is a write that clears the intent log and restores baseline — no recomputation of the recommendation itself.

This product must feel instant. If state visibility causes latency, it breaks the iterative flow state.

---


### 4. Change history (lightweight -- last 3-5 intents)

The constraint panel must show the last 3-5 applied intents as a readable change log below the
active-constraint pills. This is not a full audit trail -- it is an orientation surface: "here is
how you got to this state."

**Backend change:** `AppliedIntent` model gains a required `human_label` field (string). The field
is set by `constraint_state_service.py` using a canonical label map -- never derived on the frontend.

Canonical label map (required in `constraint_state_service.py`):

| Intent type | Human label |
| --- | --- |
| `risk_down` | "Made build safer" |
| `risk_up` | "Went more aggressive" |
| `lock_player` | "Locked {player}" |
| `ban_player` | "Banned {player}" |
| `bench_priority_up` | "Improved bench depth" |
| `bench_priority_down` | "Reduced bench depth" |
| `ban_team_combo` | "Reduced {team} exposure" |

**UI change:** `fpl-constraint-panel.tsx` renders the **last 3-5 items** from `applied_intents`
(reverse-chronological -- most recent at top) as a compact summary row:

> 3 changes: Made build safer, Locked Salah, Improved bench depth

The "Reset to Sage Recommendation" button, after first tap, shows an inline confirmation replacing
the button text:

> This will clear 3 changes. Confirm?

Second tap executes reset. No modal. No separate confirmation screen.

- `GET /draft-lab/{session_id}/constraints` returns the full constraint state after each intent application.
- `POST /draft-lab/{session_id}/reset` returns clean baseline state with empty `applied_intents`.
- `fpl-constraint-panel.tsx` renders locked players, banned players, risk posture, and bench priority from API data.
- Panel is always visible in Draft Lab — not collapsed, not behind a toggle.
- Panel updates after each intent without page navigation.
- "Reset to Sage Recommendation" button appears only when intents have been applied and resets on single confirmation tap.
- Reset does not recompute the recommendation — it restores the stored baseline snapshot.

- `human_label` is always present on every `AppliedIntent`; set by backend canonical map; never derived on frontend.
- Reset button shows inline confirmation ('This will clear N changes. Confirm?') on first tap -- no modal.
- `pytest tests/test_constraint_state_service.py` passes (minimum 15 tests: state after each intent type, reset clears all intents, baseline restoration, empty state on new session, human_label present for all intent types, history order is reverse-chronological).
