# WI-1252: Scheduler Timeout and Watchdog Recovery Hardening

**ID**: WI-1252
**Goal**: Stop scheduler timeout overlap and repeated fixed-window retry storms, restore reliable watchdog auto-restart behavior, and harden deploy cleanup so stale worker states do not require manual restarts after deploy.

**Context**:

Production showed a recurring failure mode where `cheddar-worker` stayed process-alive while scheduler heartbeats went stale. The current scheduler timeout path releases its in-process tick lock without cancelling the timed-out work, allowing overlapping long ticks. Separately, several fixed-window jobs either do not record `job_runs` success or receive malformed scheduler args, so once they become due they can re-run for the rest of the day. The watchdog detects stale heartbeats, but its restart path is not reliably recovering the worker, and deploy temporarily disables the watchdog without a guaranteed restore path on early failure.

- Cloudflare or tunnel reconfiguration

**Acceptance**:
- Scheduler timeouts fail closed instead of allowing overlapping long-running ticks to continue alongside new ticks.
- `refresh_team_metrics_daily` correctly accepts scheduler-supplied args and records a single successful daily run instead of dry-running with object-shaped input.
- Fixed-window nightly worker jobs (`fit_calibration_models`, calibration report, residual validation, CLV snapshot, daily performance report) record `job_runs` and skip once successful for the day.
- Watchdog stale-heartbeat remediation can actually restart `cheddar-worker` in production service context.
- Deploy workflow restores watchdog protection even if the deploy exits before the normal happy-path re-enable step.
- Focused regression coverage proves the corrected scheduler/job contracts.
