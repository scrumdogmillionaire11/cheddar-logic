# WI-0669: Final Recommendation Terminal Output

**ID**: WI-0669
**Goal**: Every draft workbench flow must terminate with an unambiguous "Final Recommendation" screen that makes a single, committed call. This is the product. Everything else in the workbench feeds into this screen. A weak final screen makes the entire system feel advisory rather than decisive.

**Depends on**:

- WI-0656 (draft audit scoring and comparison API must exist)
- WI-0660 (draft workbench shell must exist)

**Scope**:

- `WORK_QUEUE/WI-0669.md`
- `cheddar-fpl-sage/backend/routers/draft_sessions.py` (add `POST /draft-sessions/{id}/recommend`)
- `cheddar-fpl-sage/backend/services/recommendation_service.py`
- `cheddar-fpl-sage/backend/models/recommendation_models.py`
- `cheddar-fpl-sage/tests/test_recommendation_service.py`
- `web/src/components/fpl-final-recommendation.tsx`
- `web/src/__tests__/fpl-final-recommendation-contract.test.js`

**Out of scope**:

- Transfer/chip recommendations (post-GW1 scope)
- Price change predictions
- Multi-gameweek planning surfaces
- Changes to audit or comparison endpoints

## What this screen must do

### 1. Single primary answer

The screen opens with one committed statement from the backend. Not a range. Not a suggestion. One team.

The recommendation service must select the best team for the user's archetype and express it plainly:
"This is your best team."

The backend selects. The UI renders. No hedging.

### 2. Exactly two alternatives (max)

No more than two alternatives are allowed. Both must be present in the backend response if available:

- Alternative A: the safer version — what the user trades away for safety
- Alternative B: the upside version — what the user gains and what they risk

If only one alternative exists, return one. Never return zero alternatives (the user needs something to compare against to understand the tradeoff).

### 3. Explicit tradeoffs on every option

Tradeoffs for each alternative must be prominent — not buried in a tooltip or collapsed section.

Format (backend-set strings, not frontend derived):

    + stronger bench
    - lower captain ceiling
    - less flexibility at 7.5M slot

### 4. Profile identity alignment

Each recommendation must include a profile fit score:

    "This build matches your Balanced Climber profile (87% fit)"

Backend computes the fit percentage. Frontend renders it. No frontend estimation.

### 5. Decision moment UI

The final screen must present exactly three CTAs:

- "Lock This Team" — commits primary recommendation (fires decision receipt)
- "Use This Version" — available on each alternative (locks that version instead)
- "Go Back and Tweak" — returns user to Draft Lab with current state preserved

No "maybe later" or neutral exit. The screen requires a decision.

### 6. Non-skippable gate

The final recommendation screen is the mandatory terminal step in the `FPLProductShell` flow. A user cannot lock a team without passing through this screen. The audit must be complete before the recommendation is generated (400 if not).

---


### 7. Recommendation confidence

Every recommendation response must include a confidence signal so Sage does not always sound equally
certain. Backend sets this field -- never inferred on the frontend.

    "recommendation_confidence": "high" | "medium" | "low",
    "confidence_reason": "<plain-English sentence explaining the level>",

Rules:

- `high` -- clear archetype match, dominant lead over alternatives, no high-volatility flags on key
  players.
- `medium` -- competing alternatives are within a tight tradeoff window; the decision is real but
  not obvious. Example: "Both builds are close -- small bench depth vs slight captain ceiling tradeoff."
- `low` -- high player ownership volatility expected this GW, narrow delta between options, or
  ambiguous archetype fit (below 60% profile fit). Example: "Sage is less certain -- high-variance GW
  with no dominant captain pick."

`confidence_reason` must be a concrete sentence, never a template placeholder.

## Backend contract — `POST /draft-sessions/{id}/recommend`

    {
      "recommendation": {
        "label": "Your best team",
        "session_id": "<uuid>",
        "profile_archetype": "<archetype>",
        "profile_fit_pct": 87,
        "recommendation_confidence": "high" | "medium" | "low",
        "confidence_reason": "<plain-English sentence explaining confidence level>",
        "verdict": "<1-2 sentence plain-English summary>",
        "primary_reasons": ["<reason 1>", "<reason 2>", "<reason 3>"],
        "alternatives": [
          {
            "label": "Safer version",
            "session_id": "<uuid or null>",
            "tradeoffs": {
              "improves": ["+stronger bench", "+lower volatility"],
              "worsens": ["-captain ceiling", "-flexibility at 7.5M"]
            }
          },
          {
            "label": "Upside version",
            "session_id": "<uuid or null>",
            "tradeoffs": {
              "improves": ["+captain ceiling", "+template differential"],
              "worsens": ["-bench depth", "+volatility"]
            }
          }
        ]
      }
    }

All fields set by backend. Frontend renders only.

**Acceptance**:

- `POST /draft-sessions/{id}/recommend` returns the contract above.
- Returns 400 if session has no completed audit.
- `recommendation_confidence` is always one of `high`, `medium`, or `low` -- never absent.
- `confidence_reason` is a non-empty concrete sentence, not a placeholder string.
- Response always includes exactly one primary recommendation and 1-2 alternatives — never zero alternatives.
- `profile_fit_pct` is always an integer 0-100.
- `fpl-final-recommendation.tsx` renders recommendation, reasons, profile fit, and alternatives from props only — no fetch logic inside the component.
- "Lock This Team" and "Use This Version" CTAs fire a decision receipt (can stub if WI-0658 not yet wired).
- "Go Back and Tweak" navigates to Draft Lab with session state intact.
- Screen is non-skippable — shell does not expose a route to lock without passing through this screen.
- `pytest tests/test_recommendation_service.py` passes (minimum 17 tests: archetype-aware primary, correct alternative count, 400 on missing audit, profile fit computation, tradeoff string format, confidence high/medium/low coverage, non-empty confidence_reason).
- `node web/src/__tests__/fpl-final-recommendation-contract.test.js` passes.
