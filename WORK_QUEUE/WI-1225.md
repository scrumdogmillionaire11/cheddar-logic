# WI-1225: Replace Silent Fallback in resolvePrimaryReason with Explicit Catch-All

**ID**: WI-1225
**Goal**: Replace the unconditional `SUPPORT_BELOW_PLAY_THRESHOLD` fallback at the end of `resolvePrimaryReason()` with explicit catch-all routing so that null-edge and computed-edge residual PASS cases are not conflated.

**Context**:

`resolvePrimaryReason()` in `packages/models/src/decision-pipeline-v2.js` currently ends with:

```js
if (edgePct !== null && edgePct < thresholds.edge.play_edge_min) {
  return 'SUPPORT_BELOW_PLAY_THRESHOLD';
}
return 'SUPPORT_BELOW_PLAY_THRESHOLD';
```

That final unconditional return collapses distinct outcomes under one code. This WI changes fallback semantics to:

- `edgePct === null` -> `PASS_MISSING_EDGE`
- residual computed-edge PASS fallback -> `PASS_NO_EDGE`

Note: in normal `buildDecisionV2()` flow, many null-edge payloads are pre-routed by `classifyPrice()` (`UNPRICED` + `MARKET_PRICE_MISSING`) before this helper is reached. This WI still hardens helper correctness and future call-site safety by removing the misleading silent fallback.

**Scope**:

- `packages/models/src/decision-pipeline-v2.js`
  - replace unconditional fallback in `resolvePrimaryReason()` with explicit split (`PASS_MISSING_EDGE` vs `PASS_NO_EDGE`)
  - update `resolveTerminalReasonFamily()` so `PASS_MISSING_EDGE` and `PASS_NO_EDGE` map to `EDGE_INSUFFICIENT` (not `EXECUTION_GATE`)
  - add a test-only export surface for `resolvePrimaryReason`, or equivalent in-file test harness path
- `packages/models/src/__tests__/decision-pipeline-v2-resolve-primary-reason.test.js` (new test file)
- `WORK_QUEUE/WI-1225.md`
- `.planning/STATE.md`

**Out of scope**:

- Any label or reason-code registry changes (`PASS_MISSING_EDGE` and `PASS_NO_EDGE` already exist)
- Changes to `classifyPrice()` or `NO_PRIMARY_SUPPORT` routing (handled by WI-1226)
- `SUPPORT_BELOW_LEAN_THRESHOLD` logic; that branch is explicit, not a fallback

## Implementation Contract

### Fallback Routing Semantics

`resolvePrimaryReason()` applies the following fallback routing in order:

1. If `edgePct === null` â†’ `PASS_MISSING_EDGE` (null-edge fallback)
2. If `edgePct >= 0` and `edgePct < play_edge_min` â†’ `SUPPORT_BELOW_PLAY_THRESHOLD` (explicit below-threshold)
3. If `edgePct >= play_edge_min` and `officialStatus === 'PASS'` and no prior branch matched â†’ `PASS_NO_EDGE` (computed-edge fallback)

No unconditional return statements are permitted after branch checks.

### Residual PASS_NO_EDGE Contract

`PASS_NO_EDGE` represents a defensive fallback for logically inconsistent states where:

- `edgePct >= play_edge_min`
- `officialStatus = 'PASS'`
- no prior branch matched

This state should not occur under normal `buildDecisionV2()` flow (upstream routing should catch these cases first).

Rules:
- `PASS_NO_EDGE` is a guardrail classification, not a normative path
- Tests must treat this as "unexpected but handled"
- If `PASS_NO_EDGE` appears in normal valid decision flow, escalate as a classification bug

### Terminal Family Mapping Contract

`resolveTerminalReasonFamily()` must enforce:

- `PASS_MISSING_EDGE` â†’ `EDGE_INSUFFICIENT`
- `PASS_NO_EDGE` â†’ `EDGE_INSUFFICIENT`
- No mapping to `EXECUTION_GATE` or other families

### Test Exposure Rules

`resolvePrimaryReason` must be exposed for testing via:

- Named export with `// INTERNAL_TEST_EXPORT` marker comment, OR
- Re-export from `packages/models/src/__tests__/decision-pipeline-v2-test-harness.js` (if public export too visible)

Rules:
- Must not be imported by production runtime code outside decision-pipeline
- Must not change public API surface of models package
- Export surface must be reviewed and audited before deployment

## Acceptance

- `resolvePrimaryReason({ edgePct: null, supportScore: <above lean threshold>, officialStatus: 'PASS', ... })` returns `PASS_MISSING_EDGE`.
- `resolvePrimaryReason({ edgePct: 0.01, supportScore: <above lean threshold>, officialStatus: 'PASS', ... })` (below play_edge_min) returns `SUPPORT_BELOW_PLAY_THRESHOLD`.
- `resolvePrimaryReason({ edgePct: 0.99, supportScore: <above lean threshold>, officialStatus: 'PASS', ... })` (above play_edge_min, no earlier branch matched) returns `PASS_NO_EDGE`.
- `PASS_NO_EDGE` is treated as a defensive fallback and not a normal decision path.
- `resolveTerminalReasonFamily()` maps both `PASS_MISSING_EDGE` and `PASS_NO_EDGE` to `EDGE_INSUFFICIENT`.
- No unconditional fallback return remains in `resolvePrimaryReason()`.
- Tests enforce:
  - null-edge fallback behavior
  - computed-edge fallback behavior

  - terminal family mapping correctness (EDGE_INSUFFICIENT only, never EXECUTION_GATE)
  - absence of unconditional fallback logic in helper code
- No existing tests in `decision-pipeline-v2-*.test.js` regress.
