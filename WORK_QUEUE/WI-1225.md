# WI-1225: Replace Silent Fallback in resolvePrimaryReason with Explicit Catch-All

**ID**: WI-1225
**Goal**: Replace the unconditional `return 'SUPPORT_BELOW_PLAY_THRESHOLD'` fallback at the end of `resolvePrimaryReason()` with `return 'PASS_MISSING_EDGE'` so that the null-edgePct case is distinguishable from a genuine below-threshold result.

**Context**:

`resolvePrimaryReason()` in `packages/models/src/decision-pipeline-v2.js` terminates with:

```js
if (edgePct !== null && edgePct < thresholds.edge.play_edge_min) {
  return 'SUPPORT_BELOW_PLAY_THRESHOLD';   // line 1144 — genuine below-threshold
}
return 'SUPPORT_BELOW_PLAY_THRESHOLD';     // line 1147 — silent fallback for null edgePct
```

When `edgePct` is null (edge computation unavailable), the function falls through to the final return and emits the same reason code as a genuine edge-below-threshold result. A card with "Insufficient support" could be either "edge computed and insufficient" or "edge was never computable" — the consumer can't tell. The correct label for the null case is `PASS_MISSING_EDGE` ("Edge unavailable"), which already exists in `MODEL_REASON_CODES`.

**Scope**:

**Out of scope**:

**Acceptance**:

- `resolvePrimaryReason({ edgePct: null, supportScore: <above lean threshold>, officialStatus: 'PASS', ... })` returns `'PASS_MISSING_EDGE'`.
- `resolvePrimaryReason({ edgePct: 0.01, supportScore: <above lean threshold>, officialStatus: 'PASS', ... })` (below play_edge_min) returns `'SUPPORT_BELOW_PLAY_THRESHOLD'`.
- `resolvePrimaryReason({ edgePct: 0.99, supportScore: <above lean threshold>, officialStatus: 'PASS', ... })` (above play_edge_min) returns `'PASS_MISSING_EDGE'` (genuine fallthrough — edge computed but no explicit rule fired, so edge was not the issue either; edge unavailable is still the most accurate label).
