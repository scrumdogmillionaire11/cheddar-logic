# WI-0712: Stale DB lock auto-recovery in worker startup

**ID:** WI-0712
**Goal:** Make the worker self-healing after a crash by detecting and removing a
stale DB lock file at startup, so `pm2 restart cheddar-worker` never requires
manual `rm /opt/data/cheddar-prod.db.lock` intervention.

**Problem:**
When cheddar-worker crashes (OOM, unhandled exception, `kill -9`), it leaves
`/opt/data/cheddar-prod.db.lock` on disk. The lock file contains the PID of the
writer process. On the next PM2 restart, the worker's DB open logic reads the
PID from the lockfile, finds the PID is no longer alive, but still refuses to
open the DB and exits:

```
Error: [DB] Refusing to open /opt/data/cheddar-prod.db because another process
holds the lock (/opt/data/cheddar-prod.db.lock, pid=100093).
Set CHEDDAR_DB_ALLOW_MULTI_PROCESS=true to bypass.
```

This causes a crash loop. 100093 was dead (killed earlier), but nothing cleaned
the lock. Manual fix required: `rm /opt/data/cheddar-prod.db.lock`.

The original `ecosystem.config.js` had `pre_start_script: 'rm -f .../cheddar-prod.db.lock'`
as a guard, but `pre_start_script` is not a valid PM2 option — it was silently
ignored. That option was removed in commit `70f5eb5` without a replacement.

**Root cause location:**
Wherever the DB lock is checked/enforced before `sql.js` opens the database file.
Most likely in `packages/data/src/database.js` or the worker's DB init path.

**Proposed approach:**
In the lock-check code path, before throwing, validate that the PID in the
lockfile corresponds to a running process:

```js
const fs = require('fs');

function isProcessAlive(pid) {
  try { process.kill(pid, 0); return true; }
  catch { return false; }
}

// In the existing lock guard:
if (lockExists) {
  const lockedPid = readPidFromLock(lockPath);
  if (!isProcessAlive(lockedPid)) {
    // Stale lock — previous writer is dead, safe to remove
    console.warn(`[DB] Removing stale lock (pid=${lockedPid} is not running)`);
    fs.unlinkSync(lockPath);
  } else {
    throw new Error(`[DB] Refusing to open ... pid=${lockedPid}`);
  }
}
```

This is the standard advisory-lock pattern: the lock is only valid if the writing
process is alive.

**Scope:**
- `WORK_QUEUE/WI-0712.md`
- `packages/data/src/database.js` (or whichever file enforces the lock check —
  confirm exact path during implementation via `grep -r "Refusing to open" packages/ apps/`)

**Out of scope:**
- Any changes to `ecosystem.config.js` (the `env:` block fix was applied in WI
  commit `70f5eb5` and is not related)
- `CHEDDAR_DB_ALLOW_MULTI_PROCESS` bypass flag — do not change its semantics
- Web server DB path — web uses `closeDatabaseReadOnly()` per ADR-0002 and does
  not write locks

**Acceptance:**
1. `grep -r "Refusing to open" packages/ apps/` identifies the exact lock-check
   location, and `isProcessAlive` (or equivalent) is added there
2. When the worker is started with a lockfile containing a dead PID, it logs a
   stale-lock warning and starts successfully (no crash, no manual `rm`)
3. When the worker is started with a lockfile containing the PID of a **live**
   process (e.g., another node instance), it still refuses with the existing error
4. `npm --prefix apps/worker test` exits 0 (no regressions)
5. `pm2 restart cheddar-worker --update-env` on Pi succeeds without manual lock
   removal after a simulated crash:

```bash
# Simulate stale lock
ssh ... "echo '{"pid":99999}' > /opt/data/cheddar-prod.db.lock && \
  pm2 restart cheddar-worker --update-env && sleep 6 && \
  pm2 logs cheddar-worker --lines 10 --nostream | grep -E 'stale lock|online|Enabled sports'"
# Expected: stale lock warning then normal startup
```

**Owner agent:** unassigned
**Time window:** single session
**Coordination flag:** `needs-sync` if the lock file is in a shared `packages/data`
  module — check if web server also imports the same lock-check code path and
  ensure the fix only applies when the process is the single writer (worker)
**Tests to run:** `npm --prefix apps/worker test`
**Manual validation:** simulate stale lock as described in acceptance criterion 5

**Depends on:** nothing
**Blocks:** production reliability — crash loop is triggered every time the worker
  exits abnormally, requiring SSH + manual lock removal
