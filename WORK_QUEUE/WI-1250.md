# WI-1250: Add Scheduled Game Card Coverage Watchdog

## ID
WI-1250

## Goal
Detect when scheduled near-term games have no card payload rows at all, indicating upstream pipeline failure (ingestion, modeling, or card generation).

This fills the gap between:
- market-specific coverage checks (MLB/NBA), and
- total pipeline silence for a game.

## Problem

Current health checks:
- MLB: verifies specific market coverage
- NBA: verifies moneyline coverage

But no check ensures:
> "Did we produce any card at all for scheduled games?"

Failure mode:
- Game exists in schedule
- Pipeline silently fails upstream
- No rows in `card_payloads`
- Existing checks may still pass

This is a blind spot.

## Contract Definition

### Scheduled Game Set (DENOMINATOR)

For each sport:

Include games where:
- `game_time_utc >= now + 15 minutes`
- `game_time_utc <= now + 6 hours`
- `status IN ('scheduled', 'pre')`

Exclude:
- games starting within next 15 minutes (start buffer)
- games with status: `in_progress`, `final`, `postponed`, `canceled`

Config:
```js
lookahead_hours = 6
start_buffer_minutes = 15
```

### Coverage Definition (NUMERATOR)

A game is "covered" if:

```sql
EXISTS (
  SELECT 1 FROM card_payloads
  WHERE game_id = scheduled_game.game_id
)
```

No filtering by market, status, or play type.

This is intentionally coarse:
- We only care that something was generated.

### Threshold

```text
coverage_pct = covered_games / scheduled_games

FAIL if coverage_pct < 0.50
WARN if coverage_pct < 0.80
PASS otherwise
```

## Implementation Plan

### 1. Add new check function

```js
function checkScheduledGameCardCoverage(db, sport)
```

Returns:

```js
{
  check_name: "scheduled_game_card_coverage",
  sport: "MLB",
  status: "ok" | "warning" | "failed",
  reason: "Coverage 5/5 (100%).",
  diagnostics: {
    total_games: 5,
    covered_games: 5,
    missing_game_ids: []
  }
}
```

### 2. Wire into watchdog runner

REQUIRED:
- Register in CHECK_REGISTRY
- Add to defaultChecks
- Ensure execution in main runner flow

Specifically must flow through:
- degraded-state aggregation
- alert triggering system

### 3. Persistence contract

Do NOT change schema.

Write:

```js
reason = "Coverage 4/10 (40%). Missing: [game1, game2, game3]"
```

Diagnostics returned from function but NOT persisted as structured columns.

### 4. Tests (REQUIRED)

Add behavior tests:

#### Case 1 - PASS
- 5 scheduled games
- 5 have card_payloads
- expect `status = ok`

#### Case 2 - WARNING
- 5 scheduled
- 3 covered
- expect `warning`

#### Case 3 - FAIL
- 5 scheduled
- 2 covered
- expect `failed`

#### Case 4 - Start buffer exclusion
- game within 10 minutes
- should NOT be counted in denominator

#### Case 5 - No scheduled games
- return `ok` (avoid divide-by-zero + noise)

#### Case 6 - Runner integration
- execute full `checkPipelineHealth`
- assert new check appears in results

## Acceptance

- Scheduled-game denominator is explicitly defined and enforced.
- Coverage uses existence of `card_payloads` rows only.
- New check is wired into:
  - CHECK_REGISTRY
  - defaultChecks
  - main runner
- Degraded-state and alert system receive this check.
- `pipeline_health` row persists:
  - check_name
  - status
  - reason (string only)
- Tests cover PASS / WARNING / FAIL / buffer / no-games / runner integration.
- No schema changes required.
