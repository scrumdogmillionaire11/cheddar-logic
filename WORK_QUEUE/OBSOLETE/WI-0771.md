# WI-0771: Fix MLB K market-edge step to read existing `player_prop_lines` for strikeout market

| **Priority**       | High (live market line exists in DB but is explicitly bypassed â€” edge calculation has no anchor) |

## Problem

`player_prop_lines` is populated by `pull_odds_hourly` for all active strikeout props. The MLB K model **explicitly skips reading it** per a block introduced around WI-0733:

```js
// apps/worker/src/jobs/run_mlb_model.js line ~1245
// market lines from player_prop_lines unless a future line-sourcing WI
// explicitly reintroduces that lane.
if (forKEngine) {
  mlb.strikeout_lines = mlb.strikeout_lines ?? {};
}
```

**What this means in production:**
- `player_prop_lines` has the DK/FD strikeout over/under with vig-adjusted implied probabilities
- The K engine projects a K rate and recommends over/under against... nothing â€” there is no market line to compare
- Every K card is `PROJECTION_ONLY` even when a live market line is available
- The market-edge calculation (projection vs market) is completely absent

**Why this matters:**
- Market-edge is the primary EV signal: if model projects 7.2K and the market line is 6.5K, the over has edge
- Without the line, the K engine cannot compute EV, cannot confirm edge, and cannot produce a `PLAY` classification
- The bypass was intentional as a temporary guard. WI-0733 is complete and the guard should be lifted.

---

## Input proof

`player_prop_lines` table is populated with:
- `player_id`, `player_name`, `sport='mlb'`, `prop_type='strikeouts'`
- `line` â€” the over/under number
- `over_price`, `under_price` â€” vig-adjusted prices
- `book`, `fetched_at`

The K engine already has a `strikeout_lines` object in its input structure â€” the wiring point exists, it is just blocked.

---

## Fix spec

In `apps/worker/src/jobs/run_mlb_model.js`:

1. Remove the `if (forKEngine) { mlb.strikeout_lines = mlb.strikeout_lines ?? {}; }` bypass block
2. Add a query: `SELECT line, over_price, under_price FROM player_prop_lines WHERE player_id = ? AND prop_type = 'strikeouts' AND fetched_at > ?` (use a recency window, e.g. 3 hours)
3. Attach to `mlb.strikeout_lines.dk` (or the appropriate book key)
4. When line is present: enable market-edge calculation; K cards with edge can become `PLAY`
5. When line is absent: emit `missing_inputs: ['k_market_line']`; K card remains `PROJECTION_ONLY` (same as current behavior â€” no regression)

**Output proof:** Card payload's `prop_decision.market_line` is non-null when a live line exists; `edge_pct` is computed against that line; classification can reach `PLAY` when edge threshold is met.

**Failure proof:** When no line exists in `player_prop_lines` within the recency window, `missing_inputs: ['k_market_line']` is emitted and the card stays `PROJECTION_ONLY`. No change from current behavior in the absent-line case.
