# WI-0403: Automated PENDING_VERIFICATION Resolution via Retrospective Edge Validation

## Goal

Automatically transition cards from `PENDING_VERIFICATION` status to `CONFIRMED_PLAY` or `CONFIRMED_NO_PLAY` using post-game accuracy tracking. Build a confidence-scoring model that learns which edge-anomaly conditions historically converted to real wins/losses, enabling systematic auto-resolution of edge-sanity gates without manual human review.

## Problem Statement

WI-0402 introduced `PENDING_VERIFICATION` status to flag cards with oversized edges (>20%) on non-total markets—a legitimate safety gate. However, the current implementation requires manual human intervention to resolve each card back to a playable state.

**Gap:** Some of these edge anomalies are false alarms (market noise, data freshness issues, or real +EV spots); others indicate genuine model error. Without historical accuracy tracking, we can't distinguish which is which, forcing unnecessary manual review.

**Opportunity:** Post-game settlement data already captures actual outcomes. We can analyze which PENDING_VERIFICATION cards historically had high accuracy (edge was real signal) vs. low accuracy (edge was noise), then use that feedback to auto-resolve future cards meeting similar criteria.

## Scope

1. **Post-Game Accuracy Scoring** (worker)
   - After game settlement, extract PENDING_VERIFICATION cards for that game
   - Compare original edge percentage, market conditions (sport, market type, team), prediction strength to actual outcome
   - Record pass/fail verdict + metadata in new `pending_verification_outcomes` table
   - Track: game_id, card_id, card_type, original_edge_pct, original_reason_codes, outcome (PASS/FAIL), confidence_score

2. **Confidence Score Calculation** (models/analytics)
   - Aggregate outcomes by (sport, market_type, edge_range, strength_score_bucket)
   - Calculate rolling success rate per cohort (e.g., "NHL TOTAL edge 22-25% with 0.70+ strength → 82% pass rate")
   - Publish thresholds to new `edge_verification_cohort_thresholds` table
   - Auto-resolve threshold: success_rate > 75% triggers auto-CONFIRMED

3. **Auto-Resolution Logic** (decision-pipeline-v2.js)
   - Before returning PENDING_VERIFICATION, check cohort confidence
   - If confidence >= 75%, return CONFIRMED_PLAY (with reason code EDGE_VERIFIED_AUTO_RESOLVED)
   - If confidence < 75%, keep PENDING_VERIFICATION (human review path unchanged)
   - If insufficient cohort history (<10 samples), default to PENDING_VERIFICATION (conservative)

4. **Data Migration & Backfill**
   - Backfill `pending_verification_outcomes` using settled cards from past 2 weeks
   - Generate initial cohort thresholds from historical accuracy
   - Test on future games in shadow mode before enabling resolution

5. **Web & API Updates**
   - Add `CONFIRMED_PLAY` and `CONFIRMED_NO_PLAY` status values to game-card.ts type union
   - Distinguish auto-resolved cards in UI (badge: "Edge Verified (Automated)")
   - Update API response shape to include `auto_resolved: boolean` and `confidence_score: number`

6. **Observability & Rollback**
   - Add feature flag `ENABLE_PENDING_VERIFICATION_AUTO_RESOLUTION` (default: false)
   - Log all auto-resolution decisions with cohort hit, confidence, and score
   - Include shadow-mode reporting (log what *would* have been auto-resolved without enabling)
   - Rollback plan: disable flag, cards revert to PENDING_VERIFICATION on next run

## Out of Scope
