# WI-1016: Document and gate `initDualDb` activation path

**ID**: WI-1016
**Goal**: `packages/data/src/db-dual-init.js` implements a full dual-database architecture (record DB + local DB) that is exported and documented but never activated in any production entry point. All production code uses the single-DB path via `resolveDatabasePath`. This WI clarifies intent: add a clear header comment documenting the activation requirements and current status, and add a runtime guard so `getDb()` fails loudly (not with an opaque error) if called without initialization.

**Depends on**: none

**Scope**:
- `WORK_QUEUE/WI-1016.md`
- `.planning/STATE.md`
- `packages/data/src/db-dual-init.js`
- `packages/data/index.js` (update the JSDoc comment on the `initDualDb` export to reflect current status)

**Out of scope**:
- Actually activating `initDualDb` in any worker startup sequence (separate WI if/when needed)
- Changes to `resolveDatabasePath` or the single-DB path
- DB migrations or schema changes

**Acceptance**:
- `db-dual-init.js` has a top-of-file comment block stating: "NOT currently activated in production. Single-DB path via resolveDatabasePath is the active runtime path. This module is retained for future dual-DB deployment and is safe to call only after `initDualDb()` is invoked in the entry point."
- `getDb()` throws a clear, human-readable error message (current one says "Dual-database mode not initialized" — confirm this message is clear and points to `initDualDb` as the fix)
- `packages/data/index.js` JSDoc on the `initDualDb` export updated to include: "Currently unused in production entry points."
- No behavior changes; no test changes required

**Owner agent**: unassigned
**Time window**: unscheduled
**Coordination flag**: solo

**Tests to run**:
- `npm --prefix packages/data test -- db-path --runInBand`

**Manual validation**:
- Read the top of `db-dual-init.js` and confirm the status comment is present and accurate
