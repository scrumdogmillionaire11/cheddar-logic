## Execution Plan

### Phase 0: Evidence-First RCA (Blocking Gate)

1. Capture worker service logs for the incident window.
2. Capture DB `job_run` evidence for the same window.
3. Correlate scheduler heartbeat state transitions with job start/finish markers.
4. Identify whether failure mode is:
   - unhandled exception,
   - lock contention from live owner,
   - long-running job over deadline,
   - or process-level crash/restart loop.

**Deliverable gate:** Do not implement hardening code until this evidence pack exists and points to a concrete failure mode.

### Phase 1: Safe Hardening (Conditional on Phase 0 findings)

#### Hardening A: Lock Safety Clarification and Observability
- Keep existing `acquireDbFileLock` semantics in `packages/data/src/db/connection.js`:
  - stale lock reclaim only when owner PID is not alive
  - no age-only unlink behavior
- Add/extend structured logging around lock refusal and owner metadata to speed future RCA.

#### Hardening B: Scheduler Timeout Behavior
- Use scheduler-level timeout controls in `apps/worker/src/schedulers/main.js` and heartbeat state transitions as the enforcement point.
- If additional timeout behavior is needed, implement cooperative cancellation semantics for jobs that explicitly support abort, or escalate to process restart strategy.
- Do not add wrapper-only `Promise.race` timeout around job execution that can leave unresolved background operations.

#### Hardening C: Runtime Cleanup Invariants
- Preserve `withDb` cleanup guarantees in `packages/data/src/job-runtime.js` (`finally` closes DB).
- Only extend behavior if required by RCA and compatible with current `getDatabase`/`closeDatabase` API shape.

#### Hardening D: Health Signal Coverage
- Extend existing scheduler/pipeline-health checks so prolonged running states and missing completion markers are emitted as actionable alerts.
- Prefer extending `check_pipeline_health`/heartbeat consumption over introducing a duplicate ad-hoc health job.

### Phase 2: Verification and Rollout Safety

1. Run automated tests (below).
2. Run staging-only incident simulation with explicit guardrails.
3. Document operator runbook for lock incident triage and recovery.
4. Roll out with post-deploy monitoring window and success/failure trigger conditions.
