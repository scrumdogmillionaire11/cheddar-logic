## Horizon Contract Definition (CANONICAL)

**MLB Tomorrow Rule:** Games are published/visible when their scheduled start time falls **on or before the end of the next America/New_York calendar day** (23:59:59 ET).

**Algorithm:**
```
now_utc = current UTC time
ny_tz = America/New_York

// Get midnight ET (start of today in ET)
today_midnight_et = floor(convert_to_et(now_utc)) to start-of-day

// Get end of tomorrow (23:59:59 ET = start of day+2)
tomorrow_end_et = today_midnight_et + 2 days - 1 second

// Convert back to UTC for query
horizon_utc = convert_to_utc(tomorrow_end_et)

// All games with game_time_utc <= horizon_utc are visible
```

**Examples (all times ET, today = 2026-04-24):**
- 08:00 ET now → horizon = 2026-04-25 23:59:59 ET = 2026-04-26 03:59:59 UTC (43:59:59 hours ahead)
- 18:00 ET now → horizon = 2026-04-25 23:59:59 ET = 2026-04-26 03:59:59 UTC (29:59:59 hours ahead)
- 23:30 ET now → horizon = 2026-04-25 23:59:59 ET = 2026-04-26 03:59:59 UTC (0:29:59 hours ahead)

**Key:** The visible window is **always 24h + remainder of current day**, not a fixed 36h.

## Dev/Prod Drift Root Cause

**Current Bug (from `docs/diff-dev-prod.md`):** Tomorrow MLB games display in dev but NOT in prod.

**Root Cause:**
- Dev (`.env` or env override): `ACTIVE_GAMES_LOOKBACK_HOURS=48` + `API_GAMES_HORIZON_HOURS=48` → includes yesterday + tomorrow
- Prod (default): `API_GAMES_HORIZON_HOURS=36` (fallback in `route-handler.ts:227`) → rolls back too fast, tomorrow games can fall out

**Why it's wrong:** The contract must be **ET-day-boundary aware**, not a hardcoded hour offset. Two processes at different times with the same 36h offset will see different games.

**Fix:** Replace all hardcoded hour offsets (`36`, `24`, `48`) with a **shared contract function** that computes the ET-day boundary.

  ```typescript
  import { DateTime } from 'luxon';

  export const HORIZON_CONTRACT_VERSION = 'v1-et-boundary-aware';

  /**
   * Compute the MLB games visibility horizon end time (ET day boundary).
   * 
   * Games are visible when their game_time_utc <= horizon_end_utc.
   * The horizon extends through the end of the next America/New_York calendar day.
   * 
   * @param nowUtc Date in UTC
   * @returns ISO string of horizon end (UTC)
   */
  export function computeMLBHorizonEndUtc(nowUtc: Date): string {
    const now = DateTime.fromJSDate(nowUtc, { zone: 'utc' });
    const nowET = now.setZone('America/New_York');
    
    // Tomorrow end = start of day+2 in ET, minus 1 second
    const tomorrowEnd = nowET.plus({ days: 2 }).startOf('day').minus({ seconds: 1 });
    
    // Convert back to UTC and return ISO string
    return tomorrowEnd.toUTC().toISO();
  }

  /**
   * For backwards compatibility: convert contract end time to approximate hours offset.
   * Used only in diagnostics/logging, never for queries.
   */
  export function horizonEndToApproximateHours(nowUtc: Date): number {
    const end = computeMLBHorizonEndUtc(nowUtc);
    const endMs = new Date(end).getTime();
    const nowMs = nowUtc.getTime();
    return Math.round((endMs - nowMs) / (60 * 60 * 1000));
  }
  ```
