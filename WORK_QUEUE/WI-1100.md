# WI-1100: POTD Shadow Settlement — Root Cause Diagnosis & Fix

**Goal:** Diagnose and fix why `settle-shadow-candidates` produces zero settled rows in `potd_shadow_results` despite existing scheduler integration, candidate capture, and API/UI support.

**Scope:**
- `apps/worker/src/jobs/potd/settle-shadow-candidates.js` — add targeted diagnostics, identify the failing predicate/join/insert path, and implement the minimal fix
- `apps/worker/src/jobs/settle_game_results.js` — direct data dependency: include shadow-only POTD candidate games in result materialization so the shadow settlement job has final scores to grade
- `apps/worker/src/jobs/potd/__tests__/shadow-settlement.test.js` — extend coverage for the confirmed failure mode
- `apps/worker/src/__tests__/settlement-pipeline-integration.test.js` — extend coverage for shadow-only games if needed to lock the result-materialization fix
- `apps/worker/src/__tests__/scheduler-windows.test.js` — verify existing scheduler ordering remains intact if job execution assumptions are clarified
- `apps/worker/src/scripts/potd-sanity-check.js` — optional only if needed to expose a useful operator-facing diagnostic for shadow settlement state

**Out of Scope:**
- Scheduler wiring changes in `apps/worker/src/schedulers/main.js`, `apps/worker/src/schedulers/settlement.js`, or `apps/worker/src/schedulers/windows.js`
- UI changes in `web/src/components/play-of-the-day-client.tsx`
- API aggregation changes in `web/src/lib/potd-server.ts`
- Candidate selection logic or noise-floor tuning
- Historical backfill of pre-existing shadow candidates unless the confirmed fix requires a one-time replay step

## Current Behavior

**What is already true:**
1. `potd_shadow_candidates` rows are being written
2. `settle-shadow-candidates.js` exists and is callable standalone
3. Scheduler wiring already queues shadow settlement after canonical settlement jobs
4. API/UI already expose `nearMissSummary`, including Pending and Non-gradeable counts

**Observed failure:**
1. `potd_shadow_results` remains at or near zero settled rows in production-like usage
2. The job is likely running, but not converting candidates into settled records
3. The remaining failure is in execution, data flow, eligibility, or persistence

## Known Facts To Preserve

- Existing scheduler ordering is already covered in `apps/worker/src/__tests__/scheduler-windows.test.js`
- Existing shadow settlement happy-path/idempotency coverage lives in `apps/worker/src/jobs/potd/__tests__/shadow-settlement.test.js`
- Existing API/UI logic already distinguishes `pending` and `nonGradeable`

## Hypotheses To Test

### 1. No eligible final results
- `potd_shadow_candidates.game_id` exists, but no matching `game_results` row is present
- matching `game_results.status` is not `final`
- candidate rows stay permanently `pending` because final results never become eligible

### 2. Join mismatch between candidate and result rows
- `potd_shadow_candidates.game_id` does not match `game_results.game_id`
- sport or market normalization causes grading inputs to fail after the join
- upstream ID/casing/canonicalization drift prevents the job from finding the correct result row

### 3. Settlement predicate is too strict
- `isFinalResultRow()` rejects valid settled rows
- supported market checks exclude real candidate data
- normalization or grading throws and downgrades rows to `non_gradeable`

### 4. Job is being skipped or partially gated
- `shouldRunJobKey(jobKey)` suppresses execution in real runs
- env gating or scheduler timing assumptions prevent the expected execution path
- the scheduler queues the job, but operators cannot verify it from logs today

### 5. Upsert/persistence path is not doing what we expect
- `ON CONFLICT ... WHERE potd_shadow_results.status != 'settled'` blocks intended updates
- rows insert as `pending` or `non_gradeable` and never transition later
- transaction succeeds but reveals a state-machine issue rather than a scheduler issue

## Implementation Plan

### 1. Instrument the shadow settlement job
Add concise diagnostic logging inside `settle-shadow-candidates.js` so a single run explains the flow end to end:
- candidate rows loaded
- rows with missing `game_id`
- rows with joined `game_results`
- rows rejected as non-final
- rows rejected as unsupported/non-gradeable
- rows successfully settled
- rows skipped because an existing settled result already exists

Include at least one per-candidate debug path behind a temporary or controlled log branch if aggregate counts are not enough to identify the break.

### 2. Reproduce the failure in a controlled run
Use the standalone job and existing tests to determine whether the bad state is:
- reproducible in fixtures
- only happening with production-like data
- caused by a previously untested join/predicate edge case

### 3. Fix the confirmed root cause at the job layer
Once the failing branch is identified, make the smallest fix in `settle-shadow-candidates.js`.
Examples of acceptable fixes:
- relax or correct final-result detection
- correct a normalization path for live candidate data
- allow pending shadow rows to transition to settled when final results arrive
- improve metadata written on non-gradeable rows so operators can distinguish data quality from job failure

### 4. Lock the regression with tests
Add or update tests in `shadow-settlement.test.js` for the exact confirmed failure mode.
Only touch scheduler tests if the investigation changes assumptions about when the job is expected to run.
