# WI-0904: Dead-Feature Liquidation Classification and Action Plan

**ID**: WI-0904
**Goal**: Classify every exposed feature surface as live, partially wired, inert, dead-safe-to-delete, or dead-dangerous and create a removal/remediation plan.

**Depends on**: `.planning/STATE.md` frozen domain inventory (FPL, NFL, Auth freeze table) — treat frozen WIs as the seed list for dead-safe/dead-dangerous candidates.

**Scope**:

- `apps/worker/package.json` (read-only — audit scripts for orphan/stub detection)
- `apps/worker/src/schedulers/main.js` (read-only — audit env flags and scheduled jobs)
- `apps/worker/src/jobs/*.js` (read-only — audit implementations to distinguish stubs from live jobs; do not modify)
- `web/src/app/**/*.tsx` (read-only — enumerate pages and their nav presence)
- `web/src/app/api/**/*.ts` (read-only — enumerate API routes)
- `web/src/components/**/*.tsx` (read-only — locate user-visible copy for dead-dangerous classification; do not modify)
- `docs/audits/dead-feature-liquidation.md` (created/written — primary output)
- `docs/audits/feature-status-matrix.md` (created/written — moved from runbooks; audit artifact not runbook)

**Out of scope**:

- Full implementation of deferred sports/features
- Product roadmap reprioritization
- Runtime performance tuning
- Copy remediation (dead-dangerous entries are flagged with recommended action; actual copy changes are a follow-on WI)

**Acceptance**:

- Feature registry includes runtime route/job existence, enable flag status, and user-visible claim status.
- Dead-dangerous entries are explicitly tagged when user-facing component copy implies capability that the job/route does not implement.
- Each dead or inert surface has recommended action: delete, disable, or restore.
- Scheduler/package scripts and docs are aligned with real executable capability.
- Frozen domain WIs (FPL, NFL, Auth per STATE.md) appear in the registry with their freeze rationale and classification.
