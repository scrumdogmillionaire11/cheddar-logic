# WI-0557: Wire CLV (closing line value) feedback loop

**ID**: WI-0557
**Goal**: Activate the `ENABLE_CLV_LEDGER` flag and wire closing line value tracking so the system can measure whether model edge is real (CLV > 0 = edge is confirmed; CLV ≤ 0 = likely noise).

**Scope**:
- `packages/models/src/flags.js`
- `apps/worker/src/jobs/` (settlement / grading job that runs post-game)
- `packages/data/src/db.js` (CLV ledger table or column)
- `web/src/app/api/results/route.ts` (expose CLV fields in results API if schema supports it)

**Out of scope**:
- UI rendering of CLV metrics (separate WI)
- Retroactive CLV computation for historical plays (future)
- Automated threshold adjustment from CLV data (future)

**Background**:
`ENABLE_CLV_LEDGER` exists in `flags.js` but is not wired into any job in the audited pipeline. CLV (closing line value) is the single most reliable measure of whether a model has real edge: if you consistently beat the closing line, your edge is structural. If you don't, you're likely capitalizing on vig-favorable timing or noise. Without CLV tracking, there is no feedback loop from outcomes to model calibration — the only available feedback is win rate and ROI, which require hundreds of samples to be statistically meaningful. CLV can surface model quality issues much faster.

**Acceptance**:
- `ENABLE_CLV_LEDGER` is set to `true` in `flags.js` (or enabled via environment variable with documentation in env.example).
- A `clv_ledger` table (or `clv_open`, `clv_close`, `clv_delta` columns on `plays`) exists in the DB schema.
- The post-game settlement job records: `opening_line`, `closing_line`, `model_line` (the line the model used at decision time), `clv_delta` (model_line - closing_line, from model's side).
- `clv_delta > 0` means the model beat the closing line (positive CLV = edge confirmed).
- The results API returns `clv_delta` per play when the field is populated.
- A summary CLV metric (avg CLV by sport/market/tier) is queryable from the DB.
