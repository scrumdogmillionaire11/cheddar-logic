# WI-0636: Add turbopack/dev-chunk pattern guard to deploy public HTML verification

**ID**: WI-0636
**Goal**: After fetching public HTML for each route (post-CF-purge) and before probing any asset URL, scan the extracted JS ref list for turbopack or dev-mode chunk name patterns and call `fail_with_rollback` immediately with a human-readable message identifying the offending route and ref — eliminating the current behaviour of exhausting 12 retry attempts per chunk before surfacing a misleading chunk-404 error.

**Background**:
Deploy PR #128 rolled back because `https://cheddarlogic.com/fpl` returned HTML that referenced turbopack dev-mode chunks (`%5Bturbopack%5D_browser_dev_hmr-client_hmr-client_ts_034w1is._.js`, `063q_next_dist_client_components_builtin_global-error_077rszp.js`) from a prior development build. The deploy script spent ~3 minutes looping 12×3s per chunk before reaching rollback, and the final error message (`public JS request failed ... status=404`) gave no indication that the real problem was stale dev-build HTML. A single `grep`-style check added after HTML collection — before the per-chunk retry loop — eliminates the wasted time and makes the root cause immediately obvious in the deploy log.

This WI is complementary to WI-0635 (which adds origin-ref convergence retries). Even with convergence retries in place, having an explicit labelled guard means future incidents of this class are identified in seconds rather than minutes.

**Scope**:
- `.github/workflows/deploy-production.yml` — inline SSH script block: add turbopack pattern guard immediately after the four public HTML ref collection blocks and before `printf '%s\n' "$PUBLIC_ROUTE_JS_REFS" | while ...`

**Out of scope**:
- CF purge timing changes
- Origin (127.0.0.1) chunk verification — turbopack chunks should never appear there given `next build --webpack`
- Any application or test code outside the deploy workflow
- WI-0635 scope (public-HTML convergence retries)

**Acceptance**:
- A shell block is added after `PUBLIC_ROUTE_JS_REFS` and `PUBLIC_ROUTE_CSS_REFS` are collected and before the per-ref retry loops.
- The block greps `PUBLIC_ROUTE_JS_REFS` for at least the following patterns (case-insensitive where appropriate): `turbopack`, `%5Bturbopack%5D`, `_browser_dev_`, `hmr-client`.
- On a match, the script prints the matching route and ref path, then calls `fail_with_rollback "CF served stale dev/turbopack HTML: route=<route> ref=<ref> — likely stale CF cache or a prior dev build was cached; re-run deploy after CF propagation completes"` without entering the retry loop.
- On no match, execution continues unchanged.
- The guard adds < 100ms to deploy time when no turbopack refs are present.
- The final error message visible in the GitHub Actions log clearly says "stale dev/turbopack HTML" rather than "public JS request failed".
