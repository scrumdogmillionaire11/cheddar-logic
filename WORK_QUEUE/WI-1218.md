# WI-1218: Move Inline schema guards to numbered migrations (R-02)

**ID**: WI-1218
**Goal**: Eliminate the two inline `ALTER TABLE` guards in `packages/data/src/db/cards.js` by replacing them with a proper migration file and a CI lint check that prevents recurrence.

**Context**:

`cards.js` contains two functions that run DDL directly at job-execution time:

- `ensureCardPayloadRunIdColumn()` (line 23) тАФ adds `run_id` column + index. The column already has a proper migration (`020_add_card_payloads_run_id.sql`), so this guard is dead code that survived the migration. The migration runner already has a special `isDuplicateRunId` catch block to handle this.
- `ensureActualResultColumn()` (line 36) тАФ adds `actual_result` column. **No migration exists for this column.** This is the real gap.

Both functions race the migration runner if the worker restarts during schema initialization, are not version-tracked, and cannot be rolled back.

**Pre-work (do before any code change)**:

1. Confirm `actual_result` already exists in production by running on prod host:
   `sqlite3 /opt/data/cheddar-prod.db "PRAGMA table_info(card_payloads);" | grep actual_result`
   If it exists: new migration needs graceful handling. If it does not exist: plain `ALTER TABLE` is safe.

2. Determine if the `run_id` index created in `ensureCardPayloadRunIdColumn` is already present in `020_add_card_payloads_run_id.sql` тАФ it is, confirmed: migration 020 creates `idx_card_payloads_run_id`. The guard's index creation is therefore redundant.
