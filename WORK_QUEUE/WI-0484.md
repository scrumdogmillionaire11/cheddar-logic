# WI-0484: DRIVER_ROLES registry completeness guard

**ID**: WI-0484
**Goal**: Add an automated test that cross-references every `cardType` string emitted by the worker against `DRIVER_ROLES`, failing if any are unregistered, so the `nhl-moneyline-call`-style miss cannot happen silently again.

**Depends on**:
- `WORK_QUEUE/WI-0483.md` (all known types should be registered before the guard is added)

**Scope**:
- `web/src/__tests__/driver-roles-completeness.test.js` (new file)
- `web/package.json` (add `test:driver-roles` script)
- `WORK_QUEUE/WI-0484.md`

**Out of scope**:
- Changing role assignments (WI-0483)
- Validating worker payload schemas
- CI/CD pipeline changes

**Background**:
`nhl-moneyline-call` was produced by the NHL model job but absent from `DRIVER_ROLES`, causing all NHL ML cards to silently score as `PASS` (primary_count=0). The gap was only discovered when the `test:card-decision` test was unblocked. A completeness guard would catch this at test time whenever a new cardType is added to the worker without a corresponding registry entry.

**Implementation notes**:
- The test should import `DRIVER_ROLES` from `driver-scoring.ts` (via tsx, same pattern as `test:card-decision`).
- The known-cardTypes list should be sourced from a single canonical location — either a shared constant in the worker, or an explicit allowlist in the test itself (acceptable given the list is small and stable).
- The test should assert that every cardType in the known list has an explicit entry in `DRIVER_ROLES` (i.e., `Object.hasOwn(DRIVER_ROLES, cardType)`).
- Unknown/generic model-output types (`fpl-model-output`, `mlb-model-output`, `nfl-model-output`) that are not yet active in scoring should either be registered as CONTEXT or explicitly excluded from the check with a comment.

**Acceptance**:
1. `npm --prefix web run test:driver-roles` passes when all active worker cardTypes are registered in `DRIVER_ROLES`.
2. The test fails with a clear message listing the unregistered cardType(s) if a new type is added to the worker without a registry entry.
3. The test script is added to `web/package.json`.
