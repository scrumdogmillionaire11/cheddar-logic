# WI-1282: Live Re-Check Hook for Market Pulse Opportunities

**ID**: WI-1282
**Goal**: Let the user trigger a single on-demand odds re-fetch for a specific opportunity before acting, so they can confirm the discrepancy still exists without waiting for the next hourly scan.

**Why this is worth doing**:
- WI-1275 labels every non-FRESH item "Verify before betting â€” odds may be stale." Right now that label has no action attached to it.
- Without a re-check path, the user's only options are to act on stale data or wait up to an hour for the next scheduled refresh. Both are bad.
- A single-opportunity re-check costs one targeted odds fetch, not a full scan, and resolves the staleness question in place.

## Adversarial Constraints

- This WI does NOT trigger a full market scan. It re-fetches odds for one specific game and market only.
- The re-check result must be labeled with its own freshness timestamp. It must not silently replace the existing card's `capturedAt`.
- If the re-check finds the discrepancy gone, the card must say so explicitly. It must not remain as SCOUT or WATCH.
- This is a user-initiated read, not a background job. No scheduler, no worker involvement.
- Rate-limit re-check calls per opportunity per session to prevent abuse. One re-check per opportunity per 2 minutes is sufficient.
- Do not store re-check results in the DB. They are ephemeral, session-scoped, and must not be mistaken for tracked outcome data (that belongs to WI-1281).

## API

`POST /api/market-pulse/recheck`

Request body:
```ts
{
  gameId: string;
  marketType: string;
  side: string;
  originalCapturedAt: string; // ISO, used for rate-limit key
}
```

Response shape:
```ts
{
  recheckAt: string;           // ISO timestamp of this re-check
  freshnessStatus: FreshnessStatus;
  discrepancyFound: boolean;
  currentGapPct: number | null;
  originalGapPct: number;
  gapDelta: number | null;     // positive = widened, negative = closed
  displayTier: DisplayTier;    // re-derived from current state
  verifyBeforeBetLabel: string | null;
  recheckNote: string;         // human-readable summary, e.g. "Gap closed â€” discrepancy no longer present"
}
```

Rate-limit: one re-check per `gameId + marketType + side` per 2 minutes. Return 429 with `retryAfter` if exceeded.

## Re-Check Engine

`recheck-engine.ts` accepts the opportunity identifiers and:
1. Fetches current odds snapshots for the specified game only.
2. Runs `scanOddsDiscrepancies` from `packages/models/src/mispricing-scanner.js` on those snapshots.
3. Looks up the specific market+side result.
4. Re-derives `freshnessStatus` and `displayTier` using the same rules as the opportunity engine (freshness overrides tier, per WI-1275 contract).
5. Computes `gapDelta` relative to `originalGapPct`.
6. Produces a `recheckNote` from a fixed set of phrases:
   - `"Gap closed â€” discrepancy no longer present"`
   - `"Gap persists â€” verify with your book before acting"`
   - `"Gap widened â€” discrepancy stronger than at capture"`
   - `"Data unavailable â€” could not confirm current odds"`

`recheckNote` must come from this exhaustive list. No ad-hoc copy.

## UI

`RecheckButton` appears on opportunity cards where `freshnessStatus` is `STALE_VERIFY_REQUIRED` or `EXPIRED`.

Button states:
- Idle: "Re-check odds"
- Loading: spinner, "Checking..."
- Result: inline result block showing `recheckAt`, `recheckNote`, updated `displayTier` badge, and updated gap

On FRESH items: button is not shown (odds are current enough).

The result block replaces the stale label for that card only, for the duration of the session. It does not persist across page refreshes.

## Acceptance

- `POST /api/market-pulse/recheck` returns a correctly shaped `RecheckResult` for a valid game+market+side.
- Re-check freshness is derived at call time, not inherited from the original opportunity.
- If the discrepancy is gone, `discrepancyFound = false` and `displayTier = MARKET_ONLY` or `EXPIRED` (never `SCOUT`).
- Rate-limit returns 429 with `retryAfter` when the same opportunity is re-checked within 2 minutes.
- `recheckNote` is always one of the four canonical phrases.
- Re-check results are never written to DB.
- `RecheckButton` is not rendered on FRESH cards.
- Tests assert all four `recheckNote` outcomes, rate-limit behavior, and that FRESH items are not re-check-eligible.

## Manual Validation

- Open a Market Pulse card with `STALE_VERIFY_REQUIRED` status and confirm the "Re-check odds" button appears.
- Click re-check and confirm the result block shows `recheckAt`, `recheckNote`, and updated tier badge.
- Confirm FRESH cards do not show the button.
- Click re-check twice within 2 minutes and confirm the second call returns 429.
- Simulate a closed discrepancy (mock or wait) and confirm the card shows `discrepancyFound: false` and does not render as SCOUT.
- Reload the page and confirm the re-check result is gone (not persisted).

## Failure Modes

- Re-check fetches all odds instead of the targeted game, making it as slow as a full scan.
- `recheckNote` diverges from the canonical list due to ad-hoc copy added during implementation.
- Re-check result freshness is re-used from the original `capturedAt`, making a stale re-check look fresh.
- Rate-limit key is scoped too broadly (whole session) or too narrowly (per request), defeating its purpose.
- The result block stays visible after the game ends, showing stale re-check data with no expiry.
