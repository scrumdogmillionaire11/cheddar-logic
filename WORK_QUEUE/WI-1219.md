# WI-1219: Add Nightly Scheduled DB Backup to Worker Scheduler (R-04)

**ID**: WI-1219
**Goal**: Ensure a time-based daily DB backup runs automatically, independent of settlement job execution, so a backup exists even on days with no settlement activity.

**Context**:

`db-backup.js` is a solid backup utility but is only triggered as a side-effect of two settlement jobs:

- `settle_game_results.js` -> `backupDatabase('before-settle-games')`
- `settle_pending_cards.js` -> `backupDatabase('before-settle-cards')`

This means: if settlement jobs do not run (no games, off-season, worker bug preventing settlement), no backup is created. The DB is a single SQLite file at `/opt/data/cheddar-prod.db` shared by worker and web - corruption or storage failure is a full production outage with no automated recovery path.

The existing watchdog timer (`cheddar-worker-watchdog.timer`, every 5 min) checks scheduler liveness only. There is no backup-specific systemd timer.

**Two-part fix:**

**Part 1 (required):** Add a nightly backup job to the worker scheduler, slotted in the existing nightly sweep window alongside CLV/performance reporting.

**Part 2 (required):** Add a dedicated systemd timer `cheddar-db-backup.timer` that calls a one-shot backup script in fail-safe mode only when worker is not active. This preserves the single-writer lock contract and provides backup continuity for worker-down incidents; scheduler-down-while-worker-up remains covered by the in-process scheduler backup path plus worker watchdog auto-restart.

**Scope**:

- `apps/worker/src/schedulers/main.js` - add nightly backup job in nightly sweep block
- `apps/worker/src/schedulers/windows.js` - add `keyNightlyDbBackup` key builder if needed
- `apps/worker/src/jobs/nightly_db_backup.js` (new) - thin wrapper calling `dbBackup.backupDatabase('nightly')` with `insertJobRun` / `markJobRunSuccess` / `markJobRunFailure` guards and 20-hour dedupe via `wasJobKeyRecentlySuccessful(jobKey, 1200)`
- `deploy/systemd/cheddar-db-backup.service` (new) - oneshot service calling a backup script
- `deploy/systemd/cheddar-db-backup.timer` (new) - daily at 02:47 ET with `Timezone=America/New_York` and `Persistent=true`
- `scripts/run-db-backup.sh` (new) - lock-safe script that exits without backup when `cheddar-worker.service` is active; otherwise runs `cd /opt/cheddar-logic && node apps/worker/src/jobs/nightly_db_backup.js`
- `apps/worker/src/__tests__/` - add test for `nightly_db_backup.js` asserting job guards fire correctly
- `WORK_QUEUE/WI-1219.md`
- `.planning/STATE.md`
- `env.example`

**Out of scope**:

- Changes to `db-backup.js` internals (retention, max files - those are already env-configurable)
- Removing existing settlement-triggered backups (keep them as pre-operation snapshots)
- Changes to web or shared packages

**Acceptance**:

- `apps/worker/src/jobs/nightly_db_backup.js` exists, uses deterministic daily `jobKey` plus `wasJobKeyRecentlySuccessful(jobKey, 1200)` for 20-hour dedupe, calls `dbBackup.backupDatabase('nightly')`, and wraps with `insertJobRun` / `markJobRunSuccess` / `markJobRunFailure`
- Scheduler nightly sweep block dispatches `nightly_db_backup` job when `isFixedDue` / nightly window is true
- `deploy/systemd/cheddar-db-backup.service` and `cheddar-db-backup.timer` exist with correct `User=babycheeses11`, `EnvironmentFile`, `Persistent=true`, `Timezone=America/New_York`, and daily schedule (02:47 ET)
- `cheddar-db-backup.service` refuses concurrent execution while `cheddar-worker.service` is active (fail-closed single-writer protection)
- While worker is active, timer-triggered backup run exits with an explicit skip reason in journal logs (no concurrent DB writer)
- Backup files appear under `/opt/data/backups/` after a manual `systemctl start cheddar-db-backup.service` on the production host
- Test for `nightly_db_backup.js` passes with mocked `db-backup` and `@cheddar-logic/data`
- `CHEDDAR_DB_BACKUP_RETENTION_HOURS` and `CHEDDAR_DB_BACKUP_MAX_FILES` are documented in `env.example`
- Manual restore smoke test proves latest backup can be opened as SQLite and contains expected tables

- `sudo systemctl enable cheddar-db-backup.timer && sudo systemctl start cheddar-db-backup.timer` - confirm timer is active: `systemctl list-timers | grep cheddar-db-backup`
- Confirm `journalctl -u cheddar-db-backup.service -n 20` shows success log, not error
- Restore smoke test: copy latest backup to `/tmp`, run `sqlite3 /tmp/<backup>.db '.tables'`, confirm core tables exist (for example `card_payloads`, `job_runs`)
- Steady-state timer behavior: with worker active, run `sudo systemctl start cheddar-db-backup.service` and confirm `journalctl -u cheddar-db-backup.service -n 20` shows deterministic skip (not lock error)

**Implementation notes (non-negotiable):**

- Single-writer contract must remain intact: systemd backup service must never run concurrently with `cheddar-worker.service`.
- External timer backup path is permitted only as a fail-safe path and must not bypass DB lock semantics or mutate DB state while worker is active.
- Do not use `CHEDDAR_DB_ALLOW_MULTI_PROCESS=true` in production.
- Keep scheduler backup and timer backup on different minutes to avoid deterministic overlap.

**Guard for WI closeout**:

- `deploy/systemd/cheddar-db-backup.timer` exists in repo and is deployed to `/etc/systemd/system/`
- A nightly backup file exists at `/opt/data/backups/` after first timer fire
- At least one restore smoke test has been run successfully on the produced backup file
- Worker scheduler test suite passes without regression

**Residual risks (cannot be fully mitigated in this WI)**:

- This WI improves backup frequency but does not create off-host replication; host-level disk loss can still remove both live DB and local backups
- Timer/service deployment still depends on production systemd rollout discipline; repository changes alone do not enforce runtime activation
