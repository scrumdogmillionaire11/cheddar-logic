# WI-0316: Persist ESPN Enrichment Data to Database

**ID**: WI-0316
**Goal**: Save enriched ESPN metrics to odds_snapshots after enrichment succeeds, enabling ESPN-dependent driver features and eliminating blind model execution.

**Scope**:

- [packages/data/src/odds-enrichment.js](packages/data/src/odds-enrichment.js) — `enrichOddsSnapshotWithEspnMetrics()` return value and enrichment logic
- [apps/worker/src/jobs/run_nba_model.js](apps/worker/src/jobs/run_nba_model.js), run_nhl_model.js, run_ncaam_model.js — card writer logic to persist enriched odds
- [packages/data/src/db.js](packages/data/src/db.js) — `updateOddsSnapshot()` or new `updateOddsSnapshotRawData()` function to save enriched metrics
- Unit/integration tests for enrichment persistence

**Out of scope**:

- Sport case normalization (WI-0315)
- Schema validation constraints (WI-0317)
- ESPN API changes or fallback logic modifications
- UI layer modifications
- Changing team metrics resolution logic (already working with fallbacks)

**Acceptance**:

- ESPN enrichment results (espn_metrics) successfully persisted to odds_snapshots.raw_data for all processed games
- Verification query returns non-zero: `SELECT COUNT(*) FROM odds_snapshots WHERE json_extract(raw_data, '$.espn_metrics') IS NOT NULL;`
- All model jobs (NBA, NHL, NCAAM) persist enriched data before completing
- Enriched data includes pace metrics, scoring context, or other ESPN-derived values used by models
- Models can still function if enrichment returns null/empty (graceful degradation maintained)
- Tests pass: enrichment persistence for all three sports
- SQL verification: 100+ games have non-null espn_metrics in raw_data

**Implementation Notes**:

Current enrichment flow:
1. `enrichOddsSnapshotWithEspnMetrics(oddsSnapshot)` is called in model jobs (e.g., run_nba_model.js line ~312)
2. Function returns enriched oddsSnapshot with new `raw_data.espn_metrics` property
3. **Enriched return value is never used — discarded in place**
4. Original oddsSnapshot is inserted without enrichment

Fix approach:
- After enrichment completes, update odds_snapshots table with enriched raw_data
- Create new db function: `updateOddsSnapshotRawData(gameId, sport, enrichedRawData)` 
- Call before card generation: `updateOddsSnapshotRawData(gameId, sport, enrichedOdds.raw_data)`
- Ensure enrichment is persisted even if model card generation fails downstream

Alternative: Store enrichment in separate table with FK to odds_snapshots (if raw_data becomes too large)
