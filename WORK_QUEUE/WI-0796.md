# WI-0796: JWT revocation — persist blacklist to Redis or DB

> **STATUS: DEFERRED** — Auth infrastructure not ready. Blocked on WI-0794. Revisit when auth rollout is scoped. (2026-04-05)

**ID**: WI-0796
**Goal**: Replace the in-memory `revokedTokenIds` Set in `jwt.ts` with a persistent store (Redis or a `revoked_tokens` SQLite table) so revoked tokens remain invalid across web server restarts.

**Scope**:

- `web/src/lib/api-security/jwt.ts`
- `web/src/app/api/auth/logout/route.ts`
- `packages/data/db/migrations/` (if adding a DB table)

**Out of scope**:

- Auth middleware for admin routes (WI-0794)
- Token TTL changes
- Magic-link flow changes

**Acceptance**:

- `revokeToken()` writes the token ID to Redis (`REDIS_URL` is already configured) with a TTL equal to `AUTH_ACCESS_TTL_MS`
- `isTokenRevoked()` reads from Redis, not the in-memory Set
- After web server restart, a previously-revoked token is still rejected
- If Redis is unavailable, `isTokenRevoked()` fails open (returns `false`) with a WARN log — does not crash the server

**Owner agent**: unassigned
**Time window**: TBD
**Coordination flag**: solo

**Tests to run**:

- `npm --prefix web run build`

**Manual validation**:

- Log in, log out (revoke token), restart web server, attempt to use the revoked token — must return 401
- Confirm Redis key exists with correct TTL after logout

**Guard for WI closeout**:

- In-memory Set removed from `jwt.ts`
- Redis integration tested with restart scenario
