# WI-1015: Complete `status` → `action` migration in card transform

**ID**: WI-1015
**Goal**: The card transform emits both a legacy `status` field and the canonical `action` field on every play object (L2973 in `transform/index.ts`). The `decision.ts` code falls back to `actionFromLegacyStatus` when canonical fields are absent. This WI closes the migration: audit all UI consumers, confirm they use `action`, then remove the `status` emission and the legacy fallback path.

**Depends on**: none

**Scope**:
- `WORK_QUEUE/WI-1015.md`
- `.planning/STATE.md`
- `web/src/lib/game-card/transform/index.ts`
- `web/src/lib/game-card/decision.ts`
- `web/src/lib/types/game-card.ts`
- `web/src/components/cards/GameCardItem.tsx`
- `web/src/components/cards/shared.ts`
- `web/src/components/cards/CardsPageContext.tsx`
- Any other component files under `web/src/components/` or `web/src/app/` found to read `play.status`

**Out of scope**:
- Worker-side card_payloads schema (the `status` column in the DB is not affected)
- The `status` field on settled/result card rows (settlement uses different status semantics)
- Any changes to `decision-pipeline-v2.js` or `decision-policy.js`
- `canonical-decision.ts` (covered by WI-1014)

**Acceptance**:
- Audit complete: all UI consumers of `play.status` (display decision context) identified
- All identified consumers updated to read `play.action` instead
- Legacy `status` emission removed from `transform/index.ts` (L2973 area)
- `actionFromLegacyStatus` function and its call site removed from `decision.ts` (or retained only with explicit documentation that it handles historical DB rows that pre-date the `action` field)
- `status` field removed from the display-facing `ApiPlay` / `GameCardPlay` type if it was only there for legacy fallback
- All existing tests pass; any tests asserting `play.status` values updated to assert `play.action`
- `npm --prefix web run build` passes
