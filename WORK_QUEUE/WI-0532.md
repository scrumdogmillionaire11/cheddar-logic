# WI-0532: PP matchup edge layer — penalty rate + opposition PK quality

**ID**: WI-0532
**Goal**: Add matchup-level PP adjustment factors to `projectSogV2` so that favorable PP matchups (high-penalty opponents, weak PK units) are reflected in shot projections.

**Owner agent**: unassigned
**Time window**: TBD
**Coordination flag**: solo
**Depends on**: WI-0531 (rolling PP rates must be live)

---

## Context

Even with accurate PP shot rates and PP TOI, the model currently treats all PP opportunities as equivalent. In reality:

- Team A draws 4.5 PK/60 (penalties against) vs. Team B at 2.0 PK/60 → very different PP volume expectation
- Team A's PK kills 90% of penalties vs. Team B at 78% → very different scoring environment

A PP shot projection should factor in:
1. **Opponent PK quality** (PK%) — how dangerous is a PP against this team
2. **Opponent penalty rate** (penalties taken/60) — how many PP opportunities to expect
3. **Home/away PK adjustment** — PK performance is slightly context-dependent

---

## Scope

- `apps/worker/src/jobs/pull_nhl_team_stats.js` (new or extend) — fetch team-level PK%, penalties-against/60 from NHL Stats API
- `apps/worker/src/models/nhl-player-shots.js` — add `pp_matchup_factor` to `projectSogV2` that scales PP contribution by opponent context
- `apps/worker/src/jobs/run_nhl_player_shots_model.js` — pass opponent team stats into model job; surface `pp_matchup_factor` in drivers
- DB: `team_stats` table or extend existing team data (`team_id`, `season`, `pk_pct`, `penalties_against_per60`, `updated_at`)

### Out of scope

- EV matchup adjustments (separate WI)
- Line combination PP unit modeling (beyond current scope)
- Any UI changes

---

## Acceptance

- [ ] `team_stats` table contains `pk_pct` and `penalties_against_per60` per team per season
- [ ] `projectSogV2` accepts `pp_matchup_factor` (float, default 1.0) and applies it to PP component only
- [ ] `pp_matchup_factor` derived as: `(opponent_penalties_against_per60 / league_avg_penalties) * (1 - opponent_pk_pct) / (1 - league_avg_pk_pct)` — capped [0.5, 1.8]
- [ ] Favorable matchup (weak PK, high penalty rate) → `pp_matchup_factor > 1.0`
- [ ] Tough matchup (elite PK, low penalty rate) → `pp_matchup_factor < 1.0`
- [ ] Missing opponent data → `pp_matchup_factor = 1.0` (neutral), `PP_MATCHUP_MISSING` flag pushed
- [ ] `drivers` block surfaces `pp_matchup_factor`, `opp_pk_pct`, `opp_penalties_per60`
- [ ] Tests: same player, weak PK opponent → higher `sog_mu` than strong PK opponent
- [ ] Tests: non-PP player unaffected by matchup factor (toi_proj_pp = 0)
- [ ] Anomaly guard still fires correctly even when matchup factor is favorable
