# WI-0751: NHL Player Blocks — wire actual value resolution for projection accuracy tracking

**ID**: WI-0751
**Goal**: Add player blocked-shots data to game result metadata so the projection accuracy tracker can grade `nhl-player-blk` cards against actuals.

**Background**:
`nhl-player-blk` was added to `CARD_FAMILY_MAP`, `PROJECTION_FAMILY_LABELS`, `MODEL_FAMILY_LABELS`, and `MODEL_VERSION_MAP` in the market audit (2026-04-03). The card family (`NHL_PLAYER_BLOCKS`) is now tracked in `buildProjectionSummaries`, but `resolveProjectionActualValue` (`projection-metrics.ts:285`) has no case for `NHL_PLAYER_BLOCKS` — it falls through and returns `null`. This means sample size counts will grow but MAE, bias, and directional accuracy will always show `—` because no actual values can be resolved.

The pattern to follow is `resolvePlayerShotsActualValue` (line 237) which reads `gameResultMetadata.playerShots.fullGameByPlayerId[playerId]`. The settlement job (worker) needs to populate an equivalent `playerBlocks` object in `game_result_metadata`.

**Scope**:
- `apps/worker/src/jobs/settle_game_results.js` (or equivalent) — add `playerBlocks` data to the metadata written to `game_result_metadata` for NHL games
- `web/src/app/api/results/projection-metrics.ts` — add `NHL_PLAYER_BLOCKS` case to `resolveProjectionActualValue` that reads `gameResultMetadata.playerBlocks.fullGameByPlayerId[playerId]`

**Out of scope**:
- Adding odds or settlement (blocks remain PROJECTION_ONLY)
- First-period blocks (no 1P card type currently)
- Any DB migration (add metadata to existing column)

**Acceptance**:
- After a settled NHL game with `nhl-player-blk` cards, `/api/results?sport=NHL` returns `projectionSummaries` with `NHL_PLAYER_BLOCKS` row where `sampleSize > 0` and `mae !== null`
- `/results` page renders NHL Player Blocks row in the Projection Model Accuracy section with non-null MAE
