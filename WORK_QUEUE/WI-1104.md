# WI-1104: Web Games Read Path Debt — Isolate Legacy Repair Adapter

**ID**: WI-1104
**Goal**: Reduce fragility in the web games read path by isolating legacy repair into a versioned adapter boundary while preserving current API behavior.

**Scope**:
- `web/src/lib/games/route-handler.ts`
- `web/src/lib/game-card/transform/index.ts`
- `web/src/lib/game-card/transform/legacy-repair.ts`
- `web/src/lib/game-card/transform/adapters/v1-legacy-repair.ts` (new)
- `web/src/__tests__/api-games-repair-budget.test.js`
- `web/src/__tests__/integration/games-pipeline-contract.test.ts`
- `web/src/__tests__/game-card-transform-hardening.test.js`
- `web/src/__tests__/game-card-transform-adapters.test.js` (new)

**Out of scope**:
- UI layout/styling changes
- API route path changes
- Removal of legacy fallback behavior in this tranche
- Any worker-side payload writer changes
- `isEligibleMlbGameLineFallbackRow` and `mergeMlbGameLineFallbackRows` — these are higher-level orchestration that remain in route-handler

**Acceptance**:
- Route-handler's MLB fallback payload-probe helpers are extracted to the adapter; `transform/index.ts` re-routes its existing `legacy-repair.ts` imports through the adapter. Both files have zero direct legacy-repair logic outside the adapter boundary.
- Existing games API and transform contract tests pass without baseline fixture updates.
- New adapter has explicit interface docs in-code identifying v1 payload shape characteristics (see below).
- No net increase in route-handler branching complexity.
- `games-pipeline-contract.test.ts` repair-metadata guard (`!routeHandlerSource.includes('repair_applied')` etc.) continues to pass — adapter must not reintroduce those field names.

**v1 payload shape definition** (what `adapters/v1-legacy-repair.ts` accepts):
Payloads written by the worker before the `decision_v2` canonical envelope was stabilized. Identifying characteristics:
- May have `decision_v2.official_status` directly on the decision object (not wrapped in `canonical_envelope_v2`)
- May have `execution_gate.drop_reason` for dropped cards
- Selection side may live at `decision_v2.selection_side`, `decision_v2.canonical_envelope_v2.selection_side`, or top-level `payload.prediction`
- Market context lives at `payload.market_context.wager`, with line/price at `payload.line`/`payload.juice` or inside `wager.called_line`/`wager.called_price`
- Snapshot timestamp may live at `payload.snapshot_at`, `payload.captured_at`, or `payload.created_at`

**Specific route-handler functions to move into `v1-legacy-repair.ts`**:
- `resolveMlbFallbackOfficialStatus(payload)` — probes `decision_v2.canonical_envelope_v2.official_status` → `decision_v2.official_status`
- `hasMlbFallbackDropReason(payload)` — checks `execution_gate.drop_reason`
- `hasMlbFallbackActionableSelection(payload)` — probes selection side across 4 field paths
- `hasMlbFallbackMarketContext(payload, cardType)` — validates line/price presence for `mlb-full-game` and `mlb-full-game-ml`
- `getMlbFallbackSnapshotEpoch(row, payload)` — resolves snapshot timestamp with fallback chain
