# WI-1108: Worker Runner Decomposition — Extract Settlement Annotation + Recovery Helpers

**ID**: WI-1108
**Goal**: Reduce refactor risk in oversized worker runners by extracting **strict, deterministic helper modules** for settlement annotation and MLB recovery flow, with **provable behavior parity and zero impact on betting outcomes**.

---

## **Design Contracts (Non-Negotiable)**

### **1. settlement-annotation.js**

**Purpose**: Pure settlement interpretation layer.

**Allowed Responsibilities**:

* Map raw game results → settlement outcomes (win/loss/push/void/no_contest)
* Apply annotation fields and reason codes
* Produce deterministic settlement output or patch object

**Inputs**:

* Canonical card object
* Canonical result payload (game state / stats snapshot)

**Outputs**:

* Annotated settlement object (same shape as current runner output)

**Hard Constraints**:

* No DB access
* No logging side effects
* No environment reads
* No mutation of upstream inputs
* Must be deterministic (same input → same output)

---

### **2. recovery-flow.js**

**Purpose**: Centralized recovery + retry decision logic for MLB runner.

**Allowed Responsibilities**:

* Classify failure type (transient vs terminal)
* Determine retry eligibility
* Produce structured recovery action (retry / skip / abort)
* Emit structured warning/error codes (returned, not logged)

**Inputs**:

* Error context
* Upstream fetch state
* Retry attempt metadata

**Outputs**:

* `{ action, reason_code, retryable, metadata }`

**Hard Constraints**:

* Must NOT:

  * alter model outputs
  * alter play/pass eligibility
  * introduce new fallback behavior
  * change stale-data policy
* No direct logging (runner owns logging)
* No implicit retries (only returns decisions)

---

## **Invariants (Must Hold Post-Refactor)**

* **Zero change** to betting outcomes
* **Zero change** to play/pass/lean classification
* **Zero change** to settlement ordering/precedence
* **Zero change** to payload structure, field names, or nullability
* **Zero change** to warning/error code strings
* **Zero change** to recovery decisions and branching behavior
* No new implicit defaults introduced

If any invariant is violated → WI is incomplete

---

## **Acceptance Criteria**

* Settlement annotation logic is fully extracted into `settlement-annotation.js` with strict contract adherence
* MLB recovery logic is fully extracted into `recovery-flow.js` with strict contract adherence
* All existing tests pass **without modification to expected outputs**
* Pre- vs post-refactor outputs are **byte-for-byte identical** for controlled fixtures
* No new `TODO` / `FIXME` markers introduced
* No new side effects introduced in helpers
* Runner files are strictly thinner (logic moved, not duplicated)

* Transient failure → successful retry

Comparison must include:

* Output payload
* Ordering
* Reason codes
* Warning/error markers
