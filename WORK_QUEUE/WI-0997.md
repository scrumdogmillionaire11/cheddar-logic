# WI-0997: Add probable_date to mlb_pitcher_stats to prevent stale team-fallback pitcher lookup

**ID**: WI-0997

**Goal**: Add a `probable_date` column to `mlb_pitcher_stats` so the team-fallback query in `enrichMlbPitcherData` only matches pitchers confirmed as today's probable, eliminating the silent wrong-pitcher bug when the MLB API changes a probable intra-day.

## Background

Root cause documented in `.planning/debug/mlb-wrong-pitchers-production.md`.

Odds snapshots never embed `home_pitcher` / `away_pitcher` identity â€” both are always `null`. This means `getPitcherRow()` in `enrichMlbPitcherData` always falls through to Priority 3:

```js
SELECT * FROM mlb_pitcher_stats WHERE team = ? ORDER BY updated_at DESC LIMIT 1
```

This picks whoever was most-recently upserted for the team â€” not necessarily today's actual probable. When the MLB API changes a probable (e.g. Fisher replaces Lauer for TOR) and the new pitcher is not yet in the DB, the team fallback silently returns the wrong pitcher and corrupts K projections and downstream picks.

**Acceptance**:

- `mlb_pitcher_stats` table has a `probable_date TEXT` column (nullable for old rows)
- `upsertPitcherRows` writes `probable_date` = the local `date` variable already in scope (e.g. `'2026-04-17'`), defaulting to `todayDateString()` when called standalone
- Priority 3 team-fallback in `getPitcherRow` (inside `enrichMlbPitcherData`, L~3149) uses `AND probable_date = date('now')` â€” both `forKEngine` branches consolidated to one query
- `getPitcherEraFromDb` (L2732, WITHOUT_ODDS_MODE path) is **NOT** gated by `probable_date` â€” it keeps `ORDER BY updated_at DESC LIMIT 1` unchanged, since it is only invoked when `mlb.home_pitcher == null` and `ENABLE_WITHOUT_ODDS_MODE=true` (disabled in prod)

- Regression test: uses an in-memory SQLite DB seeded with a stale row (`probable_date` = yesterday), confirms team-fallback returns `null` instead of the stale row

## Implementation notes

### 1. `pull_mlb_pitcher_stats.js` â€” `ensurePitcherStatsTable` migrations array

Add to the existing idempotent migrations array:

```js
'ALTER TABLE mlb_pitcher_stats ADD COLUMN probable_date TEXT',
```

### 2. `pull_mlb_pitcher_stats.js` â€” `upsertPitcherRows(db, rows, date = todayDateString())`

- Add `date` parameter with default `todayDateString()`
- Add `probable_date` to the INSERT column list
- Add `?` to the VALUES list in the correct position
- Add `probable_date = excluded.probable_date` to the ON CONFLICT SET block
- In the `upsert.run(...)` call, pass `date` as the `probable_date` value

### 3. `pull_mlb_pitcher_stats.js` â€” call site of `upsertPitcherRows` (line ~771)

The `date` variable is already in scope from the `pullMlbPitcherStats` function parameter. Pass it through:

```js
// Before:
const upserted = upsertPitcherRows(db, validRows);

// After:
const upserted = upsertPitcherRows(db, validRows, date);
```

Note: the scheduler calls `pullMlbPitcherStats({ jobKey, dryRun })` with no `date` arg â€” `date` defaults to `todayDateString()` inside the function. No scheduler change needed.

### 4. `run_mlb_model.js` â€” `getPitcherRow` Priority 3 (line ~3149)

Consolidate the two-branch `forKEngine` ternary into a single query gated by `probable_date`:

```js
// Before:
const byTeam = forKEngine
  ? db.prepare('SELECT * FROM mlb_pitcher_stats WHERE team = ? ORDER BY updated_at DESC LIMIT 1')
  : db.prepare("SELECT * FROM mlb_pitcher_stats WHERE team = ? AND date(updated_at) = date('now') LIMIT 1");

// After:
const byTeam = db.prepare(
  "SELECT * FROM mlb_pitcher_stats WHERE team = ? AND probable_date = date('now') LIMIT 1"
);
```

### 5. `run_mlb_model.js` â€” `getPitcherEraFromDb` (line ~2737) â€” DO NOT CHANGE

`getPitcherEraFromDb` is the WITHOUT_ODDS_MODE synthetic projection-floor fallback. It is only called when `mlb.home_pitcher == null` and `ENABLE_WITHOUT_ODDS_MODE=true` (currently `false` in prod). Gating it by `probable_date` would cause `computeProjectionFloorF5` to return `PROJECTION_FLOOR_F5_FALLBACK` for all games whenever the pitcher pull hasn't run yet that day. Leave this query unchanged:

```js
// Leave as-is:
'SELECT siera, x_fip, x_era FROM mlb_pitcher_stats WHERE team = ? ORDER BY updated_at DESC LIMIT 1'
```

### 6. `run_mlb_model.test.js` â€” regression test (in-memory SQLite DB required)

Add a describe block that seeds an in-memory SQLite DB with a stale row (`probable_date = yesterday`) and asserts the team-fallback returns `null`. Must not be a comment-only assertion. Pattern:

```js
const Database = require('better-sqlite3');

describe('WI-0997: probable_date gates team-fallback pitcher lookup', () => {
  let db;

  beforeEach(() => {
    db = new Database(':memory:');
    db.exec(`CREATE TABLE mlb_pitcher_stats (
      id TEXT, mlb_id INTEGER, full_name TEXT, team TEXT, probable_date TEXT,
      updated_at TEXT
    )`);
  });

  afterEach(() => db.close());

  test('team-fallback returns null when no row has today probable_date', () => {
    // Seed a stale row with yesterday's probable_date
    db.prepare(
      "INSERT INTO mlb_pitcher_stats VALUES ('x', 12345, 'Old Pitcher', 'BOS', date('now', '-1 day'), datetime('now', '-1 day'))"
    ).run();

    const row = db.prepare(
      "SELECT * FROM mlb_pitcher_stats WHERE team = 'BOS' AND probable_date = date('now') LIMIT 1"
    ).get();

    expect(row).toBeUndefined();
  });

  test('team-fallback returns row when probable_date matches today', () => {
    db.prepare(
      "INSERT INTO mlb_pitcher_stats VALUES ('x', 99999, 'Today Pitcher', 'BOS', date('now'), datetime('now'))"
    ).run();

    const row = db.prepare(
      "SELECT * FROM mlb_pitcher_stats WHERE team = 'BOS' AND probable_date = date('now') LIMIT 1"
    ).get();

    expect(row).not.toBeUndefined();
    expect(row.full_name).toBe('Today Pitcher');
  });
});
```
