# WI-0671: Post-Draft Season Loop Foundation

**ID**: WI-0671
**Goal**: Build the retention foundation for the season-long loop. After a team is locked, Sage must re-engage the user after GW1 with: a concrete outcome review (expected vs actual), an honest "was Sage right?" verdict, and a pointed next-step prompt. A "come back later" stub is not enough — users need a reason to come back and a clear action when they do.

## Required surfaces

### 1. Lock team and store snapshot

`POST /api/v1/season/lock-team` accepts `{ "session_id": "<uuid>", "user_id": "<id>" }`.

Stores the complete locked team snapshot to ProductStore — formation, captain, full 15-man squad. Returns `{ "locked": true, "team_id": "<uuid>", "gw_next": <int>, "gw_next_deadline": "<ISO8601>" }`. Returns 409 on duplicate lock.

### 2. Season status

`GET /api/v1/season/status?user_id=<id>` returns:

    {
      "team_locked": true,
      "team_id": "<uuid>",
      "gw_current": 1,
      "gw_next": 2,
      "gw_next_deadline": "<ISO8601>",
      "entry_available": true,
      "outcome_pending": true
    }

`entry_available` is true when the next GW deadline is within 72 hours or the GW has started. `outcome_pending` is true if GW1 is complete but the user has not yet submitted their outcome.

### 3. GW1 outcome review (required — not a future WI)

`POST /api/v1/season/gw-outcome` accepts `{ "user_id": "<id>", "gw": 1, "actual_points": <int> }`.

Returns:

    {
      "outcome": {
        "gw": 1,
        "expected_points": <int>,
        "actual_points": <int>,
        "delta": <int>,
        "sage_verdict": "right" | "close" | "missed",
        "captain_hit": true | false,
        "weakest_slot": { "position": "<pos>", "player": "<name>", "points": <int> },
        "next_step": "<pointed action string>"
      }
    }

`sage_verdict` is set by the backend based on delta magnitude:
- `right` if actual >= expected - 4
- `close` if actual >= expected - 10
- `missed` if actual < expected - 10

`next_step` must be a concrete, pointed string. Not "consider your options". Examples the backend must be capable of generating:

- "You are 1 transfer from fixing your weakest slot (Coleman, 3pts)"
- "Your bench cost you 6 points — bench depth should increase next GW"
- "Captain pick was right. Focus on your 7.5M slot next week"

### 4. Season hub UI

`fpl-season-hub.tsx` has two modes driven by `season/status`:

**Pre-GW1-result mode** (`outcome_pending: false`): locked team summary (formation, captain, key players), countdown to next GW, "GW{N} opens in {relative date}" banner. "Prepare for GW2" CTA greyed out until `entry_available: true`.

**Post-GW1-result mode** (`outcome_pending: true` and user has submitted outcome): expected vs actual points, captain hit/miss, weakest slot call-out, Sage verdict pill (right/close/missed), next step prompt in prominent text.
