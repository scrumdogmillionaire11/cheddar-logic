## Problem

`pull_mlb_pitcher_stats.js` always stores `siera = null` (line ~189) and `x_era = null` (line ~190). No computation is implemented for either.

The MLB model reads all three and applies a weighted blend:

```js
// apps/worker/src/jobs/run_mlb_model.js
SELECT siera, x_fip, x_era FROM mlb_pitcher_stats WHERE team = ?
// weights: siera=0.4, x_fip=0.35, x_era=0.25
```

**Because `siera` and `x_era` are always NULL, the blend collapses to `x_fip` alone — but the weights (0.4 + 0.25 = 0.65) are silently voided.**

**Why this is dangerous:**
- The weight math implies a three-signal composite. Code reviewers and future agents will trust a blend that does not exist.
- `siera` carries the *largest* single weight (0.40) and is always absent. The effective `x_fip` weight should be 1.0, not 0.35 — a 3× miscalibration.
- xERA is computable from Statcast barrel/hard-hit data; SIERA is computable from K%, BB%, and GB% — both are derivable from already-fetched data.

---

## Option A — Implement SIERA estimation (preferred)

SIERA formula (simplified): `SIERA = 6.145 - 16.986*(K/PA) + 11.434*(BB/PA) - 1.858*(GB_rate)`

All three inputs are derivable from `mlb_pitcher_game_logs` (`strikeouts`, `walks`, `batters_faced`) and an estimated GB rate (use league average as fallback if not tracked).

**Benefit:** The blend becomes real; `siera` weight = 0.40 is justified.

## Option B — Remove dead branches (acceptable if Statcast integration is out of scope)

1. Drop the `siera` and `x_era` columns from the SELECT (or null-guard them explicitly)
2. Rename the function to `computeXfip()` or update the comment to say `x_fip only`
3. Set the weight to `1.0` for `x_fip` (or keep normalized weights only while signals exist)
4. Add a TODO comment for when SIERA/xERA are eventually implemented
