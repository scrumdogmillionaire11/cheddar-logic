# WI-0631: Implement refresh token persistence and revocation

**ID**: WI-0631
**Goal**: Make `createRefreshToken()` in `web/src/lib/api-security/jwt.ts` store issued refresh tokens in the `revoked_tokens` DB table (or a new `refresh_tokens` table) so they can be revoked and expire, closing the gap left by WI-0608.

**Background**:
WI-0608 added a `revoked_tokens` DB table for access token revocation. However, `createRefreshToken()` in `web/src/lib/api-security/jwt.ts` has two `TODO: Store in Redis/DB` comments (lines 109, 113) and currently returns a 32-byte random hex string stored nowhere. Refresh tokens cannot be invalidated on logout or server restart. This leaves a security gap alongside the completed IP hardening in WI-0609.

**Scope**:
- `web/src/lib/api-security/jwt.ts` — implement refresh token storage: on `createRefreshToken()`, insert a row into a `refresh_tokens` table (or reuse `revoked_tokens` style) with token hash, expiry, and userId. Add `revokeRefreshToken(token)` and `isRefreshTokenValid(token)` helpers.
- `web/src/app/api/auth/` — call `revokeRefreshToken` on logout/sign-out if a refresh token is present in the request.
- DB migration — add `refresh_tokens` table with columns: `token_hash TEXT PK`, `user_id TEXT`, `expires_at TEXT`, `created_at TEXT`, `revoked_at TEXT`.

**Out of scope**:
- Redis integration (DB persistence is sufficient for current scale)
- Refresh token rotation (single-use tokens are out of scope for now)

**Acceptance**:
- `createRefreshToken()` stores a hashed token in the DB.
- `isRefreshTokenValid(token)` returns false for expired or manually revoked tokens.
- `TODO: Store in Redis/DB` comments are removed from `jwt.ts`.
- `npx tsc --noEmit --project web/tsconfig.json` passes.
- `npm --prefix web run lint` passes.
