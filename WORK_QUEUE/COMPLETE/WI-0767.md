## Problem

The FPL model scheduler registration uses the wrong environment variable guard:

```js
// apps/worker/src/schedulers/main.js
// Current (WRONG):
env: 'ENABLE_NFL_MODEL'

// Should be:
env: 'ENABLE_FPL_MODEL'
```

This is a copy-paste error. Setting `ENABLE_FPL_MODEL=true` in `.env` has no effect — the scheduler reads `ENABLE_NFL_MODEL` and will never enable the FPL job.

**Additional gaps:**
1. No FPL data pull jobs exist in the main worker (`packages/`, `apps/worker/src/jobs/`)
2. The `cheddar-fpl-sage/` subproject is a standalone Python/React app with its own DB — no integration path to the main cheddar-logic DB exists
3. `ENABLE_FPL_MODEL=false` in `.env`

**Impact:**
- The env-var bug means even after building an FPL data layer, the model cannot be enabled without discovering this hidden bug
- The architectural split (Python standalone vs. Node worker) is undocumented — future agents will not know how FPL data could flow into the main worker

## Integration Decision (required as part of this WI)

The ADR must answer:

**Option A — Integrate into main worker**
Build `pull_fpl_data.js` fetching from FPL API (or from `cheddar-fpl-sage` DB via read-only query), write FPL-relevant rows into main DB tables, run `run_fpl_model.js` as a normal job.

**Option B — Keep standalone, add bridge**
`cheddar-fpl-sage` remains a standalone Python app. A lightweight bridge (e.g. HTTP endpoint or shared file export) feeds aggregated FPL data into the main worker's `odds_snapshots` or a new `fpl_context` table.

**Option C — Keep fully standalone, no bridge**
FPL remains completely separate. Remove the FPL scheduler entry from the main worker entirely. Add an explicit comment in main.js: `// FPL is a standalone app — see cheddar-fpl-sage/`.
