# WI-0459: Soccer Odds + Projection-Only Market Rework

**ID**: WI-0459
**Goal**: Fix soccer so it actually produces cards. Add ML/double-chance/game-total odds pulls from The Odds API. Separate odds-backed markets from projection-only player prop markets (TSOA, Player Shots, Team Totals), which the user values independently.

**Depends on**: `WORK_QUEUE/WI-0437.md` (completed)

**Root cause of "No recent SOCCER odds found"**: The model only runs when `getOddsWithUpcomingGames('SOCCER')` returns rows ŌĆö but currently that only happens if EPL games are in the DB AND the `h2h` market was stored and survived all normalizer gates. This is fragile and not the right architecture for projection-only markets anyway.

**Scope**:

- `apps/worker/src/jobs/run_soccer_model.js` ŌĆö restructure to two tracks: odds-backed and projection-only
- `packages/data/src/validators/card-payload.js` ŌĆö add/update soccer card type schemas
- `apps/worker/src/jobs/__tests__/run_soccer_model.test.js` ŌĆö update tests

**Out of scope**:

## Market Model (new)

### Odds-backed markets (pulled from The Odds API ŌåÆ emit cards with real price)

| Market | API market key | Ohio status |
|--------|---------------|-------------|
| Moneyline (h2h) | `h2h` | Tier 1 ŌĆö already being pulled |
| Game total (o/u goals) | `totals` | Tier 1 ŌĆö add to config |
| Double chance | `doubleChance` | Tier 1 ŌĆö add to config; remove from `OHIO_BANNED_MARKETS` |

### Projection-only markets (no odds from API ŌĆö user finds their own line/price)

| Market | Source | Card behavior |
|--------|--------|---------------|
| Team Totals | model projection | Emit projection card; no price; user finds value |
| Player Shots | model projection | Emit projection card; no price; user finds value |
| TSOA | model projection | Emit projection card; no price; user finds value |

Projection-only cards should still run even when `getOddsWithUpcomingGames` returns zero rows (they derive from a game schedule / fixture list, not the odds snapshots table).

## Implementation Plan

### Stage A ŌĆö config.js: add markets to SOCCER

Add `totals` and `doubleChance` to `SPORTS_CONFIG.SOCCER.markets`. Update `tokensPerFetch` from 1 ŌåÆ 3.

```js
SOCCER: {
  markets: ['h2h', 'totals', 'doubleChance'],
  tokensPerFetch: 3,
  ...
}
```

### Stage B ŌĆö run_soccer_model.js: fix banned market list

Remove `double_chance` from `OHIO_BANNED_MARKETS`. Add it (and `soccer_ml`, `soccer_game_total`) to `OHIO_TIER1_MARKETS`.

Canonical market key mapping:
- `h2h` / `moneyline` / `soccer_ml` ŌåÆ `soccer_ml`
- `totals` / `game_total` ŌåÆ `soccer_game_total`
- `double_chance` ŌåÆ `soccer_double_chance`

### Stage C ŌĆö run_soccer_model.js: two-track model runner

**Track 1 ŌĆö odds-backed** (unchanged trigger: requires odds snapshot in DB):
- Read `getOddsWithUpcomingGames('SOCCER')` as before
- For each game: emit `soccer_ml`, `soccer_game_total`, `soccer_double_chance` cards when the odds snapshot includes the matching market
- These are actionable cards with real price/line

**Track 2 ŌĆö projection-only** (new: runs independently of odds):
- Read upcoming games from `games` table directly (not filtered by odds existence)
- For each game: build TSOA, Player Shots, Team Totals projection packets using the existing `buildSoccerTier1Payload` logic
- Emit `soccer-ohio-scope` cards with `projection_only: true` in payload
- Pass reason when projection context is insufficient (existing degradation logic)

The runner should attempt both tracks for every run, not bail out if Track 1 finds no odds.

### Stage D ŌĆö validator: update schemas

- Add `soccer_ml` schema: h2h price, home/away teams, edge basis
- Add `soccer_game_total` schema: line, over/under price, edge basis
- Add `soccer_double_chance` schema: outcome (home-or-draw / away-or-draw / either-to-win), price, edge
- Update `soccer-ohio-scope` schema: add `projection_only: true` flag as valid; don't require price fields when `projection_only`

### Stage E ŌĆö update tests

- Happy path for each new odds-backed card type
- Projection-only cards emit even when no odds rows returned
- double_chance no longer rejected by normalizer
- Validator accepts `projection_only: true` with no price fields
