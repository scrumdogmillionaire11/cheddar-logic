# WI-0942: Decision-Layer Simplification (Holistic Blocking Fix)

**ID**: WI-0942
**Goal**: Reduce contradictory multi-layer decisioning by introducing one canonical block/decision contract used consistently from model output through worker publication to web surfacing.

**Depends on**: WORK_QUEUE/COMPLETE/WI-0940.md, WORK_QUEUE/WI-0941.md

**Enumerated tech debt to address in this WI**:

- TD-01: Duplicate terminal-status mutation paths across model/worker/web (`decision_v2.official_status`, `action`, `classification`, `status`) that can drift.
- TD-02: Legacy/fallback decision recomputation in web consumption paths when canonical decision data exists.
- TD-03: Inconsistent veto/demotion stamping (PASS/LEAN overrides) in worker jobs after publish-time normalization.
- TD-04: Cross-layer reason-code fragmentation (same terminal outcome represented by different code families without a canonical mapping).
- TD-05: Stale comments or compatibility branches that imply decision ownership outside canonical decision envelope flow.

**Tech debt handling rule (required)**:

- In-scope legacy/tech-debt encountered while implementing WI-0942 must be either:
  - removed in this WI when required for correctness/parity, or
  - recorded in the audit debt ledger as `retained-intentional` with rationale and follow-up WI.
- No silent carry-forward of discovered in-scope debt is allowed.

**Canonical decision envelope contract (required)**:

Every candidate must expose one canonical envelope at `decision_v2.canonical_envelope_v2` with, at minimum:

- `official_status`: `PLAY | LEAN | PASS`
- `terminal_reason_family`: single terminal family string (exactly one)
- `primary_reason_code`: single primary reason code
- `reason_codes`: ordered reason code array including `primary_reason_code`
- `is_actionable`: boolean derived from canonical status policy
- `execution_status`: `EXECUTABLE | PROJECTION_ONLY | BLOCKED`
- `publish_ready`: boolean

Rules:

- One candidate ends with exactly one terminal reason family.
- `primary_reason_code` must be present in `reason_codes`.
- No layer may recompute or override terminal outcome when canonical envelope exists.
- Web fallback logic is allowed only when canonical envelope is missing and must be logged as legacy compatibility.

**Acceptance**:

- A single canonical decision envelope defines final status, block family, primary reason, and visibility eligibility.
- Duplicate demotion logic across model, publisher, and web transform is removed or delegated to canonical helpers.
- Cross-layer parity tests prove no disagreement between worker persisted status and web surfaced status for equivalent payload classes.
- Blocking outcomes are deterministic and traceable: one candidate ends with exactly one terminal reason family.
- ADR documents migration rules and compatibility strategy for existing payload consumers.
- Audit report includes baseline vs post-change reason-family fragmentation and parity outcomes.
- Tech-debt closeout is objective and complete:
  - TD-01: parity assertions prove canonical and legacy status fields remain internally consistent where legacy fields are still emitted.
  - TD-02: web contract assertions prove canonical decision data is preferred over fallback recomputation when present.
  - TD-03: worker tests prove veto/demotion paths stamp canonical decision outcome consistently.
  - TD-04: audit includes explicit canonical mapping for dominant reason families and identifies eliminated/merged fragments.
  - TD-05: stale legacy comments/branches touched in-scope are removed or recorded as retained-intentional debt.
