# WI-0723 тАФ Build Snapshot Comparator + Truth Cleanup

**ID:** WI-0723

## Implementation Plan

### 1. `audit_rules_config.js`

Define the audit truth contract in config so comparator behavior is centralized and testable.

Required config areas:

- strict field paths, including canonical `_pricing_state.status`
- tolerant field tolerances with support for scalar and directional forms
- ignored fields
- per-card-family identity builders
- canonical selection normalization rules

### 2. `audit_invariants.js`

Define and export invariant checkers. Each returns `{ passed: boolean, violation: object | null }`.

Required invariants:

```js
// INV-001: No code mutates classification, action, status, or decision_v2.official_status
// after publishDecisionForCard returns.
checkNoPostDecisionClassificationRewrite(decisionSnapshot, publishSnapshot)

// INV-002: No executable card exists when pricing is not ready.
checkNoPricingGap(publishSnapshot)

// INV-003: No publish when watchdog block reason present.
checkNoPublishOnWatchdogBlock(publishSnapshot)

// INV-004: No projection-only payload marked as executable.
checkProjectionOnlyNotExecutable(publishSnapshot)

// INV-005: Consistency fields present for publishable totals.
checkConsistencyFieldsPresent(publishSnapshot)

// INV-006: execution_status must agree with prediction/pricing/publish state envelopes.
checkExecutionStateConsistency(publishSnapshot)
```

Each invariant must:
- Accept snapshot objects (not live card objects)
- Return a structured `violation` object with `{ invariant_id, card_key, field_path, expected, actual, severity }`
- Severity: `CRITICAL` for impossible/illegal state, `WARN` for non-blocking metadata gaps

### 3. `compare_audit_snapshot.js`

Core function: `compareSnapshots(actual, baseline, options)` тЖТ diff result

#### Comparison classes (configured per field family, NOT global):

**Strict fields** тАФ any deviation is `HIGH` severity drift:
- `classification`
- `execution_status`
- `actionable`
- `official_status` / `decision_v2.official_status`
- `reason_codes` (set equality)
- `card_type`
- `market_type`
- `selection_signature`
- `_pricing_state.status`
- `_publish_state.publish_ready`
- `official_eligible`
- NHL: `nhl_goalie_certainty_pair`, `homeGoalieCertainty`, `awayGoalieCertainty`, `homeAdjustmentTrust`, `awayAdjustmentTrust`

**Tolerant fields** тАФ deviation within tolerance is `WARN`, outside is drift:
- `confidence` (tolerance: ┬▒0.03)
- `edge` / `edge_pct` (tolerance: ┬▒0.005)
- `p_fair` / `p_implied` (tolerance: ┬▒0.02)
- `projection.total` (tolerance: ┬▒0.5)
- `projection.margin_home` (tolerance: ┬▒1.0)
- NHL: `homeExpected`, `awayExpected`, `expectedTotal` (tolerance: ┬▒0.2)

**Ignored fields** тАФ never compared:
- `generated_at`
- `run_id` / `runId`
- `captured_at`
- `job_run_id`
- `created_at` / `updated_at`
- `countdown`
- `start_time_local` (derived from game time, not model output)

Additional comparator rules:

- Normalize `reason_codes` as uppercase, deduped, sorted set equality
- Normalize selection from `selection.side`, `selection_type`, `market_context.selection_side`, or `prediction`, plus normalized period
- Use configurable identity matching for card arrays; never compare by position alone
- Emit `SPEC_DRIFT` for type changes, missing required fields, conflicting side sources, or unmatchable identities
- Collapse redundant downstream noise; report the earliest failing boundary per field/card

#### Stage-specific diffing:

For each of the five stages, collect field diffs and classify by drift category:
- Input stage drift тЖТ `INPUT_SHAPE_DRIFT`
- Enriched stage drift тЖТ `INPUT_SHAPE_DRIFT`
- Model stage drift тЖТ `MODEL_DRIFT`
- Decision stage drift тЖТ `DECISION_DRIFT`
- Publish stage drift тЖТ `PUBLISH_DRIFT`

#### Diff summary item shape:

```json
{
  "card_key": "nba-20260401-bos-nyk|nba-total-call|TOTAL|FULL_GAME|OVER",
  "field_path": "payloadData.classification",
  "expected": "PLAY",
  "actual": "LEAN",
  "drift_type": "DECISION_DRIFT",
  "severity": "HIGH",
  "comparison_class": "strict"
}
```

#### Return value:

```json
{
  "fixture_id": "...",
  "passed": false,
  "critical_count": 1,
  "high_severity_count": 1,
  "warn_count": 2,
  "invariant_violations": [],
  "diffs": []
}
```

#### Exit behavior:

`compareSnapshots` should not throw тАФ it returns results. The caller (`run_model_audit.js`) is responsible for exiting non-zero on `CRITICAL` or `HIGH` results.

### 4. Truth cleanup

- `decision_v2.official_status` and `execution_status` are the only authoritative decision truth
- `classification`, `action`, `status`, `prediction`, and legacy readiness booleans remain compatibility fields only
- Gate-held publish may add metadata, but must not silently rewrite canonical decision meaning after the decision snapshot
- `pipeline_state.pricing_ready` remains derived from explicit state envelopes and may not diverge from `_pricing_state.status`
- The audit harness must snapshot canonical state first and only derive compatibility fields second

## Acceptance

- Strict fields: any deviation returns `severity=HIGH`
- Tolerant fields: within-tolerance deviation returns no entry; outside returns `severity=WARN`
- Ignored fields: never appear in diff output
- INV-001 (post-decision rewrite): fires as `CRITICAL` when `decision_v2.official_status` changes between decision and publish snapshot
- INV-002 (executable without pricing): fires as `CRITICAL`
- INV-004 (projection-only marked executable): fires as `CRITICAL`
- INV-006 (execution-state mismatch): fires as `CRITICAL`
- Tolerance map is per-field-family, not global
- Comparator uses identity matching, not array order
- Gate-held publish keeps canonical decision truth stable and only adds non-authoritative gate metadata
- `pipeline_state.pricing_ready` is derived from canonical state and cannot silently disagree with `_pricing_state.status`
- Worker tests pass
