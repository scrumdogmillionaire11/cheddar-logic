# WI-0391: Active Plays Toggle + In-Progress Tracking

**ID**: WI-0391
**Status**: SUPERSEDED by WI-0388 (2026-03-11)
**Goal**: Add a user-facing Active Plays toggle and canonical in-progress play lifecycle so cards visibility is controlled by explicit state, not implicit time-based filtering.

**Supersession note**:
WI-0388 implemented the primary requirements: user-facing cards page toggle, derived lifecycle layer on `/api/games`, and explicit lifecycle-driven visibility for games. The net-new remaining scope is **API card endpoints parity**: applying the same lifecycle filter to `/api/cards` and `/api/cards/[gameId]` with regression tests for settled games. See successor work item (e.g., WI-0392) for the remaining parity work.

**Scope (net-new only)**:

- `WORK_QUEUE/WI-0391.md`
- `web/src/app/api/cards/route.ts` (add `lifecycle` query param support)
- `web/src/app/api/cards/[gameId]/route.ts` (add `lifecycle` query param support)
- `web/src/__tests__/api-cards-lifecycle-regression.test.js` (new regression suite)

**Out of scope**:

- Settlement grading logic (`apps/worker/src/jobs/settle_pending_cards.js`)
- ESPN/final score ingestion (`apps/worker/src/jobs/settle_game_results.js`)
- Historical backfill/resettlement of existing records
- Cards page UI toggle (completed by WI-0388)
- `/api/games` lifecycle filtering (completed by WI-0388)
- Derived lifecycle fields and display_status (completed by WI-0388)

**Acceptance**:

- `/api/cards` supports `?lifecycle=active` query param, returning only cards from non-settled games (same filtering as `/api/games?lifecycle=active`)
- `/api/cards/[gameId]` supports `?lifecycle=active` query param with same settled-game exclusion logic
- Default behavior (no `lifecycle` param) returns full history of cards including settled games
- Regression tests pass: cover at least one settled game and one in-progress game path in both modes
- Behavior is consistent with `/api/games` lifecycle filtering (uses same `ACTIVE_EXCLUDED_STATUSES` constant)
