## Context

After the MLB projections fix, `run_mlb_model.js` emits `mlb-f5` and `mlb-pitcher-k` cards into `card_payloads` and `card_results` (status=`pending`). However three gaps remain that prevent settlement and display on `/results`:

### Gap 1: No `game_results` rows for MLB games (blocks everything downstream)

`settle_game_results.js` only has `NHL`, `NBA`, and `NCAAM` in its `ESPN_SPORT_MAP` (line 49). No MLB entry exists, so `game_results` rows with `status='final'` are never created for MLB games.

Two downstream components depend on a `game_results` row with `status='final'` existing:
- `/api/results` projection tracking query: `INNER JOIN game_results gr ON gr.game_id = cr.game_id WHERE gr.status = 'final'` ŌĆö returns 0 MLB rows without it
- `settle_mlb_f5.js`: `UPDATE game_results SET metadata = json_set(..., '$.f5_total', ?)` ŌĆö the UPDATE silently affects 0 rows if no game_results row exists

**Fix:** Add `MLB: 'baseball/mlb'` to `ESPN_SPORT_MAP` in `apps/worker/src/jobs/settle_game_results.js`. The ESPN scoreboard endpoint `https://site.api.espn.com/apis/site/v2/sports/baseball/mlb/scoreboard` works the same as NHL/NBA. Verify that MLB game IDs from `pull_espn_games_direct.js` match what ESPN returns for the scoreboard ŌĆö they use the same ESPN game IDs, so the match should be direct.

If ESPN game_id mismatch is a problem (MLB's canonical game_id comes from `statsapi.mlb.com` not ESPN in some cases), alternative: add a new lightweight `settle_mlb_game_results.js` job that calls `https://statsapi.mlb.com/api/v1.1/game/${gamePk}/feed/live` (already used by `settle_projections.js` for pitcher Ks) and calls `upsertGameResult` directly. This avoids any ID-mapping concern. Register it in `apps/worker/src/schedulers/main.js` in the settlement pipeline, gated by `ENABLE_SETTLEMENT`.

### Gap 2: `mlb-pitcher-k` actual value always returns `null` on `/results`

In `web/src/app/api/results/projection-metrics.ts` line 358:
```typescript
if (cardFamily === 'MLB_PITCHER_K') {
  return null;  // hardcoded ŌĆö accuracy never tracked
}
```

`settle_projections.js` already writes striker K count per pitcher to `card_payloads.actual_result` (a JSON column: `{ pitcher_ks: N }`). But the `/api/results` projection tracking query does not SELECT `cp.actual_result`, so it is unavailable to `resolveProjectionActualValue`.

**Fix (two-part):**

Part A ŌĆö `web/src/app/api/results/route.ts`: Add `cp.actual_result` to the projection tracking SELECT (around line 733):
```sql
SELECT
  cr.sport,
  cr.card_type,
  cp.payload_data,
  cp.actual_result,           -- add this
  gr.metadata AS game_result_metadata
FROM card_results cr
LEFT JOIN card_payloads cp ON cp.id = cr.card_id
INNER JOIN game_results gr ON gr.game_id = cr.game_id
WHERE cr.card_type IN (${projTrackingPlaceholders})
  AND gr.status = 'final'
```

Part B ŌĆö `web/src/app/api/results/projection-metrics.ts`:
- Add `actualResult?: string | null` to `ProjectionMetricInputRow` type
- Pass `actualResult: row.actual_result` when mapping rows in `route.ts`
- Implement `resolveProjectionActualValue` for `MLB_PITCHER_K`:
```typescript
if (cardFamily === 'MLB_PITCHER_K') {
  if (!row.actualResult) return null;
  const parsed = safeJsonParse(row.actualResult);
  return toNumber((parsed.data as Record<string, unknown> | null)?.pitcher_ks);
}
```

### Gap 3: `mlb-pitcher-k` projection value resolution

Double-check `resolveProjectionValue` in `projection-metrics.ts` returns a meaningful number for `MLB_PITCHER_K` (the model's projected strikeout count). Look at what `payload_data` contains for `mlb-pitcher-k` cards ŌĆö it should have a `projection` or `pitcher_k_projection` numeric field. Confirm or fix as needed.
