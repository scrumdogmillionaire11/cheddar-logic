# WI-0594: Active tab regression — in-progress games missing ACTIVE label in prod

**ID**: WI-0594
**Goal**: Diagnose and fix production-only regression where games with `status = 'in_progress'` are not surfacing in the Active tab and receive no `display_status: "ACTIVE"` label. Dev environment works correctly; production does not.

**Scope**:
- `apps/worker/src/jobs/pull_schedule_nhl.js` — `normalizeStatus()`, verify `statusType.state === 'in'` branch writes `in_progress` to prod DB
- `apps/worker/src/jobs/pull_schedule_nba.js` — same
- `web/src/app/api/games/route.ts` — `resolveLifecycleMode()`, `ACTIVE_EXCLUDED_STATUSES`, `game_time_utc` predicate, `deriveDisplayStatus()`
- `web/src/__tests__/api-games-filter.test.js` — add prod-parity regression case
- `web/src/__tests__/active-tab-actionable-filter.test.js` — add regression guard

**Out of scope**:
- Pick grading logic
- Settlement pipeline (`settle_pending_cards.js`, `settle_game_results.js`)
- Card results display or styling
- Any file not listed above

**Acceptance**:
- Prod Active tab shows game cards while a game is live (status `in_progress`, `game_time_utc <= NOW`, no final result)
- `GET /api/games?lifecycle=active` in prod returns in-progress games with `display_status: "ACTIVE"` and `lifecycle_mode: "active"`
- A test in `api-games-filter.test.js` proves a game row with `status = 'in_progress'` and `game_time_utc <= NOW` is included in `?lifecycle=active` results and excluded once status transitions to `final`
- Both affected test files pass in CI

**Investigation checklist** (work through in order during execution):

1. **DB state check** — query prod DB for current game status distribution:
   ```sql
   SELECT status, COUNT(*) FROM games
   WHERE game_time_utc <= datetime('now')
   GROUP BY status;
   ```
   If `in_progress` rows are absent but games are live, the worker is not writing the status — focus on `normalizeStatus()` in the pull_schedule jobs.

2. **normalizeStatus() audit** — confirm `statusType.state === 'in'` branch is reachable AND that ESPN's live API actually returns `state: 'in'` (not e.g. `'live'`, `'inprogress'`) in the prod data feed. Check worker logs for recent `pull_schedule_*` runs.

3. **SQL predicate audit** — in `resolveLifecycleMode()` in `route.ts`, verify:
   - `game_time_utc <= datetime('now')` comparison is valid for the SQLite version in prod (no UTC offset drift)
   - `status NOT IN ACTIVE_EXCLUDED_STATUSES` covers exactly the right values and does not inadvertently exclude `in_progress` (e.g. via a stray string comparison issue)
   - The `NOT EXISTS (SELECT 1 FROM game_results ...)` sub-query is not matching in-progress games prematurely

4. **Dev vs prod diff** — run `git log --oneline` diff between last known-good prod deploy and HEAD, filtered to the four files above, to identify any change that might have introduced the regression.

5. **Environment check** — verify prod `CHEDDAR_DB_PATH` points to the correct DB containing live `card_payloads` and `games` tables (not a stale or rotated DB).
