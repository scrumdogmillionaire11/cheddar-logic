# WI-0560: Fail closed in production when AUTH_SECRET is missing or default

**ID**: WI-0560
**Goal**: Prevent production auth from silently using the insecure default secret (`dev-auth-secret-change-me`).

**Scope**:
- `web/src/lib/api-security/jwt.ts`
- `web/src/app/api/auth/token/route.ts` (error handling and user-facing messaging)
- `env.example`
- `.env.production.example`

**Out of scope**:
- Rotating existing production tokens
- External secret-manager integration
- Reworking the auth model (RBAC/resources)

**Background**:
`getAuthSecret()` currently falls back to `'dev-auth-secret-change-me'` when neither `AUTH_SECRET` nor `CHEDDAR_AUTH_SECRET` is set. In production this should be a hard error, not a warning, because any actor with the known default can forge valid tokens.

**Acceptance**:
- In production mode, token creation/verification refuses to run when secret is missing or equals the known default.
- API responses for auth failures use explicit, actionable messaging (misconfiguration, not generic unknown error).
- Example env files document required auth secret configuration for production deployments.
- Development mode retains a safe local-dev path (warning acceptable), but production path is fail-closed.
