**Acceptance**:

Captain contract -- `captain.expected_pts` and `vice_captain.expected_pts` are always present as floats (>= 0) in the completed analysis response regardless of selection path (`recommend_captaincy_from_xi`, `recommend_captaincy`, or fallback); neither field is ever `None`, missing, or derived by parsing rationale text.

Strategy path validity:

- Strategy mode auto-selects from rank bucket, then is nudged by risk posture; `>3M` rank enters `RECOVERY` by default
- Transfer hit thresholds are materially lower in `RECOVERY` than `BALANCED`
- No strategy path contains a player already in the manager's current squad as the `in` target
- No two paths in a single response share the same `(out, in)` pair (deduplicated before serialization)
- Each path carries a distinct `rationale` string reflecting its mode-specific tradeoff (safe/balanced/aggressive paths do not produce identical copy)

Projected XI / transfer linkage -- `projected_xi` and `projected_bench` in the response reflect the primary recommended transfer applied and are computed by the backend serializer; these fields are never absent when a primary transfer plan exists; format matches `starting_xi` / `bench` player object shape.

Health / risk normalization -- all risk state (`injured`, `doubtful`, `health_pct`, `critical_positions`) originates from a single normalized `squad_health` object in serialization; no separate health inference path emits conflicting counts elsewhere in the payload.

Confidence metadata -- `confidence_score` is always accompanied by `confidence_label` (string: `"HIGH"` / `"MEDIUM"` / `"LOW"`) and `confidence_summary` (one sentence naming the primary driver); raw score alone is never the only field serialized for confidence.

Transfer rationale depth -- transfer objects include `why_now` (short trigger reason), `risk_note` (one sentence on downside), and `horizon_gws` (int: projection window) alongside the existing `reason` string.
