## Background

The MLB runner currently blends these concerns in a single pass:
- Driver qualification (`projectionReady`, `cardReady`)
- Odds freshness / market availability (`pricingReady`)
- Publish eligibility (implicit: `card_ready && pricing_ready`)
- Pipeline state tracking (`buildGamePipelineState`)

This means the audit cannot distinguish "model said FIRE but pricing was absent" from "model said PASS." Both look the same from the outside.

Additionally, pitcher K cards have multiple distinct execution paths (odds-backed, projection-only, stale odds, shadow rollout, disabled rollout) that currently produce the same card shape, making it impossible to audit which path fired.

## Implementation Plan

1. **Add three explicit state objects** to each game's processing loop:

   ```js
   const predictionState = {
     status: 'QUALIFIED' | 'UNQUALIFIED',
     reason: string | null,
   };

   const pricingState = {
     status: 'FRESH' | 'STALE' | 'MISSING' | 'NOT_REQUIRED',
     reason: string | null,
     captured_at: string | null,
   };

   const publishState = {
     publish_ready: boolean,
     execution_status: 'EXECUTABLE' | 'PROJECTION_ONLY' | 'BLOCKED',
     block_reason: string | null,
   };
   ```

2. **Replace boolean flags** with the explicit state objects:
   - `pricingReady` (boolean) → `pricingState.status === 'FRESH'`
   - `cardReady` (boolean) → `predictionState.status === 'QUALIFIED'`
   - Emit both objects on every card's `payloadData` as `_prediction_state`, `_pricing_state`, `_publish_state`

3. **Hard rule: `pricing_status !== FRESH` → no executable output**:
   ```js
   if (pricingState.status !== 'FRESH') {
     publishState.execution_status = pricingState.status === 'MISSING'
       ? 'BLOCKED'
       : 'PROJECTION_ONLY';
     publishState.publish_ready = false;
     publishState.block_reason = `pricing_status=${pricingState.status}`;
   }
   ```

4. **Label projection-floor outputs explicitly**:
   - `execution_status: 'PROJECTION_ONLY'`
   - `projection_floor: true`
   - `actionable: false`
   - These must never have `execution_status: 'EXECUTABLE'`

5. **Clarify `card_ready` vs `publish_ready`**:
   - `card_ready = true` means a card object was created (output exists)
   - `publish_ready = true` means it is eligible as an executable surfaced card
   - Update `buildGamePipelineState` call to pass both

6. **Audit pitcher-K execution path explicitly** — add `k_prop_execution_path` field to pitcher K card payloads:
   - `'ODDS_BACKED'` — fresh odds, normal emit
   - `'PROJECTION_ONLY'` — no odds, projection floor path
   - `'STALE_ODDS_BLOCKED'` — odds present but beyond freshness threshold
   - `'SHADOW_ROLLOUT'` — shadow mode, no emit
   - `'DISABLED'` — `MLB_PITCHER_K_PROP_EVENTS_ENABLED` not set

7. **Add `execution_status` and `actionable` to all MLB card payloads** (additive fields, no removals).

## Acceptance

- No MLB card with `pricing_status !== FRESH` carries `execution_status=EXECUTABLE`
- Projection-floor cards carry `projection_floor=true`, `actionable=false`, `execution_status=PROJECTION_ONLY`
- All MLB cards have `execution_status` field
- Pitcher K cards have `k_prop_execution_path` field
- Worker tests pass: `npm --prefix apps/worker test`
