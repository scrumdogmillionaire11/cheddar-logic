## Background

Two problems in the current NBA projection math:

**Problem 1 — Pace-contaminated ORtg**

`projectNBACanonical` uses `avgPoints` directly as an ORtg proxy:
```js
const baseHomePPP = (homeOffRtg + awayDefRtg) / 200;  // homeOffRtg = avgPoints
const homeProjected = baseHomePPP * adjustedPace;
```

`avgPoints` per game already reflects pace. A team averaging 118 pts at 108 possessions and a team averaging 118 pts at 100 possessions have very different per-possession efficiency, but the model treats them identically. The `adjustedPace` multiplication then applies pace *a second time*.

Correct formula:
```
offRtg = (avgPoints / teamPace) * 100
defRtg = (avgPointsAllowed / opposingTeamPace) * 100
baseHomePPP = (homeOffRtg + awayDefRtg) / 200
homeProjected = baseHomePPP * adjustedPace   ← pace applied exactly once
```

**Problem 2 — Dual pace frameworks**

`projectNBA` uses raw thresholds: `>102 = Fast, <98 = Slow`  
`analyzePaceSynergy` uses 2025-26 percentile scale: PACE_MIN=99.3, PACE_MAX=107.8; Fast = 70th pct ≈105.2

The same team can be `Fast` in one and `Avg` in the other. `computeNBAMarketDecisions` in `cross-market.js` calls `projectNBA`, not `projectNBACanonical`, so the deployed cross-market decisions use the less accurate bucket thresholds.

  - `computeNBAMarketDecisions`: switch from `projectNBA` to `projectNBACanonical` + `analyzePaceSynergy` for the projected total/margin used in driver scores. **Calling order is mandatory**: `analyzePaceSynergy` must run first; its `.paceAdjustment` value is then passed as the last argument to `projectNBACanonical`. Pattern:

These adjustments will be absent from the projected total after this switch. This is an **intentional temporary regression**, explicitly deferred to WI-0836 (rest-days pipeline). The `restDaysHome`/`restDaysAway` variables must remain in `computeNBAMarketDecisions` for the `restGap` driver calculation — do not remove them.

**Out of scope:**

- Changing driver weights in `computeNBAMarketDecisions`

## Formula Fix Detail

### `projectNBACanonical` before/after

Before:
```js
const baseHomePPP = (homeOffRtg + awayDefRtg) / 200;  // homeOffRtg is raw avgPoints
```

After:
```js
// Normalize to per-100-possession rating before PPP computation
const homeOffRtgNorm = homePace > 0 ? (homeOffRtg / homePace) * 100 : homeOffRtg;
const homeDefRtgNorm = homePace > 0 ? (homeDefRtg / homePace) * 100 : homeDefRtg;
const awayOffRtgNorm = awayPace > 0 ? (awayOffRtg / awayPace) * 100 : awayOffRtg;
const awayDefRtgNorm = awayPace > 0 ? (awayDefRtg / awayPace) * 100 : awayDefRtg;

const baseHomePPP = (homeOffRtgNorm + awayDefRtgNorm) / 200;
const baseAwayPPP = (awayOffRtgNorm + homeDefRtgNorm) / 200;
```

This produces PPP values roughly in the 1.05–1.18 range (correct for NBA), rather than the inflated ~1.15–1.25 range produced by raw avgPoints / 200 on fast teams.

### Pace classification consolidation

`projectNBA`'s bucket approach should either:
- a) Call `paceToPct` from `nba-pace-synergy.js` to classify, OR
- b) Be removed from `computeNBAMarketDecisions` with `projectNBACanonical` used instead

Preferred: option (b). `projectNBA` becomes a legacy function marked `@deprecated`, retained only for callers outside cross-market.
