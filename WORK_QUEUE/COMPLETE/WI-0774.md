## Problem

Goalie state resolution in `apps/worker/src/models/nhl-goalie-state.js` uses two source types:
- `SCRAPER_NAME_MATCH` â€” matches a goalie name from a MoneyPuck HTML scrape
- `SEASON_TABLE_INFERENCE` â€” infers the likely starter from season totals

Neither reads pre-fetched confirmed starter data. No DB table stores goalie confirmation ahead of game time.

**Goalie certainty (0.06) is a weighting signal in the NHL model that gates EXECUTABLE status.** If goalie is unknown, certainty is low, and the card cannot reach EXECUTABLE.

**The NHL API provides confirmed starters ~1 hour before puck drop via:**
- `/api/v1/game/{gameId}/feed/live` or
- `/api/v1/schedule?startDate=...&endDate=...` with `expand=schedule.teams`

This data is never fetched. The model relies on a name match from an HTML parse that is fragile, delayed, and provides no certainty source tag.

**Contract violation:** A real data source exists. The model uses a noisy inference instead. Missing goalie data should produce an explicit `PASS` or downgrade, not a low-certainty EXECUTABLE based on a seasonal guess.

## Input proof

NHL API schedule/feed endpoints (no auth required):

```
GET https://api-web.nhle.com/v1/schedule/{date}
```

Returns games with `startingGoalieHome` and `startingGoalieAway` when officially confirmed; field is absent or null when not yet confirmed.

Existing types in `nhl-goalie-state.js`: `SCRAPER_NAME_MATCH`, `SEASON_TABLE_INFERENCE` â€” a new `NHL_API_CONFIRMED` type will be added.

## Fix spec

**Step 1 â€” DB migration:**
- New table `nhl_goalie_starters`:
  `game_id TEXT, team_id TEXT, goalie_id TEXT, goalie_name TEXT, confirmed BOOLEAN, source TEXT, fetched_at TEXT, PRIMARY KEY (game_id, team_id)`

**Step 2 â€” Build `pull_nhl_goalie_starters.js`:**
- Fetch today's NHL schedule from NHL API
- For each game, extract `startingGoalieHome` and `startingGoalieAway` when present
- Upsert into `nhl_goalie_starters` with `confirmed = true` when officially listed, `confirmed = false` when TBD/absent
- Schedule: run hourly in the 2-hour pre-game window (same pattern as `pull_odds_hourly`)

**Step 3 â€” Update `resolveGoalieState`:**
- First lookup: `nhl_goalie_starters` â€” if `confirmed = true`: return `NHL_API_CONFIRMED` source, certainty = 1.0
- If `confirmed = false` (listed but not confirmed): return `NHL_API_PROBABLE`, certainty = 0.7
- If no row exists (not fetched or > 2h before puck drop): fall through to existing `SCRAPER_NAME_MATCH` / `SEASON_TABLE_INFERENCE` chain
- If all sources fail: emit `missing_inputs: ['goalie_unresolved']`; cap classification at `LEAN` (no EXECUTABLE without goalie certainty)

**Output proof:** Card payload includes `raw_data.goalie_state.source = 'NHL_API_CONFIRMED'` when pre-fetched; certainty value is 1.0 when confirmed.

**Failure proof:** When goalie is unresolved from all sources: `missing_inputs: ['goalie_unresolved']`; card is explicitly capped at `LEAN`.
