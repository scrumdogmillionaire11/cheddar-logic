# WI-0559: Fix JWT HS256 signature encoding to standards-compliant base64url

**ID**: WI-0559
**Goal**: Correct JWT signature generation so tokens are RFC-compliant HS256 JWTs and interoperable with standard verifiers.

**Scope**:
- `web/src/lib/api-security/jwt.ts`
- `web/src/lib/api-security/__tests__/jwt.test.ts` (create if absent)

**Out of scope**:
- RBAC policy changes
- Token lifetime changes
- Refresh-token storage implementation

**Background**:
`createSignature()` in `jwt.ts` currently does:
- `digest().toString('binary')`
- then `Buffer.from(binaryString).toString('base64')`

This binary-string roundtrip corrupts non-ASCII bytes and yields non-standard signatures.
Repro:
- current algorithm output != `crypto.createHmac(...).digest('base64url')`

Result: internally generated tokens may verify only against this same flawed implementation and fail interoperability with standard JWT tooling.

**Acceptance**:
- Signature generation uses byte-safe base64url encoding directly from HMAC bytes (no `'binary'` string conversion).
- `verifyToken()` correctly validates tokens generated by standards-compliant HS256 implementations.
- Add tests asserting signature parity with Node crypto `digest('base64url')` for a fixed header/payload/secret vector.
- Existing token validation behavior for tampered signatures and expired tokens remains intact.
