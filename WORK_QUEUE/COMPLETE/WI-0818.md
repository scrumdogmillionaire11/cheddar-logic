# WI-0818: Emit `price_staleness_warning` on hard-locked cards with stale price

| Field | Value |
|---|---|
| **ID** | WI-0818 |
| **Goal** | When a card is in HARD lock and the current sportsbook price differs from the locked price inside T-60 (`minutesToStart < 60`), attach a `price_staleness_warning` object to the payload so downstream consumers can surface the discrepancy before a user places a bet at an unavailable price. |
| **Priority** | Medium ŌĆö prevents users from attempting bets at prices that no longer exist at any book |
| **Depends on** | None |
| **Coordination flag** | None |

---

## Background

In `publishDecisionForCard` (`apps/worker/src/utils/decision-publisher.js`), when `gateResult.allow` is false and a prior decision record exists, `applyPublishedDecisionToPayload` is called to overwrite the card with the locked decision's price:

```js
} else if (current) {
  // ...
  applyPublishedDecisionToPayload(
    card,
    current,       // ŌåÉ locked record, price may be stale
    market,
    decisionKey,
    gateResult.reason_code,
  );
}
```

`applyPublishedDecisionToPayload` then sets `payload.price = decision.recommended_price` ŌĆö the price from when the decision was first locked (potentially 2+ hours ago). The user's Discord card shows that old price. If the market has moved from -110 to -130, the user either:
- Tries to bet at -110 and it's not available ŌåÆ shopping delay ŌåÆ misses the play
- Bets at -130 unknowingly ŌåÆ pays ~5% more juice ŌåÆ eliminates the edge

The fix is purely informational: attach a warning object to the payload when the locked price diverges from the current candidate price, particularly inside T-60. No behavioral change to locking logic ŌĆö that is architectural.

## Detailed Implementation

### Task 1 ŌĆö Compute and attach `price_staleness_warning` in `applyPublishedDecisionToPayload`

**File:** `apps/worker/src/utils/decision-publisher.js`

The current signature of `applyPublishedDecisionToPayload` already receives `decision` (the locked record) and the calling context has `minutesToStart` computed just above. Pass `minutesToStart` to the function or compute it inline from the payload's `start_time_utc`.

```js
function applyPublishedDecisionToPayload(
  card,
  decision,
  market,
  decisionKey,
  gateReason,
  { minutesToStart = 9999, candidatePrice = null } = {},   // new optional arg
) {
  ...
  const lockedPrice = decision.recommended_price ?? null;
  const priceDelta = Number.isFinite(lockedPrice) && Number.isFinite(candidatePrice)
    ? Math.abs(candidatePrice - lockedPrice)
    : null;

  if (priceDelta != null && priceDelta > 0 && minutesToStart < 60) {
    payload.price_staleness_warning = {
      locked_price: lockedPrice,
      current_candidate_price: candidatePrice,
      delta_american: Number(priceDelta.toFixed(0)),
      minutes_to_start: Math.round(minutesToStart),
      reason: 'HARD_LOCK_PRICE_DRIFT',
    };
    payload.tags = Array.from(
      new Set([...(payload.tags || []), 'PRICE_STALENESS_WARNING']),
    );
  }
  ...
}
```

**Call site update in `publishDecisionForCard`:**

```js
applyPublishedDecisionToPayload(
  card,
  current,
  market,
  decisionKey,
  gateResult.reason_code,
  { minutesToStart, candidatePrice: price },  // ŌåÉ add
);
```

`price` and `minutesToStart` are already computed by this point in `publishDecisionForCard`. No additional lookups needed.

## Acceptance Criteria

- [ ] `applyPublishedDecisionToPayload` accepts `minutesToStart` and `candidatePrice` options
- [ ] When `minutesToStart < 60` and `candidatePrice !== lockedPrice`, payload contains `price_staleness_warning` with all four fields populated
- [ ] When `minutesToStart >= 60` (early lock), no `price_staleness_warning` is emitted
- [ ] When `candidatePrice === lockedPrice` (no drift), no warning
- [ ] `PRICE_STALENESS_WARNING` tag appears in `payload.tags` when warning is attached
- [ ] Existing `publishDecisionForCard` tests continue to pass; add two new tests (drift inside T-60, no drift outside T-60)
