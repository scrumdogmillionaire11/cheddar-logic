# WI-0817: Wrap delete + insert in SQLite transaction in `prepareModelAndCardWrite`

| Field | Value |
|---|---|
| **ID** | WI-0817 |
| **Goal** | Wrap the delete and insert operations in `prepareModelAndCardWrite` (and the callers' per-game card generation blocks) in explicit SQLite transactions so that a process crash between delete and insert leaves old cards intact rather than producing a card blackout for up to ~30 minutes. |
| **Priority** | Medium — data integrity; affects card availability after crashes |
| **Depends on** | None |
| **Coordination flag** | None — only touches `packages/data/src/db/cards.js` internals |

---

## Background

`prepareModelAndCardWrite` in `packages/data/src/db/cards.js` calls two separate write operations:

```js
function prepareModelAndCardWrite(gameId, modelName, cardType, options = {}) {
  ...
  const deletedOutputs = deleteModelOutputsByGame(gameId, modelName);
  const deletedCards   = deleteCardPayloadsForGame(gameId, cardType, { ...options, runId });
  return { deletedOutputs, deletedCards };
}
```

The caller then does multiple `insertCardPayload` calls for the same game — outside any transaction. The sequence is:

```
DELETE model outputs  ← separate statement
DELETE card payloads  ← separate statement
  ... some time passes ...
INSERT card 1         ← separate statement
INSERT card 2         ← separate statement
```

If the process crashes between the deletes and the inserts, that game has zero live cards until the next model run window (up to 30–60 minutes for T-minus windows, or 3+ hours for fixed-time-only sports).

SQLite's default `journal_mode` is WAL, so wrapping these in a `db.transaction()` is a zero-overhead write with no schema changes. The `better-sqlite3` synchronous API supports this directly.

## Detailed Implementation

### Task 1 — Wrap deletes in `prepareModelAndCardWrite`

**File:** `packages/data/src/db/cards.js`

```js
// BEFORE
function prepareModelAndCardWrite(gameId, modelName, cardType, options = {}) {
  const runId = normalizeRunScopeId(options);
  if (!runId) {
    throw Object.assign(
      new Error('[DB] prepareModelAndCardWrite requires a non-empty options.runId'),
      { code: 'RUN_ID_REQUIRED' },
    );
  }
  const deletedOutputs = deleteModelOutputsByGame(gameId, modelName);
  const deletedCards = deleteCardPayloadsForGame(gameId, cardType, { ...options, runId });
  return { deletedOutputs, deletedCards };
}

// AFTER
function prepareModelAndCardWrite(gameId, modelName, cardType, options = {}) {
  const runId = normalizeRunScopeId(options);
  if (!runId) {
    throw Object.assign(
      new Error('[DB] prepareModelAndCardWrite requires a non-empty options.runId'),
      { code: 'RUN_ID_REQUIRED' },
    );
  }
  const db = getDatabase();
  return db.transaction(() => {
    const deletedOutputs = deleteModelOutputsByGame(gameId, modelName);
    const deletedCards = deleteCardPayloadsForGame(gameId, cardType, { ...options, runId });
    return { deletedOutputs, deletedCards };
  })();
}
```

This is the minimal safe fix: the two deletes are atomic. If the process crashes before the transaction is committed, SQLite rolls back and the old rows survive.

### Task 2 — Wrap per-game card generation in model runners

**File:** `apps/worker/src/jobs/run_nba_model.js` (and NHL/MLB)

In the `for (const gameId of gameIds)` loop, wrap the full per-game block — from `prepareModelAndCardWrite` through all `insertCardPayload` calls — in a single outer `db.transaction()`:

```js
const db = getDatabase();
for (const gameId of gameIds) {
  try {
    db.transaction(() => {
      prepareModelAndCardWrite(gameId, modelName, cardType, { runId: jobRunId });
      // ... build cards ...
      for (const card of cardsToInsert) {
        insertCardPayload(card);
      }
    })();
    cardsGenerated += cardsToInsert.length;
  } catch (err) {
    cardsFailed++;
    errors.push({ gameId, error: err.message });
    console.error(`[NBAModel] Failed for ${gameId}:`, err.message);
    // Old cards survive — transaction rolled back automatically
  }
}
```

**Note:** `better-sqlite3` transactions are synchronous and cannot wrap `async` operations. The ESPN enrichment (`enrichOddsSnapshotWithEspnMetrics`) is async and must remain outside the transaction boundary. Only the write phase (delete + insert) goes inside.

Exact wiring will depend on where `enrichOddsSnapshotWithEspnMetrics` sits in each model runner's per-game loop. Move the transaction boundary to start after all async operations complete.
