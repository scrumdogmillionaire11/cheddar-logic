## Problem

MoneyPuck fetches and normalizes several advanced team metrics during `pull_odds_hourly` enrichment. The NHL model in `apps/worker/src/models/index.js` reads only three:
- `pk_home_pct` â€” penalty kill %
- `xgf_home_pct` â€” expected goals for %
- `goalie_gsax` â€” goalie goals saved above expected

The following fields arrive in `raw_data` and are never read by the model:

| Field | Source | What it measures |
|-------|--------|-----------------|
| `fenwick_pct` | MoneyPuck | Shot attempt quality (unblocked shots for %) |
| `hdcf_pct` | MoneyPuck | High-danger chance for % (strongest shot-quality signal) |
| `pp_toi` | MoneyPuck | Power play time on ice |
| `onice_xgf_for_per_game` | MoneyPuck | Per-player on-ice xGF (shots model only, not totals) |

**`fenwick_pct` and `hdcf_pct`** are the highest value of these â€” both are proven predictors of future goal scoring that the current model's `xgf_home_pct` signal does not fully capture.

**Contract violation:** MoneyPuck fetches are made on every `pull_odds_hourly` run. Both fields land in `raw_data`. The model reads `raw_data` for other fields. These two are silently skipped with no comment.

## Input proof

In `apps/worker/src/moneypuck.js`, both fields are parsed and normalized:
- `fenwick_pct` â€” team-level Fenwick close %
- `hdcf_pct` â€” team-level high-danger chance for %

Both appear in `raw_data.moneypuck` on the game object. `xgf_home_pct` reading in `apps/worker/src/models/index.js` provides the exact wiring pattern to follow.

## Fix spec

In `apps/worker/src/models/index.js` NHL model input construction:

1. Read `raw_data.moneypuck.fenwick_pct` (home and away values)
2. Read `raw_data.moneypuck.hdcf_pct` (home and away values)
3. Compute `shot_quality_edge = (0.5 * fenwick_diff + 0.5 * hdcf_diff)` where `*_diff = home_value - away_value`
4. Apply `shot_quality_edge` as a small modifier on the projected total (e.g. `Â±0.15` goals per unit of edge) â€” additive to the existing `xgf` adjustment
5. If both `fenwick_pct` and `hdcf_pct` are absent from `raw_data`: do not block, no downgrade â€” both are enrichment signals, not required gates. Add a note-level `pricing_context` annotation only.

**Output proof:** `raw_data.shot_quality_inputs` contains `fenwick_diff` and `hdcf_diff`; `proj_total_adjusted` reflects the modifier.

**Failure proof:** If MoneyPuck data is absent (opponent tracking gap), model behaves as today â€” no change.
