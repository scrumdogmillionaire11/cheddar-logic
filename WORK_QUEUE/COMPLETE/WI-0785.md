# WI-0785: Emit hard deploy warning when Cloudflare secrets are missing (CF purge skipped)

| Field              | Value |
|--------------------|-------|
| **ID**             | WI-0785 |
| **Goal**           | Change the `deploy-production.yml` Cloudflare cache purge step so that missing `CF_API_TOKEN` / `CF_ZONE_ID` secrets produce a visible GitHub Actions warning annotation and a deploy summary note — rather than a silent skip with only a plain `echo` in the logs |
| **Priority**       | Medium |
| **Owner agent**    | Unassigned |
| **Time window**    | TBD |
| **Coordination flag** | `needs-sync` — modifies `.github/workflows/deploy-production.yml` |

---

## Problem

From the deploy log (line 495 of `deploy-production.yml`):

```
echo "WARNING: Cloudflare secrets (CF_API_TOKEN / CF_ZONE_ID) not set — skipping CF cache purge."
```

When either secret is absent, the workflow skips the Cloudflare `purge_everything` call AND all convergence checks (`retry_public_route_refs`). The deploy completes with an exit 0 even though the CDN may still be serving stale HTML and chunk hashes.

The only signal is a plain `echo` buried in the step output. This is easy to miss during a fast deploy review.

**Impact:**
- A deploy can "succeed" in CI while users encounter stale Cloudflare-cached HTML pointing to old chunk hashes, causing 404s for new JS files
- The missing-secrets condition has previously caused confusion about whether the deploy was effective

---

## Scope

| File | Change |
|------|--------|
| `.github/workflows/deploy-production.yml` | Replace the plain `echo` missing-secrets path with: (a) `echo "::warning::CF secrets missing — Cloudflare purge skipped. Users may see stale assets until CF TTL expires."` to emit a GitHub Actions warning annotation; (b) append a note to `$GITHUB_STEP_SUMMARY` via `echo "⚠️ CF purge skipped — missing CF_API_TOKEN or CF_ZONE_ID" >> $GITHUB_STEP_SUMMARY` so it appears in the Actions job summary |

## Out of Scope

- Making the missing-secrets condition a hard failure (intentionally not done — there may be environments without CF, e.g. staging)
- Changing how CF purge itself works
- Adding CF secrets to the repo (done outside CI config)

---

## Acceptance

- A deploy run without CF secrets set shows an orange warning annotation in the GitHub Actions UI (not just a log line)
- The job summary for the deploy step includes the warning
- A deploy run WITH CF secrets still purges normally with no change to behavior

---

## Tests to Run

N/A — workflow change only; verified by manual deploy run.

## Manual Validation

- Run a deploy (or simulate the deploy step locally) without `CF_API_TOKEN` / `CF_ZONE_ID` set; confirm the warning annotation appears in the GitHub Actions run UI under the "Deploy Worker" job
- Run a deploy with secrets set; confirm purge proceeds normally
