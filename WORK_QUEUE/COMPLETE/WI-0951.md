# WI-0951: MLB T-Minus Freshness Override Schedule

**ID**: WI-0951
**Goal**: Define and implement the pregame freshness override schedule for MLB so freshness tightens by minutes-to-game band and pre-model refresh is triggered where required.

**Depends on**: WI-0950

**Scope**:

- `apps/worker/src/schedulers/mlb.js`
- `apps/worker/src/schedulers/main.js`
- `apps/worker/src/__tests__/scheduler-main-calibration.test.js`
- `apps/worker/src/__tests__/scheduler-windows.test.js`
- `WORK_QUEUE/WI-0951.md`

**Out of scope**:

- Baseline execution freshness policy (defined by WI-0950)
- Execution gate baseline threshold rewrites
- MLB model projection formulas
- Odds provider integration changes
- UI filtering and transform behavior

**Acceptance**:

1. **Override Contract Type**: define and document `TMinusFreshnessOverride` in scheduler freshness config:

   ```ts
   type TMinusFreshnessOverride = {
     minutesToGameLte: number
     requiredMaxSnapshotAgeMinutes: number
     triggerPreModelRefresh: boolean
   }
   ```

2. **Initial MLB Override Table**: scheduler uses this ordered ladder for MLB pregame windows:

   ```js
   [
     { minutesToGameLte: 180, requiredMaxSnapshotAgeMinutes: 75, triggerPreModelRefresh: false },
     { minutesToGameLte: 90,  requiredMaxSnapshotAgeMinutes: 30, triggerPreModelRefresh: true  },
     { minutesToGameLte: 45,  requiredMaxSnapshotAgeMinutes: 20, triggerPreModelRefresh: true  },
     { minutesToGameLte: 15,  requiredMaxSnapshotAgeMinutes: 10, triggerPreModelRefresh: true  }
   ]
   ```

3. **Band Resolver Rule (mandatory, deterministic)**:
   - A minutes-to-game value may satisfy multiple rows.
   - Resolver MUST choose the strictest matching row, defined as the row with the smallest `minutesToGameLte` that still satisfies `minutesToGame <= minutesToGameLte`.
   - Example: `minutesToGame=38` matches 180, 90, and 45; resolver MUST select 45.
   - Tests must prove this precedence.

4. **Contract Principle** (must be enforced exactly):

   > T-minus scheduling applies a stricter freshness override by minutes-to-game band and may trigger a pre-model odds refresh before execution. The T-minus override narrows allowable snapshot age but does not invalidate the cadence contract outside the configured pregame windows.

5. **Explicit Scheduler Behavior Order**:

   ```text
   1. Determine current minutes-to-game band.
   2. Resolve applicable override row, if any, using strictest-match precedence.
   3. If triggerPreModelRefresh = true, run odds refresh before model execution.
   4. Re-evaluate snapshot age against requiredMaxSnapshotAgeMinutes.
   5. Fall back to WI-0950 baseline policy when no override band applies.
   ```

6. **Structured Logging Required**: emit a structured T-minus freshness log for all override-band evaluations:

   ```json
   {
     "type": "EXECUTION_FRESHNESS_TMINUS",
     "minutes_to_game": 38,
     "matched_band": 45,
     "required_max_snapshot_age_minutes": 20,
     "triggered_refresh": true,
     "decision": "ALLOW_AFTER_REFRESH"
   }
   ```

   Mandatory decision enum:
   - `ALLOW` (within band threshold, no refresh needed)
   - `ALLOW_AFTER_REFRESH` (refresh triggered, then threshold satisfied)
   - `BLOCK_OVERRIDE_STALE` (still stale after required refresh/reevaluation)
   - `FALLBACK_BASELINE` (no override row applied; WI-0950 baseline path)

7. **Deterministic + Deduped Enqueue**:
   - Pre-model odds refresh enqueues must be deduped under repeated scheduler ticks in the same slot.
   - Job key schema is mandatory and stable:
     - `pull-odds:mlb:premodel:{gameId}:{matchedBandMinutes}:{slotStartIsoUtc}`
   - `slotStartIsoUtc` must be normalized to minute precision in UTC.
   - Existing hourly odds pulls remain intact; T-minus pulls augment, not replace, baseline cadence.

8. **Tests**:
   - Band-selection tests by minutes-to-game for all configured ladder rows.
   - Strictest-match precedence test (e.g., 38 must select band 45, not 90/180).
   - Refresh-trigger tests for rows where `triggerPreModelRefresh=true`.
   - Boundary tests around row cutoffs (180, 90, 45, 15).
   - Dedupe tests under repeated ticks in same slot.
   - Baseline fallback test where no override row applies.
   - Decision-enum log tests for all four decision values.
