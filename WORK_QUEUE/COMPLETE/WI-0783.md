# WI-0783: Add sequential ordering guard to the settlement pipeline

| Field              | Value |
|--------------------|-------|
| **ID**             | WI-0783 |
| **Goal**           | Make the settlement pipeline (game results → projections → MLB F5 → cards) run in a guaranteed sequential order with explicit handoff markers, so a failure in any stage is observable and later stages cannot run against partially-settled state |
| **Priority**       | High |
| **Owner agent**    | Unassigned |
| **Time window**    | TBD |
| **Coordination flag** | `needs-sync` — modifies `apps/worker/src/schedulers/main.js`; touches settlement job outputs which are part of the public API surface |

---

## Problem

Four settlement jobs exist and are scheduled independently:

- `settle_game_results.js` — resolves game outcomes
- `settle_projections.js` — settles model projections against outcomes
- `settle_mlb_f5.js` — MLB-specific F5 (first-5-innings) settlement
- `settle_pending_cards.js` — settles user-facing cards based on projections

These run as separate scheduler entries with no explicit ordering enforcement. If `settle_game_results` is still running (or fails mid-run) when `settle_pending_cards` starts, cards may be settled against incomplete game state. There is no mechanism that prevents `settle_pending_cards` from running before `settle_game_results` has finished.

**Observed risk:** Multi-market or deferred game results can create a window where some outcomes are written but not all, and card settlement reads the partial state.

---

## Scope

| File | Change |
|------|--------|
| `apps/worker/src/schedulers/main.js` | Add sequential settlement chain: after `settle_game_results` job_key resolves as `SUCCESS`, enqueue `settle_projections`; after `settle_projections` resolves as `SUCCESS`, enqueue `settle_pending_cards`. Each step checks the prior step's `job_key` status before running rather than using independent time-window conditions. |
| `apps/worker/src/jobs/settle_game_results.js` | Confirm `job_key` is written on both success and failure paths (no silent exits) |
| `apps/worker/src/jobs/settle_projections.js` | Add guard at entry: skip with log if the current game window's `settle_game_results` job_key is not in `SUCCESS` state |
| `apps/worker/src/jobs/settle_pending_cards.js` | Add guard at entry: skip with log if the current game window's `settle_projections` job_key is not in `SUCCESS` state |

## Out of Scope

- Refactoring settlement logic within any job
- Changing the settlement DB schema
- MLB F5 settlement ordering (MLB F5 is game-specific and already gated by game state)
- Adding new settlement stages

---

## Acceptance

- Running `settle_pending_cards` in a window where `settle_game_results` has not yet succeeded produces: `SKIP: settle_game_results not yet SUCCESS for this window — skipping card settlement`
- Running `settle_pending_cards` after `settle_game_results` succeeds proceeds normally
- Existing settlement tests pass
- `test:pipeline:settlement` passes
