**Timestamp Fallback Chain (Explicit Precedence)**:

Age resolver MUST use this precedence order. First field with valid value wins:

```
1. captured_at (primary, preferred)
2. pulled_at (fallback if captured_at is null)
3. updated_at (fallback if both above are null)
4. NOW() (last resort, emits DEGRADED diagnostic)
```

Valid value definition:

- Field is not null
- ISO 8601 datetime string (e.g., "2026-04-15T19:30:00Z" or "2026-04-15T19:30:00+00:00")
- UTC timezone (Z suffix or +00:00)
- Timestamp does not exceed current time (no future timestamps)

Invalid value definition:

- Null/undefined
- Non-ISO format (e.g., "April 15" or Unix timestamp as string "1713200400")
- Missing timezone info
- Future timestamp (> NOW())

Calculation:

```
age_ms = NOW() - parsedTimestamp
```

If calculation fails (unparseable timestamp) → emit MALFORMED diagnostic and use fallback or NOW().

**Non-Monotonic Timestamp Detection**:

A snapshot row is "non-monotonic" if timestamps violate causality. Example patterns:

- `captured_at > NOW()` — future capture (clock skew from odds provider)
- `pulled_at < captured_at` — pulled before capture (impossible, indicates data corruption)
- `updated_at < pulled_at` — updated before pull (corruption or replication lag)

Detect and emit MONOTONIC_VIOLATION diagnostic + capture all three fields for audit. Do NOT block on violation; log and continue with available timestamp.

**Diagnostics Schema**:

All timestamp resolution events MUST emit a structured diagnostic (JSON log):

```json
{
  "event_type": "TIMESTAMP_DIAGNOSTIC",
  "timestamp": "2026-04-15T19:30:15.123Z",
  "snapshot_id": "odds_mlb_2026-04-15_19:15:00Z",
  "sport": "mlb",
  "game_id": "mlb_2026-04-15_ws_phi",
  
  "resolved_timestamp": "2026-04-15T19:20:12Z",
  "resolved_age_ms": 123456,
  "source_field": "captured_at",
  
  "status": "VALID",
  
  "fields_inspected": {
    "captured_at": { "value": "2026-04-15T19:20:12Z", "status": "VALID" },
    "pulled_at": { "value": "2026-04-15T19:15:00Z", "status": "VALID" },
    "updated_at": { "value": "2026-04-15T19:10:00Z", "status": "VALID" }
  },
  
  "fallback_chain_executed": false,
  "violations": []
}
```

Status values:

- `VALID` — resolved directly from primary field, no fallback needed
- `DEGRADED` — fell back to secondary field or NOW()
- `MALFORMED` — could not parse one or more fields
- `MONOTONIC_VIOLATION` — causality violation detected

Emit as structured log prefixed `[TIMESTAMP_DIAGNOSTIC]` in both ingest and runner paths.
