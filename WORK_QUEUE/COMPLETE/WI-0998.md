# WI-0998: POTD Shadow Candidate Settlement Ledger

**ID**: WI-0998
**Goal**: Add a separate read/write worker path that settles POTD shadow candidates into a virtual-result ledger without contaminating official POTD bankroll or `potd_plays`.

**Scope**:
- `packages/data/db/migrations/079_create_potd_shadow_results.sql`
- `apps/worker/src/jobs/potd/settle-shadow-candidates.js`
- `apps/worker/src/jobs/potd/__tests__/shadow-settlement.test.js`
- `apps/worker/package.json`
- `apps/worker/src/jobs/settle_pending_cards.js` only if shared grading helpers must be exported or extracted

**Out of scope**:
- Official POTD firing-rule changes
- Writes to `potd_plays`
- Writes to `potd_bankroll`
- Discord publishing
- Web/API/UI changes
- Re-scoring or regenerating historical shadow candidates

**Acceptance**:
- A `potd_shadow_results` table records one settlement row per settled shadow candidate identity/run with `status`, `result`, `settled_at`, `pnl_units`, `virtual_stake_units`, and grading metadata.
- Settlement job grades eligible shadow rows from final game results for supported POTD markets: `MONEYLINE`, `SPREAD`, and `TOTAL`.
- Job is idempotent: re-running does not duplicate result rows or mutate already-settled shadow outcomes unexpectedly.
- Job leaves candidates pending or records a clear non-gradeable status when required final result data is missing.
- Virtual P&L uses candidate locked price and a fixed virtual stake, defaulting to `1.0` unit; no official bankroll tables are touched.
- Job returns summary counts by `settled`, `pending`, `non_gradeable`, `win`, `loss`, and `push`.
- Tests cover moneyline, spread, total, idempotency, missing final result, and proof that `potd_plays` / `potd_bankroll` remain untouched.
