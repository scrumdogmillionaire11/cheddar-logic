# WI-0573: Safe production snapshot export for dev parity

**ID**: WI-0573
**Goal**: Replace direct live-file prod DB copying with a worker-owned snapshot export and documented local sync flow so dev parity checks can use fresh prod data without extra odds calls.

**Scope**:
- `apps/worker/src/utils/db-backup.js` or a new sibling export utility under `apps/worker/src/utils/`
- `apps/worker/src/utils/__tests__/db-snapshot-export.test.js`
- `apps/worker/package.json`
- `scripts/sync-prod-db.sh`
- `scripts/dev-mode-snapshot.sh`
- `docs/QUICKSTART.md`
- `docs/ops-runbook.md`
- `WORK_QUEUE/WI-0573.md`

**Out of scope**:
- Automated cron/systemd timer installation on prod or local machines
- Pruned or sanitized snapshot exports
- Running local worker/scheduler jobs against the prod snapshot DB
- Schema changes, migrations, or odds/model logic changes
- Rewriting backup/restore behavior for settlement jobs beyond what is needed for the new snapshot export path

**Background**:
Current snapshot-mode tooling already exists, but `scripts/sync-prod-db.sh` copies `/opt/data/cheddar-prod.db` directly. That is convenient for parity reads, but it is not the right long-term process under the single-writer contract and better-sqlite3/WAL runtime. The safer contract is: the worker owns production snapshot export, local dev only pulls the exported artifact, and snapshot mode remains web-only/read-only.

**Acceptance**:
- A worker-owned command exports a consistent snapshot artifact from the canonical production DB using a lock-safe SQLite backup/export path, without requiring `CHEDDAR_DB_ALLOW_MULTI_PROCESS=true` and without copying the live DB file directly.
- The export command also writes a sidecar metadata file next to the snapshot artifact containing at least: `source_db_path`, `exported_at`, and snapshot file size.
- `scripts/sync-prod-db.sh` first invokes the remote export command over SSH, then copies the exported snapshot artifact plus metadata to `~/.cheddar/`, instead of copying `/opt/data/cheddar-prod.db` directly.
- `scripts/dev-mode-snapshot.sh --check` reports the local snapshot path and, when metadata exists, surfaces freshness/source details so parity users can see how old their snapshot is.
- `docs/QUICKSTART.md` and `docs/ops-runbook.md` document the safe manual workflow, explicitly distinguish `local` vs `snapshot` mode, and state that snapshot mode is for web-only parity checks and should not be used for local worker/scheduler runs.
- Repo docs/scripts no longer instruct operators to `rsync` the live production DB file directly for dev snapshot mode.
