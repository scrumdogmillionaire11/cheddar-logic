# WI-0965: Centralize MLB execution-gate override policy and surfaced-reason mapping

**ID**: WI-0965
**Goal**: Replace ad hoc MLB gate overrides with one declarative policy contract that drives both execution outcomes and surfaced reason labels.

**Depends on**: WI-0963

**Architecture constraints (must hold)**:

- ADR-0004 and WI-0963 govern this WI: worker-owned decision fields remain the only authority for MLB gate outcomes and blocker reasons; web surfaces may map labels from persisted worker reason fields but must not invent new blocker semantics.
- MLB full-game moneyline and total override policy must be defined in one source in apps/worker/src/jobs/execution-gate.js. This WI must remove or avoid duplicate threshold/override branches outside that source.
- web/src/lib/game-card/transform/decision-surface.ts must remain a pure presentation mapper from persisted worker reason/status fields to surfaced labels.
- Scope is intentionally partitioned to avoid active-file conflicts: MLB execution-gate integration now belongs to this WI and is limited to run_mlb_model execution-gate paths (no model math refactors).

- WORK_QUEUE/WI-0965.md

**Out of scope**:
- Freshness scheduler windows
- Model projection math changes
- Non-MLB execution policy redesign
- Broad refactoring of apps/worker/src/jobs/run_mlb_model.js outside execution-gate integration paths

**Acceptance**:
- MLB full-game ML and totals overrides (high-edge behavior, confidence/net-edge soft blockers) are defined in one policy source in apps/worker/src/jobs/execution-gate.js.
- PASS/LEAN surfaced reasons in web UI match the persisted worker gate blocker outcomes for MLB rows.
- Regression tests cover large-edge pass/lean boundaries, surfaced reason-label parity, and confirm non-MLB markets are unaffected.
- New override logic does not require card-type heuristics scattered across multiple code paths.
- Existing run_mlb_model integration continues to route MLB full-game ML and totals through the centralized execution-gate policy without adding a second override policy source.

**Execution plan**:
- Phase 1: Consolidate MLB execution-gate override thresholds and blocker mapping in apps/worker/src/jobs/execution-gate.js. Encode the MLB full-game ML and total override rules in one declarative source and remove duplicate or implicit policy branches encountered in-scope.
- Phase 2: Align the surfaced reason-label mapping in web/src/lib/game-card/transform/decision-surface.ts so PASS and LEAN labels are derived from persisted worker reason/status fields emitted by the centralized gate policy, not from new UI-only heuristics.
- Phase 3: Add regression coverage proving MLB override boundary behavior in apps/worker/src/jobs/__tests__/execution-gate.test.js and surfaced reason parity in web/src/__tests__/game-card-pass-surface-contract.test.js. Include at least one non-MLB guard case proving the MLB override contract does not leak into other sports.
