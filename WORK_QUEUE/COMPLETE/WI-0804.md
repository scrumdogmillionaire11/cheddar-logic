## Context

`projectBlkV1` (in `apps/worker/src/models/nhl-player-shots.js`) accepts four contextual multipliers:

| Factor | Accepted Range | What It Does |
|---|---|---|
| `opponent_attempt_factor` | `[0.90 – 1.12]` | Scales blk_mu up when opponent generates more shot attempts |
| `defensive_zone_factor` | `[0.95 – 1.08]` | Scales blk_mu up for DZ-heavy teams |
| `underdog_script_factor` | `[0.95 – 1.10]` | Scales blk_mu up when player's team is trailing/underdog |
| `playoff_tightening_factor` | `[1.00 – 1.08]` | Scales blk_mu up during playoffs where blocking intensity increases |

**None of these are currently passed** at the call site in `run_nhl_player_shots_model.js` (lines ~3090–3108). All four default to `1.0` — meaning the model ignores matchup context entirely for BLK props.

Additionally, BLK cards are only priced against the single `blkMarket.line` returned by the odds pull (typically `0.5` or `1.5`). High-volume blockers (defenders logging 2–3 blocks/game) have valid `2+` and `3+` plays that are never surfaced.

## Audit Checklist

### Gap 1 — `opponent_attempt_factor` not sourced from real data

**File:** `apps/worker/src/jobs/run_nhl_player_shots_model.js` (~L3090)

**Current state:** `opponent_attempt_factor` is not passed to `projectBlkV1` → defaults to `1.0` for every game.

**What to audit / fix:**
- Determine the correct data source for opponent shot attempts per game. Candidate: `nst_blk_rates` table (`opponent_shot_attempts_per_game`, `opponent_corsi_for_pct`, or equivalent column), or derive from `player_blk_rates` / MoneypuckBlkRates.
- If source data exists: compute `opponent_attempt_factor` as `opponentShotAttemptsPG / leagueAvgShotAttemptsPG`, clamp to `[0.90, 1.12]`, and pass to `projectBlkV1`.
- If source data does not exist: create a new ingestion job or extend `pull_nst_blk_rates.js` / `pull_moneypuck_blk_rates.js` to capture opponent shot volume per game.
- Add a test: `opponent_attempt_factor` varies per opponent and affects `blk_mu`.

### Gap 2 — `playoff_tightening_factor` always 1.0

**File:** `apps/worker/src/jobs/run_nhl_player_shots_model.js` (~L3090)

**Current state:** `playoff_tightening_factor` is not passed → defaults to `1.0` even during playoff rounds.

**What to audit / fix:**
- Detect whether the game is a playoff game. The `games` table likely has a `game_type` or `season_type` column (check `pull_nhl_schedule.js` or `games` schema). NHL API uses `gameType: 3` for playoffs.
- Playoff rounds where blocking spikes (Rounds 2–4) should receive a higher factor than Round 1. Proposed ramp:

  | Round | `playoff_tightening_factor` |
  |---|---|
  | Regular season | `1.00` |
  | Playoff Round 1 | `1.02` |
  | Playoff Round 2 | `1.04` |
  | Playoff Round 3+ | `1.06` |

- Pass this computed value to `projectBlkV1`.
- Add a test: playoff game yields `blk_mu` > equivalent regular-season game.

### Gap 3 — BLK cards only priced at a single market line

**File:** `apps/worker/src/jobs/run_nhl_player_shots_model.js` (~L3047–3060, and card output section)

**Current state:** `blkLineCandidates[0]` selects only the market's primary line. The model's `fair_over_prob_by_line` / `fair_over_prob_by_line` maps support multiple lines via `lines_to_price`, but BLK cards never populate this.

**What to audit / fix:**
- Pass `lines_to_price: [1.5, 2.5, 3.5]` (or a configurable env var `NHL_BLK_LINES_TO_PRICE`) to `projectBlkV1`.
- After projection, evaluate each line independently:
  - If `blk_mu >= 2.0` and edge at `2.5` is present → emit a `2+` card.
  - If `blk_mu >= 2.8` and edge at `3.5` is present → emit a `3+` card.
- Card `card_type` stays `nhl-player-blk`; differentiate via `market_line` field (`1.5`, `2.5`, `3.5`).
- Update `prediction` string to read e.g. `"Brent Burns 2+ BLK | Proj 2.3 · Fair 2.5"`.
- Add tests for multi-line emission: player with `blk_mu = 2.4` emits both a 1.5 and 2.5 card; player with `blk_mu = 1.1` only emits the 1.5 card.

### Gap 4 — `defensive_zone_factor` and `underdog_script_factor` also at 1.0

**In scope as part of this audit**, but lower priority than Gaps 1–3. Audit whether data exists to compute these:

- `defensive_zone_factor`: opponent DZ time / DZ starts from NST or MoneypuckBlkRates.
- `underdog_script_factor`: game moneyline odds → if player's team is ≥ +150 underdog, they may be trailing late and blocking more.

Document what data is available and whether wiring is worth the complexity. Record as a sub-finding in the SUMMARY.
