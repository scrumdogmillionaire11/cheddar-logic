# WI-0626: Fix settle_mlb_f5 doubleheader gamePk lookup bug

**ID**: WI-0626
**Goal**: Fix a silent mis-settlement bug where `settle_mlb_f5.js` looks up `game_pk` by `game_date` with `LIMIT 1`, causing cards to be graded against the wrong game on any day with multiple MLB games.

**Background**:
`settle_mlb_f5.js` (line 139) queries:
```sql
SELECT game_pk FROM mlb_game_pk_map WHERE game_date = ? LIMIT 1
```
This picks an arbitrary `game_pk` when multiple games share a date (i.e., every regular-season day). The result is F5 cards being graded against the wrong game's linescore — settled as won, lost, or push incorrectly with no error raised. The `mlb_game_pk_map` table has no `game_id` column to correlate back to the card's `game_id`.

**Fix approach**:
1. Add a `game_id` column to `mlb_game_pk_map` (or add a new join key). `pull_mlb_pitcher_stats.js` builds the map; the `game_pk_key` field needs to encode the `game_id` or home/away team pair so `settle_mlb_f5` can look up by the card's `game_id`.
2. In `settle_mlb_f5.js`, change the `mlb_game_pk_map` query to join on the card's `game_id` (or a derived key matching game date + matchup).
3. If the map key is not derivable from `game_id`, add a migration that adds `game_id TEXT` to `mlb_game_pk_map` and backfills from `games`.

**Scope**:
- `apps/worker/src/jobs/settle_mlb_f5.js` — fix `mlb_game_pk_map` lookup
- `apps/worker/src/jobs/pull_mlb_pitcher_stats.js` — update `upsertGamePkMap` / `storeGamePkMap` to store `game_id` alongside `game_pk`
- `packages/data/db/migrations/` — add migration for `game_id` column on `mlb_game_pk_map`

**Out of scope**:
- Changes to F5 grading logic
- Backfilling historical mis-settled cards (separate concern)

**Acceptance**:
- `mlb_game_pk_map` rows include a `game_id` field.
- `settle_mlb_f5.js` looks up `game_pk` by `game_id`, not `game_date LIMIT 1`.
- On a day with two MLB games, each card is graded against its own game's F5 total.
- `npm --prefix apps/worker test -- settle_mlb_f5` passes.
