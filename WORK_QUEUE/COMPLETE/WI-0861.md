# WI-0861: Calibration kill switch observability

| Field | Value |
|---|---|
| **ID** | WI-0861 |
| **Goal** | When `isMarketCalibrationEnabled` returns false, there is zero log output. Cards are silently dropped. Operators have no way to know NBA totals are suppressed without querying the DB. Add structured logging at the gate and surface kill switch state in the pipeline health report. |
| **Priority** | High |
| **Depends on** | WI-0825 (COMPLETE — gate exists), WI-0860 (schedule calibration report) |
| **Coordination flag** | solo |

---

## Context

Today the path is:

```
evaluateExecution() → isMarketCalibrationEnabled() → returns false
                                                ↓
                          blocked_by.push('CALIBRATION_KILL_SWITCH')
                                                ↓
                            card silently not written — no log
```

The scheduler logs `skip <jobKey>` for deduped jobs, but `CALIBRATION_KILL_SWITCH` happens *inside* the model run after all scheduler gates pass. The model ticks, runs the full projection, generates an execution assessment — and then silently discards the card. No `[CALIB]` line is emitted anywhere.

This created the dev/prod confusion: dev has no `calibration_reports` table so the catch in `isMarketCalibrationEnabled` always returns `true`. Production has the table with `kill_switch_active = 1`. You cannot reproduce the production suppression in dev without the table.

---

## Scope

**Files to change:**

`apps/worker/src/calibration/calibration-gate.js`
- In `isMarketCalibrationEnabled`, when returning `false` (kill switch active), emit:
  `console.warn('[CALIB_GATE] Kill switch ACTIVE for %s — ECE=%s nSamples=%s computed_at=%s — cards will be blocked', resolvedKey, row.ece, row.n_samples, row.computed_at)`
- Cache the row data alongside `enabled` so the log can fire without a second query
- Log at most once per TTL window per market (already handled by the existing 60s cache — log on cache miss only)

`apps/worker/src/jobs/check_pipeline_health.js`
- Add a `calibrationKillSwitches` section to the health report output
- Query `SELECT market, kill_switch_active, ece, n_samples, computed_at FROM calibration_reports ORDER BY datetime(computed_at) DESC` grouped by market (latest per market)
- If any market has `kill_switch_active = 1`, set a `CALIB_KILL_SWITCH_ACTIVE` warning in the health report
- If `calibration_reports` table does not exist (dev), skip gracefully

`apps/worker/src/jobs/run_nba_model.js` (and NHL, MLB equivalents)
- After `evaluateExecution` returns `shouldBet: false` with `block_reason: 'CALIBRATION_KILL_SWITCH'`, log:
  `console.log('[NBA_MODEL] %s blocked: CALIBRATION_KILL_SWITCH', jobKey)`
- This surfaces the block in the per-run log that operators already watch

**Out of scope:**
- Changing gate logic
- UI dashboard (WI-0827)
- Changing thresholds

---

## Acceptance

1. Running the NBA model when kill switch is active logs `[CALIB_GATE] Kill switch ACTIVE for NBA_TOTAL` with ECE and n_samples
2. `check_pipeline_health` output includes `calibrationKillSwitches` array; any active switch is flagged
3. Model run logs `blocked: CALIBRATION_KILL_SWITCH` for the affected card
4. `npm --prefix apps/worker test -- --testPathPattern=calibration` passes
5. `npm --prefix apps/worker test -- --testPathPattern=check_pipeline_health` passes
