# WI-0454: NHL Player Shots Prop — Hardening + Multi-Sport Prop Expansion Guide

**ID**: WI-0454
**Goal**: Close the known gaps in the NHL player shots prop pipeline that limit card volume and reliability, and produce a reusable guide for adding new player prop markets across sports so future work follows a consistent, safe pattern.

**Depends on**: WI-0453 (injury filtering — implement that first)

**Context**: The first working end-to-end run of the NHL player shots prop market surfaced only 1 qualifying card from a 2-player, synthetic-fallback pipeline. Several structural gaps limit production viability. This WI addresses those gaps and creates the process template so NBA points, rebounds, assists, and other markets can be added safely without reinventing the pipeline each time.

---

## Part 1 — NHL Player Shots Hardening

### Gap 1: Player list is a manual env var

**Problem**: `NHL_SOG_PLAYER_IDS` is a static comma-separated list maintained by hand. Missing players = missed markets. Injured players are not auto-excluded (WI-0453 covers the automated check; this WI covers expanding the list).

**Fix**:

- Expand `NHL_SOG_PLAYER_IDS` to cover the top ~30 NHL shot-volume leaders (not just stars)
- Document a quarterly review process: check shots/game leaders at trade deadline and season start, update the list
- Consider a `pull_nhl_top_shooters.js` job that auto-discovers players above a shots/game threshold from the NHL stats API and writes them to a `tracked_players` table — removing the static env var entirely

**Scope**: `apps/worker/src/jobs/pull_nhl_player_shots.js`, env config, optional `tracked_players` table

---

### Gap 2: TEAM_ABBREV_TO_NAME map is incomplete

**Problem**: `run_nhl_player_shots_model.js` uses a hardcoded `TEAM_ABBREV_TO_NAME` map with only 2 entries (EDM, TOR). Players on all other teams fail game matching silently — they are fetched, their logs stored, but no cards are ever generated.

**Fix**:

- Complete the map with all 32 NHL team abbreviations
- Add a startup warning if a player's `team_abbrev` is not in the map

**Scope**: `apps/worker/src/jobs/run_nhl_player_shots_model.js` (~30 line addition)

---

### Gap 3: Synthetic fallback produces random, unreliable edges

**Problem**: Without real Odds API lines, the fallback `Math.round((mu + (Math.random() - 0.5)) * 2) / 2` produces random lines. Cards appear or disappear between runs depending on the RNG. In production this is replaced by real lines, but in dev it makes the pipeline appear broken.

**Fix**:

- Replace the random offset with a deterministic fallback: `Math.round(mu * 2) / 2` (nearest 0.5 to projection)
- This will consistently produce 0 edge (line = projection) and generate 0 cards in dev unless a real line is available — making the behavior predictable and honest
- Document clearly in logs: `[synthetic-fallback] line=X is deterministic (no real line available)`

**Scope**: `apps/worker/src/jobs/run_nhl_player_shots_model.js`

---

### Gap 4: opponentFactor and paceFactor are hardcoded to 1.0

**Problem**: The model accepts `opponentFactor` and `paceFactor` inputs but both are hardcoded to 1.0, meaning opponent quality and team pace have no effect on projections. Stars playing a weak defensive team get the same mu as playing the league's best.

**Fix**:

- Source opponent shots-against rank from `team_metrics_cache` (already populated for other models)
- Derive `opponentFactor` as a simple ratio: opponent's shots-against-per-game / league average
- `paceFactor` can remain 1.0 until team pace data is available; add a TODO comment with the intended source

**Scope**: `apps/worker/src/jobs/run_nhl_player_shots_model.js`, `packages/data/src/db.js` (add `getTeamMetrics` query if not already accessible)

---

### Gap 5: 1P prop lines always synthetic

**Problem**: The Odds API does not reliably offer `player_shots_on_goal_1p` markets. First-period cards always use synthetic fallback and should likely be disabled by default until real lines exist.

**Fix**:

- Add env flag `NHL_SOG_1P_CARDS_ENABLED=false` (default off)
- When disabled, skip 1P card generation entirely rather than generating synthetic-fallback 1P cards
- Document in `NHL_PLAYER_SHOTS_PROP_MARKET.md`

**Scope**: `apps/worker/src/jobs/run_nhl_player_shots_model.js`, `docs/NHL_PLAYER_SHOTS_PROP_MARKET.md`

---

### Gap 6: Duplicate game ID matching (ESPN vs NHL canonical)

**Problem**: Some games in the `games` table have ESPN-style integer IDs (`401803409`) while others have NHL canonical UUID-style IDs (`5dbc5539eae23ae6f735c084b9f28cc8`) for the same game. The model runner matches players to games using both, but `game_id_map` entries are not always populated, so prop cards get stored against ESPN IDs that the `game_id_map` doesn't know about. The games API then cannot join them to canonical game records.

**Fix**:

- When the model runner finds a game with an ESPN-style ID that also has a canonical twin in the `games` table (same teams, same game_time ±15min), prefer the canonical game_id for card storage
- Add a `resolveCanonicalGameId(gameId, homeTeam, awayTeam, gameTime)` helper that checks `game_id_map` and falls back to a time+team proximity match in `games`

**Scope**: `apps/worker/src/jobs/run_nhl_player_shots_model.js`

---

### Gap 7: run_state not updated when 0 cards created

**Problem**: `setCurrentRunId` is only called when `cardsCreated > 0`. If a run produces 0 cards (e.g., no qualifying edges), the run_state stays stale and the API serves cards from the previous run — potentially cards for games that have already passed.

**Fix**:

- Always call `setCurrentRunId` on job success, regardless of card count
- This ensures stale runs are evicted from the active window even when the model finds no edges

**Scope**: `apps/worker/src/jobs/run_nhl_player_shots_model.js`

## Part 2 — Multi-Sport Prop Expansion Guide

Create `docs/PROP_MARKET_EXPANSION_GUIDE.md` documenting the standard pattern for adding a new player prop market. The guide should cover:

### The 4-file pattern

Every new prop market requires:

1. **Pull job** (`pull_{sport}_{prop}.js`) — fetches prop lines from Odds API for the market; env-gated; dry-run capable; writes to `player_prop_lines` with correct `sport`, `prop_type`, `period`
2. **Model runner** (`run_{sport}_{prop}_model.js`) — reads `player_shot_logs` or equivalent stat table; computes projection (mu); compares to real line from `player_prop_lines`; generates PROP cards; updates `run_state` with sport key
3. **npm scripts** — add `job:pull-{sport}-{prop}-props` and `job:run-{sport}-{prop}-model` to the worker's `package.json`
4. **Process doc** (`docs/{SPORT}_{PROP}_PROP_MARKET.md`) — data flow, job order, env vars, token cost, known limitations

### Registration checklist (frontend surfacing)

New prop markets must be registered in two places or cards will be silently dropped:

- `web/src/app/api/games/route.ts` → `ACTIVE_SPORT_CARD_TYPE_CONTRACT[SPORT].playProducerCardTypes` — add the new `card_type` string
- `web/src/app/api/games/route.ts` → `CORE_RUN_STATE_SPORTS` — add the sport key used in `setCurrentRunId`
- `web/src/app/api/cards/route.ts` → `CORE_RUN_STATE_SPORTS` — same sport key

### Odds API market keys by sport

Document the known market key strings for each sport so future work doesn't have to guess:

| Market | Odds API key | Notes |
|---|---|---|
| NHL shots on goal | `player_shots_on_goal` | Full game; 1P market unreliable |
| NBA points | `player_points` | Reliable coverage |
| NBA rebounds | `player_rebounds` | Reliable coverage |
| NBA assists | `player_assists` | Reliable coverage |
| NBA 3-pointers made | `player_threes` | Reliable coverage |
| MLB strikeouts (pitcher) | `pitcher_strikeouts` | Reliable coverage |
| NFL passing yards | `player_pass_yds` | Game-time injury risk is high |

### Edge classification thresholds

Document that thresholds are sport/market-specific and must be backtest-calibrated before going live. Default to the NHL SOG thresholds (HOT ≥ 0.8, WATCH ≥ 0.5 edge) only as a starting point — these should be tuned per market.

### Token cost planning

Each new market adds tokens per run. Document the formula: `events_per_day × runs_per_day × days_in_season ≈ monthly_token_budget`. Gate all new markets behind `{SPORT}_{PROP}_ENABLED=false` env flags to prevent accidental quota burn during development.
