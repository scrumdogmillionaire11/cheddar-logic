# WI-0787: Add `projection_audit` row-level table for settled projections

| Field              | Value |
|--------------------|-------|
| **ID**             | WI-0787 |
| **Goal**           | Create a `projection_audit` table that stores one row per settled projection, capturing all segmentation dimensions at settle time, replacing the lossy aggregate-only model of `tracking_stats` as the source of truth |
| **Priority**       | Medium |
| **Owner agent**    | Unassigned |
| **Time window**    | TBD |
| **Coordination flag** | `needs-sync` — adds DB migration under `packages/data/db/migrations/` |

---

## Problem

`tracking_stats` is an aggregate-only table. Its `stat_key` is `sport|marketType|direction|confidenceTier|driverKey|timePeriod` — once settled, you cannot retroactively slice by new dimensions (player_count, odds source, confidence band, etc.) without rescanning `card_results` from scratch. There is no row-level audit trail that associates a single projection outcome with all its context at settle time.

This became apparent when 1st-period total (`total_1p`) plays settled into the record during a temporary odds window and needed to be excluded. The problem is structural: `tracking_stats` cannot selectively exclude or filter once rows are accumulated.

## Table Schema

```sql
CREATE TABLE IF NOT EXISTS projection_audit (
  id                  TEXT PRIMARY KEY,        -- card_result_id
  card_result_id      TEXT NOT NULL UNIQUE,    -- FK reference to card_results.id
  sport               TEXT NOT NULL,
  market_type         TEXT NOT NULL,           -- 'total', 'moneyline', 'spread', etc.
  period              TEXT,                    -- NULL, '1P', '2P', etc.
  player_count        INTEGER,                 -- 1, 2, 3 ... number of players in card
  confidence_score    REAL,                    -- raw model confidence (0-1)
  confidence_band     TEXT,                    -- '<40', '40-50', '50-60', '60+'
  odds_american       INTEGER,                 -- opening line at projection time
  sharp_price_status  TEXT,                    -- 'CONFIRMED', 'ESTIMATED', 'UNTAGGED'
  direction           TEXT,                    -- 'OVER', 'UNDER', 'HOME', 'AWAY', etc.
  result              TEXT NOT NULL,           -- 'win', 'loss', 'push'
  pnl_units           REAL NOT NULL DEFAULT 0,
  settled_at          TEXT NOT NULL,           -- ISO8601
  job_run_id          TEXT,
  metadata            TEXT,                    -- JSON blob for any extra context
  created_at          TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_projection_audit_sport_market
  ON projection_audit (sport, market_type);
CREATE INDEX IF NOT EXISTS idx_projection_audit_settled_at
  ON projection_audit (settled_at);
CREATE INDEX IF NOT EXISTS idx_projection_audit_player_count
  ON projection_audit (player_count);
CREATE INDEX IF NOT EXISTS idx_projection_audit_sharp_price_status
  ON projection_audit (sharp_price_status);
```

---

## `confidence_band` Derivation

Derived at write time from `confidence_score`:

| Score range | Band label |
|-------------|------------|
| < 0.40      | `<40`      |
| 0.40–0.50   | `40-50`    |
| 0.50–0.60   | `50-60`    |
| ≥ 0.60      | `60+`      |
| NULL/unknown | `unknown` |

---

## `player_count` Derivation

Attempt to extract from card payload metadata. Fallback to NULL if unavailable. Do not block settlement on a missing value.

---

## Write Point

In `settle_pending_cards.js` Step 2 loop (same place `incrementTrackingStat` is called), add a call to `insertProjectionAudit` for every row in `aggregateRows`. This runs at settle time, not projection time.

Use `INSERT OR IGNORE` so re-runs are safe (idempotent on `card_result_id`).

## Acceptance

- Migration runs cleanly with no errors
- After a settlement run, `SELECT COUNT(*) FROM projection_audit` equals the number of rows settled in that run
- Row for a settled 1P total has `period = '1P'`, `market_type = 'total'`
- Row for a settled 2-player NBA total has `player_count = 2` (or NULL if not derivable)
- `INSERT OR IGNORE` allows settlement re-runs without duplicates
- No change to `tracking_stats` behavior (WI-0788 handles that)
