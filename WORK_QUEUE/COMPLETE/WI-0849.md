## Observed Failure

Every NBA model run fails for game `01f002eb30dfd24045740d4fbf68207e`:

```
[gate] 01f002eb30dfd24045740d4fbf68207e [nba-totals-call]: HARD_LOCKED
[err]  01f002eb30dfd24045740d4fbf68207e: FOREIGN KEY constraint failed
```

The `HARD_LOCKED` gate fires correctly (a settled `card_results` row exists for the call card). The FK error fires immediately after in the same game's write transaction.

## Diagnosis Steps

**Step 1 — Find orphan card_results:**

```sql
SELECT cr.id, cr.card_id, cr.status, cr.card_type
FROM card_results cr
LEFT JOIN card_payloads cp ON cp.id = cr.card_id
WHERE cp.id IS NULL;
```

If rows are returned: migration 062 deleted non-canonical `card_payloads` but left `card_results` behind for that game's driver cards. These orphaned rows cause FK violations when `prepareModelAndCardWrite` issues a DELETE against `card_payloads` and SQLite's FK engine finds a child `card_results` row pointing to NULL.

Wait — `prepareModelAndCardWrite` deletes from `card_payloads`. SQLite FK enforcement requires that `card_results` rows whose `card_id` references a `card_payloads` being deleted are handled first. `deleteCardPayloadsByGameAndType` does delete pending `card_results` first, but if orphaned `card_results` (pointing to an already-deleted `card_payloads`) exist, they may cause a FK violation through a different path.

**Step 2 — Isolate affected game:**

```sql
SELECT cp.id, cp.card_type, cp.created_at, cr.status
FROM card_payloads cp
LEFT JOIN card_results cr ON cr.card_id = cp.id
WHERE cp.game_id = '01f002eb30dfd24045740d4fbf68207e'
ORDER BY cp.created_at;
```

**Step 3 — Check PRAGMA foreign_keys state in the write path**

Confirm `foreign_keys = ON` is set consistently. If it is, the orphan cleanup in step 1 is the fix.

## Likely Fix

If orphaned `card_results` rows are confirmed (step 1 returns rows):

**Option A — Migration 063 (cleanup):**
Add `063_cleanup_orphaned_card_results.sql`:
```sql
-- Remove card_results rows whose card_id no longer exists in card_payloads.
-- These are left-over orphans typically from the migration 062 dedup run.
DELETE FROM card_results
WHERE card_id NOT IN (SELECT id FROM card_payloads);
```

**Option B — Guard in deleteCardPayloadsByGameAndType:**
Before deleting `card_payloads`, first delete any orphaned `card_results` referencing the target cards. This makes the function self-healing but is broader than needed.

Prefer Option A unless the orphan pattern is expected to recur.

## Notes

- This only affects one game per run consistently (`01f002eb30dfd24045740d4fbf68207e`), suggesting game-specific state corruption rather than a systematic code bug.
- The HARD_LOCKED gate is working correctly — the call card is legitimately locked. The FK failure is in the driver card write path for the same game.
- WI-0843 (card_results duplication) is related context but addresses a different symptom.
