# WI-0608: JWT revocation persistence — move from in-memory Set to DB table

**ID**: WI-0608
**Goal**: Replace the in-memory token revocation Set in `jwt.ts` with a persistent `revoked_tokens` DB table so that revoked tokens remain invalid across server restarts.

**Background**:
`web/src/lib/api-security/jwt.ts` stores revoked token IDs in a module-level `Set<string>`. A server restart clears the Set, making previously-revoked tokens valid again. An explicit TODO in the file (lines 97–101) says to move this to Redis or DB. This is a security regression: any admin-triggered revocation is silently undone on deploy or crash-restart.

**Scope**:
- `packages/data/db/migrations/049_create_revoked_tokens.sql` (new) — `revoked_tokens (jti TEXT PRIMARY KEY, revoked_at INTEGER, expires_at INTEGER)`
- `packages/data/src/db.js` — add `insertRevokedToken(jti, expiresAt)`, `isTokenRevoked(jti)`, `pruneExpiredRevokedTokens()`
- `web/src/lib/api-security/jwt.ts` — replace in-memory Set with DB calls; call prune on a background interval or at startup

**Out of scope**:
- Redis integration (DB is sufficient for current scale)
- Revocation UI or admin endpoint changes

**Acceptance**:
- After calling `revokeToken(jti)`, the token remains invalid after a simulated server restart (new process reading from DB).
- `isTokenRevoked(jti)` returns `true` for tokens inserted by `insertRevokedToken`.
- Expired tokens are pruned on startup (or cron) so the table does not grow unbounded.
- TypeScript compiles cleanly.
