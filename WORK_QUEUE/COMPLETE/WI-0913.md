# WI-0913: News Signal Foundation - Registry, Schema, and Reason Codes

**ID**: WI-0913
**Goal**: Establish the canonical cross-sport news input contract, source registry, freshness rules, and reason-code vocabulary so all later polling, model, and watchdog work flows through structured `slate_context` data instead of raw text side channels.

**Depends on**: none

**Replan note**:

- Treat WI-0913 as the umbrella foundation for cross-sport news input architecture.
- Execute implementation via WI-0915 (polling + target resolution + slate_context attachment), WI-0916 (sport-specific impact translation + input gate), and WI-0917 (watchdog + validator enforcement).
- Keep playoff overlay work in WI-0914 after WI-0917 lands because the same scheduler/gate/validator surfaces are shared.

**Execution split**:

- WI-0913 defines the shared contract and registry foundation only.
- WI-0915 implements polling orchestration, target resolution, and `slate_context` attachment.
- WI-0916 implements sport-specific impact translation and input-gate integration.
- WI-0917 implements decision/watchdog/validator enforcement.

**Out of scope**:

- Scheduler polling cadence and impacted-game refresh wiring
- Resolving normalized items into runner payloads or model inputs
- Sport runner, model, watchdog, or payload-validator behavior changes
- Live FOOTIE worker runtime integration (registry/schema support only in this WI)

**Acceptance**:

- A deterministic `SourceRegistryEntry` contract exists with role-based source classes, trust tiers, freshness/poll frequency metadata, and sport/category coverage.
- A canonical `NewsSignal` contract exists with normalized category/status/severity/source/targets/timestamps/payload/derived fields and explicit reason-code support.
- Source trust rules are documented and enforced: low-trust or warn-only sources cannot confirm critical play-killing events.
- Freshness semantics are defined per source/category and normalized into stable freshness bands without relying on ad hoc runner logic.
- Duplicate and contradictory news rules are documented and test-covered before scheduler/model integration begins.
- FOOTIE is represented in the shared docs/registry contract, but no worker runtime polling is required in this WI.
