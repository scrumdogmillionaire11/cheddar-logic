# WI-0997: POTD Shadow Candidate Settlement Keys and Complete Capture

**ID**: WI-0997
**Goal**: Make POTD shadow candidate rows consistently captured and machine-gradeable without changing official POTD firing behavior.

**Scope**:
- `apps/worker/src/jobs/potd/run_potd_engine.js`
- `apps/worker/src/jobs/potd/__tests__/run-potd-engine.test.js`
- `packages/data/db/migrations/078_add_potd_shadow_candidate_settlement_fields.sql`

**Out of scope**:
- Official POTD firing-rule changes
- Writes to `potd_plays` beyond the existing fired official POTD path
- Writes to `potd_bankroll`
- Shadow settlement grading or outcome computation
- Web/API/UI changes

**Acceptance**:
- `potd_shadow_candidates` has gradeable fields for future rows, including canonical `selection` (`HOME`, `AWAY`, `OVER`, `UNDER`), `game_time_utc`, and a stable candidate identity key sufficient for idempotent daily shadow tracking.
- Shadow candidates are persisted on every terminal POTD engine path after candidates are scored: official fired, no best candidate, confidence-below-HIGH, zero-wager, and stake-below-minimum.
- Repeated same-day runs do not create ambiguous duplicate current-state shadow candidates for the same candidate identity.
- Existing official `potd_plays`, `potd_bankroll`, and card payload behavior is unchanged.
- Tests cover fired-day capture, no-pick capture, and at least one late-gate rejection path.
