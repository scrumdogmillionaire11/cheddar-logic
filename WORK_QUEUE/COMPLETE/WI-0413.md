## Summary

Build WI-0378 as a **fault-aware audit**, not a polite documentation exercise. The real job is to expose where the NHL totals stack can lie to itself about goalie certainty, apply model math on shaky starter assumptions, and still accidentally surface a clean-looking FIRE/HOLD/PASS output. The files already show that goalie status drives parsing, model adjustment, wrapper gating, consistency checks, and watchdog vetoes, but the semantics are split across tags, user inputs, and inferred status instead of one canonical contract.   

## Why This Is Needed

The current NHL totals system has the exact kind of architecture that breeds fake confidence.

The parser emits goalie tags and warnings, the workflow expects user-supplied goalie certainty, the pace model applies goalie-tier adjustments including an `Unknown` fallback, consistency checks use goalie tiers to shape volatility and bias, and the watchdog can force totals to PASS if the goalie is unconfirmed. That means one bad or ambiguous goalie state can contaminate the pipeline in multiple ways while still looking structurally valid.    

The biggest faulty-model behaviors to design around are these:

1. **Semantic collapse:** ‚ÄúConfirmed,‚Äù ‚ÄúExpected,‚Äù and ‚ÄúUnknown‚Äù can behave too similarly downstream, even though the wrapper explicitly treats them differently. 
2. **Math on bad identity:** the pace model still applies a goalie adjustment when the goalie is unknown, which is useful as a fallback but dangerous if downstream logic forgets that the number is low-trust. 
3. **Split-brain inputs:** parser-derived goalie tags and user-supplied goalie status can disagree, with no clearly documented arbitration rule.  
4. **Contradiction masking:** consistency checks can label a game with a clean event direction while uncertainty in goalie identity should really cap it harder.  
5. **Late wrapper heroics:** the wrapper can look disciplined with HOLD/FIRE labels while the underlying model total was produced from bad assumptions. That‚Äôs lipstick on a pig, buddy. The pig still bites.

### Step 3 ‚Äî Explicitly enumerate faulty model behaviors

This is the part you asked for. The agent should include at least these named behaviors.

**FM-01: Expected-as-Confirmed drift**
The parser emits `NHL_GOALIE_EXPECTED`, but downstream logic may still let the pace model use a specific goalie tier while the wrapper only partially downgrades. Result: model total looks sharper than it is.  

**FM-02: Unknown-is-bad-goalie contamination**
The pace model applies an `Unknown: +0.10` adjustment. That is a modeling convenience, not evidence the goalie environment is over-friendly. If analysts or downstream code read it as ‚Äúweak goalie environment,‚Äù overs get fake juice. 

**FM-03: Parser/User split-brain**
The orchestrator and workflow require user-supplied goalie status, while the parser can also infer goalie context from text. No canonical merge contract is shown. Result: one layer says Confirmed, another says Expected, and the system has no single truth source.  

**FM-04: Name-resolution tier mismatch**
Parser confidence rules allow missing goalie metrics or mismatches without hard failure. That means a goalie can be identified as starter but fail metric linkage, causing tier degradation or fallback behavior that is easy to miss. 

**FM-05: Environment over-certainty under missing goalie trust**
Consistency checks derive `vol_env` partly from goalie tier combinations and output `OVER/UNDER/NEUTRAL` directionality. If goalie tier itself is uncertain, the derived environment can look cleaner than the evidence supports. 

**FM-06: Under bias from public heuristics**
The fade-public model explicitly warns against ‚Äúeveryone likes the over, so auto-under‚Äù logic in NHL totals. The WI should treat any under recommendation that leans on public fade without goalie confirmation as a fault pattern. 

**FM-07: Wrapper rescue illusion**
The wrapper can correctly output HOLD because a goalie is not confirmed, but the model total beneath it may already be biased by a wrong starter assumption. HOLD helps operationally, but it does not fix the corrupted number. 

**FM-08: Watchdog too late**
The watchdog can force totals to PASS if goalie/player is unconfirmed, but by then upstream classifications and notes may already have been generated around compromised model math. The WI should call this a ‚Äúlate veto architecture‚Äù issue. 

**FM-09: Contradictory signals not escalated enough**
Slate context and consistency rules say contradictory or missing data should cap confidence. The WI should verify whether goalie contradictions specifically become strong flags, not just generic warnings.

## Failure Modes & Guards

### Failure mode: one goalie state, five interpretations

The current architecture lets parser tags, workflow inputs, model adjustments, consistency checks, and watchdog rules each interpret goalie certainty a little differently. That is how bugs get dressed up as process.

**Guard:** require the WI to recommend a single pre-model canonical goalie object.

### Failure mode: uncertainty gets numericized too early

The pace model‚Äôs `Unknown +0.10` fallback is useful, but it can become a fake signal if not labeled as uncertainty-driven. 

**Guard:** split ‚Äúprojection adjustment‚Äù from ‚Äúconfidence in adjustment.‚Äù

### Failure mode: wrapper looks disciplined while number is rotten

A HOLD label can hide bad underlying assumptions.

**Guard:** add a model-trust field that wrapper must surface in notes or use in gating.

### Failure mode: watchdog catches things only after ranking

If the watchdog is the first strong gate, your model and wrapper already spent effort ranking junk. 

**Guard:** move goalie trust checks earlier in the eventual implementation WIs.

### Failure mode: contradictions become soft warnings

Parser warnings are useful, but warnings alone do not stop picks. 

**Guard:** define contradiction-specific fatal or near-fatal flags for NHL totals.

## Open Questions ‚Äî Resolved

All four decisions are stamped. These govern how WI-0379 through WI-0384 are scoped.

### Q1: Should `EXPECTED` starters influence goalie tier math, or only wrapper logic?

**Decision:** Yes, but only through a degraded-trust path.

- `CONFIRMED` ‚Üí full goalie tier math allowed; `adjustment_trust: FULL`
- `EXPECTED` + named starter resolved ‚Üí math runs, `adjustment_trust: DEGRADED`
- `UNKNOWN` ‚Üí no directional goalie tier adjustment; `adjustment_trust: NEUTRALIZED`
- `CONFLICTING` ‚Üí no goalie tier adjustment, official totals eligibility blocked; `adjustment_trust: BLOCKED`

Rationale: forcing `EXPECTED` to wrapper-only discards genuinely useful information and flattens too many spots. Treating `EXPECTED` as `CONFIRMED` is fake precision. Degraded-trust is the correct middle lane.

### Q2: When parser-derived and user-supplied goalie states disagree, which source wins?

**Decision:** User-supplied wins, but disagreement must be preserved as an explicit conflict artifact.

Precedence order:
1. User-supplied explicit status
2. Trusted scraper / name-match
3. Weak inference / season-table fallback

If sources disagree, emit `CONFLICTING_SOURCE_EVIDENCE` into `evidence_flags`. The final canonical state must not look clean when sources conflict. Stale or malformed user input must also be flagged.

Rationale: This system operates with late human-confirmed inputs in NHL mode; user input is intended to be authoritative. But precedence resolves authority, not truth ‚Äî conflict evidence must survive arbitration.

### Q3: Should "unknown goalie" produce a neutral adjustment or a conservative uncertainty band instead of the current directional over nudge?

**Decision:** Neutral adjustment plus uncertainty escalation. No over nudge from unknown state.

- Math layer: neutral goalie adjustment (no directional factor applied)
- Consistency layer: increase volatility, reduce trust (`vol_env = VOLATILE`, `total_bias = INSUFFICIENT_DATA`)
- Wrapper / watchdog: default HOLD/PASS unless edge is oversized and other factors are overwhelming

Rationale: `UNKNOWN` means missing information, not bullish information. Treating unknown as an over nudge is fake certainty. This is consistent with the project's consistency-check doctrine: missing inputs ‚Üí downgrade confidence, treat conservatively.

### Q4: Should totals be hard-blocked on `CONFLICTING` goalie state at model time, not just watchdog time?

**Decision:** Yes for official-play eligibility; no for raw descriptive projection generation.

- Model may still compute a **descriptive neutralized projection** for observability and debugging
- Official market classification must be blocked at the model-time eligibility layer (`official_eligible: false`)
- Watchdog enforces again as a second barrier via `GOALIE_CONFLICTING` reason code

Rationale: avoids black-box blindness while preventing downstream nonsense. `CONFLICTING` must never produce a normal-looking total output.

## Faulty Model Behavior Inventory

> This audit treats faulty model behavior as a first-class deliverable. The objective is not only to document where goalie inputs exist, but to identify where the NHL totals system can generate a plausible yet incorrect recommendation due to ambiguous starter confirmation semantics, delayed uncertainty handling, conflicting input sources, or confidence inflation caused by fallback goalie assumptions.

### Current-State Goalie Pipeline Map

Each hop documents: Input ‚Üí Transform ‚Üí Assumptions ‚Üí Output ‚Üí Failure if wrong ‚Üí Silent or explicit.

```
raw goalie note / user status
  -> parser tags + warnings (moneypuck.js)
  -> player/game objects (oddsSnapshot.raw_data goalie fields)
  -> merged game goalie state (NO canonical merge ‚Äî goalie_home_gsax wins by default)

  -> wrapper HOLD/FIRE/PASS gate (decision-publisher.js deriveAction via tier)

```

### Semantic Faults

**FM-01: Expected-as-Confirmed drift**

`homeGoalieConfirmed` is derived from `goalieHomeGsax !== null`. GSaX presence in MoneyPuck's season table is not game-day starter confirmation. When a goalie is "Expected" (rumor, not official), their season GSaX may still be present, causing the model to treat them as confirmed. The pace-model adjustment runs at full trust. The confidence cap (`goalieConfidenceCapped`) only triggers when BOTH goalies are null ‚Äî a single partially-trusted goalie keeps the cap from firing.

Severity: **High**

**FM-02: Unknown-is-bad-goalie contamination**

The pace model applies `goalieAdjFactor(savePct)` when savePct is present. There is no numeric fallback for "Unknown goalie" in `nhl-pace-model.js` ‚Äî when savePct is null, no adjustment fires at all (lines 252-261). However, when a backup goalie's save-pct IS present in the data but their starter status is actually unknown or rumored, the model applies a legitimate-looking weak-goalie adjustment (+uplift on opponent goals). This can produce a nudge toward OVER without any flag that the goalie identity is unverified. Downstream analysis may interpret the over-nudge as a weak-goalie read rather than an uncertainty artifact.

Severity: **High**

**FM-03: Parser/User split-brain**

`moneypuck.js` enriches `raw_data` with GSaX from a season table. There is no user-supplied starter confirmation field in the current pipeline. The `homeGoalieConfirmed` boolean is derived purely from the scrape result ‚Äî there is no merge contract between a user-supplied goalie status and the scraper output. If a user or upstream system annotates a game with "Expected" or "CONFLICTING" starter intent, that signal has no path into the model. The scrape wins by default.

Severity: **Critical**

**FM-04: Name-resolution tier mismatch**

MoneyPuck's `parseGoalies()` matches by team name (canonical). If the team's primary goalie is out and the backup plays, the backup's GSaX is not necessarily the top row for that team ‚Äî but the current scraper takes `goaliesByTeam[canonical]` without filtering for starter. The goalie with the best recent game may not be the actual game-day starter. No metric join validation is performed.

Severity: **Medium**

**FM-05: Environment over-certainty under missing goalie trust**

`cross-market.js` uses `goalieSignal` as a driver in the TOTAL net score. The signal is `eligible: goalieSum !== null`. When both GSaX values are present (even if one is wrong-identity), the driver fires at full weight (effective weight contribution to net score). The consistency envelope derived in `decision-publisher.js` uses `vol_env` from conflict score ‚Äî not from goalie certainty. A game with shaky goalie identity can still receive a clean `STABLE` vol_env and a directional `FAVOR_OVER` or `FAVOR_UNDER` event_direction_tag.

Severity: **High**

**FM-06: Under bias from public heuristics**

The cross-market goalie_quality signal `goalieSignal = clamp(-goalieSum / 6, -1, 1)` maps high combined GSaX to an UNDER signal. If public consensus is "strong goalies tonight ‚Üí UNDER" and the model's goalie data reinforces this, a fade-public logic overlay could be double-counting: the model already baked in the goalie signal, and then the fade-public heuristic adds further under pressure based on public sentiment that was itself driven by the goalie narrative. This is most dangerous when goalie identity is only partially confirmed.

Severity: **Medium**

---

### Data-Contract Faults

**FM-07: Wrapper rescue illusion**

`decision-publisher.js` maps `tier ‚Üí action` deterministically. A HOLD output correctly signals caution, but the underlying `expectedTotal` and `expected1pTotal` in the pace model were computed before any goalie uncertainty check. The card payload has no field that surfaces "this total was computed from low-confidence goalie input." A user sees HOLD and interprets it as "sharp but cautious" ‚Äî not as "model math may be corrupted." HOLD does not fix the number.

Severity: **Critical**

**FM-08: Watchdog too late**

`WATCHDOG_REASONS` in `decision-pipeline-v2.js` does not include a `GOALIE_UNCONFIRMED` code. The only goalie-adjacent protection is the confidence cap in `nhl-pace-model.js` (`confidence = Math.min(confidence, 0.35)` when `goalieConfidenceCapped = true`). But `goalieConfidenceCapped` fires only when BOTH goalies are unconfirmed (null GSaX). The watchdog is downstream of projection, driver computation, card assembly, and validation. By the time the watchdog could theoretically veto on goalie grounds, the model has already run, the card has been assembled, and the notes have been generated with goalie-adjusted numbers baked in.

Severity: **High**

### Model-Math Faults

The model applies goalie save-pct adjustment at Hop 3 regardless of the `homeGoalieConfirmed` boolean (see Pipeline Hop 3 above). # The 7 Most Common Reasons NHL Totals Run Too High

Here‚Äôs what I will be looking for immediately when you send the outputs.

## 1Ô∏è‚É£ Goalie Adjustment Overweight

A classic model mistake.

Example:

```
Elite goalie   -0.35
Backup goalie  +0.30
Poor goalie    +0.45
```

Those look harmless but if:

* both teams project **+0.25 offense**
* both goalies are downgraded

You get:

```
+0.5 offensive environment
+0.6 goalie effect
---------------------
+1.1 goals inflation
```

That‚Äôs enormous.

If your **1P totals are also high**, this is often the cause.

---

## 2Ô∏è‚É£ Pace Model Double Counting

Very common when models use both:

```
shot rate
expected goals
pace
```

These are **not independent variables**.

If pace already drives shot rate, and shot rate drives xG, you are counting the same signal **twice or three times**.

Symptoms:

* Full totals ~0.4‚Äì0.8 goals high
* 1P totals especially inflated

---

## 3Ô∏è‚É£ 1P Scaling Incorrect

The typical mistake:

```
1P = Full Total √ó 0.33
```

But NHL scoring **is not evenly distributed**.

Typical league averages:

```
1P ‚âà 2.05 goals
Full ‚âà 6.1 goals
Ratio ‚âà 0.335
```

But models often forget:

* empty net goals inflate full totals
* penalties late in game inflate 3P

A better ratio is usually:

```
1P ‚âà 0.31‚Äì0.32 √ó full total
```

If you‚Äôre using **0.33‚Äì0.34**, your 1P numbers will run hot.


If your model does something like:
