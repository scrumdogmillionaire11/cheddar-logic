# WI-0584 — Line-Change Dedup Gap: Two Cards for Same Player/Side After Odds Update

**ID:** WI-0584
**Goal:** Fix the dedup gap that allows two cards for the same player + side to coexist after an odds-line change, caused by `dedupeLine` differing between model runs and `seenNhlShotsPlayKeys` not catching the existing card.
**Severity:** MEDIUM
**Owner agent:** unassigned
**Status:** COMPLETE — Done in quick task 77 (2026-03-24, 974f114)
**Time window:** —
**Coordination flag:** none

## Problem

Two failure modes interact:

1. **`purgePlayerCardsForGame` silent failure** — if the purge query throws or returns 0 rows deleted when it should have deleted the old card, the old card persists in `card_payloads`.
2. **`seenNhlShotsPlayKeys` dedup keyed on `dedupeLine`** — if the old card was published at line `2.5` and the new run sees line `3.0`, the dedup key differs and both cards are inserted. The `/api/games` route then returns two cards for the same player/side on the same game.

The result is duplicate prop cards surfacing to the user for the same bet.

## Scope

- `apps/worker/src/jobs/run_nhl_player_shots_model.js`:
  - Wrap `purgePlayerCardsForGame` in a try/catch with `console.warn` on failure.
  - Log the number of rows deleted; warn if 0 when cards are expected.
  - Change the dedup key for `seenNhlShotsPlayKeys` to `${gameId}:${playerId}:${side}` (no line) so a line change still deduplicates within a single run.
- `web/src/app/api/games/route.ts`:
  - Add a secondary dedup pass in the prop-card assembly that keeps only the most-recent card per `(gameId, playerId, propType, side)` tuple before returning.
- Add a test: two cards for same player/side/game with different `dedupeLine` values → only the newest card is returned.

## Out of Scope

- Refactoring the purge function beyond error handling.
- Changes to other prop types.
