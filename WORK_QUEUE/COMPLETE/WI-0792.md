## Context

WI-0761 shipped the Model Health Dashboard at `/admin`. It currently shows:

- **Pipeline Health** — recent `pipeline_health` rows (phase / check / status / reason / timestamp)
- **Recent Model Outputs** — last 50 rows from `model_outputs`

The page is functional but deliberately minimal. Two open questions need design decisions before further investment:

1. **Log presentation** — Raw rows in reverse-chron is a start, but it may not surface signal well.
2. **Alert reaction** — `pipeline_health` rows with `status=warning|failed` exist, but there is no defined playbook for what a human should do when they see one.

---

## Design Decisions (SETTLED)

### Log Presentation

**Decision: Current snapshot card + history table is the right structure. Keep it. Add two improvements:**

1. **Consecutive failure count on snapshot cards** — The snapshot already shows latest-per-phase, but doesn't tell you how long a phase has been failing. The history table already contains the data; derive a streak count client-side by counting the leading same-status rows per (phase, check_name) from the ordered history. Display it on the card as e.g. "failed × 3" when streak ≥ 2.

2. **Stale check-runner detection** — Some checks (notably `mlb:f5_market_availability`) intentionally skip writing rows when no games are in window. A 23h-old snapshot card that shows `ok` looks identical to a 1m-old one. Cards where `created_at` age > 35min (i.e., older than one check cycle + buffer) should be visually muted and labeled "no active games" or "check dormant" rather than presenting the stale result as current state.

**Rejected:**

- Sparklines — not worth the complexity for this internal tool at current scale
- `job_runs` integration on this page — too noisy; `job_runs` is a separate diagnostic surface
- Auto-refresh interval change — 60s is fine; health checks run every 5min so polling faster adds no signal

### Alert Reaction Playbook

**Threshold definitions:**

| Severity | Meaning | Default action |
|---|---|---|
| `ok` | All clear | None |
| `warning` | Degraded but not critical | Monitor; intervene if still warning after 1h |
| `failed` × 1 | One check cycle failed | Investigate — check logs |
| `failed` × 3 (≈15min) | Sustained failure | Intervene immediately — re-run job |
| `failed` × 12 (≈1h) | Prolonged failure | Escalate — possible systemic issue |

**Per-phase runbook:**

| Phase | Check | Status | SLA | Recovery steps |
|---|---|---|---|---|
| `schedule` | `freshness` | `failed` | 24h | 1. Verify games DB has upcoming rows. 2. Re-run `pull_schedule.js`. 3. Check sport API connectivity. |
| `odds` | `freshness` | `failed` | **15min / immediate** | 1. Identify which game_ids are stale (reason string says N/M). 2. Check `pull_odds.js` logs for rate-limit or API errors. 3. Re-run `pull_odds.js`. |
| `cards` | `freshness` | `failed` | 30min | 1. Check which games within T-2h are missing cards. 2. Identify which sport model job produces those cards. 3. Re-run `run_[sport]_model`. |
| `settlement` | `backlog` | `warning` | 1h | 1. Check if count is static across consecutive checks (stalled) vs growing/shrinking (slow). 2. Check `settle_game_results.js` logs. 3. Verify those game_ids have final scores in `games` table. 4. Re-run settler. |
| `mlb` | `f5_market_availability` | `failed` | 1 day (informational) | 1. Check `odds_snapshots` for those game_ids — is the F5 line present? 2. If no F5 lines in odds data, this is an upstream feed gap, not a code issue — monitor. |
| `nba` | `model_freshness` | `failed` | 4h threshold | 1. Check `job_runs` for `run_nba_model` — did it fail or not run? 2. Re-run `run_nba_model`. |
| `nhl` | `model_freshness` | `failed` | 4h threshold | Re-run `run_nhl_model`. |
| `nhl` | `shots_model_freshness` | `failed` | 4h threshold | Re-run `run-nhl-player-shots-model`. |
| `mlb` | `model_freshness` | `failed` | 3h threshold | Re-run `run_mlb_model`. |

**Special case — `settlement backlog` never escalates to `failed`:**
The current code always writes `warning` for settlement backlog > 0, never `failed`. This is intentional (backlog of unsettled games is normal intraday) but means the streak counter won't catch a stalled settler. The dashboard improvement (WI-0799) should highlight when the backlog count is unchanged across ≥ N consecutive checks.

**Special case — `mlb:f5_market_availability` skips writing when no MLB games in window:**
By design, no row is written when there are no upcoming MLB games within T-6h. A stale snapshot card for this check means "no MLB games right now," not a runner failure. The stale-card visual treatment (WI-0799) handles this.
