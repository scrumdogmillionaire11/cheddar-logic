# WI-0455: Projection-Only View — Separate 1P and Player Props from Play Cards

**ID**: WI-0455
**Status**: IDEA — not yet scoped/prioritized
**Goal**: Give 1P totals and player props their own dedicated view so users aren't mixing informational projections with actionable play cards that have real odds and edge calculations.

---

## The Problem

Right now there are two fundamentally different objects being rendered in the same card surface:

**Play cards** (full pipeline)
- Spread, moneyline, full-game total
- Odds fetched → edge calculated → PLAY/LEAN/PASS decision
- Users act on these

**Projection-only cards** (no odds pipeline)
- 1P totals (`nhl-pace-1p`)
- Player props (SOG, points, etc.)
- No odds fetched, no edge computed, no actionable decision
- Users read these for context/research

Mixing them creates friction: a user scanning for plays sees projection cards that look similar but have no odds, no edge, and no action. Conversely, users who want the projection context have to hunt through play cards to find it.

---

## Ideas / Options

### Option A — Tab-based split on the existing cards page

Add a tab bar to the cards page:

```
[ Plays ]  [ Projections ]  [ Props ]
```

- **Plays**: current full-game card view (spread/ML/total with odds + edge)
- **Projections**: 1P cards (`nhl-pace-1p`, any future 2H/alt-period cards)
- **Props**: player prop cards (SOG, points, rebounds, assists)

Low friction — same page, same URL base, just filtered by market_type/cardType bucket.

### Option B — Separate route per surface

```
/cards          → plays only (status quo minus projections)
/cards/1p       → first-period projections
/cards/props    → player props
```

Cleaner URL semantics, easier to deep-link and share. Each route can have its own filter defaults optimized for that content type. Higher implementation cost.

### Option C — Collapsible sections within the existing game card

Within each game's card group, stack sections:

```
▶ PLAYS        (spread/ML/total — collapsed or expanded)
▶ 1P OUTLOOK   (first-period projection — collapsed by default)
▶ PROPS        (player lines — collapsed by default)
```

Keeps everything game-scoped. Useful when users want the full picture for one game without navigating away.

### Option D — "Context panel" alongside the play card

1P and props rendered as a sidebar or bottom drawer when a play card is selected/expanded. The projection informs the play rather than competing with it for attention.

---

## Implementation Considerations

- **Market type discriminator**: `market_type === 'FIRST_PERIOD'` and `market_type === 'PROP'` already exist — no new schema work needed to separate them
- **Odds absence**: Projection cards have no `edge`, no `sharp_price`, no `price_home/away`. The projection view should not render those columns at all — different card template needed
- **Card component**: `card.tsx` currently assumes odds/edge present. A `ProjectionCard` variant would strip the odds/edge section and emphasize the projected value (e.g., projected total: 1.4 goals) instead
- **API**: `GET /api/cards` already returns all card types. Filtering can happen client-side by `market_type` bucket, or a `?view=projections` query param added server-side to reduce payload
- **Feature flag**: Player props are already behind `NEXT_PUBLIC_ENABLE_PLAYER_PROPS`. The projection view could re-use this pattern for a `NEXT_PUBLIC_ENABLE_PROJECTIONS_VIEW` flag during rollout

---

## Open Questions

1. Should 1P and player props be in the same "Projections" tab or separate tabs? (Props are game-keyed; 1P is also game-keyed but a different market — probably separate)
2. When no odds are available for props, what does the card show? Projected value + historical hit rate? Just the line?
3. Does the projection view need its own filter panel or can it inherit the game/sport/time filters from the play view?
4. How does this interact with the results page — do projection-only cards participate in settlement tracking?

---

## Rough Effort Estimate

- Option A (tabs): Small — 1-2 days frontend, no backend changes
- Option B (routes): Medium — 2-4 days frontend, minor API changes
- Option C (collapsible sections): Medium — 2-3 days frontend
- Option D (context panel): Medium-Large — depends on how much the card component needs to change
