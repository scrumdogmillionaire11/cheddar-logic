# WI-0908: MLB F5 Projection Settlement Contract and Status Normalization

**ID**: WI-0908
**Goal**: Make MLB F5 projection cards settle through a single deterministic path and store terminal outcomes in card_results using the canonical status/result contract.

**Depends on**: none

**Scope**:

- apps/worker/src/jobs/settle_mlb_f5.js
- apps/worker/src/jobs/settle_pending_cards.js
- apps/worker/src/schedulers/settlement.js
- `apps/worker/src/jobs/__tests__/settle_mlb_f5.test.js`
- `apps/worker/src/jobs/__tests__/settle_pending_cards.phase2.test.js`
- `apps/worker/src/__tests__/scheduler-main-calibration.test.js`

**Out of scope**:

- Repricing MLB models
- Adding new MLB markets
- Tracking dashboard/UI redesign

**Acceptance**:

- MLB F5 cards are never terminally voided by settle_pending_cards before F5 grading eligibility is evaluated.
- settle_mlb_f5 writes canonical terminal rows (status='settled', result in win/loss/push) instead of non-standard status values.
- Hourly and nightly scheduler ordering and gating produce one authoritative F5 settlement outcome per card.
- Terminal writes for F5 are idempotent at the write boundary (a second settlement attempt cannot overwrite an already-terminal row with a conflicting outcome).
- Non-dry-run integration test proves DB row mutation for a representative F5 card.
- End-to-end integration test proves the same F5 row flows through settle_pending_cards + settle_mlb_f5 with exactly one terminal settlement path.

**Execution tasks**:

1. **Task 1: Protect F5 rows from projection-only auto-close path**

   - **Files**: apps/worker/src/jobs/settle_pending_cards.js, apps/worker/src/jobs/__tests__/settle_pending_cards.phase2.test.js
   - **Action**: Enforce an explicit early F5 guard in pending-card settlement so rows matching ANY canonical F5 signature are skipped from terminal auto-close/error-void branches and left pending for settle_mlb_f5. Preserve existing non-F5 projection-only behavior.
     - Canonical F5 signatures to guard:
       - `card_type IN ('mlb-f5','mlb-f5-ml')`
       - `market_type IN ('FIRST_5_INNINGS','F5_TOTAL','F5_ML')`
       - `market IN ('f5_total','f5_ml')` (when present in payload/context)
     - Add table-driven test coverage for each signature branch so no F5 token variant can leak to pending-card terminal void/error.
   - **Verify**: `npm --prefix apps/worker run test -- --testPathPattern="settle_pending_cards.phase2"`
   - **Done**: F5 rows cannot transition to terminal void/error in settle_pending_cards before F5 eligibility check; all canonical F5 signature variants are explicitly covered by tests.

2. **Task 2: Normalize F5 settlement contract to canonical terminal tokens**

   - **Files**: apps/worker/src/jobs/settle_mlb_f5.js, apps/worker/src/jobs/__tests__/settle_mlb_f5.test.js
   - **Action**: Update F5 grading/write path so persisted terminal rows always use status='settled' and result in {'win','loss','push'}. Normalize any internal legacy tokens ('won'/'lost') to canonical result values before DB writes.
   - **Verify**: `npm --prefix apps/worker run test -- --testPathPattern="settle_mlb_f5"`
   - **Done**: Settled F5 rows in card_results contain only canonical status/result values; no non-standard terminal status tokens are written.

3. **Task 3: Enforce scheduler ordering and single authoritative F5 outcome**

   - **Files**: apps/worker/src/schedulers/settlement.js, apps/worker/src/__tests__/scheduler-main-calibration.test.js
   - **Action**: Ensure hourly/nightly settlement scheduling and MLB F5 enqueue ordering cannot produce conflicting terminal outcomes for the same card within one tick window. Keep deterministic ordering/gating assertions in scheduler calibration tests.
   - **Verify**: `npm --prefix apps/worker run test -- --testPathPattern="scheduler-main-calibration"`
   - **Done**: Scheduler tests assert one authoritative F5 settlement path per card per run window.

4. **Task 4: Add explicit non-dry-run DB mutation proof**

   - **Files**: apps/worker/src/jobs/__tests__/settle_mlb_f5.test.js
   - **Action**: Add/adjust integration test to run settleMlbF5 with dryRun=false against fixture DB and assert card_results row mutates from pending to status='settled' with result in {'win','loss','push'} and settled_at populated.
   - **Verify**: `npm --prefix apps/worker run test -- --testPathPattern="settle_mlb_f5" -t "non-dry-run"`
   - **Done**: Test failure occurs if write-path mutation does not happen or if canonical contract is violated.

5. **Task 5: Add cross-job end-to-end single-path integration proof**

   - **Files**: apps/worker/src/jobs/__tests__/settle_mlb_f5.test.js, apps/worker/src/jobs/__tests__/settle_pending_cards.phase2.test.js
   - **Action**: Add an end-to-end integration test that executes settle_pending_cards followed by settle_mlb_f5 against the same fixture F5 row and asserts:
     - settle_pending_cards does not terminally mutate the F5 row (remains pending)
     - settle_mlb_f5 performs the single terminal transition to status='settled'
     - exactly one terminal write occurs for the row (no conflicting terminal history)
   - **Verify**: `npm --prefix apps/worker run test -- --testPathPattern="settle_(pending_cards\.phase2|mlb_f5)" -t "e2e single path"`
   - **Done**: One test proves deterministic cross-job flow and fails if either job violates single-path settlement.

6. **Task 6: Enforce idempotent terminal write guard in settle_mlb_f5**

   - **Files**: apps/worker/src/jobs/settle_mlb_f5.js, apps/worker/src/jobs/__tests__/settle_mlb_f5.test.js
   - **Action**: Add write-boundary idempotency guard so F5 settlement updates only apply when current row state is pending (or other explicitly non-terminal state). If row is already terminal, skip update and log deterministic skip reason; do not overwrite with a second terminal outcome.
   - **Verify**: `npm --prefix apps/worker run test -- --testPathPattern="settle_mlb_f5" -t "idempotent terminal write guard"`
   - **Done**: Re-running settle_mlb_f5 on an already-terminal F5 row cannot change settled result/outcome and cannot produce conflicting terminal write.
