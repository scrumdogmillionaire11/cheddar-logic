# WI-0456: Migrate DB layer from sql.js to better-sqlite3

**ID**: WI-0456
**Goal**: Replace the sql.js in-memory SQLite engine with `better-sqlite3` (native binding) to eliminate the write-tear corruption race and the full-file-serialize-on-every-write performance cliff.

---

## Why

`sql.js` is a WebAssembly port of SQLite designed for browsers. It has no file I/O — every read loads the entire DB file into memory, every write serializes the entire DB back to disk with `fs.writeFileSync`. On a Pi with a ~66 MB DB and two processes running concurrently (worker + web server), this produces:

1. **Write-tear corruption** — `writeFileSync` is not atomic. A concurrent `readFileSync` in the web server can read a file that is half-written, producing `database disk image is malformed`. This is the root cause of the production outage on 2026-03-15.
2. **Full-file write amplification** — every single row insert rewrites 66 MB to disk. This gets worse as the DB grows.
3. **No WAL** — sql.js ignores `PRAGMA journal_mode = WAL`. The write-tear workaround (`.tmp` + `renameSync`) is a band-aid that only prevents torn reads; it does nothing for write contention.

`better-sqlite3` uses the native SQLite C library with real file I/O, WAL journal mode, and OS-level read/write locking. Concurrent reads are safe by default. Writes touch only the changed pages, not the whole file.

## Scope

- `packages/data/package.json` — swap `sql.js` for `better-sqlite3`
- `packages/data/src/db.js` — replace sql.js init, `DatabaseWrapper`, `saveDatabase()`, `getDatabase()`, `getDatabaseReadOnly()` with better-sqlite3 equivalents
- `packages/data/src/sqlite-wrapper.js` — same replacement (simpler wrapper used by some jobs)
- `packages/data/src/db-dual-init.js` — replace `saveDbFile()` atomic write shim with better-sqlite3 open
- `packages/data/src/db-multi.js` — audit for any sql.js usage; update if present
- `scripts/run-migrations.js` — verify migration runner works with better-sqlite3 exec API (interface is compatible)

## Out of scope

- Schema changes — no migrations needed; better-sqlite3 opens the same `.db` file format
- API route or worker logic changes — callers of `getDatabase()` / `getDatabaseReadOnly()` must not need to change
- Moving off SQLite entirely (Postgres, Turso, etc.)
- WAL checkpoint tuning beyond enabling WAL mode at startup

---

## API compatibility notes

better-sqlite3 is synchronous. The query API changes but `db.js` abstracts all of this behind `DatabaseWrapper` — callers use `exec()`, `prepare()`, etc. already. The main internal changes:

| sql.js | better-sqlite3 |
|---|---|
| `new SQL.Database(buffer)` | `new Database(filePath)` |
| `db.run(sql, params)` | `db.prepare(sql).run(params)` |
| `db.exec(sql)` | `db.exec(sql)` (same) |
| `db.prepare(sql); stmt.step()` | `db.prepare(sql).all()` / `.get()` / `.run()` |
| `stmt.getAsObject()` | stmt returns plain objects natively |
| `db.export()` → `writeFileSync` | not needed — writes go to file directly |
| `getDatabaseReadOnly()` loads full file | `new Database(path, { readonly: true })` |
| `PRAGMA journal_mode = WAL` ignored | `db.pragma('journal_mode = WAL')` works |

## Acceptance

1. `packages/data` has no `sql.js` dependency remaining — `better-sqlite3` is the only SQLite engine.
2. `saveDatabase()` is deleted (or a no-op stub) — no callers need updating because `db.js` exports the same public surface.
3. WAL mode is enabled at `init()` time: `db.pragma('journal_mode = WAL')`.
4. Concurrent reads during a write do not produce `database disk image is malformed` — verified by running the web server and worker simultaneously in dev and hitting `/api/cards` during a model run.
5. All existing migrations apply cleanly against the live production DB file with no schema changes.
6. All tests that were passing before this WI still pass after.
7. The production deploy pipeline (`scripts/run-migrations.js`) completes successfully on a clean pull to the Pi.
