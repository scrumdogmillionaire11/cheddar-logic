| Field | Value |
|---|---|

## Background

The root `layout.tsx` exports a single catch-all metadata object:

```ts
export const metadata: Metadata = {
  title: 'Cheddar Logic | Signal-Qualified Analytics',
  description: 'Outputs based on confidence thresholds and uncertainty controls.',
  metadataBase: new URL('https://cheddarlogic.com'),
  openGraph: { title: 'Cheddar Logic', ... },
};
```

Every page currently inherits this verbatim. When a user shares `/play-of-the-day`, `/cards`, or an education article, the og:title is just "Cheddar Logic" — no page context. This WI adds page-level overrides.

**Note on `results/page.tsx`:** It is marked `'use client'` and can't export `metadata` directly. The correct approach is to add `web/src/app/results/layout.tsx` (a Server Component wrapper) that exports `metadata` for the `/results` route while leaving the existing client page untouched.

## Implementation Notes

### Metadata shape for each static page (example)

```ts
import type { Metadata } from 'next';

export const metadata: Metadata = {
  title: 'Play of the Day | Cheddar Logic',
  description: 'Daily signal-qualified pick with confidence context and stake recommendation.',
  openGraph: {
    title: 'Play of the Day | Cheddar Logic',
    description: 'Daily signal-qualified pick with confidence context and stake recommendation.',
    url: 'https://cheddarlogic.com/play-of-the-day',
  },
};
```

### `generateMetadata` for education/[slug]

```ts
import type { Metadata } from 'next';
import { getEducationArticleBySlug } from '@/lib/education/content';

export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>;
}): Promise<Metadata> {
  const { slug } = await params;
  const article = getEducationArticleBySlug(slug);
  if (!article) return {};
  return {
    title: `${article.title} | Cheddar Logic`,
    description: article.summary ?? `${article.title} — Cheddar Logic educational content.`,
    openGraph: {
      title: article.title,
      description: article.summary ?? `${article.title} — Cheddar Logic educational content.`,
      url: `https://cheddarlogic.com/education/${slug}`,
    },
  };
}
```

> If `article.summary` does not exist on the type, use `article.title` only. Do not modify the education content data model.

### `results/layout.tsx` (new file)

## Acceptance Criteria

1. Every route listed in Scope has a non-default `<title>` tag rendered in the HTML `<head>` — confirmed by `curl https://cheddarlogic.com/<route> | grep '<title>'` or local `next dev` inspection.
2. Each page's `og:title` and `og:description` differ from the root layout defaults and match the per-page values in the table above.
3. `/education/getting-started-fpl-analytics` renders `<title>Getting Started with FPL Analytics | Cheddar Logic</title>`.
4. `/results` renders its page-specific title despite the page file being `'use client'` (served via `results/layout.tsx`).
5. `npm --prefix web run build` completes with no type errors.
6. No existing page UI or behavior is changed (metadata addition only).
