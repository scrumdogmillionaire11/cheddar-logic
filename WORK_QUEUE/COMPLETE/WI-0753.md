# WI-0753: NHL 1P Totals — audit CARD_FAMILY_MAP placement and clarify odds-backed vs projection-only

**ID**: WI-0753
**Goal**: Clarify whether `nhl-pace-1p` / `nhl-1p-call` should remain in `CARD_FAMILY_MAP` (odds-backed settlement grouping) or be removed from it, given the stated rule that only full-game totals, ML, and spreads fetch odds.

**Background**:
Per requirements, "full game totals, moneyline and spreads are the only markets we fetch odds for — other markets are PROJECTION_ONLY." NHL 1P totals are a sub-period market. However, `nhl-pace-1p` and `nhl-1p-call` both appear in `CARD_FAMILY_MAP` (projection-metrics.ts:313–314) — the map used for odds-backed settlement grouping. They also correctly appear in `PROJECTION_FAMILY_LABELS` (line 34).

Two possibilities:
1. 1P totals are always PROJECTION_ONLY → they should be removed from `CARD_FAMILY_MAP` (their presence there is harmless because `deriveResultCardMode` correctly reads `basis/line_source` and will mark them PROJECTION_ONLY before they reach settlement, but it's confusing dead declaration)
2. 1P totals DO have live odds in some cases → they legitimately belong in both maps and can settle

This WI is an audit/decision task, not necessarily a code change.

**Scope**:
- `apps/worker/src/jobs/` — check whether any NHL worker fetches 1P odds from the odds provider
- `web/src/app/api/results/projection-metrics.ts:308–354` — CARD_FAMILY_MAP
- `web/src/lib/types/game-card.ts` — CanonicalMarketType definitions

**Out of scope**:
- Changing the projection tracking path for 1P (it's already correct)
- Any UI changes

**Acceptance**:
- Document whether 1P odds are ever fetched (yes/no) with evidence
- If always PROJECTION_ONLY: remove `nhl-pace-1p` and `nhl-1p-call` from `CARD_FAMILY_MAP` and add a comment in `PROJECTION_FAMILY_LABELS` explaining the dual-map design
- If sometimes odds-backed: add a note in code clarifying when and ensure the model sets `basis=ODDS_BACKED` in those cases
