# WI-0666: Worker job — pull_public_splits + scheduler registration + DB write layer

**ID**: WI-0666
**Goal**: Add a `pull_public_splits` worker job that fetches splits from the ActionNetwork adapter, matches them to our game IDs, writes the splits columns to `odds_snapshots` via a new DB function, and registers on a 60-minute cadence in the scheduler.

**Background**:
The DB schema (WI-0664) and the ActionNetwork adapter (WI-0665) provide the raw ingredients. This job wires them under the single-writer pattern: only the worker writes to the DB (ADR-0002). Splits are attached to the most recent `odds_snapshots` row per game via an UPDATE rather than a new INSERT — they are a property of the current snapshot moment. Unmatched games that have a high-confidence book consensus (`spread_consensus_confidence = 'HIGH'`) are flagged with `splits_source = 'pinnacle_proxy'` so the pipeline gate (WI-0667) can use Pinnacle-divergence as a fallback signal.

**Depends on**: WI-0664, WI-0665

**Scope**:
- `apps/worker/src/jobs/pull_public_splits.js` (new file)
- `apps/worker/src/schedulers/main.js` (register job — `needs-sync`)
- `packages/data/src/db.js` (add `updateOddsSnapshotSplits` and `getActiveGamesForSplits` — `needs-sync`)

**Out of scope**:
- Model or pipeline changes (WI-0667)
- Web UI display of split percentages
- Writing splits directly to `card_payloads`
- Any web route changes (web server is read-only per ADR-0002)

**Acceptance**:
- `getActiveGamesForSplits(sports[])` in `packages/data/src/db.js` returns rows `{ game_id, home_team, away_team, sport, game_time_utc }` for games starting within the next 48h
- `updateOddsSnapshotSplits({ gameId, splitsData, jobRunId })` in `packages/data/src/db.js` issues an `UPDATE odds_snapshots SET ... WHERE id = (SELECT id FROM odds_snapshots WHERE game_id = ? ORDER BY captured_at DESC LIMIT 1)`, writing all 8 splits columns
- `pull_public_splits.js` exports `runPullPublicSplits({ db, logger })` which:
  1. Calls `getActiveGamesForSplits(['MLB','NBA','NHL','NFL'])`
  2. Groups games by sport
  3. Calls `fetchSplitsForDate` for each sport for today + tomorrow dates
  4. Calls `matchSplitsToGameId` to align to our `game_id`s
  5. Writes `splits_source = 'action_network'` for matched games
  6. Writes `splits_source = 'pinnacle_proxy'` (pct fields null) for unmatched games where `spread_consensus_confidence = 'HIGH'`
  7. Logs match rate per sport; ≥ 70% is INFO, < 70% is WARN
- Job soft-fails per-sport on ActionNetwork non-200 (logs error, skips sport, continues others)
- Job registered in `main.js` on a **60-minute cadence**, active-hours-only guard (same pattern as existing odds jobs)
- Single-writer contract maintained: no `db.exec()`, `stmt.run()`, `runMigrations()`, or `closeDatabase()` called from any web route as a side effect of this WI
