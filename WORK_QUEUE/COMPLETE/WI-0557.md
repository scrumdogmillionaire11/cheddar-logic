# WI-0557: Wire CLV (closing line value) feedback loop

**ID**: WI-0557
**Goal**: Complete the existing odds-based `ENABLE_CLV_LEDGER` feedback loop so the system can measure whether the model consistently beats the market close using additive CLV telemetry.

**Background**:
`ENABLE_CLV_LEDGER` already gates additive CLV telemetry in the repo, and `clv_ledger` plus calibration reporting already depend on the odds-based fields `odds_at_pick`, `closing_odds`, and `clv_pct`. The remaining gap is operational: settlement currently records the opening entry but does not compute and persist real closing odds, open CLV rows are closed too early when no close is available, and `/api/results` does not expose CLV telemetry. This WI preserves the existing odds-based contract and completes the feedback loop without changing settlement outcomes or the single-writer DB architecture.

**Acceptance**:
- `ENABLE_CLV_LEDGER` remains environment-controlled and documented in `env.example`; no hardcoded default is flipped on in `packages/models/src/flags.js`.
- `clv_ledger` remains the additive source of truth with the existing odds-based fields `odds_at_pick`, `closing_odds`, and `clv_pct`.
- The post-game settlement path resolves a closing odds value from `odds_snapshots` using the same market-selection precedence as the locked market contract.
- `clv_pct` is computed as same-side implied-probability improvement: `implied_prob(closing_odds) - implied_prob(odds_at_pick)`.
- `clv_pct > 0` means the model beat the closing odds (positive CLV = edge confirmed).
- CLV rows remain open (`closed_at IS NULL`) when no usable close is available so later settlement sweeps can reconcile them.
- `/api/results` returns an optional `clv` object per ledger play when CLV telemetry exists, without changing existing result or PnL fields.
- A summary CLV metric remains queryable from the DB via the existing `clv_ledger` / calibration reporting path.
