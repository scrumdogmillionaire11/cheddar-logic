# WI-0711: Port 3000 pre-start cleanup for cheddar-web

**ID:** WI-0711
**Goal:** Prevent the recurring `EADDRINUSE :3000` crash loop on cheddar-web by
adding a startup wrapper that kills any process squatting on port 3000 before
`next start` runs, making PM2 restarts self-healing.

**Problem:**
When cheddar-web is restarted via `pm2 restart`, orphaned `next-server` processes
that were started outside of PM2 (manual `next start`, pre-PM2 sessions, or prior
crash residues) continue holding port 3000. PM2's new cheddar-web instance
immediately fails with `EADDRINUSE :3000` and enters a crash loop.

This has happened twice in production:
- pid 100093 — held port 3000 for >3 run cycles before manual `kill -9`
- pid 102144 — held port 3000 after a later restart, same symptom

The removed `pre_start_script` option in `ecosystem.config.js` was supposed to
address a related issue (DB lock cleanup) but is not a valid PM2 option and was
silently ignored. PM2 provides no built-in `pre_start` hook for arbitrary shell
commands.

**Proposed approach:**
Replace the bare `next start` invocation in `ecosystem.config.js` with a thin
shell wrapper script at `scripts/start-web.sh`:

```bash
#!/bin/bash
set -e
# Free port 3000 if held by an orphan (best-effort, not fatal if nothing to kill)
fuser -k 3000/tcp 2>/dev/null || true
exec node_modules/.bin/next start
```

Update `ecosystem.config.js` cheddar-web entry:
```js
{
  name: 'cheddar-web',
  cwd: '/opt/cheddar-logic/web',
  script: '../scripts/start-web.sh',         // replaces node_modules/.bin/next
  // args: 'start'                           // REMOVE — now baked into script
  interpreter: 'bash',
  env_file: '/opt/cheddar-logic/.env.production',
  instances: 1,
  autorestart: true,
  watch: false,
}
```

**Scope:**
- `WORK_QUEUE/WI-0711.md`
- `scripts/start-web.sh` (new file, must be `chmod +x`)
- `ecosystem.config.js` (cheddar-web `script` + `interpreter` + remove `args`)

**Out of scope:**
- Any Next.js build or config changes
- cheddar-worker process definition
- Any systemd / Caddy changes

**Acceptance:**
1. `scripts/start-web.sh` exists, is executable (`chmod +x`), and contains the
   `fuser -k 3000/tcp || true; exec next start` pattern
2. `ecosystem.config.js` cheddar-web entry points to the wrapper and sets
   `interpreter: 'bash'`
3. `pm2 restart cheddar-web --update-env` succeeds without EADDRINUSE even when
   an orphan process is parked on port 3000 beforehand (manual test: start a
   `sleep 9999` or `nc -l 3000` to occupy the port, then run `pm2 restart`)
4. cheddar-web reaches `online` status and `curl -s -o /dev/null -w '%{http_code}' http://localhost:3000/` returns 200 within 30 s

**Owner agent:** unassigned
**Time window:** single session
**Coordination flag:** none — isolated to scripts/ and ecosystem.config.js
**Tests to run:** manual — see acceptance criterion 3 above
**Manual validation:**
  ```bash
  # On Pi — simulate squatter, confirm self-healing restart
  ssh -i ~/.ssh/cheddar-deploy babycheeses11@cheddarpi.local \
    "nc -l 3000 & sleep 1 && pm2 restart cheddar-web --update-env && \
     sleep 10 && pm2 status --no-color | grep cheddar-web"
  # Expected: cheddar-web shows 'online' despite nc squatter
  ```

**Depends on:** nothing
**Blocks:** production reliability — this same crash loop has required manual
  intervention twice in one session
