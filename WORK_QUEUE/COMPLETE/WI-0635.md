# WI-0635: Stabilize deploy verification for public `/fpl` HTML convergence

**ID**: WI-0635
**Goal**: Prevent false production deploy rollbacks when the public site briefly serves stale or split-traffic HTML after Cloudflare purge, especially on `/fpl`, by making deploy verification retry public route HTML until it matches origin asset references before probing asset URLs.

**Background**:
The deploy workflow currently fetches public HTML for `/`, `/cards`, `/fpl`, and `/results` once after Cloudflare purge, extracts `_next/static` asset refs from that single response, and then verifies those refs. If Cloudflare returns one stale or mixed response during propagation, the workflow can keep chasing bad refs even after the route has converged to the correct release, causing a rollback despite the origin being healthy.

**Scope**:
- `.github/workflows/deploy-production.yml`
- `WORK_QUEUE/WI-0635.md`

**Out of scope**:
- Cloudflare dashboard or tunnel configuration changes
- Application runtime code in `web/**`, `apps/**`, or `packages/**`
- Deploy workflow changes unrelated to public HTML/static asset verification

**Acceptance**:
- Public route verification retries each HTML route until extracted JS/CSS refs match the origin route for the same path, or fails with a clear mismatch error after retries.
- Asset URL verification uses route refs from a converged public HTML response rather than a single potentially stale response.
- Deploy workflow still fails closed if public HTML never converges to the origin release.
