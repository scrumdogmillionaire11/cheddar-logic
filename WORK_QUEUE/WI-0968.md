# WI-0968: Fix PASS projection filtering—prevent PASS projections from surfacing to UI

**ID:** WI-0968

**Goal:** Fix bug where projections marked PASS ("No edge at current price") are still being surfaced to the UI with edge data visible, breaking the play/pass contract.

**Status:** CREATED

**Owner agent:** (awaiting claim)

**Time window:** 1-2 hours

**Coordination flag:** None (isolated to filtering/surfacing logic)

---

## Problem Statement

A projection for **Colorado Rockies @ Houston Astros (Apr 16)** shows:

- **Status:** PASS
- **Label:** "No edge at current price"
- **But underlying model data visible:** finalEdge +0.90, drivers=[SP_MISMATCH|OFFENSE_MATCHUP_EDGE]

This creates confusion: PASS projections should not be surfaced or their edge data should not be visible. Either:

1. The projection should be a PLAY (edge is meaningful), or
2. The surfacing logic should fully exclude PASS projections (including edge breakdowns)

**Incident Context:**

- WI-0967 excluded PASS rows from over/under record tallying but did not address rendering/surfacing
- This suggests the filter boundary is incomplete (only affects one API endpoint, not the card/display layer)

- `apps/worker/src/models/**` — gating logic determining PASS vs PLAY assignment
- Tests: `web/src/app/api/__tests__/**`, `apps/worker/src/jobs/__tests__/**`

**Acceptance Criteria:**

3. Projection-metrics API filters correctly by actionable_call status
4. Test confirms PASS rows are excluded from card iteration on play date April 16
