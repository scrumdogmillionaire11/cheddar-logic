## Functional Requirements

### 1. Profile Onboarding (Controlled Input Only)

Capture exactly five inputs that map to decision behavior:

* Risk posture (Conservative / Balanced / Aggressive)
* Bench preference (light vs strong bench)
* Template vs differential bias
* Time horizon (short vs medium)
* Entry mode (manual draft vs screenshot import)

**Rules:**

* No open-ended questions
* No personality-style inputs
* Output must map directly to backend-supported parameters

---

### 2. Draft Lab (Constraint-Based Editing Only)

User can modify a draft using **only supported actions**:

* Lock player
* Ban player
* Adjust bench priority
* Adjust risk/profile setting
* Submit constrained intent (via API)

**Supported intent examples:**

* “lock Saka”
* “ban Jackson”
* “reduce risk”
* “favor upside”
* “avoid double Arsenal defense”

**Rules:**

* All actions must resolve to a known intent contract
* Unsupported inputs must be rejected (not interpreted loosely)
* No frontend-generated squad logic

---

### 3. Screenshot Upload + Parse Review (Hard Gate)

Flow:

1. Upload screenshot(s)
2. Display parsed squad with confidence indicators
3. Highlight ambiguous/low-confidence slots
4. Require user correction for all flagged slots
5. Only then allow audit

**Rules:**

* No bypass for low-confidence parsing
* No “accept all” shortcut
* Audit is blocked until all ambiguity is resolved

---

### 4. Draft Audit (Explainable, Structured)

Audit must render a verdict using fixed explanation categories:

* Structure quality
* Fixture quality
* Minutes/security
* Volatility
* Flexibility
* Profile fit

**Rules:**

* Must use backend-provided outputs
* No invented frontend scoring
* No raw dumps of data or logs
* Must be readable in < 10 seconds

---

### 5. Draft Comparison (Two Candidates Only)

Compare two draft candidates across fixed axes:

* Upside
* Safety
* Flexibility
* Profile fit

**Output:**

* Winner (from backend or defined contract)
* Per-axis deltas
* Clear visual comparison

**Rules:**

* No frontend “guessing” winner logic unless contract defines it
* No more than two drafts compared at once

---

## Non-Negotiable Constraints

* No frontend math for projections, transfers, or scoring
* No unsupported natural-language execution
* No audit completion with unresolved parsing ambiguity
* No independent decision logic outside backend contracts
* All actions must map to explicit, typed intents
