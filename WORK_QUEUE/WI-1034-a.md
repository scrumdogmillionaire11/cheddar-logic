# WI-1034-a

## Goal
Formalize the verification contract in the vegas-agent doc layer. Define `GATE_CHECK` and `STANDARD_AUDIT` as named audit levels with explicit boundaries - `STANDARD_AUDIT` is the existing 8-step `## Audit Sequence` in `bet_review.md`, renamed and anchored. Define every blocker code, action type, severity, promotion rule, and expiry rule. Keep verdict vocabulary as `PLAY/LEAN/PASS/FADE`, but require `verification_state` as companion context so pending verification (`LEAN + verification_state=PENDING`) is never conflated with a true weak-edge lean.

## Depends on
None. Doc-layer only — no worker or pipeline code changes.

## Acceptance
- `vegas-agent/core/verification_contract.md` exists and defines: all `VerificationBlockerCode` values (including which map to existing codes vs new), all `VerificationActionType` values, hard/soft severity taxonomy, the three GATE_CHECK checks by name (PRICE_STALE, STARTER_UNCONFIRMED/STARTER_MISMATCH, LINE_MOVE_ADVERSE), threshold policy (defaults 5 cents but configurable by sport/market), promotion rules (`CLEARED` never implies `PLAY`; `CLEARED -> PLAY` only after re-running threshold logic), expiry rules, resolver terminal state output (CLEARED/FAILED/EXPIRED each have defined emit behavior), and user-facing/internal output format rules including required `verification_state` companion semantics for `LEAN`.
- `vegas-agent/workflows/pre_flight.md` exists, is named `GATE_CHECK` formally, enumerates exactly the three checks from WI-1034.md in order with their fail codes, emits `PASS — [REASON_CODE]: [sentence].` on failure (no other output), emits `GATE_CHECK: CLEAR` on pass, and explicitly states: first-fail output is for user-facing brevity while internal state may collect all cheap-to-detect blockers in the same pass.
