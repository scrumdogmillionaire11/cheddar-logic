# WI-1034-a

## Goal
Formalize the verification contract in the vegas-agent doc layer with explicit Slight Edge semantics. Define `GATE_CHECK` and `STANDARD_AUDIT` as named audit levels with strict boundaries, where `STANDARD_AUDIT` is the existing 8-step `## Audit Sequence` in `bet_review.md`, renamed and anchored. Preserve verdict vocabulary (`PLAY/LEAN/PASS/FADE`) while requiring `verification_state` as companion truth so pending verification (`LEAN + verification_state=PENDING`) is never conflated with a true Slight Edge lean (`LEAN + verification_state=CLEARED|NOT_REQUIRED`).

## Depends on
None. Doc-layer only - no worker or pipeline code changes.

## Acceptance
- `vegas-agent/core/verification_contract.md` exists and defines: all `VerificationBlockerCode` values (including which map to existing codes vs new), all `VerificationActionType` values, hard/soft severity taxonomy, the three GATE_CHECK checks by name (PRICE_STALE, STARTER_UNCONFIRMED/STARTER_MISMATCH, LINE_MOVE_ADVERSE), threshold policy (defaults 5 cents but configurable by sport/market), promotion rules (`CLEARED` never implies `PLAY`; `CLEARED -> PLAY` only after re-running threshold logic), expiry rules, resolver terminal state output (CLEARED/FAILED/EXPIRED each have defined emit behavior), and output semantics that separate pending verification from true Slight Edge lean via `verification_state`.
- `vegas-agent/workflows/pre_flight.md` exists, is named `GATE_CHECK` formally, enumerates exactly the three checks from WI-1034.md in order with fail codes, emits `PASS - [REASON_CODE]: [sentence].` on failure (no other output), emits `GATE_CHECK: CLEAR` on pass, and explicitly states: first-fail output is for user-facing brevity while internal state may collect all cheap-to-detect blockers in the same pass.
- `vegas-agent/workflows/verification_resolver.md` exists, defines the resolver loop (consume pending requirements, dispatch action_type, update status), and for each terminal state (CLEARED, FAILED, EXPIRED) specifies exactly what is emitted and which layer receives it. `CLEARED` is explicitly documented as eligibility for re-evaluation, not an automatic `PLAY` verdict.
- `vegas-agent/workflows/bet_review.md` contains `## Audit Levels` section with `GATE_CHECK` and `STANDARD_AUDIT` as named subsections; `STANDARD_AUDIT` subsection explicitly states it is implemented by `## Audit Sequence` below; `STANDARD_AUDIT` must not recreate gate-failure logic unless resolver delivered new data; undefined audit-level synonyms (`full audit`, `quick audit`, `mini audit`, `deep dive`) are disallowed unless explicitly mapped to a defined level.
