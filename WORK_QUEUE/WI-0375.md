**Acceptance**:

Rendering correctness:

- `expected_pts` label renders when value is `0` or any non-null numeric; falsy guard (`if (captain.expected_pts)`) is replaced with `captain.expected_pts !== undefined && captain.expected_pts !== null` in `CaptaincySection`
- Results and Dashboard consume the same `projected_xi` / `projected_bench` fields from the payload; no component recomputes lineup state from raw player flags
- `SquadSection` derives `hasTransfers` from `projected_xi` presence and non-empty primary transfer plan, not from action-string parsing

Contradiction removal:

- `getRiskStatement` in `Results.tsx` and any equivalent in `Dashboard.tsx` derive from `squad_health` only; no independent inference from raw injury/doubt counts outside that object
- `DecisionBrief` justification text and `RiskNote` text cannot simultaneously assert opposite health states (e.g. "squad is healthy" AND "3 players injured")

Planner score display:

- `confidence_label` and `confidence_summary` from payload surface in `DecisionBrief`; raw numeric score is not displayed without those metadata fields present
- Timeline detail from `fixture_planner.gw_timeline` renders in `DataTransparency` if present in payload

Strategy path display:

- Duplicate `(out, in)` path entries do not render; display is deduplicated by `(out, in)` pair in the UI even if backend sends duplicates (defensive render guard)
- No strategy path option displays a player already shown in the active squad list as the `in` target

Results / Dashboard parity:

- Both pages render captain, transfer, and projected XI from the same payload field names; no page uses a private fallback field the other has dropped
