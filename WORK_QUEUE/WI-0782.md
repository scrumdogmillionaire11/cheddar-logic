# WI-0782: Cache getUpcomingGames() in scheduler tick loop to reduce DB read pressure

| Field              | Value |
|--------------------|-------|
| **ID**             | WI-0782 |
| **Goal**           | Add a short-lived in-process cache (TTL ~5 minutes) for the `getUpcomingGames()` call in the scheduler tick loop so that the DB is not queried on every 30-second tick; invalidate the cache immediately after `syncGameStatuses` or `pullEspnGamesDirect` completes |
| **Priority**       | Low-Medium |
| **Owner agent**    | Unassigned |
| **Time window**    | TBD |
| **Coordination flag** | `needs-sync` — modifies `apps/worker/src/schedulers/main.js` |

---

## Problem

`apps/worker/src/schedulers/main.js` calls `getUpcomingGames()` on every scheduler tick (default tick interval is 30 seconds). This is a synchronous SQL query against the production DB that runs 2× per minute, 120× per hour, ~2,880× per day even when no games are scheduled.

On a Raspberry Pi with a single-writer sql.js DB, every unnecessary read adds lock contention and I/O pressure. More critically, because the DB is sql.js (in-memory with periodic saves), each read re-hydrates and re-queries the same largely-unchanged games table.

---

## Scope

| File | Change |
|------|--------|
| `apps/worker/src/schedulers/main.js` | Add an in-process cache object: `{ data: null, fetchedAt: null }`; wrap `getUpcomingGames()` call to return cached result if `Date.now() - fetchedAt < 5 * 60 * 1000`; set `fetchedAt = null` immediately after any job that mutates game statuses completes (i.e. after `syncGameStatuses`, `pullEspnGamesDirect` job run callbacks) |

## Out of Scope

- Changes to `getUpcomingGames()` implementation in `@cheddar-logic/data`
- Changing the tick interval itself
- Any other scheduler logic

---

## Acceptance

- A `console.debug` log (or equivalent) confirms cache hits vs misses over a 10-minute `DRY_RUN` run
- After `syncGameStatuses` fires, the next tick re-queries the DB (cache busted)
- Without a game-status-changing job, 10 ticks at 30s each result in 1 or 2 DB reads (not 10)
- `test:scheduler:windows` passes
