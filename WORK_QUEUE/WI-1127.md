# WI-1127: API Error Redaction And Response Finalizer Consistency

**ID**: WI-1127
**Goal**: Remove internal error leakage from client responses and enforce consistent rate-limit and security headers across all scoped route exit paths.

**Scope**:
- web/src/app/api/cards/route.ts
- web/src/app/api/cards/[gameId]/route.ts
- web/src/app/api/results/route.ts
- web/src/lib/api-security/index.ts
- web/src/__tests__/api-error-redaction.test.js
- web/src/__tests__/api-rate-limit-header-consistency.test.js

**Out of scope**:
- Full structured logging platform migration
- API schema redesign

**Acceptance**:
- Client-facing 4xx/5xx errors in scoped routes return an opaque user-safe message plus a `correlationId` field, and never include stack traces, SQL fragments, or raw exception class names in response JSON.
- `correlationId` is generated per failed request, returned in response JSON, and asserted by automated tests for presence/non-empty string shape.
- All success and error responses in scoped routes (cards, cards/[gameId], results) include expected rate-limit headers and this required API security-header set:
  - `Content-Security-Policy`
  - `Strict-Transport-Security`
  - `X-Frame-Options`
  - `X-Content-Type-Options`
  - `Referrer-Policy`
  - `X-XSS-Protection`
  - `Permissions-Policy`
  - `X-Permitted-Cross-Domain-Policies`
- Regression tests include explicit route-family matrix coverage:
  - `/api/cards`: at least one success path + one forced error path
  - `/api/cards/[gameId]`: at least one success path + one forced error path
  - `/api/results`: at least one success path + one forced error path
