# WI-1136: Add Production-Grade Liveness And Readiness Endpoints

**ID**: WI-1136
**Goal**: Provide explicit production health endpoints for incident diagnosis and platform probes.

**Scope**:
- web/src/app/api/healthz/route.ts
- web/src/app/api/readyz/route.ts
- web/src/lib/db-init.ts
- web/src/lib/api-security/config.ts
- docs/DEPLOYMENT.md
- web/package.json
- web/src/__tests__/api-healthz.test.js
- web/src/__tests__/api-readyz.test.js

**Out of scope**:
- Full observability platform migration
- Admin diagnostics UI changes
- README.md changes

**Acceptance**:
- healthz returns 200 `{ status: "ok" }` always (pure liveness, no DB or heavy dependencies).
- readyz returns 200 `{ status: "ok" }` when DB probe passes; returns 503 `{ status: "unavailable", reason: "<string>" }` when DB is unreachable or malformed.
- `/api/healthz` and `/api/readyz` are registered in `PUBLIC_ROUTES` in `config.ts` so they are never blocked by the fail-closed auth policy (WI-1124).
- `db-init.ts` exports a synchronous `checkDbReady(): { ok: boolean; reason?: string }` that opens a real read-only connection, runs `SELECT 1`, and closes it - capable of returning `ok: false` if the DB file is missing, locked, or malformed.
- Deployment docs include probe examples and expected status semantics.
