# WI-0382: Consistency Check Goalie Uncertainty Escalation

**ID**: WI-0382
**Goal**: When `CanonicalGoalieState.starter_state` is `UNKNOWN` or `CONFLICTING` for either team, force the consistency layer to output `vol_env = 'VOLATILE'` and `total_bias = 'INSUFFICIENT_DATA'` before driver signal eligibility is evaluated. This prevents a game with shaky goalie identity from receiving a clean `STABLE` environment and a directional `FAVOR_OVER` / `FAVOR_UNDER` classification.

## Design

### Existing enum contracts (do not invent new values)

Per actual code in `decision-publisher.js` and `cross-market.js`:

- `vol_env` values: `'VOLATILE'` | `'STABLE'` — both already exist; use `'VOLATILE'`
- `total_bias` values: `'OK'` | `'INSUFFICIENT_DATA'` — both already exist; use `'INSUFFICIENT_DATA'`
- `event_direction_tag` values: `'FAVOR_OVER'` | `'FAVOR_UNDER'` — do NOT add `'INSUFFICIENT_DATA'` here

### Escalation trigger

```js
function goalieUncertaintyBlocks(homeGoalieState, awayGoalieState) {
  const blocking = ['UNKNOWN', 'CONFLICTING'];
  return (
    blocking.includes(homeGoalieState?.starter_state) ||
    blocking.includes(awayGoalieState?.starter_state)
  );
}
```

### Forced outputs when escalation fires

In `cross-market.js` — before or in place of `computeTotalBias()`:
```js
if (goalieUncertaintyBlocks(homeGoalieState, awayGoalieState)) {
  return 'INSUFFICIENT_DATA';
}
```

In `decision-publisher.js` — in `deriveVolEnv()`:
```js
if (goalieUncertaintyBlocks(homeGoalieState, awayGoalieState)) {
  return 'VOLATILE';
}
```

Note: `EXPECTED` state does NOT trigger escalation. It allows normal computation with the understanding that the pace model has already applied a degraded-trust adjustment.

## Tests Required

```
goalieUncertaintyBlocks — UNKNOWN either side → true
goalieUncertaintyBlocks — CONFLICTING either side → true
goalieUncertaintyBlocks — EXPECTED either side → false (no escalation)
goalieUncertaintyBlocks — CONFIRMED both sides → false
goalieUncertaintyBlocks — null state → false (null-safe)

computeTotalBias — UNKNOWN goalie → INSUFFICIENT_DATA (overrides driver signal)
computeTotalBias — CONFLICTING goalie → INSUFFICIENT_DATA
computeTotalBias — EXPECTED goalie → normal computation runs
computeTotalBias — CONFIRMED both → normal computation runs (existing tests must pass)

deriveVolEnv — UNKNOWN goalie → VOLATILE
deriveVolEnv — CONFLICTING goalie → VOLATILE
deriveVolEnv — CONFIRMED both + low conflict score → STABLE (existing behavior preserved)
```

## Failure Modes & Guards

### Failure mode: event_direction_tag receives INSUFFICIENT_DATA
**Guard:** tests must assert `event_direction_tag` is never set to `INSUFFICIENT_DATA`; it only takes `FAVOR_OVER` / `FAVOR_UNDER`.

### Failure mode: EXPECTED starter triggers escalation
**Guard:** test explicitly asserts that EXPECTED state does NOT force VOLATILE or INSUFFICIENT_DATA.

### Failure mode: escalation fires after driver scores are already computed
**Guard:** `goalieUncertaintyBlocks()` check must happen at the start of `computeTotalBias()` / `deriveVolEnv()`, before any driver signal is evaluated.
