# WI-0802: Audit wind_dir usage in MLB model — used or decorative metadata?

**ID**: WI-0802
**Goal**: Determine whether `wind_dir` (fetched by `pull_mlb_weather.js` and stored in `mlb_game_weather`) influences any coefficient or gate in `mlb-model.js` or `run_mlb_model.js`; either wire it into the model or document that it is payload metadata only.

**Scope**:

- `apps/worker/src/models/mlb-model.js` (read-only audit)
- `apps/worker/src/jobs/run_mlb_model.js` (read-only audit, lines around `wind_dir` usage)
- `apps/worker/src/jobs/pull_mlb_weather.js` (read-only audit)
- Documentation update in whichever file is authoritative (one-line comment)

**Out of scope**:

- Changes to weather fetch logic
- Changes to model coefficients
- Any new model features

**Acceptance**:

- A comment in `run_mlb_model.js` near the `mlb.wind_dir = weatherRow.wind_dir` assignment explicitly states whether `wind_dir` is used in any downstream computation or is payload metadata only
- If it IS used: a test confirms the coefficient changes when wind_dir is headwind vs tailwind
- If it is NOT used: a comment says "payload metadata only — not modeled" so future developers don't add handling expecting it to matter

**Owner agent**: unassigned
**Time window**: TBD
**Coordination flag**: solo

**Tests to run**:

- `npm --prefix apps/worker run test -- --testPathPattern run-mlb-model`

**Manual validation**:

- Read `mlb-model.js` and trace every reference to `wind_dir` through to any final coefficient calculation

**Guard for WI closeout**:

- Comment added explaining `wind_dir`'s role (or lack thereof)
- No undocumented use of `wind_dir` in model computation
