# WI-0859: POTD fallback publish path and hard-deadline error log

| Field | Value |
|---|---|
| **ID** | WI-0859 |
| **Goal** | Add a fallback publish path that fires `run_potd_engine` once if the primary window was missed, plus a hard-deadline error log when no successful publish is recorded by 4:30 PM ET. |
| **Priority** | High — production silent failure |
| **Depends on** | WI-0858 (requires `postDeadlineEt` in the schedule metadata shape) |
| **Coordination flag** | solo |

---

## Context

Even with WI-0858's 15-tick primary window (`[targetPostTimeEt, 4:15 PM)`), the entire window can still be missed if the worker is restarted mid-afternoon or a previous tick is long-running. When that happens there is currently no recovery path — POTD silently produces no pick for the day.

Additionally, `run_potd_engine` succeeding as a job does not guarantee a pick was actually selected and posted. A failed primary attempt leaves a `failed` status in `job_runs`, which `shouldRunJobKey` allows to retry, but only within the primary window. Once the window closes, the failed record blocks any new attempt.

The fix is a second guard that activates after `postDeadlineEt` passes with no success recorded, and uses a distinct job key so `shouldRunJobKey` cannot block it.

---

## Fix

In the POTD block of `computeDueJobs` (`apps/worker/src/schedulers/main.js`), after the primary window block, add:

```js
const forceFallbackDeadline = DateTime.fromISO(meta.postDeadlineEt, { zone: TZ }).plus({ minutes: 15 });
// = 4:30 PM ET

const fallbackJobKey   = `${publishJobKey}:fallback`;
const alreadySucceeded = wasJobKeyRecentlySuccessful(publishJobKey, 300) ||
                          wasJobKeyRecentlySuccessful(fallbackJobKey, 300); // within 5 hours

// Fallback window: [postDeadlineEt, forceFallbackDeadline)
if (
  nowEt >= postDeadlineEt &&
  nowEt <  forceFallbackDeadline &&
  !alreadySucceeded &&
  shouldRunJobKey(fallbackJobKey)
) {
  console.warn('[POTD] Primary window missed — firing fallback publish');
  jobs.push({
    jobName: 'run_potd_engine',
    jobKey:  fallbackJobKey,
    execute: runPotdEngine,
    args:    { jobKey: fallbackJobKey, dryRun, schedule: meta },
    reason:  `potd fallback engine (primary window missed, ${nowEt.toISO()})`,
  });
}

// Hard deadline: past 4:30 PM with no success — log for alerting
if (nowEt >= forceFallbackDeadline && !alreadySucceeded) {
  console.error(
    `[POTD] Hard deadline passed with no successful publish — ` +
    `manual review needed (date=${meta.playDate})`
  );
}
```

**Import**: `wasJobKeyRecentlySuccessful` is already exported from `@cheddar-logic/data` and destructured at the top of `main.js` — add it to the existing destructure if not already present.

**Key design rationale**:
- `fallbackJobKey = publishJobKey + ':fallback'` is critical. The primary attempt may have a `running` or `failed` record under `publishJobKey`. `shouldRunJobKey` would suppress a retry under the same key. The distinct suffix creates a clean slot.
- `alreadySucceeded` checks both `publishJobKey` and `fallbackJobKey` — either a successful primary or a successful fallback silences both the fallback job and the hard-deadline log.
- The hard-deadline log at 4:30 PM fires every tick until EOD. This is intentional: it creates repeated log entries that an alerting rule (Datadog, CloudWatch, etc.) can fire on once per day.

## Acceptance

1. At `nowEt = 4:15 PM` with no `success` record for `potd|<date>`: `run_potd_engine` is queued with key `potd|<date>:fallback`
2. At `nowEt = 4:16 PM` with a `success` record for `potd|<date>`: no fallback job queued, no error log
3. At `nowEt = 4:30 PM` with no success record: no job queued, `[POTD] Hard deadline passed` appears in console output
4. Fallback job key is `potd|<date>:fallback` — distinct from primary `potd|<date>`
5. If fallback already ran successfully (record under `potd|<date>:fallback`): `alreadySucceeded` is `true` (dual-key check), so neither a second fallback job nor the hard-deadline log fires on subsequent ticks
6. Existing `scheduler-windows.test.js` tests remain green
7. New tests added (see below)
