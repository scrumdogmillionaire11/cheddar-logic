# WI-1236: Fix `/api/games` Dedupe and Degraded Visibility

**ID**: WI-1236
**Goal**: Prevent matchup-collision merges in `/api/games` and make degraded or stale response modes explicit to `/wedge` consumers.

**Context**:

The current `/api/games` read path dedupes on a matchup-style key rather than canonical `game_id`, which can merge separate same-day matchups into one response row. The route also serves degraded and stale-cache payloads, but the UI mostly logs this rather than surfacing it clearly. This WI hardens the read contract first without changing model thresholds or settlement rules.

**Scope**:
- `web/src/lib/games/service-layer.ts`
- `web/src/lib/games/route-handler.ts`
- `web/src/lib/game-card/filters.ts`
- `web/src/__tests__/api-games-dedupe-and-degraded-state.test.js` (new)
- `WORK_QUEUE/WI-1236.md`

**Out of scope**:
- Any change to betting thresholds, canonical decision policy, or model scoring
- Any change to `/api/cards`, `/api/results`, or POTD behavior
- Any change to worker ingestion or settlement jobs

**Acceptance**:
- `/api/games` dedupe uses canonical `game_id` rather than `sport|away|home|date`.
- Same-day rematches no longer merge into one response row.
- Degraded response states are explicitly surfaced in the response contract, including stale-cache and fallback-data conditions.
- `/wedge` consumers can distinguish healthy, stale, and degraded game payloads without relying on console-only warnings.
- Regression coverage proves duplicate-matchup rows stay separate and degraded flags propagate.
