## API contract decisions

- `/api/admin/model-health` must preserve the existing response shape `{ success, data: sportSnapshots[] }` and keep every existing sport snapshot field unchanged.
- `/api/admin/model-health` must add a top-level `potd_health` object derived read-only from existing POTD tables: `potd_daily_stats`, `potd_plays`, `potd_nominees`, `potd_shadow_candidates`, and `potd_shadow_results`.
- `potd_health` must include at minimum `status`, `last_run_at`, `today_state`, `candidate_count`, and `near_miss` freshness/count fields. Sparse or missing data must return deterministic `no-data`/`stale` semantics rather than throwing.
- `/api/admin/pipeline-health` must preserve the existing `data` row array and add either read-only virtual POTD lane rows/checks or an additive `potd_lanes` field.
- `/api/admin/pipeline-health` POTD lane visibility must be derived from existing POTD tables, not by editing `check_pipeline_health.js` or writing canonical `pipeline_health` rows.
- POTD lane checks must attribute failures by phase for run recency, today fired/no-pick state, candidate volume, and near-miss shadow settlement freshness.
- Any retained legacy/compatibility branch must include an `INTENTIONAL_RETENTION: WI-1158` rationale; otherwise superseded POTD/near-miss health branches must be removed.

- Operators can see POTD lane health in Dr. Claire output with explicit status and reasons, not generic freshness text.

- Missing or sparse historical POTD/near-miss data degrades safely to deterministic `no-data`/`stale` semantics.

- Superseded legacy POTD/near-miss health branches in scoped files are removed or intentionally retained with explicit rationale.
