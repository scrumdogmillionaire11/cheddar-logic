# WI-1210: Enforce /api/results Performance Budget + High-Cardinality Regression Harness

**ID**: WI-1210
**Goal**: Add enforceable latency and allocation regression coverage for `/api/results` under
high-cardinality datasets so performance regressions fail CI before production impact.
Execution is phased: harness first, baseline second, budgets third.

**Scope**:

- `web/src/__tests__/api-results-performance-budget.test.ts` (new — primary deliverable)
- `web/package.json` (test script wiring only)
- `web/src/__tests__/integration/results-behavioral-contract.test.ts` (extend if appropriate)

**Referenced but NOT modified** (read to import/call only):

- `web/src/lib/results/query-layer.ts` — import `queryResultsReportingData`, `ResultsRequestFilters`
- `web/src/lib/results/transform-layer.ts` — import `buildResultsAggregation`, `buildResultsResponseBody`
- `web/src/app/api/results/projection-metrics.ts` — import `PROJECTION_TRACKING_CARD_TYPES`

**Out of scope**:

- `web/src/app/api/results/route.ts` — HTTP layer is NOT tested here; see architecture note
- Full pagination/cursor redesign of `/api/results`
- Worker settlement logic changes
- UI rendering/virtualization changes on `/results`

**Architecture note — why no HTTP layer**:

The route has a 15-second LRU cache keyed on filter params (`cache.ts`). There is no clean,
cache-neutral HTTP path for repeatable latency measurement: `?_diag=1` bypasses the cache but
also activates `countFilteredRowsForDiagnostics()`, changing the profile under test. The
`startIsolatedNextServer` harness spawns Next.js as a child process, making in-process heap
measurement impossible.

**The correct approach**: call `queryResultsReportingData` and `buildResultsAggregation` /
`buildResultsResponseBody` directly against an isolated test DB. This gives repeatable,
cache-free, in-process timing and allocation measurement. The HTTP route is already covered
by behavioral and segmentation contract tests.

## Phase 1 — Implement Harness (No Budget Assertions Yet)

Build the full test harness as specified below. **Do NOT add failing assertions in this phase.**
Instead, log structured results for inspection.

**Fixture**: 500 settled `card_payloads` rows with realistic `payload_data` JSON blobs
(~1–2 KB each), spread across 3 sports (`nba`, `nfl`, `mlb`), each with a matching `games`
row, `card_display_log` entry, and `card_results` row. Seeded via a deterministic helper
(no random IDs) so runs are repeatable. Use `setupIsolatedTestDb` from `db-test-runtime.js`.

Each `payload_data` JSON must include: `confidence_pct`, `decision_basis` (enum: ODDS_BACKED),
`play.decision_v2.official_status` (enum: PLAY|LEAN|PASS), `play.period` (enum: FULL_GAME|1P),
`recommended_bet_type`, `market_type`, `selection`, `line`, `locked_price`, `home_team`,
`away_team`. This ensures the query-layer fixtures exercise realistic payload filtering.

Insert fixture rows deterministically (e.g., card IDs = "perf-fixture-001" through
"perf-fixture-500") so repeated runs are reproducible. Seed across sports evenly:
~167 rows per sport (nba, nfl, mlb). For each row, insert:
1. One `games` row (status='final')
2. One `card_payloads` row with full payload
3. One `card_display_log` entry (one per game)
4. One `card_results` row (result='WIN' or 'LOSS' randomly but deterministically seeded)

**Variants to measure**: For each variant, construct a `ResultsRequestFilters` object:
```javascript
{
  limit: 50 or 200,           // per variant
  sport: 'NBA' | 'NFL' | 'MLB' | null,  // null for default (all sports)
  cardCategory: null,
  minConfidence: null,
  market: null,
  includeOrphaned: false,
  dedupe: true or false,      // per variant
  diagnosticsEnabled: false,  // always false for perf test
}
```

Measure these 4 variants (in order, so heap warms progressively):
- default: limit=50, dedupe=true, sport=null (all sports)
- limit=200: limit=200, dedupe=true, sport=null
- limit=200 nodedupe: limit=200, dedupe=false, sport=null
- sport=NBA: limit=50, dedupe=true, sport='NBA'

**Timing methodology**: for each variant, run 5 times, discard the first (JIT/warm-up),
record the max of the remaining 4. Log to stdout in a structured table:

```
[perf] variant=default         runs=5 discard=1 max_ms=NNN heap_delta_mb=NNN
[perf] variant=limit200        runs=5 discard=1 max_ms=NNN heap_delta_mb=NNN
[perf] variant=limit200_nodupe runs=5 discard=1 max_ms=NNN heap_delta_mb=NNN
[perf] variant=sport_filter    runs=5 discard=1 max_ms=NNN heap_delta_mb=NNN
```

**DB wiring**: after `setupIsolatedTestDb`, seed rows using `db.getDatabase()` (the
write client from the data package). To call `queryResultsReportingData`, obtain the read-only
instance via `getDatabaseReadOnly()` imported from `@cheddar-logic/data` — this is the correct
type (`ResultsDb`) for the function signature. Call `closeReadOnlyInstance(dbInstance)` after
each timing run. Construct `ResultsRequestFilters` objects directly (do not use
`parseResultsRequestFilters`, which requires `URLSearchParams`).

**Allocation measurement**: for the `limit200_nodupe` variant, capture
`process.memoryUsage().heapUsed` before and after the call. Call `global.gc?.()` before the
before-measurement if available. Log the delta. No assertion yet.

**Phase 1 deliverable**: `api-results-performance-budget.test.ts` passes (exits 0), prints
the table above, and makes no budget assertions.

## Phase 3 — Set Budgets From Baseline Data

**Rule**: `budget = max(CI observed) × 1.5`, rounded up to the nearest clean number
(e.g., 372 ms → 400 ms). Heap budget similarly derived from CI observed × 1.5.

Then:
- Replace log-only output with assertions
- Failure message must include measured value and budget:
  `"query+agg took 612 ms, budget is 400 ms (variant: limit=200 dedupe=false)"`
- Document derivation in test source comments (not bare constants)

**Phase 3 deliverable**: assertions are live; test fails if any variant exceeds budget.

## Phase 4 — Stabilize
## Phase 4 — Stabilize

Before adding to any CI gate:
- Run test suite 5+ times sequentially (same process, same heap warm-up path)
- For each run, collect the 4-variant max times
- Compute coefficient of variation (std-dev / mean) across all runs: must be < 25%
- If CV > 25%: widen budget to `max(CI observed) × 2.0` OR increase run count to 7 (discard first 2)

This ensures the test isn't flaky — timing variance is expected due to OS scheduling, but
if it exceeds 25%, the budget is too tight for the environment.
