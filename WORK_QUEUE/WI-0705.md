# WI-0705: Fix Build Lab "New session" — frontend/backend contract mismatch

**ID**: WI-0705
**Goal**: Fix the Build Lab "New session" action so it stops returning `manager_id: Field required; gameweek: Field required`. The frontend `DraftSessionCreateRequest` sends `user_id` but the backend expects `manager_id` + `gameweek` (both required, no default).

**Depends on**: none

**Scope**:
- `WORK_QUEUE/WI-0705.md`
- `web/src/lib/fpl-api.ts` — align `DraftSessionCreateRequest` and `DraftSession` types to the backend contract
- `web/src/components/fpl-draft-lab.tsx` — add a gameweek input and pass `manager_id` + `gameweek` to `createDraftSession`

**Out of scope**:
- Backend model changes (backend contract is correct)
- Changes to any other component

**Acceptance**:
- Clicking "New session" in Build Lab no longer returns a 422 with `manager_id: Field required; gameweek: Field required`.
- The component collects a target gameweek (number input, 1–38, default current GW) before calling `createDraftSession`.
- `DraftSessionCreateRequest` in `fpl-api.ts` matches the backend: `{ manager_id: string; gameweek: number; }` (removes `user_id`, removes `candidates`, removes `label`).
- `DraftSession` response type reflects `{ session_id, manager_id, gameweek, status, constraints, started_at }`.

**Owner agent**: unassigned
**Time window**: TBD
**Coordination flag**: solo
**LOE**: S

**Root cause**:
`fpl-api.ts` `DraftSessionCreateRequest` was defined with `user_id`/`label`/`candidates` (matching an older or different backend). The actual `cheddar-fpl-sage/backend/models/draft_api_models.py` `DraftSessionCreateRequest` requires `manager_id: str` and `gameweek: int` with no defaults. The component passes `{ user_id: userId }` which maps to neither required field.

**Tests to run**:
- `npx tsc --noEmit --project web/tsconfig.json`
- `npm --prefix web run build`

**Manual validation**:
- Navigate to FPL Sage → Build Lab → click "New session"
- Session is created and session ID appears without error
- Gameweek input (or sensible default) is visible before creating the session
