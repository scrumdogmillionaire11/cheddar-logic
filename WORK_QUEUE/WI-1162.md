# WI-1162: Canonical Discord Publish Status Contract Across Lines And Props

**ID**: WI-1162
**Goal**: Implement a single cross-market publish-status field in publisher output so Discord eligibility semantics are unified across game lines, game props, and player props while preserving `LEAN` as the internal canonical token and `Slight Edge` as user-facing copy.

**Requirements**: [DISCORD-HOOK-01, DISCORD-HOOK-02]

**Roadmap traceability**:
- Executes phase plan: `.planning/phases/discord-hook-eligibility/discord-hook-eligibility-01-PLAN.md`
- This WI introduces the canonical publisher contract used by downstream Discord formatting and posting behavior.
- Canonical upstream prerequisites land in `WI-1160` and `WI-1161`; this WI consumes those fields and must fail closed when they are absent.

**Scope**:

- `apps/worker/src/utils/decision-publisher.js`
- `apps/worker/src/utils/__tests__/decision-publisher.v2.test.js`
- `WORK_QUEUE/WI-1162.md`

**Out of scope**:

- Modifying Discord message rendering layout
- Removing all legacy fallback logic from Discord formatter
- Backfilling historical payload rows in storage

**Acceptance**:

- [DISCORD-HOOK-01] Publisher stamps `webhook_publish_status` on all card payloads with canonical domain: `PLAY`, `SLIGHT_EDGE`, `PASS_BLOCKED`.
- [DISCORD-HOOK-02] Alias/status normalization for lines/props is centralized in publisher logic; Discord formatter does not gain new inference branches.
- Existing compatibility fields (`webhook_bucket`, `webhook_eligible`, `webhook_lean_eligible`, `webhook_display_side`) remain populated in a backward-compatible manner.
- Tests include market-family fixtures for game lines, game props, and player props plus exhaustive coverage for every allowed canonical `final_play_state` mapping.
- Canonical token contract is enforced:
  - `decision_v2.official_status` may only be `PLAY`, `LEAN`, or `PASS`.
  - `final_play_state` may only be `OFFICIAL_PLAY`, `LEAN`, `WATCH`, `BLOCKED`, or `NO_PLAY`.
  - `webhook_publish_status` may only be `PLAY`, `SLIGHT_EDGE`, or `PASS_BLOCKED`.
  - `webhook_bucket` remains compatibility-only with values `official`, `lean`, or `pass_blocked`.
- User-facing copy may say `Slight Edge`, but no publisher/model/filter/API logic compares against that copy token. Public label mapping is derived from canonical `LEAN`.
- Canonical `WATCH` is treated as a blocked positive signal and must never route into clean `lean`/`SLIGHT_EDGE` publish status.
- Canonical precedence is enforced for publish routing:
  - `final_play_state` wins when present.
  - `decision_v2.official_status` is used only when `final_play_state` is missing.
  - Legacy fields (`action`, `classification`, `status`, including `WATCH`/`HOLD`) are fallback-only and must never override canonical fields.
- Canonical-to-publish mapping is exact:
  - `OFFICIAL_PLAY` -> `webhook_publish_status=PLAY`, `webhook_bucket=official`
  - `LEAN` -> `webhook_publish_status=SLIGHT_EDGE`, `webhook_bucket=lean`
  - `WATCH` -> `webhook_publish_status=PASS_BLOCKED`, `webhook_bucket=pass_blocked`
  - `BLOCKED` -> `webhook_publish_status=PASS_BLOCKED`, `webhook_bucket=pass_blocked`
  - `NO_PLAY` -> `webhook_publish_status=PASS_BLOCKED`, `webhook_bucket=pass_blocked` for Discord routing only; `NO_PLAY` is not a hard block.
- Missing canonical fields fail closed:
  - If neither `final_play_state` nor `decision_v2.official_status` is present, emit `webhook_publish_status=PASS_BLOCKED`, `webhook_bucket=pass_blocked`, append `MISSING_CANONICAL_STATUS`, and set `decision_trace.source_of_truth=none`.
- No silent default mapping is allowed:
  - Every `webhook_publish_status` must be derived from an explicit mapping.
  - If mapping cannot be resolved, emit `PASS_BLOCKED` and append `INVALID_PUBLISH_MAPPING`.
- Publisher detects and records status drift instead of silently normalizing conflicts:
  - `decision_v2.official_status=LEAN` and `final_play_state=WATCH` is allowed, with `final_play_state` winning.
  - `decision_v2.official_status=PLAY` and `final_play_state=LEAN` records `STATUS_DOWNGRADED_AFTER_MODEL`.
  - `decision_v2.official_status=PLAY`, `final_play_state=WATCH`, and legacy positive tokens (`classification=LEAN`, `action=HOLD`, `status=WATCH`) still resolve to `webhook_publish_status=PASS_BLOCKED`; `final_play_state` wins, drift is preserved in diagnostics, and legacy fields do not regain precedence.
  - Any derived publish status that disagrees with canonical mapping records `CANONICAL_STATUS_CONFLICT` or `INVALID_PUBLISH_MAPPING`.
- Publisher stamps `publish_status_flags` and `decision_trace` with enough detail to debug routing without reading code.
