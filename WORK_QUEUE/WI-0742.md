## Required Output Contract

Every card with `execution_status=PROJECTION_ONLY` must include these four fields
in its published payload:

```json
{
  "numeric_projection": 4.7,
  "projection_unit": "runs_f5",
  "recommended_direction": "OVER",
  "confidence": 0.61,
  "actionable": false,
  "execution_status": "PROJECTION_ONLY"
}
```

### Field definitions

| Field | Type | Constraint |
|-------|------|------------|
| `numeric_projection` | `number` | finite, positive, > 0 |
| `projection_unit` | `string` | one of `"goals_1p"`, `"shots_on_goal"`, `"strikeouts"`, `"runs_f5"` |
| `recommended_direction` | `"OVER" \| "UNDER"` | must be one of those two tokens |
| `confidence` | `number` | 0 < confidence <= 1 |
| `actionable` | `false` | hardcoded false â€” already enforced by INV-006, kept explicit |
| `execution_status` | `"PROJECTION_ONLY"` | already enforced by INV-004 |

### Per-family units

| Family | `projection_unit` | Example value |
|--------|-------------------|---------------|
| NHL_1P_TOTAL | `"goals_1p"` | `1.72` |
| NHL_PLAYER_SHOTS | `"shots_on_goal"` | `3.8` |
| MLB_PITCHER_K | `"strikeouts"` | `6.4` |
| MLB_F5_TOTAL | `"runs_f5"` | `4.7` |

## Implementation Notes

### 1. Model job changes (emit the contract)

Each job must populate the required fields on the card payload before it is
written to the database. Use existing model computation values â€” do not add new
models. **Never use a degraded fallback or floor value as `numeric_projection`
unless it is the only signal available, and if so, log a warning.**

- **`run_nhl_model.js` / NHL_1P_TOTAL**: `numeric_projection` = the full 1P
  expected-goals estimate from the F1P driver (not a floor). Because the line
  is always posted at 1.5, `recommended_direction` = `OVER` if
  `numeric_projection > 1.5`, else `UNDER`. Emit `projection_low` / `projection_high`
  if the driver produces a variance estimate.
  `projection_unit="goals_1p"`.

- **`run_nhl_player_shots_model.js` / NHL_PLAYER_SHOTS**: `numeric_projection`
  = the full-model shots-on-goal estimate for that player in that matchup
  (season avg adjusted for opponent allow rate, home/away split, pace). This is
  the number you'd compare against any SOG line a book posts.
  `recommended_direction` follows the model's existing selection logic.
  Emit `projection_low` / `projection_high` from the model's variance if available.
  `projection_unit="shots_on_goal"`.

- **`run_mlb_model.js` / MLB_PITCHER_K**: `numeric_projection` = `k_mean` from
  the pitcher-K driver (starter K rate Ă— innings cap Ă— batter K rate â€” the same
  value WI-0734 already surfaces in the web UI). **Do NOT use `projectionFloor`
  or a static fallback here.** `recommended_direction` follows the existing
  publish-gate selection. Emit `projection_low` / `projection_high` if the
  driver produces a confidence interval.
  `projection_unit="strikeouts"`.

- **`run_mlb_model.js` / MLB_F5_TOTAL**: `numeric_projection` = the full-model
  F5 run-total estimate (starter ERA/xFIP Ă— opponent offense Ă— weather
  adjustment). **Do NOT use `projectionFloorF5` as the primary value** â€” that
  is a defensive floor for degraded-model cards. If the full model result is
  available, use it. Only fall back to `projectionFloorF5` when the degraded
  path is active, and flag `projection_is_floor: true` in that case.
  Emit `projection_low` / `projection_high` from the model's range estimate.
  `projection_unit="runs_f5"`.

### 2. INV-007 (new invariant)

```
INV-007 â€” PROJECTION_ONLY_NUMERIC_CONTRACT
Severity: CRITICAL
Trigger: execution_status === 'PROJECTION_ONLY'
Check: numeric_projection is a finite positive number
       projection_unit is a non-empty string
       recommended_direction is 'OVER' or 'UNDER'
       confidence is a number in (0, 1]
```

This invariant fires on every PROJECTION_ONLY card. It does not fire on LIVE or
EXECUTABLE cards.

### 3. Comparator changes

Add a `checkProjectionContract` method to `compare_audit_snapshot.js`:
- If `execution_status === 'PROJECTION_ONLY'`, verify the four fields are present.
- If the fixture `expected` block includes `numeric_projection_min` /
  `numeric_projection_max`, enforce the range (tolerance check).
- If no range is set in the fixture, only verify the field is present and
  structurally valid (positive finite number).

### 4. Fixture updates

Each of the four PROJECTION_ONLY fixtures must be updated to include in
`expected`:

```json
{
  "numeric_projection_present": true,
  "projection_unit": "<expected unit string>",
  "recommended_direction_present": true,
  "confidence_range": [0.0, 1.0]
}
```

After the first audit run with the new model output, set a tighter
`confidence_range` and add `numeric_projection_min` / `numeric_projection_max`
from the actual output, then re-lock with `baseline_reviewed: true`.

For NHL_1P_TOTAL fixtures: add an assertion that `recommended_direction`
correctly reflects which side of 1.5 the projection falls on.
