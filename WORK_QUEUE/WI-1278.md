# WI-1278: Expose Stale/Degraded Freshness State on /wedge

**ID**: WI-1278
**Goal**: Surface backend freshness state in the `/wedge` UI so users can distinguish fresh data from degraded/stale fallback responses.

**Scope**:
- `web/src/components/cards/CardsPageContext.tsx`
- `web/src/components/cards/CardsHeader.tsx`
- `web/src/components/cards/types.ts`
- `web/src/components/cards/shared.ts`
- `web/src/__tests__/cards-freshness-state.test.tsx` (new)

**Out of scope**:
- Backend fallback semantics changes in `/api/games`
- Polling interval tuning
- Any betting/model behavior changes

**Acceptance**:
- UI renders explicit freshness indicator for `full`, `degraded_base_games`, and `stale_cache` response modes.
- Indicator clears automatically when fresh mode resumes.
- No extra network requests are introduced.
- Test coverage validates mode-to-banner mapping and recovery behavior.

**Owner agent**: unassigned
**Time window**: 2-4 hours
**Coordination flag**: none
**Depends on**: WI-1273

**Tests to run**:
- `npm --prefix web exec tsx src/__tests__/cards-freshness-state.test.tsx`
- `npm --prefix web exec tsx src/__tests__/api-games-market-smoke.test.js`

**Manual validation**:
- Simulate `X-Games-Mode` values and confirm corresponding user-facing indicator text.
- Verify no stale indicator appears on normal `full` responses.

**Failure modes**:
- Freshness state lags behind due to stale local state.
- Banner appears in non-wedge contexts unexpectedly.
- UX noise from transient degraded states without debounce.
