# WI-0907: Recovery-Path Policy Audit and Failure-Mode Classification

**ID**: WI-0907
**Goal**: Classify all missing/inconsistent input recovery paths as hard fail, soft pass, degraded output, hidden output, retry, or fallback, and remove silent inconsistency.

**Depends on**: WI-0900, WI-0901

**Scope**:

Core job and decision logic:

- `apps/worker/src/jobs/execution-gate.js` — core gate logic + reason codes
- `apps/worker/src/jobs/run_nhl_model.js` — NHL recovery paths
- `apps/worker/src/jobs/run_mlb_model.js` — MLB recovery paths
- `apps/worker/src/jobs/run_nba_model.js` — NBA recovery paths
- `packages/models/src/decision-pipeline-v2.js` — decision watchdog + price logic

Web surface and transform:

- `web/src/lib/games/route-handler.ts` — API surface for classification exposure
- `web/src/lib/game-card/transform/index.ts` — transform metadata injection

Audit documentation:

- `docs/audits/recovery-path-classification.md` (new)
- `.planning/codebase/HARDENING_AUDIT.md` (new or update)

**Scope expansion rule**:
If additional paths are discovered during enumeration (Phase 1), append exact file paths to this work item before editing them. Do not broaden to repository-wide globs without documented justification tied to one acceptance criterion (per WI-0900 protocol).

**Out of scope**:

- Incident tooling overhaul
- Provider contract renegotiation
- New alert transport integrations

**Acceptance**:

- Every major failure mode is classified into one recovery bucket (hard fail, soft pass, degraded output, hidden output, retry, fallback).
- Classification matrix documented in `recovery-path-classification.md` with columns: scenario, path entry point, bucket, reason-code, reason-layer.
- Silent degradations are eliminated or explicitly marked with reason code and visibility policy.
- Retry/fallback behavior is consistent for equivalent failure classes across NBA/NHL/MLB.
- Route-handler and transform wire classification and reason metadata to API/debug surfaces.
- Mapper prompt updated with Config and State Truth Audit directive for future audits.

**Execution plan**:

## Phase 1: Audit initialization and enumeration

- Enumerate all failure scenarios in NBA, NHL, MLB model runners (missing key stats, data freshness mismatches, timestamp disagreements from WI-0900, reason-code gaps from WI-0901).
- List all decision-gate checkpoints: execution-gate, decision-v2 watchdog/price logic, publish, transform.
- Cross-reference each scenario against reason-code taxonomy from WI-0901 (docs/audits/reason-code-taxonomy.md).
- Deliverable: Enumeration table linking scenario → current handler → current output.

## Phase 2: Classification mapping

- For each scenario, assign exactly 1 of 6 recovery buckets: hard-fail, soft-pass, degraded-output, hidden-output, retry, fallback.
- Verify consistency across sports (equivalent scenarios must map to same bucket).
- Identify any silent degradations (masked errors, swallowed exceptions, no reason-code emission).
- Deliverable: Draft classification matrix (scenario → bucket → reason-code layer origin).

## Phase 3: Documentation

- Author `recovery-path-classification.md` with matrix: path → checkpoint → bucket → reason-code → visibility policy.
- Author or update `.planning/codebase/HARDENING_AUDIT.md` with recovery-path classification checklist for future audits.
- Deliverable: Audit docs committed.

## Phase 4: Code updates and wiring

- Wire recovery classification into route-handler.ts debug metadata output (ensure classification is machine-readable, not just prose).
- Wire recovery classification into transform/index.ts transform_meta so cards carry classification context.
- Confirm execution-gate.js emits reason-code for all recovery paths (no silent swallows).
- Update mapper system prompt (tools/ai or docs/decisions) with "Config and State Truth Audit" directive.
- Deliverable: Code changes committed with test updates.

## Phase 5: Validation

- Inject one missing-input scenario (e.g., missing key player from stats feed) for each sport; verify deterministic recovery class, reason code, layer.
- Inject one inconsistent-input scenario (e.g., timestamp newer in data but older in write); verify deterministic classification.
- Run full test suite.
- Deliverable: Manual validation evidence and test results.
