# WI-0833: Feature Correlation Cluster Detection in CI

| Field | Value |
|---|---|
| **ID** | WI-0833 |
| **Goal** | Add a CI check that computes pairwise Pearson correlation across all numeric features fed to each sport model, flags any pair with |r| ≥ 0.80 as a redundancy alert, and fails the build if a new pair exceeds the threshold without an explicit suppression annotation. This prevents correlated features from silently inflating apparent model confidence. |
| **Priority** | Medium |
| **Depends on** | WI-0828 (standardised model output interface; ablation test infrastructure) |
| **Coordination flag** | None — CI tooling only |

---

## Background

WI-0828 adds per-feature ablation tests (does nulling a feature degrade performance?). A complementary check is whether two features carry the same signal — if so, the model double-counts that signal and overstates its confidence.

Known candidates from current codebase:
- `xwoba_vs_hand` and `iso` (MLB) — both measure contact quality; |r| likely > 0.75
- `hard_hit_pct` and `xwoba` — hard contact is a component of xwOBA
- `goalie_gsax` and `homeGoalieSavePct` (NHL) — both measure goalie quality; addressed structurally by WI-0823 but CI should confirm post-fix correlation drops

The check runs on a set of fixture snapshots (real or synthetic) that represent typical model inputs.

---

## Scope

**Files to create:**
- `apps/worker/src/audit/feature_correlation_check.js` — loads feature fixture data for each sport, computes pairwise Pearson r matrix, emits a JSON report, exits non-zero if any pair |r| ≥ threshold without suppression

**Files to change:**
- `.github/workflows/ci.yml` (or equivalent) — add `feature_correlation_check` step after unit tests
- `apps/worker/src/audit/fixtures/correlation_suppressions.json` — allowlist for known acceptable high-correlation pairs with required rationale comment

**Out of scope:**
- Automatic feature removal — this WI only detects and alerts; removal is done via subsequent WIs (WI-0821 already removes ISO/hard_hit from MLB)
- VIF (variance inflation factor) — Pearson |r| is sufficient and interpretable at this scale
- Prop models

---

## Correlation Thresholds

| Level | |r| range | Action |
|---|---|---|
| INFO | 0.60 – 0.79 | Log warning; do not fail build |
| ALERT | 0.80 – 0.89 | Fail build unless suppressed with rationale |
| CRITICAL | ≥ 0.90 | Always fail build; no suppression allowed |

## Suppression format

```json
{
  "suppressions": [
    {
      "sport": "NHL",
      "feature_a": "goalie_gsax",
      "feature_b": "homeGoalieSavePct",
      "threshold_override": 0.85,
      "rationale": "Being unified by WI-0823; suppressed until that WI lands",
      "expires_after_wi": "WI-0823"
    }
  ]
}
```

Suppressions with `expires_after_wi` must be automatically invalidated (build fails) once the target WI's commit is detected in history (check by scanning for `WI-0823` in `git log`).

## Acceptance

- [ ] `feature_correlation_check.js` produces correct Pearson r matrix for synthetic fixture with known correlation
- [ ] Build fails when pair |r| ≥ 0.80 without suppression

- [ ] Suppression with `expires_after_wi` set to a merged WI causes build to fail (expiry enforcement)
- [ ] CI step integrated and passing on `main`
