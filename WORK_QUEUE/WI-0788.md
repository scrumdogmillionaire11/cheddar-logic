# WI-0788: Refactor `tracking_stats` to be a materialized cache over `projection_audit`

| Field              | Value |
|--------------------|-------|
| **ID**             | WI-0788 |
| **Goal**           | Replace the live-increment pattern in `tracking_stats` with a recompute-from-source pattern using `projection_audit` as the authoritative row-level store, so that `tracking_stats` becomes a queryable cache that can be filtered, segmented, and fully regenerated at any time |
| **Priority**       | Medium |
| **Owner agent**    | Unassigned |
| **Time window**    | TBD — depends on WI-0787 complete |
| **Coordination flag** | None — no shared touchpoints beyond files already owned by settlement pipeline |

---

## Depends On

**WI-0787** must be complete and `projection_audit` must have forward-written rows before this WI is executed.

---

## Problem

`tracking_stats` is currently written via `incrementTrackingStat` at settle time as the primary record. This has several weaknesses:

1. **No way to exclude classes of plays retroactively** — e.g., `total_1p` rows that settled during a temporary odds window had to be excluded by a code-level `continue` guard (see WI-0787 context). Already-accumulated rows remain dirty.
2. **Resettlement is destructive** — `resettle_historical_cards.js` wipes and re-aggregates `tracking_stats` by rescanning `card_results`, which is slow and loses dimension fidelity.
3. **Segmentation is fixed** — the `stat_key` schema bakes in dimensions at write time; new dimensions (player count, confidence band) cannot be added without a full rescan.
4. **`tracking_stats` and `projection_audit` will diverge** — without a single source of truth, running both in parallel risks inconsistency.

---

## Proposed Approach

After WI-0787 is in place and `projection_audit` has been backfilled:

1. **Remove `incrementTrackingStat` calls from `settle_pending_cards.js`** — `projection_audit` writes replace them as the settle-time write.
2. **Add a `recomputeTrackingStats(db, filters?)` function** that:
   - Optionally scopes to a time window or sport
   - GROUPs `projection_audit` by `(sport, market_type, direction, sharp_price_status, confidence_band, player_count)`
   - UPSERTs the resulting aggregates into `tracking_stats` with appropriate `stat_key` values
3. **Call `recomputeTrackingStats`** at the end of each settlement run (full replace for the affected time window), replacing the current per-row increment loop.
4. **Update `resettle_historical_cards.js`** to call `recomputeTrackingStats` instead of its own rescan loop.

## Scope

| File | Change |
|------|--------|
| `apps/worker/src/jobs/settle_pending_cards.js` | Remove `incrementTrackingStat` loop in Step 2; add call to `recomputeTrackingStats` at end of Step 2 |
| `packages/data/src/db/tracking.js` | Add `recomputeTrackingStats(db, opts)` function |
| `packages/data/src/db/index.js` | Export `recomputeTrackingStats` |
| `packages/data/index.js` | Re-export `recomputeTrackingStats` |
| `apps/worker/src/jobs/resettle_historical_cards.js` | Replace re-aggregate block with `recomputeTrackingStats` call |

## Out of Scope

- Changing the `tracking_stats` DB schema
- Backfilling `projection_audit` from history (separate follow-on)
- Removing `upsertTrackingStat` (still needed for external callers)
- Any UI or API changes to how `tracking_stats` is queried

---

## Key Invariants

- `tracking_stats` rows must remain queryable by all existing callers without schema change
- Recompute must be **idempotent** — running it twice produces the same result
- `total_1p` rows in `projection_audit` must be **excluded** from `tracking_stats` recompute (by `WHERE period != '1P' OR period IS NULL` filter in the GROUP BY query), honoring the WI-0787 intent
- Recompute should run in a single DB transaction per sport/window to avoid partial state

## Acceptance

- After a settlement run, `tracking_stats` rows match what you would compute by hand from `SELECT ... GROUP BY` on `projection_audit`
- `total_1p` rows do NOT appear in `tracking_stats`
- `recomputeTrackingStats` called twice in a row produces identical `tracking_stats` state
- `resettle_historical_cards` still works end-to-end using the new recompute path
- Existing tests for `tracking_stats` incrementing are updated or replaced with recompute equivalents
