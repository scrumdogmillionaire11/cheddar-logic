# WI-1204: Assess & Clean Up Redundant Deploy Steps

**ID**: WI-1204
**Goal**: Assess the five redundant/dead-code patterns identified in `deploy-production.yml`, remove or consolidate the safe ones, and document the intentional ones — estimated to save 30–60 seconds per deploy and reduce CI confusion.

**Scope**:
- `.github/workflows/deploy-production.yml`

**Out of scope**:
- Any runtime worker/web code
- CI job structure changes beyond `deploy-production.yml`
- Tailscale, ssh-action, or secrets configuration

**Acceptance**:
- The two permanently-disabled jobs (`deploy-web-vercel`, `deploy-backend-railway`) are removed or have a clearly dated comment explaining a concrete re-enable condition
- Duplicate npm cache clear (lines 221–223: `npm cache clean --force` followed by `rm -rf ~/.npm/_cacache`) is collapsed to one command
- Static archive pruning policy is unified into a single pass with one consistent retention value (currently 3 vs 20 in two separate blocks)
- Duplicate cgroup worker-kill block (second pass ~line 340) has an explicit comment confirming it is intentional (watchdog re-acquire guard), or is removed if analysis shows it is unreachable given the mask approach
- Hardcoded lock-file path (`/opt/data/cheddar-prod.db.lock`, ~line 207) is removed or replaced with a `${CANONICAL_DB_PATH}.lock` reference; no removal before `.env.production` is sourced
- No behavior changes to worker startup, DB migration, or web smoke checks

**Estimated time savings per deploy**:
| Item | Estimated saving |
|------|-----------------|
| Remove redundant `rm -rf ~/.npm/_cacache` (already done by `npm cache clean`) | ~5–10 s |
| Collapse duplicate cgroup kill + sleep passes (removes sleep 2 + sleep 1) | ~3 s |
| Remove dead `if: false` job definitions (parse overhead only) | < 1 s |
| Unified archive pruning (eliminates redundant `find` pass) | ~1–2 s |
| **Total** | **~10–15 s direct; ~30–60 s if avoided disk-space abort on constrained Pi** |

The most meaningful gain is correctness: the conflicting archive pruning limits (3 vs 20) mean up to 17 extra snapshot directories can accumulate between deploys, risking the disk-space abort gate (`< 512 MB free`). Fixing that prevents a full deploy failure, not just a slowdown.
