# WI-0667: Pipeline gate — computePublicSplitsGate + wire PASS_SHARP_MONEY_OPPOSITE

**ID**: WI-0667
**Goal**: Add `computePublicSplitsGate` to `decision-pipeline-v2.js` that uses live public bet %/handle % data to demote PLAY→LEAN when heavy public money is on our recommended side, producing the `PASS_SHARP_MONEY_OPPOSITE` reason code that already exists as a dead display label in Discord and the web UI.

**Background**:
`PASS_SHARP_MONEY_OPPOSITE` has display labels in `apps/worker/src/jobs/post_discord_cards.js` (L62) and `web/src/components/cards/game-card-helpers.tsx` (L245) but is never produced by any pipeline code. The splits data (WI-0664–0666) provides the missing input. The gate is opt-in: when no splits data is present the gate is a no-op, so this change cannot regress any existing card decisions.

The philosophical contract: heavy public betting % on a side is historically correlated with the "square" side — the side that sharp/professional money is against. When our model agrees with the public > 72% / 68% threshold, we demote to LEAN rather than block outright, since our edge signal still holds but the public-vs-sharp signal adds caution.

**Acceptance**:
- New function `computePublicSplitsGate(payload, oddsSnapshot)` added to `decision-pipeline-v2.js`:
  - **No-op** when `oddsSnapshot.splitsSource` is null/absent — gate skips, current status unchanged
  - **Fade-positive signal**: when recommended side's `publicBetsPct < FADE_PUBLIC_BET_THRESHOLD (40)` AND `publicHandlePct < FADE_PUBLIC_HANDLE_THRESHOLD (35)` → appends `FADE_PUBLIC_POSITIVE` to `payload.tags`
  - **Contrarian demotion**: when recommended side's `publicBetsPct > SHARP_PUBLIC_FADE_BET_THRESHOLD (72)` AND `publicHandlePct > SHARP_PUBLIC_FADE_HANDLE_THRESHOLD (68)` AND current status is `PLAY` → demotes to `LEAN`, sets `primary_reason_code = 'PASS_SHARP_MONEY_OPPOSITE'`
  - LEAN→PASS demotion is intentionally NOT included (one step at a time; tune after live data calibration)
- Threshold constants defined at top of file alongside `PLAY_EDGE_MIN`/`LEAN_EDGE_MIN`:
  ```
  FADE_PUBLIC_BET_THRESHOLD = 40
  FADE_PUBLIC_HANDLE_THRESHOLD = 35
  SHARP_PUBLIC_FADE_BET_THRESHOLD = 72
  SHARP_PUBLIC_FADE_HANDLE_THRESHOLD = 68
  ```
- Gate is inserted in `buildDecisionV2` **after** `applyNbaTotalQuarantine` and **before** `resolvePrimaryReason`
- `oddsSnapshot` context is extended to pass `publicBetsPctHome/Away`, `publicHandlePctHome/Away`, `splitsSource` from the DB read at card-build time (the `context.oddsSnapshot` object already flows in from run model jobs)
- Unit tests (`public-splits-gate.test.js`) cover:
  - No splits data → status unchanged, no tags added
  - Heavy public (>72%/68%) on recommended side, status=PLAY → demotes to LEAN, reason PASS_SHARP_MONEY_OPPOSITE
  - Heavy public on recommended side, status=LEAN → no change (already LEAN)
  - Fade-positive signal (<40%/<35%) → FADE_PUBLIC_POSITIVE in tags, status unchanged
  - Pinnacle proxy source (pct fields null) → no-op (treated as no splits data)
- `npm test --workspace=packages/models` passes
