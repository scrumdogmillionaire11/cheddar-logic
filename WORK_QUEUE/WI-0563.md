# WI-0563: Enforce API security checks on `/api/cards/[gameId]` and add SQLi regression tests

**ID**: WI-0563
**Goal**: Ensure the per-game cards endpoint applies the same request validation/rate-limit guardrails as other API routes, and lock this behavior with malicious-input regression tests.

**Scope**:
- `web/src/app/api/cards/[gameId]/route.ts`
- `web/src/__tests__/api-cards-sport-filter.test.js` (or new dedicated security test under `web/src/__tests__/`)
- `web/package.json` (optional: add a dedicated security test script)

**Out of scope**:
- Schema changes in `packages/data`
- Business logic changes to card selection/dedupe semantics
- Auth model redesign beyond existing `requireEntitlementForRequest` usage

**Background**:
`/api/cards`, `/api/games`, and `/api/results` call `performSecurityChecks(...)`, but `/api/cards/[gameId]` currently does not. The route uses bound SQL parameters, so immediate SQLi risk appears low, but the missing centralized validation/rate-limit check is a hardening gap and increases future regression risk.

**Acceptance**:
- `GET /api/cards/[gameId]` calls `performSecurityChecks(request, '/api/cards/[gameId]')` before DB access.
- Rejected validation/rate-limit requests return structured error responses consistent with other guarded routes.
- Add regression coverage using malicious probe values (for `cardType`, `dedupe`, `lifecycle`, and path `gameId`) to confirm no SQL injection and no unhandled exceptions.
- Add/adjust query-param allowlist for `/api/cards/[gameId]` so accepted params match route behavior (`cardType`, `card_type`, `dedupe`, `limit`, `offset`, `lifecycle` as applicable).
