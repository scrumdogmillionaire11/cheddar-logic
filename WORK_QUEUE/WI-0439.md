# WI-0439: Eliminate dual-repo deployment drift

**ID**: WI-0439
**Goal**: Consolidate FPL Sage to a single source of truth so the uvicorn server always runs the monorepo code without manual file syncing.

**Background**:
During WI-0432 debugging (2026-03-13), we discovered that the uvicorn DEV server was running from a **separate standalone repo** at `/Users/ajcolubiale/projects/cheddar-fpl-sage/` rather than the monorepo at `/Users/ajcolubiale/projects/cheddar-logic/cheddar-fpl-sage/`. This caused all 7 FPL dashboard regressions because the standalone repo held an old (972-line) version of `result_transformer.py` while the monorepo had the new (1555-line) version.

The fix was a one-time file sync (cp) from monorepo → standalone. This is brittle: any future changes to the monorepo backend/src will again diverge from what the server runs.

**Tech debt created**:

1. **Dual-repo drift**: The standalone repo (`/Users/ajcolubiale/projects/cheddar-fpl-sage/`) is a manually-synced copy, not linked to the monorepo. Changes to `cheddar-logic/cheddar-fpl-sage/` must be manually copied to `cheddar-fpl-sage/` for the DEV server to pick them up.

2. **No enforcement**: There is no CI check, Makefile target, symlink, or git hook ensuring the two repos stay in sync. Drift will recur silently.

3. **`_cached_result_meets_fpl_contract()` guard** (analyze.py): A validation function was added to detect legacy cached results missing FPL dashboard contract fields. This guard works but is a symptom of the root problem (stale code). Once the dual-repo issue is resolved, this guard can be simplified or removed.

4. **Debug markers removed but pattern established**: During investigation, `TRANSFORMER_VERSION_CHECK_2026_03_13` and `DEBUG_INITIAL_DICT_KEYS` log lines were added and later removed from `result_transformer.py`. The pattern of adding version-check log lines is a workaround for not having a proper deployment verification step.

5. **Editable install maps to standalone**: The `cheddar_fpl_sage` editable install (`__editable__.cheddar_fpl_sage-0.1.0.pth`) maps to `/Users/ajcolubiale/projects/cheddar-fpl-sage/src/cheddar_fpl_sage` — the standalone repo — not the monorepo. Any Python process using the system pyenv will always load the standalone repo's analysis engine.

**Scope**:
- `/Users/ajcolubiale/projects/cheddar-fpl-sage/` (standalone repo — to be deleted or linked)
- `/Users/ajcolubiale/projects/cheddar-logic/cheddar-fpl-sage/` (monorepo source of truth)
- `~/.pyenv/versions/3.12.10/lib/python3.12/site-packages/__editable__.cheddar_fpl_sage-0.1.0.pth`
- Makefile or dev scripts for starting the DEV server

**Out of scope**:
- PROD deployment changes (separate concern)
- Frontend (`web/`) files — already part of monorepo only

**Acceptance**:
- The uvicorn DEV server starts from `cheddar-logic/cheddar-fpl-sage/` (or a symlink to it)
- The editable install resolves `cheddar_fpl_sage` to the monorepo's `src/` directory
- `diff -rq --exclude=__pycache__ cheddar-logic/cheddar-fpl-sage/backend cheddar-fpl-sage/backend` returns no output (or standalone repo is deleted)
- A Makefile `dev` target or shell script documents the correct way to start the server
