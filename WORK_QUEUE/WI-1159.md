## Must-haves (goal-backward)

**truths**:

- NHL family coverage on `/results/projections` depends on both API inclusion and UI family filters; both surfaces must align.
- Existing projection-settled behavior for currently visible families must remain backward-compatible.
- If proxy-backed rows are unavailable for a supported NHL family, the API must use a deterministic read path rather than returning empty silently.
- UI family filters must be derived from canonical supported families (or an explicitly maintained superset) to prevent future hidden-family drift.
- Production verification must include both endpoint payload inspection and live page rendering checks.

**artifacts**:

- `web/src/app/api/results/projection-settled/route.ts` includes deterministic NHL family row inclusion logic.
- `web/src/components/results/ProjectionAccuracyClient.tsx` includes NHL family options/filters aligned to supported families.
- `web/src/components/results/ProjectionResultsTable.tsx` renders user-facing labels/sections for NHL families without fallback-only token text.
- `web/src/__tests__/api-results-projection-settled-contract.test.js` and related tests enforce NHL-family presence expectations.
- `web/src/__tests__/projection-results-family-filter.test.js` asserts behavioral filtering/rendering of NHL families on the projections surface.

**key_links**:

- API family support contract (`PROJECTION_ANALYTICS_CONTRACT_BY_MARKET_FAMILY`) -> projection-settled query/filter logic.
- projection-settled response rows -> ProjectionAccuracyClient supported family filter set.
- filtered settled rows -> grouped table sections and family labels.
- deployment verification checklist -> prod endpoint payloads -> live `/results/projections` rendering.
