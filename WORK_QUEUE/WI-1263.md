# WI-1263: Audit Support Score Calibration For High-Edge Non-PLAY Candidates

**ID**: WI-1263

**Goal**: Explain why high-edge candidates remain `SLIGHT_EDGE` or `LEAN` by instrumenting and auditing the decision path around support score, blockers, price verification, and model health.

**Context**:

High-edge candidates such as Ducks ML at `+145` should not remain in a lower decision tier without a clear, inspectable reason. Before changing thresholds, the system needs an audit pass that measures which gate is actually suppressing promotion.

The audit must measure, at minimum:

- `edge_pct`
- `support_score`
- blocker reason codes
- price verification state
- model health state
- decision reason code
- realized outcome and CLV when available

This work item is diagnostic and calibration-oriented. It should improve visibility into the decision boundary before any threshold changes are proposed.

**Scope**:

- `apps/worker/src/jobs/helpers/ensure-canonical-decision-v2.js`
- `apps/worker/src/jobs/potd/run_potd_engine.js`
- `apps/worker/src/jobs/potd/__tests__/run-potd-engine.test.js`
- `packages/data/src/decision-outcome.builders.js`
- `packages/data/src/decision-outcome.ts`

**Out of scope**:

- Lowering the `PLAY` bar without audit evidence
- Reclassifying fallback rows as official POTD
- UI work on `/cards` or `/potd`
- Long-range bankroll or staking adjustments

**Acceptance**:

- High-edge non-PLAY candidates emit enough structured diagnostics to explain why they failed promotion
- Audit output can distinguish support-score suppression from blocker suppression, price verification failures, and stale model-health conditions
- The targeted test suite covers at least one high-edge `SLIGHT_EDGE` candidate and verifies the emitted diagnostics
- No official POTD contract changes are introduced as part of the audit
