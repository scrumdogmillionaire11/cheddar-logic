## Problem

`insertModelOutput()` is called by two model jobs but its return value is never used by any consumer.

**Writers:**
- `apps/worker/src/jobs/run_mlb_model.js` line ~1979 â€” calls `insertModelOutput()` after each model run
- `apps/worker/src/jobs/run_nfl_model.js` line ~254 â€” same

**Getters defined but never called anywhere in `apps/` or `web/src/`:**
- `packages/data/src/db/models.js` exports `getLatestModelOutput`, `getModelOutputs`, `getModelOutputsBySport`
- Zero call sites confirmed via grep

**Additional gap:** `run_nhl_model.js` and `run_nba_model.js` header comments claim to write `model_outputs` but neither imports nor calls `insertModelOutput`. The comment is misleading.

**Impact:**
- Every MLB model run writes prediction records invisibly. The table grows without bound.
- As-built, the `model_outputs` data layer provides zero value. It is pure overhead.

---

## Decision required before implementation

**Option A â€” Wire (preferred if prediction audit is desired)**
Build a `/api/model-outputs` route and display per-model prediction history in the admin panel or results dashboard.

**Option B â€” Remove (preferred if `card_payloads` is sufficient)**
Remove `insertModelOutput()` calls from `run_mlb_model.js` and `run_nfl_model.js`, correct the misleading header comments in `run_nhl_model.js` and `run_nba_model.js`, and mark the getter functions as deprecated (or delete them).

Document the decision as a comment in `packages/data/src/db/models.js`.
