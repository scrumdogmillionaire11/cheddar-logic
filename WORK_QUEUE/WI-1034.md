# WI-1034 - Design Brief: Verification as a Blocker-Resolution System

> **This is a design brief, not an executable work item.**
> Execution is split into: WI-1034-a (doc layer), WI-1034-b (NHL worker), WI-1034-c (MLB + watchdog).
> Do not claim this file. Claim the child WIs.

## Problem

Every "Verification Required" state in the system currently means "stop" - it does not encode what is missing, what event clears it, how to fetch it, when to retry, or what to do if it never clears. That produces dead-card noise and human babysitting.

The NHL model already implements the core pattern: `GATE_GOALIE_UNCONFIRMED` -> `HOLD` -> retry via `applyExecutionGateWithStaleRecoveryToNhlCard`. This design brief formalizes that pattern into a cross-sport contract.

## Code Taxonomy Alignment

Before introducing any new codes, verify against the canonical taxonomy in `packages/models/src/decision-pipeline-v2.js`. That file calls `_assertPipelineCodesRegistered()` at startup - unregistered codes throw.

### Existing codes that cover new concepts (do NOT create duplicates)

| New concept | Use existing code | Registry |
|---|---|---|
| Injury status blocking a bet | `INJURY_UNCERTAIN` | `WATCHDOG_REASONS` |
| Market moved against thesis | `LINE_MOVE_ADVERSE` | `PRICE_REASONS` |
| Model needs repricing at new inputs | `EDGE_RECHECK_PENDING` | `PRICE_REASONS` |

### New codes genuinely required (not covered by existing codes)

| New code | Registry | Semantic distinction |
|---|---|---|
| `STARTER_UNCONFIRMED` | `WATCHDOG_REASONS` | SP/goalie not yet confirmed - no mismatch yet, just pending |
| `STARTER_MISMATCH` | `WATCHDOG_REASONS` | SP/goalie confirmed but differs from priced assumption |
| `PRICE_STALE` | `PRICE_REASONS` | Signal price and current best-available diverge beyond drift threshold |

Note: `GOALIE_UNCONFIRMED` already exists in `WATCHDOG_REASONS` and covers the NHL goalie case. `STARTER_UNCONFIRMED` covers the MLB SP equivalent. These are parallel codes, not duplicates.

## Core Contract

Every provisional candidate must carry a `CandidateVerification` object before audit or output:

```ts
type VerificationState = "NOT_REQUIRED" | "PENDING" | "CLEARED" | "FAILED" | "EXPIRED";

type VerificationBlockerCode =
  // New codes (to be registered in WI-1034-b)
  | "STARTER_UNCONFIRMED"
  | "STARTER_MISMATCH"
  | "PRICE_STALE"
  // Existing codes (already registered - use string values directly)
  | "INJURY_UNCERTAIN"        // maps to WATCHDOG_REASONS.INJURY_UNCERTAIN
  | "LINE_MOVE_ADVERSE"       // maps to PRICE_REASONS.LINE_MOVE_ADVERSE
  | "EDGE_RECHECK_PENDING"    // maps to PRICE_REASONS.EDGE_RECHECK_PENDING
  // Soft blockers (new, registered in WI-1034-b)
  | "BEST_LINE_UNCONFIRMED"
  | "WEATHER_STATUS_PENDING"
  | "MARKET_SOURCE_UNCONFIRMED";

type VerificationActionType =
  | "FETCH_MARKET_SNAPSHOT"
  | "FETCH_BEST_LINE"
  | "FETCH_STARTER_STATUS"
  | "FETCH_LINEUP_STATUS"
  | "FETCH_WEATHER_STATUS"
  | "REPRICE_MODEL"
  | "RECHECK_MOVEMENT_WINDOW"
  | "MARK_EXPIRED";

interface VerificationRequirement {
  blocker_code: VerificationBlockerCode;
  severity: "HARD" | "SOFT";
  status: "PENDING" | "CLEARED" | "FAILED";
  unblock_condition: string;
  action_type: VerificationActionType;
  source_needed: string;
  retry_policy: {
    mode: "ONCE" | "UNTIL_START" | "WINDOWED";
    interval_minutes?: number;
    stop_at?: string;
  };
  resolved_value?: string | number | boolean;
  resolved_at?: string;
  failure_reason?: string;
}

interface CandidateVerification {
  verification_state: VerificationState;
  requirements: VerificationRequirement[];
  can_promote_to_play: boolean;
  next_action_summary: string | null;
}
```

## Blocker Taxonomy

### Hard blockers (prevent PLAY)
| Blocker | Action | Clear condition |
|---|---|---|
| `PRICE_STALE` | `FETCH_MARKET_SNAPSHOT`, `FETCH_BEST_LINE` | Best price within configured drift threshold (default 5 cents) or model repriced and edge survives |
| `STARTER_UNCONFIRMED` | `FETCH_STARTER_STATUS` | Official starter/goalie posted and matches priced assumption |
| `STARTER_MISMATCH` | `REPRICE_MODEL` | Repriced edge still clears threshold at actual starter |
| `LINE_MOVE_ADVERSE` | `RECHECK_MOVEMENT_WINDOW` + `REPRICE_MODEL` | Move stabilizes and repriced edge survives |
| `EDGE_RECHECK_PENDING` | `REPRICE_MODEL` | Repriced edge clears threshold |
| `INJURY_UNCERTAIN` | `FETCH_LINEUP_STATUS` | Injury clears or repriced edge survives confirmed absence |

### Soft blockers (cap at LEAN)
| Blocker | Action | Clear condition |
|---|---|---|
| `BEST_LINE_UNCONFIRMED` | `FETCH_BEST_LINE` | Best available line captured and edge survives there |
| `WEATHER_STATUS_PENDING` | `FETCH_WEATHER_STATUS` | Status no longer materially alters projection |
| `MARKET_SOURCE_UNCONFIRMED` | `FETCH_MARKET_SNAPSHOT` | Source confirmed and price validated |

## Audit Level Definitions

### GATE_CHECK
Three blocking checks run in order before any analysis. Stop on first failure for user-facing output.
On failure: emit `PASS - [REASON_CODE]: [one sentence].` and stop - no further output.
On all three passing: emit `GATE_CHECK: CLEAR` and proceed to STANDARD_AUDIT.

Internal-state requirement:
- User-facing output remains first-fail only.
- Resolver and worker internals should collect all currently detectable blockers in the same pass when checks are cheap, for telemetry/debugging and retry prioritization.

The three checks:
1. **Price freshness**: current best-available line is within the configured drift threshold (default 5 cents) of the signal price. Fail code: `PRICE_STALE`.
2. **Starter confirmed** (MLB/NHL only): starting pitcher or goalie is confirmed and matches who the model priced. Fail code: `STARTER_UNCONFIRMED` or `STARTER_MISMATCH`.
3. **Line movement direction**: market has not moved against the signal side by more than the configured adverse-move threshold (default 5 cents) since signal generation. Fail code: `LINE_MOVE_ADVERSE`.

Threshold policy:
- Defaults in this brief are 5 cents for examples and initial rollout.
- Runtime thresholds should be configurable by sport and market in implementation.

### STANDARD_AUDIT
The 8-step analysis defined in `bet_review.md § Audit Sequence`. Reached only after `GATE_CHECK: CLEAR`. Produces thesis summary, edge summary, contradictions, missing data, risk notes, and a final verdict.

Boundary rule:
- `STANDARD_AUDIT` consumes post-gate candidate state and must not independently recreate or reinterpret `GATE_CHECK` failures unless new data arrives through resolver flow.

## Verdict Contract

VEGAS verdicts are: `PLAY` / `LEAN` / `PASS` / `FADE`.

`HOLD` is a worker-layer action that surfaces as `LEAN` in the audit layer.
User-facing output for a pending verification state uses `LEAN`, not `HOLD`:

```
LEAN - Verification pending
Waiting on:
- [unblock_condition per requirement]
Auto-recheck: every Nm until T-Xm
```

Internal: full structured `CandidateVerification` object.

Required semantic guardrail:
- `LEAN` must never be interpreted alone for verification status.
- `verification_state` is required companion context:
  - `LEAN + verification_state=PENDING` = candidate blocked pending verification
  - `LEAN + verification_state=CLEARED|NOT_REQUIRED` = true low-confidence lean

Reporting and analytics must segment by `verification_state` so unresolved pending candidates are not blended with true weak-edge leans.

## Promotion Rules

- `PENDING -> CLEARED`: all HARD blockers resolved
- `CLEARED` never implies `PLAY` by itself
- `CLEARED -> PLAY`: only after re-running threshold logic and confirming edge/policy thresholds still pass
- `PENDING -> FAILED`: any HARD blocker resolves against thesis
- `PENDING -> EXPIRED`: start time or retry window reached without resolution

## Resolver Terminal State Output Contract

For each terminal state, the resolver emits:
- `CLEARED` -> trigger STANDARD_AUDIT; pass `CandidateVerification` to audit layer
- `FAILED` -> emit `PASS - [blocker_code]: [failure_reason].`; no further processing
- `EXPIRED` -> emit `PASS - EXPIRED: Verification window closed without resolution.`; archive candidate

## Watchdog Enforcement Rules

Status note: this section is target-state behavior specification for downstream implementation. Enforcement lands in WI-1034-b and WI-1034-c, not in WI-1034-a doc-only work.

- `Verification Required` with `requirements.length === 0` -> fatal contract violation
- `LEAN` (verification pending) with no `next_action_summary` -> invalid
- `PENDING` past expiry still surfaced as actionable -> invalid stale candidate
