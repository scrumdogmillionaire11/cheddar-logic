# WI-1226: Reroute NO_PRIMARY_SUPPORT Out of classifyPrice to Correct Terminal Family

**ID**: WI-1226
**Goal**: Move the `hasPrimarySupport` check out of `classifyPrice()` so that `NO_PRIMARY_SUPPORT` routes to `EDGE_INSUFFICIENT` (not `PRICING_UNAVAILABLE`) and is visible as a model signal failure rather than a market data failure, while also closing the adjacent promotion regression where `EDGE_SANITY_NON_TOTAL` high-edge gates can be promoted back to `PLAY`.

**Context**:

`classifyPrice()` in `packages/models/src/decision-pipeline-v2.js` currently handles missing driver support as a pricing outcome:

```js
if (!hasPrimarySupport) {
  return {
    sharp_price_status: 'UNPRICED',
    price_reason_codes: [PRICE_REASONS.NO_PRIMARY_SUPPORT],
    proxy_capped: false,
  };
}
```

Because `sharp_price_status: 'UNPRICED'` is returned, `resolveTerminalReasonFamily()` routes the card to `PRICING_UNAVAILABLE` — the same family as missing odds or missing model probability. This is wrong: "no drivers available" is a model signal failure, not a market data failure. The correct terminal family is `EDGE_INSUFFICIENT`.

Downstream effects of the current misrouting:
- Cards with no drivers read as "waiting on market data" to any consumer that branches on `PRICING_UNAVAILABLE`.
- `PRICING_UNAVAILABLE` is the family watched for WATCH-state recovery logic; driver-absent cards could be incorrectly held in WATCH when they should be PASS-terminal.

**Proposed fix**: Lift the `hasPrimarySupport` check into `buildDecisionV2()` as a pre-flight before `classifyPrice()` is called. When `hasPrimarySupport` is false, set `priceDecision` to a synthetic result with `sharp_price_status: 'NO_SUPPORT'` (new status) and `price_reason_codes: ['NO_PRIMARY_SUPPORT']`. Update `resolveTerminalReasonFamily()` to route `NO_SUPPORT` to `EDGE_INSUFFICIENT`.

**Adjacent regression to close in same file lane**:
- `WI-1228` introduced high-end `LEAN -> PLAY` promotion for selected markets.
- The promotion helper currently ignores the `EDGE_SANITY_NON_TOTAL` gate signal.
- Result: a high-edge non-TOTAL card with only moderate support can correctly classify as `LEAN` first, then get promoted back to `PLAY`, defeating the edge-sanity guard intent and failing `decision-pipeline-v2-edge-sanity-gate.test.js`.
- Fix: treat `EDGE_SANITY_NON_TOTAL` as promotion-blocking so direct `PLAY` outcomes with strong support still survive, but gated `LEAN` outcomes do not promote through the sanity check.
