# WI-1017: Exclude no_contest from avg_pnl_per_card denominator; lock win-rate denominator as regression test

**ID**: WI-1017
**Goal**: After WI-1007 ships `no_contest` as a first-class settlement result, ensure: (1) `avg_pnl_per_card` in `tracking_stats` uses `wins + losses + pushes` as its denominator (not `COUNT(*)` which silently includes no_contest rows), and (2) a regression test locks the win-rate formula as `wins / (wins + losses)` so push and no_contest can never inflate it.

> **Note on scope:** Win-rate denominators are already correct everywhere (`wins / (wins + losses)`). The actual data-integrity bug is in `avg_pnl_per_card` / ROI denominators. This WI fixes the bug and adds the regression test that was missing.

**Depends on**: WI-1007 (F5 ML settlement must be live so `no_contest` rows exist in `projection_audit` to test against)
**Release classification**: required before WI-1007 results are surfaced in any analytics or reporting view

**Scope**:

- `WORK_QUEUE/WI-1017.md`
- `packages/data/src/db/tracking.js` — `recomputeTrackingStats` (Query A + Query B): fix `avgPnl = totalPnl / COUNT(*)` to use `wins + losses + pushes` as denominator
- `packages/data/src/db/tracking.js` — `incrementTrackingStat`: fix `avg_pnl_per_card` SQL CASE to use `wins + losses + pushes` (not `settled_cards`) as denominator
- `apps/worker/src/audit/performance_drift_report.js` — verify `metricRows.length` excludes `no_contest` rows from ROI denominator; add explicit no-op branch

- `packages/data/src/__tests__/tracking.test.js` — create if absent; add denominator regression tests

**Acceptance**:

- `recomputeTrackingStats` (both Query A and Query B in `tracking.js`) computes `avgPnl = totalPnl / (wins + losses + pushes)` — not `totalPnl / COUNT(*)`.
- `incrementTrackingStat` SQL CASE for `avg_pnl_per_card` divides by `wins + losses + pushes` — not `settled_cards`.
- `performance_drift_report.js` ROI denominator (`metricRows.length`) either excludes `no_contest` rows or the code documents why they cannot reach `metricRows`.
- `web/src/app/api/results/route.ts` result counter loop has an explicit `no_contest` branch (even if a no-op skip) so behavior is documented, not accidental.

- A test proves: seed W wins, L losses, P pushes, N no_contests with known `pnl_units` → assert `avg_pnl_per_card = totalPnl / (W + L + P)` unaffected by N.

## Implementation Steps

### Step 1: DONE — Scope pre-investigated

Skip the discovery grep commands. Use the Summary table above as the source of truth for what to fix.

### Step 2: Fix `recomputeTrackingStats` avg_pnl denominator (`tracking.js`)

In **both** Query A (market-level, ~line 472) and Query B (verification-segmented, ~line 524) inside `recomputeTrackingStats`:

Change:
```js
const avgPnl = total > 0 ? totalPnl / total : 0;
```
To:
```js
const decided = wins + losses + pushes;
const avgPnl = decided > 0 ? totalPnl / decided : 0;
```

The `total = COUNT(*)` variable is still needed for `totalCards`/`settledCards` — keep it. Only the `avgPnl` divisor changes.

### Step 3: Fix `incrementTrackingStat` avg_pnl SQL CASE (`tracking.js`)

The incremental UPDATE currently divides `avg_pnl_per_card` by `settled_cards + deltaTotal`. Change the SQL CASE to divide by `wins + losses + pushes` after the increment:

```sql
avg_pnl_per_card = CASE
  WHEN (wins + ? + losses + ? + pushes + ?) > 0
  THEN (total_pnl_units + ?) / (wins + ? + losses + ? + pushes + ?)
  ELSE 0
END,
```

Bind params in order: `deltaWins, deltaLosses, deltaPushes, deltaPnl, deltaWins, deltaLosses, deltaPushes` — reusing the delta params already available in the function.

### Step 4: Verify ROI denominator in `performance_drift_report.js`

Locate the row loop that builds `metricRows` and the line `pnlSum / metricRows.length`. Determine no_contest reachability:

- **If** no_contest rows have `pnl_units = null` → `Number.isFinite(row.pnl_units)` is false → `roiAvailable = false` → ROI already suppressed → add a comment documenting this, no code change.
- **If** no_contest rows could have `pnl_units = 0` (finite) → add filter to exclude `result` values outside `['WIN', 'LOSS', 'PUSH']` before contributing to `metricRows`.

Either way, add an explicit no-op branch to the result counter loop:
```js
else if (row.result === 'NO_CONTEST') { /* excluded from all denominators */ }
```

### Step 5: Add explicit no_contest branch in `web/src/app/api/results/route.ts`

In the result counter loop (around line 853), add:
```ts
else if (row.result === 'no_contest') { /* excluded: not counted in win_rate or avg_pnl denominators */ }
```

This converts silent drop to documented exclusion. No behavioral change.

### Step 6: Add denominator regression tests

Create `packages/data/src/__tests__/tracking.test.js` (or extend if it exists). Seed an in-memory sql.js DB via the existing test helpers.

**Test 1 — win_rate lock:**
```
seed: 3 wins, 2 losses, 4 pushes, 5 no_contests (pnl_units = 0, result = 'no_contest')
call: recomputeTrackingStats({ fullReplace: true })
assert: win_rate = 3 / 5 = 0.6
assert: pushes = 4 (not 9)
assert: wins = 3, losses = 2
```

**Test 2 — avg_pnl_per_card denominator:**
```
seed: 3 wins (pnl_units=1), 2 losses (pnl_units=-1), 4 pushes (pnl_units=0), 5 no_contests (pnl_units=0)
call: recomputeTrackingStats({ fullReplace: true })
assert: avg_pnl_per_card ≈ (3 - 2) / (3 + 2 + 4) = 1/9 ≈ 0.111
assert NOT: avg_pnl_per_card ≈ 1/14  ← would indicate no_contests in denominator
```
