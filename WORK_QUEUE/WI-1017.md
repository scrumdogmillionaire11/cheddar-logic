# WI-1017: Update MLB settlement analytics to exclude push and no_contest from win-rate denominator

**ID**: WI-1017
**Goal**: Ensure analytics queries that compute MLB win-rate use `wins / (wins + losses)` — not `wins / total_plays` — so that `push` and `no_contest` results introduced by WI-1007 F5 ML settlement do not silently distort reported performance.

**Depends on**: WI-1007 (F5 ML settlement must be live so push/no_contest rows exist to test against)
**Release classification**: required before WI-1007 results are surfaced in any analytics or reporting view

**Scope**:

- `WORK_QUEUE/WI-1017.md`
- Analytics/reporting file(s) that own win-rate and ROI queries (identify in Step 1)
- Tests covering push and no_contest denominator exclusion

**Out of scope**:

- F5 ML settlement logic (owned by WI-1007)
- Any settlement file changes
- Adding new analytics metrics beyond denominator correctness

**Acceptance**:

- Win-rate query uses `wins / (wins + losses)` — confirmed by inserting push and no_contest rows into a test dataset and asserting the denominator is unchanged.
- ROI query excludes push and no_contest rows from the denominator.
- Push rows have `result = 'push'` in DB and are excluded from both numerator and denominator.
- `no_contest` rows have `result = 'no_contest'` in DB and are excluded from both numerator and denominator.
- A test proves: insert N push rows + M no_contest rows alongside W wins and L losses; assert `win_rate = W / (W + L)` regardless of N and M.

**Owner agent**: unassigned
**Time window**: unscheduled (schedule after WI-1007 is complete)
**Coordination flag**: solo

**Tests to run**:

- Determine during Step 1 — depends on which analytics file is in scope

**Manual validation**:

- Insert push and no_contest rows into a local DB with known win/loss counts.
- Run the analytics query directly.
- Confirm denominator equals `wins + losses` only, not `wins + losses + pushes + no_contests`.

**CLAIM**: unassigned

## Summary

WI-1007 introduces `push` and `no_contest` as first-class settlement results. The settlement layer contract (WI-1007) is complete when those values are stored correctly. This WI closes the downstream gap: ensuring reporting queries treat them correctly.

The risk without this WI: `push` rows silently inflate the denominator, making win-rate appear lower than it is. `no_contest` rows would do the same.

## Implementation Steps

### Step 1: Identify the analytics file(s) in scope

Find the query or function that computes win-rate for MLB cards:

```bash
grep -r "win_rate\|wins.*losses\|total_plays\|win.*percent\|settlement.*summary" \
  apps/worker/src --include="*.js" -l
grep -r "win_rate\|wins.*losses\|total_plays" \
  web/src --include="*.ts" --include="*.tsx" -l
```

Add the owning file(s) to the `Scope` field above before writing any code.

### Step 2: Confirm current denominator

Read the existing query. Determine whether it currently uses:
- `wins / total_plays` (broken — needs fix)
- `wins / (wins + losses)` (correct — add a test to lock it)
- Something else (document and decide)

### Step 3: Fix or confirm the denominator

If broken: change to `wins / (wins + losses)` filtering out `result NOT IN ('push', 'no_contest', 'void', NULL)` from the denominator.

If correct: add a regression test that fails if the denominator ever includes push or no_contest rows.

### Step 4: Add push/no_contest exclusion test

Write a unit or integration test that:
1. Seeds W wins, L losses, P pushes, N no_contests
2. Calls the win-rate function
3. Asserts result equals `W / (W + L)` — not affected by P or N
