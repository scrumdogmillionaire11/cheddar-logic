# WI-1276: Collapse /results Initial Double Fetch

**ID**: WI-1276
**Goal**: Remove duplicate backend work on initial `/results` load by collapsing the summary+ledger two-call bootstrap into one measured request path without changing API contract.

**Scope**:
- `web/src/app/results/page.tsx`
- `web/src/app/api/results/route.ts`
- `web/src/lib/results/query-layer.ts`
- `web/src/__tests__/integration/results-behavioral-contract.test.ts`
- `web/src/__tests__/integration/results-initial-load-fetch-count.test.ts` (new)

**Out of scope**:
- Any change to `/api/results` response schema
- Any change to settlement logic or betting/model behavior
- Any UI redesign beyond minimal loading-state adjustments

**Acceptance**:
- Initial `/results` render performs one primary `/api/results` request for summary+ledger.
- Existing filters and ledger-limit behavior remain intact.
- API payload shape remains backward-compatible.
- New test asserts initial-load request count and data parity.

**Owner agent**: unassigned
**Time window**: 3-5 hours
**Coordination flag**: needs-sync
**Depends on**: none

**Tests to run**:
- `npm --prefix web exec tsx src/__tests__/integration/results-behavioral-contract.test.ts`
- `npm --prefix web exec tsx src/__tests__/integration/results-initial-load-fetch-count.test.ts`

**Manual validation**:
- Load `/results` with browser network tab open and confirm one initial `/api/results` call.
- Toggle filters and confirm summary and ledger stay consistent with pre-change behavior.

**Failure modes**:
- Hidden second fetch still fires due to effect dependency churn.
- Single-call path increases TTFB too much for large ledgers.
- Regression in loading/error state ordering on slow networks.
