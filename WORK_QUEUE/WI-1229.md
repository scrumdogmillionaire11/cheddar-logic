# WI-1229: Audit Historical Display-Log Misses and Add Visibility Integrity Monitoring

**ID**: WI-1229
**Goal**: Preserve a one-time record of known historical `DISPLAY_LOG_NOT_ENROLLED` rows, add ongoing monitoring for new display-eligible `Play` / `Slight Edge` rows that miss `card_display_log`, and document a proof-gated remediation path without authorizing blanket backfill.

**Context**:

WI-1232 (formerly WI-1228 before renumbering on 2026-05-03) fixed the write-path bug where `card_display_log` kept a single winner per `game_id + run_id`, which silently dropped legitimate sibling actionable cards before `/results` could ever track them. That fix is forward-looking. The remaining follow-up is operational:

- Historical misses should be written down once so they are explicitly known-bad history rather than silent anomalies.
- Future misses should trigger an operator-visible signal if a display-eligible modern row is created without `card_display_log` enrollment.
- Any historical remediation must remain proof-gated. A row being `Play` or `Slight Edge` is not sufficient proof that it was actually shown live.

This WI is audit/observability/runbook work only. It must not change model thresholds, decision production, settlement strictness, `/api/games` authority rules, or `/results` inclusion rules.

**Out of scope**:

- Any change to how `Play`, `Slight Edge`, or `Pass` is decided
- Any relaxation of the canonical `decision_v2` requirement for modern `/wedge` rows
- Any blanket `card_display_log` backfill for historical rows
- Any change that allows main `/results` to include settled odds-backed rows without display-log proof
- Any mixing of projection-only families into main `/results`

**Acceptance**:

- A one-time audit artifact exists under `docs/audits/` and records:
  - the audit window/date
  - total actionable recent rows
  - counts for `ENROLLED`, `PROJECTION_ONLY`, `NOT_DISPLAY_ELIGIBLE`, and `DISPLAY_LOG_NOT_ENROLLED`
  - representative card IDs for each bucket
  - a note that historical `DISPLAY_LOG_NOT_ENROLLED` rows are known-bad history unless separate live-display proof exists
- Settlement or pipeline health reporting exposes a dedicated visibility-integrity section/query for new display-eligible actionable rows missing `card_display_log`, including count and sample IDs
- The monitoring path can alert when the count is non-zero; zero remains quiet
  - explicit alert contract: when visibility-integrity missing-enrollment count > 0, `apps/worker/src/jobs/check_pipeline_health.js` emits `VISIBILITY_INTEGRITY_ALERT` with count + sample IDs and exits non-zero
  - explicit quiet contract: when count = 0, no `VISIBILITY_INTEGRITY_ALERT` marker is emitted and process exits zero
- A runbook exists under `docs/runbooks/` that defines:
  - acceptable forms of live-display proof
  - the exact SQL/command sequence for proof-gated backfill of specific rows only
  - an explicit prohibition on bulk backfill
- No live authority path changes are introduced
- No threshold changes are introduced
