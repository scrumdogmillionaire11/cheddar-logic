# WI-0780: Decompose schedulers/main.js into per-sport schedule config modules

| Field              | Value |
|--------------------|-------|
| **ID**             | WI-0780 |
| **Goal**           | Break `apps/worker/src/schedulers/main.js` (1,331 lines) into per-sport schedule config modules, leaving `main.js` as a thin orchestrator that imports and merges them — following the existing pattern established by `schedulers/fpl.js` and `schedulers/player-props.js` |
| **Priority**       | Medium |
| **Owner agent**    | Unassigned |
| **Time window**    | TBD |
| **Coordination flag** | `needs-sync` — modifies `apps/worker/src/schedulers/main.js` |

---

## Problem

`apps/worker/src/schedulers/main.js` registers all sports (NHL, NBA, MLB, NFL, Soccer, NCAAM), all job types (ingestion, model runs, settlement, reporting, health checks), and all window/T-minus logic in a single 1,331-line file. The `fpl.js` and `player-props.js` sub-schedulers show this can be decomposed — but only those two have been extracted.

**Impact:**
- Adding a new sport requires touching a 1,300-line file with no isolation
- Per-sport scheduling tests must import and run the entire scheduler
- T-minus window logic is duplicated inline rather than derived from a shared config object
- Sports that share no games (e.g. MLB vs NHL) have their job registrations co-mingled, making season-gating harder to read and test

---

## Scope

| File | Change |
|------|--------|
| `apps/worker/src/schedulers/nhl.js` | New — extract NHL job registrations and T-minus window config from `main.js`; export `computeNhlDueJobs(windows, games)` |
| `apps/worker/src/schedulers/nba.js` | New — extract NBA job registrations; export `computeNbaDueJobs(windows, games)` |
| `apps/worker/src/schedulers/mlb.js` | New — extract MLB + MLB F5 job registrations; export `computeMlbDueJobs(windows, games)` |
| `apps/worker/src/schedulers/nfl.js` | New — extract NFL model registration; export `computeNflDueJobs(windows, games)` |
| `apps/worker/src/schedulers/settlement.js` | New — extract settlement, backfill, and reporting job registrations (sport-agnostic); export `computeSettlementDueJobs(windows, games)` |
| `apps/worker/src/schedulers/main.js` | Slim to: env setup, tick loop, `getUpcomingGames()`, window computation, then calls each sub-scheduler's `compute*DueJobs` and merges results |
| `apps/worker/src/schedulers/__tests__/` | Update or add test files for each new sub-scheduler; existing `main.js` tests must still pass |

## Out of Scope

- Changing scheduling cadence, window definitions, or T-minus offsets
- Changing any job implementations under `src/jobs/`
- FPL and player-props sub-schedulers (already extracted — do not touch)
- Adding new jobs to the scheduler

---

## Acceptance

- `wc -l apps/worker/src/schedulers/main.js` is under 300 lines after decomposition
- Each per-sport file exports its `compute*DueJobs` function
- `npm run scheduler:dev` runs without error
- `npm run test:scheduler:windows` passes
- Existing jobs register and fire at the same windows as before (verified by DRY_RUN output comparison before/after)
