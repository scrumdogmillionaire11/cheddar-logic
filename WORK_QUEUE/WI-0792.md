## Context

WI-0761 shipped the Model Health Dashboard at `/admin`. It currently shows:

- **Pipeline Health** — recent `pipeline_health` rows (phase / check / status / reason / timestamp)
- **Recent Model Outputs** — last 50 rows from `model_outputs`

The page is functional but deliberately minimal. Two open questions need design decisions before further investment:

1. **Log presentation** — Raw rows in reverse-chron is a start, but it may not surface signal well.
2. **Alert reaction** — `pipeline_health` rows with `status=warning|failed` exist, but there is no defined playbook for what a human should do when they see one.

---

## Brainstorm Prompts

### Log Presentation

- Should rows be grouped by phase or flat reverse-chron?
- Auto-refresh vs manual reload — health check runs every 5 min
- Show only the **latest** row per phase (current-state summary) vs full history?
- Distinguish first-occurrence vs repeat failures — how long has a phase been failing?
- Color-code rows by age: `failed` from 3 hours ago is more urgent than 30 sec ago
- Trend sparkline per phase (ok / warning / failed over last N checks)?
- Surface `job_runs` table too? (last run time, duration, success/failure per job key)
- Consider a dedicated 'Current Health Snapshot' card (1 row per phase = latest state) above the full history table

### Alert Reaction Playbook

Proposed SLAs:

| Phase | Check | warning SLA | failed SLA | Recovery |
|---|---|---|---|---|
| schedule | freshness | monitor | 24h — re-run `pull_schedule.js` | |
| odds | freshness | monitor | 15 min — check `pull_odds.js` logs, re-run | |
| cards | freshness | monitor | 30 min — identify stalled model job, re-run | |
| settlement | backlog | 1h | 1h — check `settle_game_results.js` + game status | |
| mlb | f5_market_availability | informational | monitor 1 day — check F5 line presence in `pull_odds.js` | |

Open questions:
- Should `ENABLE_PIPELINE_HEALTH_WATCHDOG=true` + Discord emit (deferred from WI-0761) replace manual page-checking as the primary alert channel?
- What threshold triggers 'needs immediate action' vs 'check tomorrow'?
- Should sustained `failed` (N consecutive checks) escalate differently than a one-off?
