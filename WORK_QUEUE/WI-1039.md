# WI-1039: Discord Webhook Optimization — Market Filtering, POTD Integration, Rate-Limit Retry

**ID**: WI-1039

**Goal**: Make it easy to add/remove markets from Discord hooks at runtime, surface POTD play results in the correct channels, and fix the known 429 rate-limit failures in production.

---

## Background & Motivation

Three problems discovered during brainstorm (Apr 19 2026):

1. **Market filter UX is bad**: `DISCORD_CARD_WEBHOOK_MARKETS` works but is opaque — no log of active filters, no deny-list mode, and env parsing runs per-card instead of once per run. Debugging "why is nothing posting?" is painful.
2. **POTD is invisible on no-pick days**: All four `noPlay` paths in `run_potd_engine.js` return silently. Operators can't distinguish "no play fired" from "job didn't run" and have no context on how far off the slate was.
3. **Rate-limit retries are missing**: Production logs show 429 errors. The `retry_after` field is in the response body but `sendDiscordMessages` does not use it.

---

## Agent directive (summary)

Implement this WI with the following rules:

1. Add normalized filter logging and a zero-card warning when filters eliminate all output.
2. Improve no-pick alert to show required vs actual edge and highest edge on the slate.
3. Prevent NO_PICK POTD entries from appearing in snapshot feed to avoid duplicate messaging.
4. Ensure POTD cards render as a distinct section when included in snapshot — never mixed with market cards.
5. Add jitter to 429 retry sleep and log each retry attempt.
6. Add a single "engine run complete" heartbeat log per POTD execution.

---

### Sub-task A — Market filter hygiene

**Files**:

- `apps/worker/src/jobs/post_discord_cards.js`
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js`
- `env.example`

**Changes**:

1. **Parse filters once per run, not per card**

   In `buildDiscordSnapshot`, resolve `DISCORD_CARD_WEBHOOK_MARKETS`, `DISCORD_CARD_WEBHOOK_SPORTS`, and `DISCORD_CARD_WEBHOOK_BUCKETS` into a `filters` object once at the top of the function. Pass `filters` into `cardMatchesWebhookFilters(card, bucket, filters)` instead of re-reading `process.env` on every card.

2. **Add deny-list mode via `DISCORD_CARD_WEBHOOK_MARKETS_DENY`**

   Parse new env var `DISCORD_CARD_WEBHOOK_MARKETS_DENY` (CSV) — same token normalization as the allow-list. If set, cards whose market tag is in the deny set are excluded when no allow-list is configured. Allow-list takes precedence if both are set (deny is ignored when allow-list is active). Document in `env.example`.

3. **Startup log of active filters**

   At the top of `postDiscordCards()` (after the enabled/URL guards), emit one log line:

   ```
   [post-discord-cards] Filters — sports:<value|all> markets:<value|all> buckets:<value|all> deny:<value|none>
   ```

4. **Normalized filter log after parsing**

   After resolving the filters object in `buildDiscordSnapshot`, log the resolved token sets:

   ```
   [post-discord-cards] Normalized filters → markets:[TOTAL, ML] deny:[SHOTS]
   ```

   Emitted once per snapshot build, not per card. If all filters are unset, log `markets:[all] deny:[none]`. This prevents "why is nothing posting?" confusion from normalization mismatches.

5. **Zero-card warning**

   After applying filters, if the filtered card set is empty, log:

   ```
   [post-discord-cards] WARNING — 0 cards matched filters (markets:<value> sports:<value> buckets:<value>)
   ```

6. **`KNOWN_MARKET_TAGS` constant**

   Export a frozen `KNOWN_MARKET_TAGS` array matching the branches in `normalizeMarketTag`:
   `['TOTAL', 'ML', 'Spread', '1P', 'TEAM TOTAL', 'TSOA', 'ANYTIME', 'SOT', 'SHOTS', 'PROP', 'POTD']`.

   At startup, if `DISCORD_CARD_WEBHOOK_MARKETS` is set, `console.warn` for any token not in `KNOWN_MARKET_TAGS`. Do not crash. Add to `env.example` comment with the full valid token list.

**Acceptance**:

- All three `parseCsvTokens(process.env.DISCORD_CARD_WEBHOOK_*)` calls inside `cardMatchesWebhookFilters` are removed; filters object is resolved once per `buildDiscordSnapshot` call.
- Setting `DISCORD_CARD_WEBHOOK_MARKETS_DENY=SHOTS` excludes SHOTS cards when no allow-list is set.
- Setting both allow-list and deny-list: allow-list wins, deny is ignored.
- Startup log is emitted every time `postDiscordCards()` runs (after enabled/URL guards).
- Normalized filter log is emitted once per `buildDiscordSnapshot` call.
- When filtered card set is empty, the zero-card warning is logged (not a throw).
- Unknown market token in `DISCORD_CARD_WEBHOOK_MARKETS` produces `console.warn`, not a crash.

#### B0 — Engine heartbeat (every run)

At the end of every `runPotdEngine` execution — regardless of outcome — emit a single structured heartbeat immediately before each `return`:

```
[POTD] Engine run complete — candidates:72 viable:0 status:NO_PICK
[POTD] Engine run complete — candidates:14 viable:3 status:FIRED play:NBA TOTAL OVER 227.5
```

Distinguishes "no play found" from "job never ran." Pass `{ candidatesCount, viableCount, status, selectionLabel }` as context. All existing `return` paths must emit this before returning.

#### B1 — POTD no-pick → `DISCORD_ALERT_WEBHOOK_URL`

All four `noPlay` return paths in `runPotdEngine` currently return silently. Add a `sendPotdNopickAlert()` helper in `run_potd_engine.js` that:

- Posts to `DISCORD_ALERT_WEBHOOK_URL` (not `DISCORD_POTD_WEBHOOK_URL`)
- Only fires if `DISCORD_ALERT_WEBHOOK_URL` is set — silent if unset (no crash)
- Uses the existing `sendDiscordMessages` import already in the file

**Message format**:

```
⚠️ POTD — No Pick (Apr 19)

Reason: confidence_below_high_gate

Top candidate:
NBA TOTAL OVER 227.5
Edge: +3.2% (below required 4.0%)
Score: 0.28 (LOW)

Highest qualifying edge today: +1.4%
Required minimum edge: 2.0%

Candidates scored: 8
Viable: 2
```

Rules for the message:

- Always show actual edge AND required edge — makes "how far off was the slate?" immediately obvious
- "Highest qualifying edge" = best `edgePct` among all scored candidates (not just viable), for context
- "Required minimum edge" = effective noise floor (`resolveNoiseFloor(sport, marketType, POTD_MIN_EDGE)`); use the lowest floor across candidates when they span multiple sports
- Show `bestCandidate` fields when available; fall back to `topByEdge` fields when `bestCandidate` is null
- `reason` maps to the return path: `no_viable_candidates`, `confidence_below_high_gate`, `zero_wager`, `min_stake_rejected`

Four no-pick paths to hook:

1. `bestCandidate` is null — reason: `no_viable_candidates`
2. `lowConfidenceCandidate` gate — reason: `confidence_below_high_gate`
3. `zero_wager` gate — reason: `zero_wager`
4. `rawWager / bankrollAtPost < POTD_MIN_STAKE_PCT` gate — reason: `min_stake_rejected`

#### B2 — POTD play card in `post_discord_cards` snapshot (opt-in)

Currently `fetchCardsForSnapshot` hard-excludes `potd-call` via SQL. Change to opt-in:

- Add env var `DISCORD_INCLUDE_POTD_IN_SNAPSHOT=false` (default false — backward-compatible)
- When `true`, remove the `AND LOWER(cp.card_type) != 'potd-call'` SQL filter
- Add `'potd'`/`'potd-call'` branch in `normalizeMarketTag` returning `'POTD'`
- When `DISCORD_INCLUDE_POTD_IN_SNAPSHOT=true` AND `DISCORD_POTD_WEBHOOK_URL` is set, `run_potd_engine.js` must suppress its own POTD Discord post to avoid double-posting
- Document in `env.example`

**Critical guard — no NO_PICK cards in snapshot**:

Add a defensive guard: any `potd-call` card with `payloadData.final_play_state !== 'OFFICIAL_PLAY'` must be excluded from snapshot output. The alert channel already handles no-pick; snapshot must never show a no-pick state. No duplicate messaging.

**POTD formatting isolation**:

When `market_tag === 'POTD'`, the card must render in a **distinct leading section**, never grouped with regular market cards. Pull `POTD`-tagged cards out of the per-sport grouping logic and render them first:

```
🔥 POTD
[potd card row]

────────────────────

🏀 NBA | 7:10 PM ET
...
```

**Acceptance**:

- When POTD fires NO_PICK, `DISCORD_ALERT_WEBHOOK_URL` receives the alert with required vs actual edge, highest edge on slate, and reason.
- If `DISCORD_ALERT_WEBHOOK_URL` is unset, no-pick paths remain silent (no crash).
- `DISCORD_INCLUDE_POTD_IN_SNAPSHOT=false` (default): snapshot excludes `potd-call` — existing behavior unchanged.
- `DISCORD_INCLUDE_POTD_IN_SNAPSHOT=true`: `potd-call` OFFICIAL_PLAY cards appear in a distinct leading section, never mixed into per-sport market groups.
- A `potd-call` card with `final_play_state !== 'OFFICIAL_PLAY'` never appears in snapshot.
- When both `DISCORD_INCLUDE_POTD_IN_SNAPSHOT=true` and `DISCORD_POTD_WEBHOOK_URL` are set, the engine skips its own Discord post.
- Heartbeat log is emitted on every `runPotdEngine` return path (all 5: 4 no-pick + 1 fired).

### Sub-task C — Rate-limit retry in `sendDiscordMessages`

**Files**:

- `apps/worker/src/jobs/post_discord_cards.js`
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js`

**Changes**:

Current `sendDiscordMessages` throws on 429. Discord returns `{ retry_after: <seconds> }` in the 429 body.

Add retry logic:

- On 429: parse `retry_after` from the JSON body
- Sleep for `Math.ceil(retry_after * 1000) + random jitter between 50–150ms` — jitter prevents sync collisions when multiple jobs hit rate limit simultaneously
- Log before sleeping: `[post-discord-cards] Rate limited — retrying after ${retryAfterMs}ms (retry 1/1)`
- Retry the single failed message once
- If it 429s again, throw immediately — do not loop
- Add `DISCORD_MAX_RETRY_AFTER_MS=5000` env guard: if `retry_after * 1000 > DISCORD_MAX_RETRY_AFTER_MS`, throw immediately without sleeping — prevents indefinitely long runs when Discord is overloaded
- Log the guard trip: `[post-discord-cards] Rate limit retry_after ${retryAfterMs}ms exceeds guard — failing fast`

**Acceptance**:

- A 429 with `retry_after: 0.3` causes sleep of ~300ms + 50–150ms jitter + one retry before failing.
- A 429 with `retry_after: 10` (> 5s default guard) throws immediately without sleeping.
- A second 429 on the retry throws immediately without a second sleep.
- Log line `Rate limited — retrying after Xms (retry 1/1)` is emitted before each sleep.
- No change to behavior on 2xx or other 4xx/5xx responses.
