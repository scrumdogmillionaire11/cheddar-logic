# WI-1039: Discord Webhook Optimization — Market Filtering, POTD Integration, Rate-Limit Retry

**ID**: WI-1039

**Goal**: Make it easy to add/remove markets from Discord hooks at runtime, surface POTD play results in the correct channels, and fix known 429 failures without creating misleading or duplicate Discord output.

---

## Background & Motivation

Three problems discovered during brainstorm (Apr 19 2026):

1. **Market filter UX is bad**: `DISCORD_CARD_WEBHOOK_MARKETS` works but is opaque. No normalized log, no deny-list mode, and env parsing currently happens per-card.
2. **POTD operational visibility is weak**: No-pick behavior can be misread as scheduler failure unless timing states are explicit.
3. **Rate-limit handling is brittle**: Discord 429 responses include `retry_after`, but current send path fails immediately.

---

## Agent directive (summary)

Implement this WI with the following rules:

1. Add normalized filter logging and a zero-card warning with total pre-filter card count.
2. Improve no-pick alert wording to `Highest edge observed` and show required vs actual edge.
3. Prevent NO_PICK POTD entries from appearing in snapshot feed.
4. Ensure POTD rows render as a distinct section and bypass market filters in snapshot mode.
5. Add jitter to one-time 429 retry and log retry attempt.
6. Add hard cumulative Discord timeout guard.
7. Add heartbeat log per POTD run with timestamp + run id.
8. Implement explicit POTD timing states: `PENDING_WINDOW`, `OFFICIAL_PLAY`, `NO_PICK_FINAL`.
9. Emit no-pick alert only after finalization (`NO_PICK_FINAL`), never during `PENDING_WINDOW`.

### Sub-task A — Market filter hygiene

**Files**:

- `apps/worker/src/jobs/post_discord_cards.js`
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js`
- `env.example`

**Changes**:

1. **Parse filters once per run, not per card**

   In `buildDiscordSnapshot`, resolve `DISCORD_CARD_WEBHOOK_MARKETS`, `DISCORD_CARD_WEBHOOK_SPORTS`, and `DISCORD_CARD_WEBHOOK_BUCKETS` into a single `filters` object. Pass `filters` into `cardMatchesWebhookFilters(card, bucket, filters)`.

2. **Add deny-list mode via `DISCORD_CARD_WEBHOOK_MARKETS_DENY`**

   Parse new CSV env var with same normalization as allow-list. If allow-list is set, deny-list is ignored (allow-list precedence).

3. **Startup log of active filters**

   Emit once per `postDiscordCards()` run after enabled/URL guards:

   ```
   [post-discord-cards] Filters — sports:<value|all> markets:<value|all> buckets:<value|all> deny:<value|none>
   ```

4. **Normalized filter log after parsing**

   Emit once per snapshot build:

   ```
   [post-discord-cards] Normalized filters -> markets:[TOTAL, ML] deny:[SHOTS]
   ```

5. **Actionable zero-card warning**

   If filters drop everything, emit:

   ```
   [post-discord-cards] WARNING — 0 cards matched filters (markets:<value> sports:<value> buckets:<value>)
   [post-discord-cards] Total cards before filter: <count>
   ```

6. **`KNOWN_MARKET_TAGS` constant**

   Add frozen tags list matching `normalizeMarketTag` branches:

   `['TOTAL', 'ML', 'Spread', '1P', 'TEAM TOTAL', 'TSOA', 'ANYTIME', 'SOT', 'SHOTS', 'PROP', 'POTD']`

   Warn (do not throw) when allow-list includes unknown tokens.

**Acceptance**:

- `cardMatchesWebhookFilters` no longer reads env directly.
- `DISCORD_CARD_WEBHOOK_MARKETS_DENY=SHOTS` excludes SHOTS when allow-list is unset.
- Allow-list overrides deny-list when both are set.
- Startup filter log and normalized filter log both emit.
- Zero-card warning includes pre-filter total card count.
- Unknown market tokens warn without crashing.
- **Tech debt additions:**
  - `KNOWN_MARKET_TAGS` constant defined at module top (frozen, reusable)
  - Env var validation helper `validateDiscordEnvVars()` runs once at job start, warns if numeric bounds violated
  - Parameterized test cases for all filter combinations: no filters, allow-list only, deny-list only, both set (allow precedence), unknown token warning
  - Shared test helper `makeFilterTestCases()` centralizes test fixtures

#### B0 — Heartbeat traceability

Every `runPotdEngine` return path must emit heartbeat with timestamp and run id:

```
[POTD] Engine run complete — ts:2026-04-19T13:02:11Z run:potd-1713531731 candidates:72 viable:0 status:PENDING_WINDOW
[POTD] Engine run complete — ts:2026-04-19T15:14:45Z run:potd-1713539685 candidates:84 viable:0 status:NO_PICK_FINAL
[POTD] Engine run complete — ts:2026-04-19T12:47:12Z run:potd-1713536832 candidates:14 viable:3 status:OFFICIAL_PLAY play:NBA TOTAL OVER 227.5
```

Required fields: `ts`, `run`, `candidates`, `viable`, `status`, optional `play`.

---

#### B0.5 — POTD timing state machine (required)

Use this exact state model:

- `PENDING_WINDOW`: no final decision yet
- `OFFICIAL_PLAY`: official POTD exists
- `NO_PICK_FINAL`: decision window closed with no official play

Window contract (ET):

- Before 12:00 PM ET -> `PENDING_WINDOW`
- 12:00 PM to 4:00 PM ET -> `OFFICIAL_PLAY` if qualified else `PENDING_WINDOW`
- At/after 4:00 PM ET -> `OFFICIAL_PLAY` if exists else `NO_PICK_FINAL`

Alerting contract:

- Never emit no-pick alert during `PENDING_WINDOW`
- Emit no-pick alert only when status becomes `NO_PICK_FINAL`
- If NO_PICK is excluded from snapshot, this state machine is the operational source of truth

**Tech debt addition:**

- Export `POTD_TIMING_STATES` constant at module top (frozen object with `PENDING_WINDOW`, `OFFICIAL_PLAY`, `NO_PICK_FINAL` keys)
- Export `POTD_WINDOW_ET` constant with `OPENS_HOUR: 12` and `CLOSES_HOUR: 16` for future window adjustments
- Use these in all state checks and heartbeat logging (prevents magic strings, enables compile-time validation in TypeScript migrations)

#### B1 — POTD no-pick alert to `DISCORD_ALERT_WEBHOOK_URL`

Add `sendPotdNopickAlert()` in `run_potd_engine.js`:

- Sends to `DISCORD_ALERT_WEBHOOK_URL`
- Uses injected sender path (`sendDiscordMessagesFn`) for testability
- Silent no-op if alert webhook unset
- Fires only in `NO_PICK_FINAL`

**Message format**:

```
⚠️ POTD — No Pick (Apr 19)

Reason: below_noise_floor

Top candidate:
NBA TOTAL OVER 227.5
Edge: +1.4% (below required 2.0%)
Score: 0.28 (LOW)

Highest edge observed: +1.4%
Required minimum edge: 2.0%

Candidates scored: 8
Viable: 0
```

Rules:

- Use phrase `Highest edge observed` (not `Highest qualifying edge`)
- `requiredEdge` must be computed from the candidate being explained:
  `resolveNoiseFloor(alertCandidate.sport, alertCandidate.marketType, POTD_MIN_EDGE)`
- Do not use lowest cross-sport floor
- Use `bestCandidate` when present, otherwise fallback to top-by-edge candidate

Reason mapping must include:

- `below_noise_floor`
- `no_viable_candidates`
- `confidence_below_high_gate`
- `zero_wager`
- `min_stake_rejected`

Implementation contract:

- `sendPotdNopickAlert({ sendDiscordMessagesFn, webhookUrl, message })`
- `runPotdEngine` passes injected `sendDiscordMessagesFn`
- Do not call raw imported `sendDiscordMessages` in alert path

**Tech debt addition:**

- Export `POTD_NOPICK_REASONS` constant (frozen object mapping reason codes to reason strings)
- Use in alert rendering for compile-time validation of reason strings
- Prevents mismatched reason values between alerting and logging

#### B2 — POTD play card in snapshot (opt-in)

Current snapshot SQL excludes `potd-call`. Make this opt-in:

- Add `DISCORD_INCLUDE_POTD_IN_SNAPSHOT=false` (default)
- When true, remove SQL exclusion on `potd-call`
- Add `potd` / `potd-call` branch in `normalizeMarketTag` -> `POTD`
- If snapshot include is true and `DISCORD_POTD_WEBHOOK_URL` is set, suppress engine's direct POTD post to prevent duplicate messaging

Critical guard:

- Any `potd-call` with `final_play_state !== 'OFFICIAL_PLAY'` is excluded from snapshot

POTD ordering + filter bypass:

- POTD renders in distinct leading section
- POTD rows bypass market filters in snapshot mode:
  if `market_tag === 'POTD'`, `bypassFilters = true`

**Acceptance**:

- NO_PICK never appears in snapshot
- `DISCORD_INCLUDE_POTD_IN_SNAPSHOT=true` shows OFFICIAL_PLAY POTD in leading section
- POTD section is not hidden by market allow-list filters
- Included POTD card must still satisfy SQL join/time requirements (`game_id` resolves to `games.game_id` with non-null `game_time_utc`)

### Sub-task C — Discord 429 resilience + hard timeout

**Files**:

- `apps/worker/src/jobs/post_discord_cards.js`
- `apps/worker/src/jobs/__tests__/post_discord_cards.test.js`
- `env.example`

**Changes**:

- Parse 429 `retry_after`
- Retry once only
- Sleep for `ceil(retry_after_ms) + jitter(50..150ms)`
- Log retry attempt:
  `[post-discord-cards] Rate limited — retrying after <ms> (retry 1/1)`
- Keep `DISCORD_MAX_RETRY_AFTER_MS=5000` fast-fail guard
- Add cumulative time guard `TOTAL_DISCORD_TIMEOUT_MS=10000`
- If total retry/send time exceeds timeout budget, abort and throw

Testability contract:

- `sendDiscordMessages` accepts `sleepFn` injection (default real timer) alongside `fetchImpl`
- Retry path uses `await sleepFn(...)`

**Acceptance**:

- 429 `retry_after: 0.3` -> one sleep + one retry
- 429 `retry_after: 10` -> immediate fail via max-retry-after guard
- second 429 after retry -> fail immediately
- cumulative retry/send time over `TOTAL_DISCORD_TIMEOUT_MS` -> fail fast
- no behavior changes for 2xx or non-429 errors
- **Tech debt additions:**
  - Discord resilience constants at module top (frozen): `DISCORD_MAX_RETRY_AFTER_MS`, `TOTAL_DISCORD_TIMEOUT_MS`, `RETRY_JITTER_MIN_MS`, `RETRY_JITTER_MAX_MS`, `MAX_RETRIES`
  - Webhook URL validation helper `validateDiscordWebhookUrl(url)` runs once at job start (protocol + hostname check)
  - Prevents misconfigured webhooks from consuming timeout budget
