# WI-0899: Database Truth Ownership Audit for Stateful Tables

**ID**: WI-0899
**Goal**: Classify every stateful table as canonical, derived/cache, display-only, or mixed-risk, and identify where read paths treat derived state as truth.

**Depends on**: none

**Scope**:

- WORK_QUEUE/WI-0899.md
- docs/audits/db-truth-ownership.md
- docs/audits/db-truth-ownership.inventory.md

**Out of scope**:

- Schema redesign or migration execution
- New table creation
- Settlement formula changes
- Any code changes in `packages/data/**`, `apps/worker/**`, or `web/**` (read-only analysis only)

**Acceptance**:

- Every stateful table has owner process, write jobs, mutability policy, and truth classification.
- A lie-risk list identifies tables that can diverge from canonical model output.
- Read-path map shows where API/UI consumes derived tables as authoritative.
- Post-write mutation permissions are explicitly documented per table.
- At least one guard proposal is documented for each high-risk table.
- Completeness proof is documented: table universe extracted from migrations is reconciled against the final per-table audit registry (no missing tables).

**Execution plan**:

- Phase 1: Build deterministic table universe inventory from `packages/data/db/migrations/**/*.sql` (CREATE TABLE and ALTER TABLE targets) and store it in `docs/audits/db-truth-ownership.inventory.md`.
- Phase 2: Build ownership and mutability registry in `docs/audits/db-truth-ownership.md` for every table in the inventory: owner process, write paths, post-write mutability policy, and truth class (canonical, derived/cache, display-only, mixed-risk).
- Phase 3: Build read-path authority map from `web/src/app/api/**/*.ts` and `web/src/lib/games/**/*.ts`, marking where derived/cache tables are treated as authoritative and identifying lie-risk paths.
- Phase 4: For each high-risk table/path, document at least one concrete guard proposal (read-only route hardening, source-priority contract, drift detector, freshness gate, or explicit fallback policy).
- Phase 5: Run automated checks and manual trace validation; mark complete only if inventory reconciliation is exact and all high-risk items have guard proposals.
