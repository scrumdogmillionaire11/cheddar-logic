**ID**: WI-0670
**Goal**: Elevate comparison from a discrete feature to an ambient behavior. After every state change in the workbench, the user sees relative position — not just absolute output. Tradeoff chips must be atomic, directional, and visually consistent so users can compare faster than they can think.

## Tradeoff chip specification

Chips must be:

- Atomic: one idea per chip. Never combine axes.
- Directional: prefix with `+` (positive) or `-` (negative). Always.
- Consistent: same chip labels used across all surfaces in the workbench.

Chip label format (backend-set label, not raw number):

    +Bench Strength
    -Captain Ceiling
    +Flexibility
    -Volatility

Show only axes where the delta is non-zero. Maximum 3 chips per state change. Chips from API only — no frontend derivation.

Color rules: green chip for `+` direction, amber for `-`. Grey chip is not allowed (if magnitude is zero, omit the chip).

---


## Single source of truth for chip generation

All tradeoff chip data -- whether on the Draft Lab after an intent, on the Final Recommendation
alternatives, or on the post-onboarding side-by-side comparison -- must originate from one function:

`comparison_service.py::compute_delta_axes(state_before, state_after) -> list[TradeoffAxis]`

Rules:

- This is the **only** place axis names, directions, and labels are computed.
- No other module, route handler, or frontend component may derive or infer chip labels.
- Any surface that renders chips must call this function (via the `/delta` endpoint) and render the
  result verbatim.
- `recommendation_service.py` (WI-0669) must call `compute_delta_axes` when generating alternative
  tradeoffs -- not duplicate the axis logic independently.

This is enforced by `test_comparison_delta.py::test_chip_labels_only_from_compute_delta_axes`: any
code path that produces a `TradeoffAxis` outside `compute_delta_axes` must fail this test.

## Delta endpoint contract

`GET /api/v1/draft-sessions/{id}/delta` returns the axis-level change vs the immediate previous session state.

    {
      "delta": {
        "session_id": "<uuid>",
        "baseline_label": "Before last change",
        "current_label": "After: risk_down",
        "axes": [
          { "name": "Bench Strength",   "direction": "+", "label": "+Bench Strength" },
          { "name": "Captain Ceiling",  "direction": "-", "label": "-Captain Ceiling" },
          { "name": "Volatility",       "direction": "-", "label": "-Volatility" }
        ]
      }
    }

Returns 404 if no baseline exists (first edit with no prior state). Frontend handles 404 gracefully — no chips rendered on first change.

## When comparison triggers (automatically — not user-triggered)

After onboarding completes: shell fetches 2-3 profile-matched builds via existing `/draft-sessions/compare` and shows them side-by-side. User enters the lab already knowing their relative options.

After any Draft Lab intent is confirmed: shell calls `GET /draft-sessions/{id}/delta`. Tradeoff chips render inline below the confirmation without page navigation.

After screenshot parse completes: backend comparison of parsed team vs profile-optimal reference. Same delta contract.

The goal: every state transition immediately answers "compared to before, what did I just do?"
