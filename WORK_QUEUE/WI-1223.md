# WI-1223: Resolve FPL frozen-domain mismatch in web surface (R-10)

**ID**: WI-1223
**Goal**: Remove or gate stale `/fpl` web exposure so the UI/API contract matches the current operational status of the frozen FPL domain.

**Context**:

Risk R-10 identifies that the FPL domain is effectively frozen operationally, but the web layer still serves `/fpl` routes. This creates product confusion, operational noise, and a maintenance trap where stale endpoints appear healthy despite no active roadmap ownership.

This WI does not remove FPL code. It aligns runtime exposure with current policy using a controlled gate and explicit response behavior.

**Scope**:

- `web/src/middleware.ts` — add gate behavior for `/fpl` paths behind an explicit env flag (for example `ENABLE_FPL_ROUTES`)
- `web/src/app/fpl/**` — update top-level route behavior to return a frozen/disabled response when gate is off
- `web/src/lib/api-security/config.ts` — include FPL route gate config + default value
- `web/src/__tests__/fpl-route-gating.test.ts` (new) — assert `/fpl` returns disabled response when gate off, normal response when on
- `web/package.json` — add `test:fpl-route-gating` script (for example: `node --import tsx/esm src/__tests__/fpl-route-gating.test.ts`) for deterministic CI execution
- `.github/workflows/ci.yml` — run FPL route gating test in validate job
- `.planning/STATE.md`
- `WORK_QUEUE/WI-1223.md`

**Out of scope**:

- Rebuilding or reactivating FPL feature functionality
- Changes to `cheddar-fpl-sage` service runtime
- Deleting FPL codepaths

**Acceptance**:

- New env flag controls `/fpl` route availability with safe default `false`
- With flag off: `/fpl` returns consistent disabled response (410 or 404, pick one and document)
- With flag on: existing `/fpl` behavior is unchanged
- CI has a deterministic test for both flag states
- Deployment configuration documents the flag and expected production value (service env or checked-in env template)

**Tests to run**:

- `npm --prefix web run test:fpl-route-gating`
- `npm --prefix web run build`

**Manual validation**:

- Run web with `ENABLE_FPL_ROUTES=false` and verify `/fpl` disabled response
- Run web with `ENABLE_FPL_ROUTES=true` and verify `/fpl` loads existing page

**Guard for WI closeout**:

- CI test proves both enabled/disabled states
- Production default is explicitly set and documented
- No unrelated behavior changes in non-FPL routes

**Residual risks (cannot be fully mitigated in this WI)**:

- Route gating reduces accidental exposure but does not remove legacy FPL code; dormant code still requires periodic dependency/security maintenance
- Environment drift can re-enable routes unexpectedly if deployment config is modified outside git-controlled workflow
