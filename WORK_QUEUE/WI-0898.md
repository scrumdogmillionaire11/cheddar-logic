# WI-0898: Env-Var Truth Table for Qualification, Downgrade, and Execution Gating

**ID**: WI-0898
**Goal**: Produce a deterministic, machine-readable config truth table for environment variables that change play qualification, downgrade behavior, execution gating, and sizing. This is a prerequisite for WI-0906 (continuity audit) which requires a definitive map of where edge/confidence/size signals are gated by config.

**Depends on**: none

**Scope** (NARROWED — qualification and execution gating only):

**Must-audit files (qualification, downgrade, gating logic)**:
- `packages/models/src/edge-calculator.js` — edge measurement and edge-related toggles
- `packages/models/src/decision-pipeline-v2.js` — qualification gates, downgrade decision branches
- `apps/worker/src/jobs/execution-gate.js` — execution eligibility and quality state gating
- `apps/worker/src/jobs/run_nba_model.js` — sport-specific downgrade and qualification gates
- `apps/worker/src/jobs/run_nhl_model.js` — sport-specific downgrade and qualification gates
- `apps/worker/src/jobs/run_mlb_model.js` — sport-specific downgrade and qualification gates
- `apps/worker/src/jobs/potd/run_potd_engine.js` — POTD-specific qualification and downgrade gates
- `apps/worker/src/jobs/potd/signal-engine.js` — POTD signal tier and confidence gates
- `docs/audits/env-var-truth-table.md` — output artifact (human-readable markdown table)
- `docs/audits/env-var-truth-table.json` — output artifact (machine-readable JSON, for WI-0906 parsing)
- `WORK_QUEUE/WI-0898.md` — this work item

**Out of scope** (explicitly deferred):
- UI visibility toggles (web layer env vars) — separate future work item WI-0898b
- Projection-only modes — separate future work item WI-0898b

- Removing variables from runtime
- Altering deployment secrets
- Creating new tests (model test coverage belongs to WI-0825/WI-0831/WI-0819)

**Acceptance Criteria** (linked to WI-0906 requirements):

1. **Machine-readable truth table exists in two formats** (supports WI-0906-REQ-01: edge→confidence→size lineage)
   - **Markdown table** (human review): `docs/audits/env-var-truth-table.md`
     - Schema columns: `Var Name | Default | Read Location (module:function) | Qual Impact | Downgrade Impact | Exec Gate Impact | Safe Range | Dangerous Values | Test Coverage | Status | Notes`
     - 25–35 rows (comprehensive audit of NBA/NHL/MLB/POTD/core gates)
   - **JSON export** (WI-0906 parsing): `docs/audits/env-var-truth-table.json`
     - JSON schema: `{ metadata: { generated_date, auditor, git_revision }, vars: [{ name, default, read_location, qual_impact, downgrade_impact, exec_gate_impact, safe_range, dangerous_values, test_coverage, status, notes }] }`
     - Must be valid JSON, parseable by `JSON.parse()` without errors
   - Example rows provided in Output Format Specification section below

2. **All vars with qualification impact are tagged and searchable** (supports WI-0906-REQ-02: breaks where size ≠ 0)
   - Each row: `qual_impact: true` OR `qual_impact: false`
   - `qual_impact: true` = changing this var prevents pickup from surfacing OR forces existing pickup to NO_BET
   - Examples: `NO_BET_IF_STALE_EDGE_HOURS`, `QUALIFICATION_MIN_EDGE_PCT`, any var read in `execution-gate.js:evaluateExecution()`
   - Searchable in JSON: `vars.filter(v => v.qual_impact === true)`

3. **All vars with downgrade impact are tagged** (supports WI-0906-REQ-02: fragile confidence breaks)
   - Each row: `downgrade_impact: true` OR `downgrade_impact: false`
   - `downgrade_impact: true` = changing this var reduces stake/confidence without zeroing (status → DEGRADED, stake reduced but >0)
   - Examples: `DEGRADED_IF_EDGE_MOVE_PCT`, `STALE_EDGE_HOURS`, any var that modifies `quality_status` without forcing NO_BET
   - Note: One var can have `qual_impact=true AND downgrade_impact=true` (e.g., `NO_BET_IF_STALE_EDGE_HOURS`)

4. **All vars with execution-gating impact are tagged** (supports WI-0906-REQ-04: remediation guards for zero-size)
   - Each row: `exec_gate_impact: true` OR `exec_gate_impact: false`
   - `exec_gate_impact: true` = changing this var can force stake to zero OR prevent execution entirely
   - Examples: `ENABLE_NBA_MODEL`, `FORCE_NO_BET`, `NO_BET_IF_STALE_EDGE_HOURS`

5. **Test coverage is verified (existing tests only; no new tests created)** (replaces unmeasurable criterion)
   - Each row lists EITHER:
     - `test_coverage: "file:test_name"` (e.g., `"execution-gate.test.js:should reject NO_BET_IF_STALE_EDGE_HOURS"`)
       - Auditor must verify: `grep -rn "NO_BET_IF_STALE_EDGE_HOURS" apps/worker/src/jobs/__tests__/` to confirm test file references this var
     - OR `test_coverage: "none_found"` (auditor searched; no test references this specific var)
   - Auditor appends PR comment with grep command and results for all "none_found" entries
   - **Out of scope**: Creating new tests (belongs to model validation WI-0825/WI-0831/WI-0819)

6. **Dead/stale vars are marked (no deletion action)** (supports future config cleanup)
   - Each row tagged: `status: "live" | "overridden_in_prod" | "stale" | "dead_candidate"`
   - `dead_candidate` rows include `dead_reason` field (e.g., "var read in conditional branch disabled since 2026-Q1")
   - No deletion recommendations in this WI (out of scope)
   - Row notes may suggest remediation (e.g., "consider removal after migration to new var X")

7. **Config aliases and duplicates are flagged** (supports detecting continuity breaks from config confusion)
   - Search strategy: For each var, grep all 8 must-audit files
   - If two vars implement the same logical gate: flag one as "alias/duplicate" in notes (e.g., "OLD_THRESHOLD and NEW_THRESHOLD both gate downgrade; recommend removing OLD_THRESHOLD")
   - If none found: state in markdown doc footer: "No config aliases detected"

8. **Existing test gate coverage is validated (unit tests only; no end-to-end manual)** (supports acceptance)
   - For `qual_impact: true` vars, confirm test file exists and is referenced in `test_coverage` column
   - For `downgrade_impact: true` vars, confirm test file exists and is referenced in `test_coverage` column
   - For `exec_gate_impact: true` vars, confirm test file exists and is referenced in `test_coverage` column
   - Manual spot-check: Run `npm --prefix apps/worker run test -- execution-gate.test.js` to confirm tests execute without errors
   - **No end-to-end model runs**: Keep validation at unit-test level only
