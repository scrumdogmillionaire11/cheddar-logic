# WI-0846: Pipeline Watchdog Self-Check — Alert When check_pipeline_health Gap > 2h

| Field | Value |
|---|---|
| **ID** | WI-0846 |
| **Goal** | The `check_pipeline_health` job is the primary health watchdog. As of 2026-04-06 it ran at 00:59 UTC and then not again until manually triggered ~21 hours later — a full day with no watchdog writes to `pipeline_health`. Nothing alerted on this gap. Add a self-referential check: on each `check_pipeline_health` run, query `job_runs` for the previous `check_pipeline_health` success timestamp. If the gap is > 2h, write a Discord alert and a `failed` row to `pipeline_health` with phase=`watchdog` and check_name=`heartbeat`. This is the "watchdog watching the watchdog." |
| **Scope** | `apps/worker/src/jobs/check_pipeline_health.js` |
| **Out of scope** | Changes to scheduler config, other jobs, frontend, creating a new job |
| **Acceptance** | (1) At the start (or end) of each `check_pipeline_health` run, query `job_runs` for the most recent prior success row where `job_name='check_pipeline_health'` and `status='success'`. (2) If gap is > 2h, call `shouldSendAlert` / `buildHealthAlertMessage` with a watchdog-heartbeat payload and send Discord alert (if `ENABLE_PIPELINE_HEALTH_WATCHDOG=true`). (3) Write a `pipeline_health` row: `phase='watchdog'`, `check_name='heartbeat'`, `status='warning'`, `reason='Last watchdog run was Xh ago'`. (4) On normal gap (< 2h), write `status='ok'` heartbeat row. (5) Unit test: mock job_runs with a 3h gap → heartbeat row is warning + alert fires. (6) Unit test: mock job_runs with a 30m gap → heartbeat row is ok, no alert. |
| **Owner agent** | unassigned |
| **Time window** | unassigned |
| **Coordination flag** | none |
| **Tests to run** | Structural verify first: `grep -c "checkWatchdogHeartbeat\|heartbeat" apps/worker/src/jobs/check_pipeline_health.js` must return >= 2. Then: `npm --prefix apps/worker test -- --testPathPattern=check_pipeline_health` |
| **Manual validation** | Temporarily pause the scheduler on Pi for 2.5h. Restart — first run should send a Discord alert and write a warning heartbeat row. Check `/admin` page for the watchdog:heartbeat card showing warning. |

## Context

On 2026-04-06 the scheduler stopped writing `pipeline_health` rows after 00:59 UTC. By 21:34 UTC when `job:dr-claire` was run, the pipeline_health section showed all checks as stale/1d ago — no alerts had fired. The `ENABLE_PIPELINE_HEALTH_WATCHDOG=true` flag is set and WI-0797 implemented Discord alerting, but nothing checks whether the watchdog itself is still running.

This is the minimal viable "watchdog heartbeat" — a self-check that fires exactly when it's needed (when the watchdog has been silent too long).

## Implementation Notes

Add at the start of the main `checkAndWrite` or `main()` function in `check_pipeline_health.js`:

```javascript
async function checkWatchdogHeartbeat(db) {
  const prev = db.prepare(`
    SELECT started_at FROM job_runs
    WHERE job_name = 'check_pipeline_health' AND status = 'success'
    ORDER BY started_at DESC LIMIT 1
  `).get();

  if (!prev) return; // First ever run — skip

  const gapMs = Date.now() - new Date(prev.started_at).getTime();
  const gapH = (gapMs / 3600000).toFixed(1);
  const isGap = gapMs > 2 * 60 * 60 * 1000;

  db.prepare(`
    INSERT INTO pipeline_health (phase, check_name, status, reason)
    VALUES ('watchdog', 'heartbeat', ?, ?)
  `).run(
    isGap ? 'warning' : 'ok',
    isGap ? `Last watchdog run was ${gapH}h ago` : `Heartbeat OK (${gapH}h since last run)`
  );

  if (isGap && shouldSendAlert('watchdog_heartbeat')) {
    await sendDiscordAlert(buildHealthAlertMessage([{
      phase: 'watchdog', check_name: 'heartbeat',
      status: 'warning',
      reason: `check_pipeline_health gap: ${gapH}h (threshold: 2h)`
    }]));
  }
}
```
