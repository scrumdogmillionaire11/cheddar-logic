## Functional Requirements

### FR1. Legal XI selection

The system must always return:

- exactly 11 starters
- exactly 1 starting goalkeeper
- a legal FPL formation
- exactly 4 bench players

Illegal formations must never be returned.

Allowed formations:

- 3-4-3
- 3-5-2
- 4-4-2
- 4-3-3
- 4-5-1
- 5-4-1
- 5-3-2
- 5-2-3

Rule:

> The optimizer must enumerate only these formations and select the highest scoring candidate available from the current squad.

### FR2. Hard exclusions

Players marked as `OUT` or `BANNED` must never be included in the Starting XI.

If insufficient playable players exist, the optimizer must still return the best legal XI possible and emit a data-quality or availability warning.

### FR3. Risk-aware selection

The optimizer must support three risk profiles:

- Conservative
- Balanced
- Aggressive

Risk profile must influence close lineup decisions as follows:

- Conservative: prefer expected minutes and role certainty; penalize volatility and minutes risk more heavily; favor floor over upside.
- Balanced: projected-points led with moderate penalties; blend floor and upside.
- Aggressive: tolerate volatility/minutes uncertainty more; favor ceiling and attacking upside when projection gap is meaningful; still respect hard exclusions.

### Lineup Scoring Inputs

The optimizer must derive a StartScore for each player using:

- `projected_points`
- `expected_minutes`
- `fixture_adjustment` (if available)
- `volatility`
- availability flags

Penalty rules:

- `OUT` → hard exclusion
- `BANNED` → hard exclusion
- `DOUBT` → penalty
- `MINUTES_RISK` → posture-scaled penalty

Posture weighting:

- Conservative: minutes weight up; volatility penalty up; risk-flag penalty up
- Balanced: projected points primary; moderate penalties
- Aggressive: volatility penalty down; ceiling bias up; minutes penalty down

Guardrail:

> `projected_points` must remain the dominant StartScore component in all risk modes.

### FR4. Formation evaluation

The optimizer must evaluate all legal candidate formations available from the squad and choose the formation with the highest total adjusted lineup score.

Formation selection must not be hardcoded.

Fallback rule:

> If all candidate formations include a player with expected_minutes < 20, select the formation with the highest combined expected_minutes across the XI.

### FR5. Deterministic Bench Order

Bench ordering must follow this fixed rule set:

1. Bench slot 1 = highest `projected_points` outfield substitute.
2. Bench slot 2 = second highest `projected_points` outfield substitute.
3. Bench slot 3 = lowest `projected_points` outfield substitute.
4. Bench slot 4 = goalkeeper.

Tie-breaker order:

1. higher `expected_minutes`
2. lower volatility
3. alphabetical `player_id` fallback

Rule:

> Identical inputs must always produce identical bench order.

### FR6. Captain and vice alignment

Final captain and vice captain must both be members of the finalized Starting XI.

If an upstream captain/vice selection conflicts with finalized XI, the system must recalculate or promote valid fallbacks rather than returning inconsistent output.

### FR7. Explanation metadata

The optimizer must provide explanation metadata for:

- why a player started
- why a player was benched
- why a formation was chosen
- how risk posture affected close calls

Explanation text must be short, concrete, and derived from optimizer logic.

### Lineup Confidence

Confidence is derived from minutes certainty across the XI:

- `HIGH`: 9+ players with expected_minutes >= 70
- `MEDIUM`: 6-8 players with expected_minutes >= 70
- `LOW`: 5 or fewer players with expected_minutes >= 70

### FR8. API contract

Backend API must expose the canonical lineup decision payload including:

- formation
- starters
- bench
- captain
- vice captain
- lineup confidence
- explanation metadata
- risk profile used
- warnings/notes when relevant

### FR9. UI rendering

Dashboard and Results must render finalized Starting XI using pitch-style layout.

UI requirements:

- one visual row each for GK, DEF, MID, FWD
- row population adapts to formation
- display captain and vice badges
- display projected points
- display bench in explicit order
- visually distinguish flagged/risky/benched players
- maintain readability on smaller screens

Frontend rule:

> The frontend must not compute formation, starters, or bench logic.
> It must render the lineup strictly from the canonical API payload.

### FR10. End-to-end reachability

The enriched lineup payload must flow through:

- optimizer
- integration layer
- result transformer
- API response model
- frontend API typings
- Dashboard/Results rendering

No backend-only implementation counts as done.

## Canonical Contract (Locked)

```ts
type OptimizedLineupResponse = {
  formation: string;
  risk_profile: "CONSERVATIVE" | "BALANCED" | "AGGRESSIVE";
  lineup_confidence: "HIGH" | "MEDIUM" | "LOW";
  formation_reason: string;
  risk_profile_effect?: string;
  notes: string[];
  starters: LineupPlayer[];
  bench: BenchPlayer[];
  captain_player_id: number | string;
  vice_captain_player_id: number | string;
};

type LineupPlayer = {
  player_id: number | string;
  name: string;
  team: string;
  position: "GK" | "DEF" | "MID" | "FWD";
  projected_points: number;
  expected_minutes?: number;
  flags?: string[];
  badges?: string[];
  start_reason?: string;
};

type BenchPlayer = {
  player_id: number | string;
  name: string;
  position: "GK" | "DEF" | "MID" | "FWD";
  bench_order: 1 | 2 | 3 | 4;
  projected_points: number;
  expected_minutes?: number;
  flags?: string[];
  bench_reason?: string;
};
```

Terminology:

> Use “canonical lineup decision payload” in code/docs, not “canonical finalized lineup payload”.

## Acceptance Criteria

- Same input always returns same Starting XI and same bench order.
- Returned XI is always legal and drawn from allowed formation set.
- `OUT` and `BANNED` players never appear in XI.
- Captain and vice always appear in XI.
- Changing posture can change close lineup decisions.
- Explanation metadata exists for formation and player-level decisions.
- Dashboard and Results render the same finalized XI from the same payload.
- Frontend performs no independent lineup inference.
- API contract snapshot test fails if lineup payload shape changes.
