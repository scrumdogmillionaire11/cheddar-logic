# Starting XI Logic Requirements for FPL Sage

The Starting XI logic must pick the optimal 11-man lineup from the user’s 15-man squad each gameweek, using projection data, fixture context, player risk, and user-selected risk posture. It must output both a valid FPL formation and a pitch-style visual representation of the lineup. This logic should sit downstream of the team model, fixture model, and projection engine, and must respect the existing contracts around player projections, volatility, minutes risk, and squad structure.   

## 1. Goal

Select the best legal Starting XI for the current gameweek by balancing:

* projected points
* expected minutes
* fixture quality
* volatility
* clean sheet potential
* attacking upside
* user risk posture

The output must not just maximize raw projected points blindly. It must apply different lineup-selection behavior depending on whether the manager is Conservative, Balanced, or Aggressive. The team model already defines those risk balance modes and the projection engine already exposes volatility and expected minutes, so the XI logic must consume those rather than inventing new fields.  

## 2. Inputs

The Starting XI selector must consume, at minimum:

* current 15-man squad from `FplTeamInput`
* player-level projections from `PlayerProjection`
* team model outputs including player tiers and flags
* fixture difficulty context from `FixtureProfile` / `FixtureProjection`
* selected user risk posture override:

  * `CONSERVATIVE`
  * `BALANCED`
  * `AGGRESSIVE`

Relevant existing fields include:

* `position`
* `status_flag`
* `nextGW_pts`
* `xMins_next`
* `volatility`
* `tier`
* flags such as `MINUTES_RISK`, `NAILED_90`, `FIXTURE_BOOST`, `FIXTURE_TRAP`, `INJURY_SHADOW`
* fixture attack/defense difficulty
* clean sheet probability where available.    

## 3. Core Functional Requirements

### 3.1 Legal Formation Enforcement

The selector must always return a legal FPL formation:

* exactly 1 GK
* at least 3 DEF
* at least 2 MID
* at least 1 FWD
* total of 11 starters

Allowed shapes include:

* 3-4-3
* 3-5-2
* 4-4-2
* 4-3-3
* 4-5-1
* 5-4-1
* 5-3-2
* 5-2-3

No illegal shape can ever be returned. If raw point optimization would create an illegal shape, the selector must fall back to the highest-scoring legal shape.

### 3.2 Availability Filtering

The selector must downgrade or exclude players based on availability:

* `OUT` or `BANNED`: cannot start
* `DOUBT` or `UNKNOWN`: may start only if no clearly better safe option exists
* `INJURY_SHADOW`: apply a risk penalty
* `MINUTES_RISK`: apply a stronger penalty for Conservative and Balanced than for Aggressive. Existing status and flag structures already support this behavior.  

### 3.3 Base Start Score

Every player should receive a `StartScore` used for XI selection.

Baseline formula should be conceptually:

`StartScore = nextGW_pts + fixture_adjustment + role_adjustment - risk_penalty`

The exact weights can be configurable, but the logic must use:

* `nextGW_pts` as the primary driver
* `xMins_next` as a floor/safety input
* fixture difficulty and clean sheet context
* volatility as a penalty or upside modifier depending on risk mode
* player role signals such as `SET_PIECES`, `PEN_TAKER`, `FIXTURE_BOOST`, `FIXTURE_TRAP`. These fields already exist in the model contracts and should be reused.  

### 3.4 Position-Specific Logic

The selector must not treat all positions the same.

For GKs:

* prioritize expected points, clean sheet potential, and minutes certainty
* ignore attacking upside
* strongly penalize minutes uncertainty

For DEFs:

* combine clean sheet odds with attacking upside
* defenders with set pieces or strong attacking threat may beat safer clean-sheet-only defenders under Balanced or Aggressive modes

For MIDs and FWDs:

* prioritize attacking projection
* apply fixture and minutes adjustments
* allow upside-based selection under Aggressive mode

This aligns with the fixture model’s separation of attack and defense difficulty and the projection engine’s player-level point output.  

## 4. Risk Profile Behavior

This is where the logic stops being generic sludge and becomes useful.

### 4.1 Conservative

Conservative mode must prefer floor over ceiling.

Rules:

* prefer players with higher `xMins_next`
* heavily penalize `HIGH` volatility
* heavily penalize `MINUTES_RISK`
* prefer nailed starters over explosive punts
* prefer defenders/GKs with stronger clean sheet odds
* break close decisions in favor of safety, even if upside is lower

Conservative mode should be willing to bench:

* a flashy attacker with minutes risk
* a player returning from injury
* a DGW-type trap with uncertain starts
  if a safer starter projects similarly.

### 4.2 Balanced

Balanced mode must act as the default tradeoff layer.

Rules:

* prioritize `nextGW_pts`
* apply moderate minutes and volatility penalties
* accept some volatility if upside meaningfully beats safer options
* prefer fixture-boosted players when projection gaps are close
* use sensible blend of floor and upside

Balanced is the default “don’t be stupid, don’t be scared” mode.

### 4.3 Aggressive

Aggressive mode must lean into upside.

Rules:

* tolerate more volatility
* tolerate lower minutes floor when upside is materially higher
* prefer explosive attackers in strong fixtures
* allow attacking defenders to beat safer but passive defenders
* break close calls toward higher ceiling options
* be less conservative about benching steady low-ceiling players

Aggressive should still obey hard blocks:

* do not start `OUT` or `BANNED`
* do not ignore obviously poor minutes projection if the player is unlikely to play enough to return value

The team model already defines risk balance categories and the projection engine already tags volatility, so this requirement is about applying those outputs to lineup decisions instead of just displaying them.  

## 5. Bench Logic Requirements

The Starting XI logic must also produce a legal bench order.

Bench output must include:

* bench goalkeeper
* first bench outfield player
* second bench outfield player
* third bench outfield player

Bench order must reflect:

* strongest likely substitute first
* legal autosub compatibility
* expected minutes
* position flexibility

Bench rules by mode:

* Conservative: bench order should favor players most likely to actually appear
* Balanced: blend minutes and projection
* Aggressive: first bench can favor upside if still reasonably likely to play

Dead or near-dead assets should never sit first bench unless there is literally no better alternative.

## 6. Formation Selection Logic

The engine must evaluate candidate legal formations rather than assuming one.

Required behavior:

1. generate all legal formations available from the squad
2. compute total `StartScore` for each formation
3. choose highest-scoring formation
4. if two formations are very close, use risk posture as tie-breaker

Tie-break examples:

* Conservative: favor the shape with more secure-minute players
* Balanced: favor total score
* Aggressive: favor the shape with more attacking ceiling

This prevents the classic dumb-agent behavior of hardcoding 3-4-3 because nostalgia told it so.

## 7. Captaincy Alignment Requirement

The Starting XI selector must be compatible with captain/vice logic.

Requirements:

* captain and vice must always be selected from the chosen XI
* if the current captain logic proposes a benched player, the lineup engine must either:

  * promote that player into the XI if justified, or
  * force captaincy recalculation
* final XI and captaincy output must never conflict

This is necessary because captaincy is projection-led and already defined globally in the orchestrator and transfer workflow.  

## 8. Display Requirements: Pitch View

The output must render visually as a football pitch with the Starting XI placed by position.

### 8.1 Layout

The UI must show:

* goalkeeper nearest the bottom
* defenders in one row
* midfielders in one row
* forwards in the highest row
* substitutes shown below the pitch in bench order

The pitch must adapt to shape dynamically:

* 3 defenders = 3 cards on defender row
* 5 midfielders = 5 cards on midfield row
* 1 forward = 1 card centered up top

### 8.2 Player Card Content

Each player card on the pitch should display:

* player name
* team
* position
* projected points
* optional minutes indicator
* captain / vice badge where relevant

Optional but useful:

* fixture badge
* risk icon
* injury/doubt flag
* bench reason tooltip

### 8.3 Visual Encoding

Use clean visual cues:

* green glow or border for strongest starts
* amber for moderate-risk starts
* red/grey for benched or flagged players
* captain badge clearly distinct from vice badge

Risk posture should visibly affect explanations, not the pitch layout itself.

### 8.4 Responsiveness

The pitch display must:

* work on desktop first
* degrade cleanly on mobile
* avoid card overlap
* keep formation readable without horizontal chaos

The user asked for it to “look like a pitch with starting 11,” so this is not optional garnish. It is the main event.

## 9. Explanation Requirements

For each lineup decision, the engine must be able to explain:

* why a player started
* why a player was benched
* why a formation was chosen
* how risk posture affected close calls

Example explanation fields:

* `start_reason`
* `bench_reason`
* `formation_reason`
* `risk_profile_effect`

These explanations should be short and concrete:

* “Started for higher nextGW projection and secure 85+ expected minutes”
* “Benched due to minutes risk and difficult fixture”
* “4-4-2 beat 3-5-2 by safety-adjusted score in Conservative mode”

No fake philosopher mode. Just the actual reason.

## 10. Edge Cases

The selector must handle:

* multiple flagged or injured players
* only one playable forward
* one or more non-playing bench assets
* tied projections between players
* double gameweek upside with uncertain minutes
* blank gameweek assets that remain in squad
* conflicting safety vs upside cases

Fallback rules:

* always prioritize legality
* always prioritize likely appearance over zero-minute punts unless Aggressive mode still shows clear projected edge
* if data is incomplete, return best legal XI with explicit `DATA_GAP` note rather than pretending certainty

That behavior matches the orchestrator’s general “no silent magic” principle when data is incomplete. 

## 11. Output Contract

Recommended output shape:

```text
StartingXIResult {
  formation: string                 # e.g. "3-4-3"
  starters: StartingPlayer[11]
  bench: BenchPlayer[4]
  captain_player_id: string
  vice_captain_player_id: string
  lineup_confidence: "HIGH" | "MED" | "LOW"
  risk_profile: "CONSERVATIVE" | "BALANCED" | "AGGRESSIVE"
  formation_reason: string
  notes: string[]
}

StartingPlayer {
  player_id: string
  name: string
  position: "GK" | "DEF" | "MID" | "FWD"
  row_slot: int
  projected_points: float
  expected_minutes: float
  start_score: float
  flags: string[]
  badges: string[]                 # e.g. ["CAPTAIN", "SET_PIECES"]
  start_reason: string
}

BenchPlayer {
  player_id: string
  name: string
  bench_order: 1 | 2 | 3 | 4
  projected_points: float
  expected_minutes: float
  bench_reason: string
}
```

## 12. Acceptance Criteria

The feature is complete when:

* the engine always returns a legal XI
* risk profile changes at least some close decisions in a predictable way
* bench order is sensible and legal
* captain and vice always exist in the XI
* pitch view renders correctly for all legal formations
* users can tell at a glance who is starting, benched, captain, vice, and risky
* decision reasons are exposed for each bench/start call
* data gaps are surfaced explicitly, not hidden under fake confidence

## 13. Implementation Guidance for the Agent

Build this in three layers:

1. **Lineup scoring layer**

   * computes `StartScore` by player under each risk posture

2. **Formation optimizer**

   * evaluates all legal formations and selects best one

3. **Presentation mapper**

   * converts XI into pitch rows and bench cards for UI rendering

Do not bury display logic inside projection logic. Do not hardcode one formation. Do not let the UI independently infer lineup structure. That road leads to bugs and ugly lies.
