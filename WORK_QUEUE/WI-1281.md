# WI-1281: Opportunity Outcome Tracking

**ID**: WI-1281
**Goal**: Record what happens to surfaced Market Pulse opportunities after they appear — did the discrepancy close, persist, or widen — so we can distinguish signal from noise over time.

**Why this is worth doing**:
- WI-1275 surfaces discrepancies and labels their freshness. WI-1281 answers the follow-on question: were those discrepancies real?
- Without outcome data, there is no way to know whether SCOUT items lead to line movement, whether WATCH items quietly vanish, or which books are consistently slow.
- This is the minimum feedback loop needed to evaluate signal quality.

## Adversarial Constraints

- This WI does not evaluate win/loss on bets. It evaluates whether the market moved in the direction the discrepancy indicated. No P&L, no settlement, no bet tracking.
- Outcome tracking must be strictly read-only from the web side. The worker is the sole DB writer per ADR-0002.
- Outcome snapshots must tolerate missing data: games may end before the next scan, books may go offline, odds may not refresh. Missing = unknown, not failure.
- Do not backfill historical odds. Tracking starts from when this WI ships; prior discrepancies have no outcome record.

## Worker Job

`market-pulse-outcome-tracker.js` runs after each regular mispricing scan.

For each recent opportunity (captured_at within last 4 hours, not yet checked):
1. Re-fetch current odds for the same game and market.
2. Compute gap at check time.
3. Classify outcome:
   - `CLOSED`: gap dropped below 50% of original gap_pct
   - `PERSISTED`: gap within 50-150% of original
   - `WIDENED`: gap grew beyond 150% of original
   - `NOT_FOUND`: game or market not present in current scan
   - `UNKNOWN`: data missing or scan error
4. Write one `market_pulse_outcomes` row.

Job must be idempotent on `opportunity_id` + `checked_at` window.

## Web Read Surface

`GET /api/market-pulse/outcomes`

Response shape:
```ts
{
  outcomes: OutcomeRecord[];
  summary: {
    total: number;
    closed: number;
    persisted: number;
    widened: number;
    notFound: number;
    unknown: number;
    closedPct: number;   // closed / (closed + persisted + widened)
    byTier: Record<DisplayTier, { closed: number; persisted: number; widened: number }>;
    byBook: Record<string, { moved: number; total: number }>;
  };
  asOf: string; // ISO
}
```

This route is read-only. It never writes.

## Acceptance

- `market_pulse_outcomes` table is created by migration and does not touch existing tables.
- Worker job checks each recent opportunity exactly once per scan cycle.
- Outcomes are classified using the three-threshold rule (CLOSED / PERSISTED / WIDENED) and stored with the display tier and freshness at capture time.
- `/api/market-pulse/outcomes` returns outcome summary including per-tier breakdown.
- Missing data (NOT_FOUND, UNKNOWN) does not cause job failure or missing rows.
- No bet or settlement data is referenced.
- Worker is the sole writer; web route is read-only.
- Tests assert outcome classification logic for each threshold boundary.
- Tests assert idempotency: re-running the job does not duplicate rows for the same opportunity+window.

- Run the worker job twice for the same opportunity window and confirm no duplicate rows.
- Confirm no settlement or bet columns exist in `market_pulse_outcomes`.

## Failure Modes

- Outcome checker re-scans data that has aged out, producing `NOT_FOUND` for everything and making results look like signal disappears.
- Gap threshold constants (50%/150%) are too loose or too tight to be meaningful for the book spread ranges we see.
- `opportunity_id` hash collides across different game+market+side combinations, corrupting outcome attribution.
- Worker job latency grows because outcome checking runs inline with the main scan.
