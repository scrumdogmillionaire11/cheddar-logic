# WI-1262: Add Next Best Fallback Surface

**ID**: WI-1262

**Goal**: Add a separately labeled fallback lane when no official POTD exists, sourced from high-quality monitored candidates without weakening the `POTD = PLAY only` contract.

**Context**:

The system needs a product-safe answer for dead POTD days without collapsing official and fallback semantics. When no official `PLAY` exists, the user experience should still surface the best monitored candidate from `LEAN` or `SLIGHT_EDGE`, but it must remain explicitly distinct from POTD.

Fallback candidates must:

- come from model-backed signals only
- have positive edge
- have no hard blockers
- remain outside `potd_plays`

Valid presentation labels include `Next Best`, `Watchlist`, or `Monitored`, but they must not be called `POTD`.

**Scope**:

- `web/src/app/api/potd/route.ts`
- `web/src/lib/potd-server.ts`
- `web/src/__tests__/api-potd.test.js`
- `web/src/__tests__/potd-near-miss-best-edge.test.js`
- `web/src/__tests__/ui-potd-smoke.test.js`

**Out of scope**:

- Writing fallback selections into `potd_plays`
- Counting fallback selections as official POTD history
- Altering official POTD selection from `PLAY` to `best available`
- Bankroll or staking changes for fallback rows

**Acceptance**:

- When no official POTD exists, the API returns a separately labeled fallback lane sourced from `LEAN | SLIGHT_EDGE`
- Fallback candidates require positive edge, model-backed provenance, and no hard blockers
- Fallback presentation is explicitly distinct from official POTD in both API naming and UI copy
- Fallback rows are tracked separately from official POTD records and never persisted as `potd_plays`
- Existing official POTD behavior remains unchanged when a `PLAY` exists

**Manual validation**:

- On a no-PLAY day, verify the fallback lane appears with the chosen non-POTD label
- Confirm the fallback lane disappears or de-prioritizes when an official POTD exists
- Verify no fallback selection is written or displayed as official POTD history
