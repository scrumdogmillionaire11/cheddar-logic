# WI-0392: API Card Endpoints Lifecycle Parity

**ID**: WI-0392
**Goal**: Apply the same lifecycle filtering (from WI-0388) to `/api/cards` and `/api/cards/[gameId]` endpoints so card visibility is consistent with games endpoint behavior across all API surfaces.

**Scope**:

- `WORK_QUEUE/WI-0392.md`
- `web/src/app/api/cards/route.ts`
- `web/src/app/api/cards/[gameId]/route.ts`
- `web/src/__tests__/api-cards-lifecycle-regression.test.js` (new regression test suite)

**Out of scope**:

- Settlement grading logic (`apps/worker/src/jobs/settle_pending_cards.js`)
- ESPN/final score ingestion (`apps/worker/src/jobs/settle_game_results.js`)
- Historical backfill/resettlement of existing records
- Cards page UI toggle (completed by WI-0388)
- `/api/games` lifecycle filtering (completed by WI-0388)
- Derived lifecycle fields and display_status (completed by WI-0388)

**Acceptance**:

- `/api/cards` supports `?lifecycle=active` query param, returning only cards from non-settled games
- `/api/cards/[gameId]` supports `?lifecycle=active` query param with same settled-game exclusion logic
- Default behavior (no `lifecycle` param) returns full history of cards including settled games
- Both endpoints use the same `ACTIVE_EXCLUDED_STATUSES` constant from WI-0388 (postponed, cancelled, final, closed, complete)
- Settled games (with status=final or status=complete) are excluded when `?lifecycle=active`
- Regression tests pass: cover at least one settled game and one in-progress game path in both modes
