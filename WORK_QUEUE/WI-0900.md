**Out of scope**:

- Rewriting scheduler cadence
- Changing external provider timestamps
- New timezone libraries
- Non-timestamp business logic tuning (edge, confidence, staking)

**Acceptance**:

- Field-level registry identifies which timestamp controls eligibility, surfacing, settlement, and stale blocking.
- Latest-selection logic is classified as data-time or write-time for each audited path.
- Timezone handling is documented with one canonical runtime policy.
- Conflict matrix documents behavior when timestamps disagree.
- At least three high-risk disagreement paths have explicit remediation tasks proposed.

**Execution plan**:

- Phase 1: Build timestamp field registry from audited worker/model/data/web paths; classify each timestamp as provider data-time, ingest write-time, or derived runtime-time.
- Phase 2: Trace controlling timestamp for each decision surface (eligibility, card surfacing, settlement, stale blocking) and record whether selection is data-time-first or write-time-first.
- Phase 3: Document canonical timezone/runtime policy (UTC parsing/storage/comparison boundaries) and identify any local-time-dependent ordering risk points.
- Phase 4: Build disagreement conflict matrix covering at least: data-time newer but write-time older, write-time newer with stale data-time, missing data-time with present write-time, and timezone-ambiguous records.
- Phase 5: Propose remediations for at least three highest-risk disagreement paths with explicit owner surface, guard type, and verification command.
- Phase 6: Run automated checks and manual replay validation; mark complete only when all acceptance bullets map to explicit evidence in the audit doc.
