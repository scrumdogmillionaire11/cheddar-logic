# WI-0938-06: THE BOARD - Edge Reasoning Enricher + Backend Integration

**ID**: WI-0938-06  
**Parent**: WI-0938  
**Goal**: Implement the edge reasoning enricher function that computes market errors, action states, quality scores, and trigger conditions. Integrate into card publishing pipeline without breaking existing systems.

**Depends on**: WI-0938-02 (endpoints must exist) + WI-0938-01 (architecture plan must be complete)

**Scope**:

- `web/src/lib/edge-reasoning/compute-edge-reasoning.ts` (main enricher function)
- `web/src/lib/edge-reasoning/market-error-framing.ts` (market error logic)
- `web/src/lib/edge-reasoning/edge-type-classifier.ts` (classify edge type)
- `web/src/lib/edge-reasoning/quality-score-formula.ts` (deterministic quality scoring)
- `web/src/lib/edge-reasoning/action-state-logic.ts` (FIRE/WAIT/PASS rules)
- `web/src/lib/edge-reasoning/trigger-system.ts` (unlock conditions)
- Integration point: decision-publisher or web API layer (per WI-0938-01 decision)
- API endpoints: populate `/api/board/opportunities`, `/api/board/blocked-edges`, `/api/board/edge-types` with real data
- Backend aggregation: Edge Type Tracker statistics (win/loss, ROI, trends)
- Settlement tracking: wire up settlement results to pattern tracking (ongoing)

**Out of scope**:

- Worker-side changes (enrichment happens in web layer only)
- Database migrations (no new tables required, uses existing card data)
- Discord formatting (that's WI-0938-07)
- UI component rendering (that's WI-0938-03/04/05)

**Acceptance**:

- `computeEdgeReasoning()` function takes a card payload and returns populated `edge_reasoning` object
- Function is deterministic and testable
- All fields match schema in `WORK_QUEUE/THE_BOARD_EDGE_SCHEMA_SPEC.md`
- Market Error framing captures "why is market wrong" (not just "model disagrees")
- Edge Type classification correctly assigns STALE_PRICE, INFORMATION, STRUCTURAL, BEHAVIORAL, or TIMING
- Quality Score is deterministic and explainable (formula is documented in code comments)
- Action State logic correctly outputs FIRE / WAIT / PASS (no other values)
- Trigger conditions are explicitly formatted and actionable
- `/api/board/opportunities` returns cards with `action_state` = "FIRE" or "WAIT" (excludes PASS)
- `/api/board/blocked-edges` returns cards with `action_state` = "PASS" and `quality_score` ≥ 5.0
- `/api/board/edge-types` aggregates historical performance by edge type (win/loss, ROI, trend)
- All enrichment fields are optional (backward compatible, existing cards still work)
- Enricher is called post-watchdog, doesn't break existing routing/decision logic
- Unit tests: each enrichment function tested independently with fixtures
- Integration tests: full pipeline with mock card data
- No breaking changes to existing card payloads or API contracts
