# WI-1240: Unify Projection Results Contract

**ID**: WI-1240
**Goal**: Align projection grading, materialization, and frontend projection surface visibility under one explicit family contract.

**Context**:

The backend analytics tables can grade more projection families than `/results/projections` currently surfaces. Materialization semantics and inclusion rules differ across storage, settlement, and UI contracts. This WI makes the allowlist and visibility semantics authoritative in one place.

**Scope**:
- `packages/data/src/db/projection-accuracy.js`
- `apps/worker/src/jobs/settle_projections.js`
- `web/src/lib/results/projection-results-contract.ts`
- `web/src/app/api/results/projection-accuracy/route.ts`
- `web/src/app/api/results/projection-settled/route.ts`
- `packages/data/__tests__/projection-results-contract-unification.test.js` (new)
- `web/src/__tests__/projection-results-family-contract.test.js` (new)
- `WORK_QUEUE/WI-1240.md`

**Out of scope**:
- Any change to betting-results contract in `/api/results`
- Any threshold or model-quality tuning
- Any POTD changes

**Acceptance**:
- One shared projection-family allowlist governs storage/read eligibility semantics.
- Materialization semantics for each supported family are explicit and consistent across backend and frontend reads.
- Projection family drift between analytics tables and `/results/projections` is eliminated.
- Regression coverage proves multi-family inclusion/exclusion behavior is consistent across routes.
