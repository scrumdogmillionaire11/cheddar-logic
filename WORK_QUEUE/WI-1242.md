# WI-1242: Upgrade `/api/readyz` to Contract-Aware Health

## ID

WI-1242

## Goal

Prevent false-positive readiness by making `/api/readyz` validate the DB contracts required by enabled web surfaces, not just DB reachability.

## Problem

Current readiness only proves the DB can open and answer `SELECT 1`. That allows `/api/readyz` to return ready even when `/api/games`, `/api/cards`, or `/api/results` would fail because required tables/columns are missing.

Silent ready on a broken schema is a production lie.

## Required Readiness Contract

`/api/readyz` must validate:

### 1. DB Reachable
- open connection succeeds
- `SELECT 1` succeeds

### 2. Enabled Web Surfaces

The following surfaces are considered enabled for all deployments (not configurable):

- `/api/games`
- `/api/cards`
- `/api/results`

### 3. Required Schema Per Enabled Surface

#### `/api/games` requires:
- **Tables:**
  - `games`: must exist
  - `odds_snapshots`: must exist
  - `run_state`: must exist
  - `job_runs`: must exist

- **Columns in `odds_snapshots` (fallback if missing—warn but don't fail):**
  - `spread_consensus_line`, `spread_consensus_confidence`
  - `spread_dispersion_stddev`, `spread_source_book_count`
  - `total_consensus_line`, `total_consensus_confidence`
  - `total_dispersion_stddev`, `total_source_book_count`
  - `h2h_consensus_home`, `h2h_consensus_away`, `h2h_consensus_confidence`

#### `/api/cards` requires:
- **Tables:**
  - `run_state`: must exist
  - `job_runs`: must exist
  - `card_payloads`: must exist
  - `cards`: must exist (inherited from games route via deduplication)

#### `/api/results` requires:
- **Tables:**
  - `game_results`: must exist
  - `card_results`: must exist
  - `card_payloads`: must exist

- **Columns in `card_results` (fallback if missing—warn but don't fail):**
  - `market_key`, `market_type`, `selection`, `line`, `locked_price`

- **Optional Table (doesn't fail readiness if missing):**
  - `clv_ledger`: if missing, results reporting degrades gracefully

## Design Requirements

### Readiness Response Shape

**Healthy (200):**
```json
{
  "ok": true,
  "ready": true,
  "db": {
    "reachable": true
  },
  "contracts": {
    "checked": ["games", "cards", "results"],
    "failures": [],
    "warnings": []
  }
}
```

**Not Ready (503):**
```json
{
  "ok": false,
  "ready": false,
  "db": {
    "reachable": true
  },
  "contracts": {
    "checked": ["games", "cards", "results"],
    "failures": [
      {
        "surface": "results",
        "table": "card_results",
        "column": "market_key",
        "reason": "MISSING_COLUMN"
      }
    ],
    "warnings": [
      {
        "surface": "games",
        "table": "odds_snapshots",
        "column": "spread_consensus_line",
        "reason": "MISSING_COLUMN_FALLBACK"
      }
    ]
  }
}
```

**DB Unreachable (503):**
```json
{
  "ok": false,
  "ready": false,
  "db": {
    "reachable": false,
    "reason": "DB_FILE_NOT_FOUND | DB_OPEN_FAILED | DB_LOCKED"
  },
  "contracts": {
    "checked": [],
    "failures": [],
    "warnings": []
  }
}
```

### Status Codes

- **200**: DB is reachable AND all required tables/columns for enabled surfaces exist (warnings allowed).
- **503**: DB is reachable but any required table/column is missing; OR DB is unreachable.

### Contract Rules

1. Validate only enabled surfaces.
2. Validate required tables and columns using read-only SQL introspection.
3. Missing optional tables do not fail readiness.
4. Missing optional columns (fallback columns) produce warnings, not failures.
5. Do not call migrations.
6. Do not mutate the DB.
7. Do not treat missing critical schema as an empty successful 200 response.

## Implementation Plan

### 1. `web/src/lib/db-init.ts`

Add a read-only readiness helper:

```ts
type ContractFailure = {
  surface: string;
  table: string;
  column?: string;
  reason: 'MISSING_TABLE' | 'MISSING_COLUMN';
};

type ContractWarning = {
  surface: string;
  table: string;
  column: string;
  reason: 'MISSING_COLUMN_FALLBACK';
};

type ReadinessResult = {
  ok: boolean;
  ready: boolean;
  db: {
    reachable: boolean;
    reason?: string;
  };
  contracts: {
    checked: string[];
    failures: ContractFailure[];
    warnings: ContractWarning[];
  };
};

export function checkDbReady(): ReadinessResult
```

It must:

- run DB reachability check
- run contract validation for all enabled surfaces
- return structured failures and warnings
- never throw for expected missing-table/missing-column cases
- only throw or fail hard for unexpected DB access errors

Use read-only introspection only:

- table existence via `SELECT name FROM sqlite_master WHERE type='table' AND name=?`
- column existence via `PRAGMA table_info(table_name)` or equivalent read-only query

### 2. `web/src/app/api/readyz/route.ts`

Update route to:

- call `checkDbReady()`
- return `200` when `ready=true`
- return `503` when `ready=false`
- include structured failure/warning reasons in JSON response body

### 3. Test File

Create: `web/src/__tests__/api-readyz-contract-aware-health.test.js`

Test matrix (use isolated test DBs for each):

1. **Healthy DB with all required tables/columns** → `200`, `ready=true`, `failures=[]`
2. **DB reachable but missing required table** (e.g., `games` table missing) → `503`, `ready=false`, reason `MISSING_TABLE`
3. **DB reachable but table missing required column** (e.g., `card_results.market_key` missing) → `503`, `ready=false`, reason `MISSING_COLUMN`
4. **DB reachable with unrelated historical tables missing** (e.g., legacy `fpl_matches` table missing) → still `200`, `ready=true`
5. **DB reachable but optional column missing** (e.g., `odds_snapshots.spread_consensus_line` missing) → `200`, `ready=true`, but `warnings` contains the missing column
6. **DB unreachable/open failure** → `503`, `ready=false`, reason in `db.reason` field

**Test constraint:** No contract failure may return `ok=true` unless `ready=true`.

## Acceptance

- `/api/readyz` distinguishes DB reachability from operational readiness.
- Enabled surfaces have explicit table/column contracts defined in code (not inferred).
- Missing required tables return `503` with `MISSING_TABLE` in failures.
- Missing required columns return `503` with `MISSING_COLUMN` in failures.
- Missing optional/fallback columns return `200` with `MISSING_COLUMN_FALLBACK` in warnings.
- Historical/unenabled tables are not checked.
- Web process performs no migrations or writes.
- Partial-schema DB (missing any required element) cannot return `200`.
- Regression tests cover: healthy, missing-table, missing-column, unrelated-missing-table, missing-optional-column, and unreachable-DB cases.
- Response includes structured `failures` and `warnings` arrays.
