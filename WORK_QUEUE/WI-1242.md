# WI-1242: Upgrade `/api/readyz` to Contract-Aware Health

**ID**: WI-1242
**Goal**: Prevent false-positive readiness by making `/api/readyz` validate required schema/contracts for enabled surfaces rather than only DB reachability.

**Context**:

Today `/api/readyz` proves the DB file can open and `SELECT 1` can run. That is not sufficient for operational readiness when key tables for `/api/games`, `/api/results`, or other enabled surfaces are missing. This WI makes readiness surface-aware.

**Scope**:
- `web/src/app/api/readyz/route.ts`
- `web/src/lib/db-init.ts`
- `web/src/__tests__/api-readyz-contract-aware-health.test.js` (new)
- `WORK_QUEUE/WI-1242.md`

**Out of scope**:
- Any change to liveness/healthz behavior beyond readiness semantics
- Any change to worker job success criteria
- Any change to public surface filtering rules

**Acceptance**:
- `/api/readyz` distinguishes DB reachability from operational readiness.
- Readiness checks validate required tables/contracts for enabled surfaces.
- Missing critical tables or schema requirements return NOT READY with a clear failure reason.
- Regression coverage proves partial-schema scenarios fail readiness.

**Tests to run**:
- `node web/src/__tests__/api-readyz-contract-aware-health.test.js`

**Manual validation**:
- Run against a partial-schema DB and confirm `/api/readyz` reports not ready with an explicit reason.
- Run against a healthy DB and confirm readiness still succeeds.

**Guard for WI closeout**:
- Do not silently downgrade critical schema failures to empty `200` readiness responses.
- Keep readiness scoped to enabled/required surfaces rather than every historical table in the DB.
