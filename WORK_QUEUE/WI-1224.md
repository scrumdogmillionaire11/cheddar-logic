# WI-1224: Persist Explicit MLB F5 Settlement Call in Card Payload Contract

**ID**: WI-1224
**Goal**: Stop inferring MLB F5 proxy settlement side from historical projection values by persisting an explicit settlement call and grading mode on each card payload at creation time, then using that persisted contract everywhere downstream.

**Context**:

The current MLB F5 proxy-results flow still depends on reconstructing intent from `proj_value` after the fact. That is fragile for three reasons:

- The producer knows the intended call when the card is created, but that intent is not persisted in a canonical machine-readable field.
- Settlement/reporting code is forced to reverse-engineer side intent from numeric ranges (`< 3.5`, `> 4.5`, gray zone), which makes historical cleanup and replay awkward.
- Old gray-zone rows can remain visible in `projection_proxy_evals` because the UI reads materialized rows directly; restarting services does not rewrite that historical interpretation.

For MLB F5 projection-only cards, the source of truth should live on the card payload itself. The producer should persist whether the card is:

- an official `UNDER_3_5` call
- an official `OVER_4_5` call
- a gray-zone `TRACK_ONLY` / `NO_CALL`

Downstream settlement, backfill, and results APIs should consume that explicit contract instead of re-deriving the call from projection math.

**Proposed payload contract**:

Add an explicit object on MLB F5 card payloads, for example:

```json
{
  "projection_settlement_policy": {
    "market_family": "MLB_F5_TOTAL",
    "grading_mode": "OFFICIAL",
    "official_call": "UNDER_3_5",
    "reason_code": "CLEAR_UNDER"
  }
}
```

Gray-zone example:

```json
{
  "projection_settlement_policy": {
    "market_family": "MLB_F5_TOTAL",
    "grading_mode": "TRACK_ONLY",
    "official_call": null,
    "reason_code": "GRAY_ZONE_NO_CALL"
  }
}
```

**Payload evolution strategy (backward compatibility)**:

- `projection_settlement_policy` is **optional** on the payload (fields may be omitted)
- New payloads generated by `run_mlb_model.js` **MUST** include this object
- `settle_projections.js` **MUST** handle three cases:
  1. **Policy present and valid:** Use the persisted grading_mode and official_call
  2. **Policy missing (old card):** Log WARN, do NOT infer settlement window; skip proxy row write with reason `LEGACY_PAYLOAD_NO_POLICY`
  3. **Policy malformed:** Log ERROR, do NOT materialize proxy row; record error for monitoring
- Results API filters out gray-zone rows using grading_mode, so legacy rows (which lack the field) are implicitly suppressed after a transition period

**Validator schema for projection_settlement_policy**:

Add to `packages/data/src/validators/card-payload.js`:

```javascript
const projectionSettlementPolicySchema = z.object({
  market_family: z.literal('MLB_F5_TOTAL'),
  grading_mode: z.enum(['OFFICIAL', 'TRACK_ONLY']),
  official_call: z.enum(['UNDER_3_5', 'OVER_4_5']).nullable(),
  reason_code: z.enum([
    'CLEAR_UNDER',      // projection clearly < 3.5
    'CLEAR_OVER',       // projection clearly > 4.5
    'GRAY_ZONE_NO_CALL' // 3.5 <= projection <= 4.5
  ])
}).strict();

// Validation rules for consistency:
// - If grading_mode === 'OFFICIAL', official_call must be 'UNDER_3_5' or 'OVER_4_5'
// - If grading_mode === 'TRACK_ONLY', official_call must be null
// - reason_code must align with grading_mode

const mlbF5PayloadSchema = basePayloadSchema.extend({
  projection_settlement_policy: projectionSettlementPolicySchema.optional()
});
```

**Out of scope**:

- Reworking the underlying MLB F5 model thresholds themselves
- Retrofitting every historical card payload in one migration (handled via backward-compat logic in settlement)
- UI redesign of the projection breakdown table
- Non-MLB-F5 markets

**Acceptance**:

- Every newly created `mlb-f5` card payload contains a validated `projection_settlement_policy` object
- Producer writes one of exactly three explicit states for MLB F5 cards:
  - `OFFICIAL` + `UNDER_3_5` + reason_code = `CLEAR_UNDER`
  - `OFFICIAL` + `OVER_4_5` + reason_code = `CLEAR_OVER`
  - `TRACK_ONLY` + `official_call=null` + reason_code = `GRAY_ZONE_NO_CALL`
- `settle_projections.js` no longer determines MLB F5 official side by comparing `projected_total` to thresholds; **all** reads of grading mode and side come from persisted contract
- Results API filters using `grading_mode` field: TRACK_ONLY rows are excluded from official settled projection outcomes
- Old cards (pre-deployment) lack `projection_settlement_policy`; settlement skips these with explicit `LEGACY_PAYLOAD_NO_POLICY` log
- Missing/invalid `projection_settlement_policy` payloads on new cards fail closed: error logged, no proxy W/L rows written, monitoring alert triggered

**Guard for WI closeout**:

- Validator rejects malformed `projection_settlement_policy` payloads for `mlb-f5` cards
- No MLB F5 settlement path reads `proj_value < 3.5` / `> 4.5` inference; all reads use persisted policy
- Gray-zone semantics are represented explicitly in payload data (`grading_mode: TRACK_ONLY`), not implied by absence
- Settlement explicitly logs `LEGACY_PAYLOAD_NO_POLICY` for old cards; monitoring can track transition
- All three settlement state combinations (OFFICIAL/UNDER, OFFICIAL/OVER, TRACK_ONLY) are tested and working

**Residual risks (deferred or cannot be fully mitigated in this WI)**:

- Historical rows already materialized in `projection_proxy_evals` table will continue to exist. To suppress them:
  - **Short-term (during transition):** Results API filter already excludes them if they lack grading_mode or have grading_mode=TRACK_ONLY
  - **Long-term:** A separate cleanup WI should remove legacy gray-zone rows once fresh cards have been under the new contract for a deployment cycle
- **Transition period (1-2 days):** Until enough fresh MLB F5 cards are produced under the new contract, stale legacy rows may co-exist in the DB. The results API filter handles this transparently.
