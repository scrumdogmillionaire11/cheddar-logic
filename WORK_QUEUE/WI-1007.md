**Acceptance**:

- `settle_mlb_f5` identifies F5 ML cards via strict market identity rules before grading logic.
- F5 ML settlement uses the same canonical `resolveF5Snapshot()` function as F5 totals — not similar logic, the *same function*.
- `resolveF5Snapshot()` returns `READY` only when: 5 innings completed OR 4.5 innings completed with home team leading, AND `is_verified = true`.
- `is_verified` is computed deterministically: `home_runs !== null && away_runs !== null && inning_state_consistent && !partial_play_flag && (snapshot_fetched_at - last_event_timestamp) > MIN_STABILITY_WINDOW`. `MIN_STABILITY_WINDOW = 8000` (ms, pinned) and guards against race conditions where feed lag causes a mid-update snapshot to appear stable.
- `resolveF5Snapshot()` returns `UNGRADABLE` when status would otherwise be READY but `is_verified = false` (dirty/mid-play feed state).
- F5 ML grading rules are deterministic:
  - selected side leading after F5 snapshot = `win`
  - selected side trailing after F5 snapshot = `loss`
  - tied after F5 snapshot = `push`
- `push` is a first-class result: must not count as win or loss, must not appear in win-rate denominator, and must be preserved in analytics.
- `normalizeF5MlSelection()` maps all known side variants to `HOME` | `AWAY` | `INVALID`. Team-name-to-side resolution requires 100% confidence via `game_id` mapping; any ambiguity returns `INVALID`, not a guess.
- `game_id` must uniquely identify a single event instance; doubleheader games must use distinct IDs.
- `isF5TotalCard(card)` and `isF5MlCard(card)` are mutually exclusive; both returning true is a hard assertion failure.
- Missing or invalid `selection_side` emits explicit reason code and does not settle.
- UNGRADABLE game state settles to terminal state with `result = 'no_contest'` and is excluded from win%, ROI, and push count but included in volume tracking and failure diagnostics.
- Settlement begins with hard idempotency guard: `card.status === 'settled'` is checked AND `card.result !== null` is asserted; already-settled rows are skipped without mutation; feed corrections on settled rows are detected and logged as `F5_CORRECTION_DETECTED` without overwriting.
- `result = 'no_contest'` must be a valid value in the DB `result` column before deployment; migration required if column is enum-constrained.
- Existing F5 totals settlement remains unchanged, with negative assertion tests proving totals rows never enter the ML branch and ML rows never read `line` or totals-specific fields.
- Settlement emits structured per-run observability log including snapshot status distribution and result distribution.
- WI-1007 must emit `push` and `no_contest` results with explicit contracts compatible with downstream analytics (result values stored, reason codes emitted). Analytics denominator correction (ensuring `wins / (wins + losses)`, excluding push and no_contest from denominator) is tracked in follow-on **WI-1017**.
- Integration tests cover: HOME win, AWAY win, tie-push, not-yet-eligible, missing-side, unverified snapshot (UNGRADABLE), already-settled (idempotency), feed-correction detection, doubleheader isolation, mutual exclusion predicate, and UNGRADABLE paths.

## Summary

WI-1007 addresses a real problem: F5 ML is one of the few projection-only MLB markets that can be settled cleanly without odds. The following sections lock down every place where "mostly works" would become the outcome.

**Killshot rule:**

> There must be exactly one deterministic function that returns the F5 score snapshot, and both totals and ML settlement must depend on it.

---

## Why This Is Needed

- F5 ML is projection-only.
- It does **not** require a spread or price threshold to determine outcome.
- The scoreboard after 5 innings is sufficient to grade `HOME` / `AWAY`.

Ambiguity that must be closed before implementation:

- How a card is identified as F5 ML.
- How selected side is encoded (and what variants exist in production payloads).
- When the F5 snapshot is considered officially gradeable AND internally consistent.
- What happens on suspended/shortened/doubleheader games.
- How "tied after 5" and "no_contest" are handled downstream in analytics.
- What idempotency enforcement looks like in code, not just in intent.
- What feed corrections on already-settled rows should do.

## Design

### Contract 1: F5 Snapshot (shared by totals and ML)

```ts
resolveF5Snapshot(gameState) -> {
  status: "READY" | "NOT_READY" | "UNGRADABLE",
  home_runs: number | null,
  away_runs: number | null,
  is_verified: boolean,
  reason_code?: string
}
```

**`gameState` input type — mapped from MLB Stats API `/api/v1.1/game/{gamePk}/feed/live`:**

```ts
type F5GameState = {
  // linescore.currentInning
  current_inning: number;
  // true when isTopInning === false (bottom half in progress)
  is_bottom_inning: boolean;
  // sum of linescore.innings[0..4].home.runs; null if < 5 innings fetched
  home_runs: number | null;
  // sum of linescore.innings[0..4].away.runs; null if < 5 innings fetched
  away_runs: number | null;
  // linescore.outs — used to verify top-5 is fully complete (must === 3)
  current_outs: number | null;
  // gameData.status.abstractGameState: 'Final' | 'In Progress' | 'Suspended' | 'Cancelled'
  abstract_game_state: string;
  // !liveData.plays.currentPlay.about.isComplete when abstractGameState === 'In Progress', else false
  partial_play_flag: boolean;
  // liveData.plays.currentPlay.startTime (ISO string); null if no current play
  last_event_timestamp: string | null;
  // wall-clock time the snapshot was fetched (injected by caller, not from API)
  snapshot_fetched_at: string;
};
```

**Field derivation rules:**
- `inning_state_consistent`: `current_inning >= 5 && (current_outs === 3 || current_inning > 5)` — top-5 complete only when `current_outs === 3` at `current_inning === 5` top half
- `partial_play_flag`: `!liveData.plays.currentPlay.about.isComplete` when `abstract_game_state === 'In Progress'`; else `false`

**READY conditions (mirrors book grading):**

- 5 innings completed for both teams, OR
- 4.5 innings completed AND home team is leading (bottom of 5th not played per standard rules)

**AND `is_verified = true`**, computed as:

```ts
is_verified = (
  home_runs !== null &&
  away_runs !== null &&
  inning_state_consistent &&            // no top-5 with outs < 3, etc.
  !partial_play_flag &&                  // no mid-event ambiguity from feed
  (snapshot_fetched_at - Date.parse(last_event_timestamp)) > MIN_STABILITY_WINDOW  // race-condition guard
)
// MIN_STABILITY_WINDOW = 8000 (ms, pinned — do not change without updating tests)
// Prevents is_verified=true on a snapshot where the feed batched score + inning
// updates together but last_event_timestamp arrived slightly ahead of the score.
```

All five conditions must hold. This definition is canonical — different developers must not interpret `is_verified` differently.

**NOT_READY:** game in progress, F5 threshold not yet reached

**UNGRADABLE:** game suspended, canceled, corrupted/missing score data, asymmetric inning state, OR game state meets READY conditions but `is_verified = false`

Enforcement:

```ts
if (wouldBeReady && !is_verified) {
  return { status: "UNGRADABLE", reason_code: "F5_SCORE_UNVERIFIED_STATE" }
}
```

This function is the single gate. Both `settleF5Total` and `settleF5Ml` call it. Neither reimplements it.

### Contract 2: Selection Normalization

```ts
normalizeF5MlSelection(card) -> "HOME" | "AWAY" | "INVALID"
```

Maps ALL known variants:

| Input | Output |
|---|---|
| `"HOME"`, `"home"`, `"H"` | `"HOME"` |
| `"AWAY"`, `"away"`, `"A"` | `"AWAY"` |
| team name with 100% confidence mapping via game_id | `"HOME"` or `"AWAY"` |
| team name with any ambiguity or mapping confidence < 100% | `"INVALID"` |
| anything else | `"INVALID"` |

**Critical rule:** Never guess HOME/AWAY from ambiguous team strings. Fail closed > wrong settlement.

**Doubleheader rule:** `game_id` must uniquely identify a single event instance. If two same-day same-matchup games share an ID, normalization and snapshot resolution are both unreliable. This must be caught upstream; settlement fails closed if game_id is ambiguous.

Raw payload is never used directly. Every card goes through this function before any grading logic.

### Contract 3: Settlement Input

```ts
type MlbF5MlSettlementInput = {
  card_type: "mlb-f5-ml";
  market_key: "F5_ML" | "FIRST_5_MONEYLINE";
  selection_side: "HOME" | "AWAY";   // post-normalization only
  game_id: string;                   // must be globally unique per event instance
};
```

### Contract 4: Push as First-Class Result

Push is not a fallback or error state. It must:

- Not count as a win or loss in ROI calculations.
- Not appear in win-rate denominator: `win_rate = wins / (wins + losses)`, never `wins / total_plays`.
- Be stored as `result = 'push'` in the DB row.
- Be preserved in reporting pipelines.

Required analytics layer test: insert push rows into dataset, compute win-rate, confirm denominator is unchanged.

### Contract 5: UNGRADABLE Terminal State

UNGRADABLE settles to:

- `status = 'settled'`
- `result = 'no_contest'`
- `primary_reason_code` (existing DB column) = one of the UNGRADABLE codes below

**Pre-deployment check:**

```sql
SELECT DISTINCT result FROM card_results;
```

**✅ CONFIRMED (2026-04-18):** `result` column is unconstrained `TEXT`. Current values: `void`, `loss`, `win`, `push`. No migration needed — `no_contest` can be written immediately.

| ROI | ❌ No |

| Failure diagnostics | ✅ Yes |

### Contract 6: Feed Correction Detection

Feeds can issue stat corrections after initial ingest. The idempotency guard must not overwrite settled results, but must surface drift:

```ts
if (card.status === "settled") {
  assert(card.result !== null, "settled card has null result — data corruption");

  if (CORRECTIONS_WINDOW_ACTIVE) {
    const latest = resolveF5Snapshot(gameState);
    if (latest.status === "READY" && snapshotDiffersFromStoredResult(latest, card)) {
      log("F5_CORRECTION_DETECTED", { card_id, stored_result: card.result, latest_snapshot: latest });
    }
  }

  return SKIP_ALREADY_SETTLED;
}
```

Rule: detect and log. Never overwrite. Manual review handles corrections if required.

### Contract 7: Mutual Exclusion of Market Predicates

```ts
assert(!(isF5TotalCard(card) && isF5MlCard(card)),
  "card classified as both F5 total and F5 ML — payload classification error");
```

**Predicate implementations (keyed on `card_results.card_type` — must be SELECT'd explicitly):**

```js
function isF5TotalCard(card) { return card.card_type === 'mlb-f5'; }
function isF5MlCard(card)    { return card.card_type === 'mlb-f5-ml'; }
```

`card_type` is written at card-insertion time from the model output. The SQL query in `settleMlbF5` must include `cr.card_type` in its SELECT list for these predicates to function. Currently it does not — this must be added in Step 7.

This assertion runs before routing, not inside each branch.

### Branch Structure

```ts
// Mutual exclusion guard
assert(!(isF5TotalCard(card) && isF5MlCard(card)));

// Idempotency guard — with correction detection
if (card.status === "settled") {
  assert(card.result !== null);
  detectFeedCorrection(card, gameState);  // logs F5_CORRECTION_DETECTED if drift found
  return SKIP_ALREADY_SETTLED;
}

// Market routing — before any grading
if (isF5TotalCard(card)) {
  return settleF5Total(card, gameState);
}

if (isF5MlCard(card)) {
  return settleF5Ml(card, gameState);
}

return SKIP_NOT_F5_MARKET;
```

Inside `settleF5Ml`:

```ts
const selection = normalizeF5MlSelection(card);
if (selection === "INVALID") {
  return fail_closed("F5_ML_INVALID_SELECTION_SIDE");
}

const snapshot = resolveF5Snapshot(gameState);
if (snapshot.status === "NOT_READY") {
  return PENDING("F5_NOT_ELIGIBLE_YET");
}
if (snapshot.status === "UNGRADABLE") {
  return settleAsNoContest(card, snapshot.reason_code ?? "F5_UNGRADABLE_GAME_STATE");
}

// snapshot.status === "READY" && is_verified === true
if (snapshot.home_runs > snapshot.away_runs) {
  return selection === "HOME" ? win("F5_ML_HOME_LEADING_AFTER_5") : loss("F5_ML_HOME_LEADING_AFTER_5");
}
if (snapshot.away_runs > snapshot.home_runs) {
  return selection === "AWAY" ? win("F5_ML_AWAY_LEADING_AFTER_5") : loss("F5_ML_AWAY_LEADING_AFTER_5");
}
return push("F5_ML_TIED_AFTER_5_PUSH");
```

### Reason Codes

| Code | Meaning |
|---|---|
| `F5_ML_HOME_LEADING_AFTER_5` | Home ahead at F5 snapshot |
| `F5_ML_AWAY_LEADING_AFTER_5` | Away ahead at F5 snapshot |
| `F5_ML_TIED_AFTER_5_PUSH` | Tied at F5 snapshot → push |
| `F5_NOT_ELIGIBLE_YET` | Snapshot NOT_READY, card remains pending |
| `F5_UNGRADABLE_GAME_STATE` | Game suspended/canceled/corrupted |
| `F5_SCORE_UNVERIFIED_STATE` | Snapshot READY conditions met but feed state is dirty/mid-play |
| `F5_ML_SELECTION_SIDE_MISSING` | `selection.side` absent from payload |
| `F5_ML_INVALID_SELECTION_SIDE` | Side present but cannot be normalized with certainty |
| `F5_CORRECTION_DETECTED` | Feed issued a correction on an already-settled row (logged, not applied) |

## Invariants

F5 ML settlement **requires**:

- Valid, globally unique `game_id`
- `selection_side` resolvable to `HOME` or `AWAY` via `normalizeF5MlSelection()` with 100% confidence
- `resolveF5Snapshot()` returning READY with `is_verified = true`

F5 ML settlement does **not** require:

- `line`
- `odds`
- `price`

F5 ML never reads:

- `card.line`
- `payload.projection.total`
- Any totals-specific fields

Totals settlement never reads:

- `selection.side` for grading
- Any side-score comparison logic

`isF5TotalCard` and `isF5MlCard` are mutually exclusive by assertion.

## Full Test Matrix

### Pre-deploy

- `result` column accepts `no_contest` value (or migration confirmed run).

### Idempotency

- Already settled F5 ML row is skipped — `status`, `result`, `primary_reason_code` unchanged on second run.
- Repeated run emits observability log with `settled: 0` for already-settled rows.
- Settled row with `result = null` triggers assertion failure (data corruption guard).
- Settled row re-run with different snapshot → result unchanged, `F5_CORRECTION_DETECTED` logged.

### Identity / Normalization

- `card_type='mlb-f5-ml'` with `selection.side='HOME'` routes to ML branch.
- `card_type='mlb-f5-ml'` with `selection.side='AWAY'` routes to ML branch.
- `card_type='mlb-f5-ml'` with `selection.side='home'` (lowercase) normalizes correctly.
- `card_type='mlb-f5-ml'` with missing `selection.side` → `F5_ML_SELECTION_SIDE_MISSING`, no settlement.
- `card_type='mlb-f5-ml'` with unresolvable side → `F5_ML_INVALID_SELECTION_SIDE`, no settlement.
- Team name with ambiguous game_id mapping → `INVALID`, no settlement (never guesses HOME/AWAY).

### Doubleheader

- Two games: same teams, same day, distinct game_ids → normalization and snapshot resolve independently per game_id.
- Ambiguous game_id (doubleheader collision) → settlement fails closed.

### Mutual Exclusion

- Card matching both `isF5TotalCard` and `isF5MlCard` predicates → assertion failure.

### Snapshot / Eligibility

- Game at inning 4, both teams batting → NOT_READY → pending with `F5_NOT_ELIGIBLE_YET`.
- Game at 4.5 innings, home trailing → NOT_READY (bottom of 5th still to play).
- Game at 4.5 innings, home leading → READY (bottom of 5th not played per F5 rules).
- Game at 5 complete innings, all `is_verified` conditions met → READY.
- Game at inning 5, `home_runs = null` → `is_verified = false` → UNGRADABLE with `F5_SCORE_UNVERIFIED_STATE`.
- Game at inning 5, `partial_play_flag = true` → `is_verified = false` → UNGRADABLE.
- Game at inning 5, `last_event_timestamp` within `MIN_STABILITY_WINDOW` of `now` → `is_verified = false` → UNGRADABLE with `F5_SCORE_UNVERIFIED_STATE` (feed lag race condition guard).
- Game suspended at 3 innings → UNGRADABLE → `F5_UNGRADABLE_GAME_STATE`.
- Missing score fields in feed → UNGRADABLE → `F5_UNGRADABLE_GAME_STATE`.

### Grading

- HOME selection, home leads after F5 → `win`, reason `F5_ML_HOME_LEADING_AFTER_5`.
- HOME selection, home trails after F5 → `loss`, reason `F5_ML_HOME_LEADING_AFTER_5`.
- AWAY selection, away leads after F5 → `win`, reason `F5_ML_AWAY_LEADING_AFTER_5`.
- AWAY selection, away trails after F5 → `loss`, reason `F5_ML_AWAY_LEADING_AFTER_5`.
- Either selection, tied after F5 → `push`, reason `F5_ML_TIED_AFTER_5_PUSH`.

### UNGRADABLE terminal state

- UNGRADABLE row settles to `result = 'no_contest'` with appropriate reason code.
- `no_contest` row is not counted in win%, ROI, or push count.
- `no_contest` row is included in volume tracking and failure diagnostics.

### Isolation (negative assertions)

- `mlb-f5` totals row with same game_id never enters ML branch.
- F5 totals row with tied score but total over line settles using totals result, not push.
- F5 ML grading function never reads `card.line`.
- F5 ML grading function never reads `payload.projection.total`.
- F5 totals grading function never reads `selection.side` for outcome.

### Push / analytics layer

- Push row has `result='push'` in DB.
- Push row is not included in win-rate numerator or denominator: insert push rows, compute `wins / (wins + losses)`, confirm result is identical to without push rows.
- `no_contest` row is not included in win-rate denominator.
