# WI-0761: Surface `pipeline_health` records — route or Discord webhook

| Field              | Value |
|--------------------|-------|
| **ID**             | WI-0761 |
| **Goal**           | Make health check results actually observable: either expose them via a `/api/admin/pipeline-health` route or emit them through the existing Discord webhook so the `pipeline_health` table stops being a silent dead sink |
| **Priority**       | Medium (ops visibility — silent accumulation of health data with no consumer) |
| **Owner agent**    | Unassigned |
| **Time window**    | TBD |
| **Coordination flag** | solo |

---

## Problem

`check_pipeline_health.js` runs on a schedule and writes rows to `pipeline_health` (line ~42, `INSERT INTO pipeline_health`). It is scheduled in `apps/worker/src/schedulers/main.js` (line ~302).

**Zero consumers exist:**
- No `FROM pipeline_health` anywhere in `web/src/`
- No SELECT from `pipeline_health` anywhere in `apps/` or `packages/`

**Impact:**
- Health check results are computed and stored but never seen
- Ops cannot observe pipeline degradation without querying the DB directly
- The table accumulates rows indefinitely

---

## Decision required before implementation

**Option A — Web route (preferred for dashboard integration)**
Add `GET /api/admin/pipeline-health` in `web/src/app/api/admin/pipeline-health/route.ts` returning the N most recent rows per pipeline. Optionally surface in an admin page.

**Option B — Discord emit (preferred for low-overhead alerting)**
In `check_pipeline_health.js`, after the DB write, emit results through the existing Discord webhook (already used elsewhere in the worker). No new web route needed.

**Option C — Both**
Write to DB (existing behavior) + emit Discord alert only when status changes from `ok` to degraded.

---

## Scope

| File | Change |
|------|--------|
| `apps/worker/src/jobs/check_pipeline_health.js` | Option B/C: add Discord emit after DB insert |
| `web/src/app/api/admin/pipeline-health/route.ts` | Option A/C: new GET route returning recent pipeline health rows |
| `packages/data/src/db/` | Add `getPipelineHealth(limit)` getter if not present |

## Out of Scope

- Changes to what `check_pipeline_health.js` checks or how it determines health status
- Schema changes to the `pipeline_health` table
- Authentication/authorization on the admin route (treat as follow-on)

---

## Acceptance

- Option A: `GET /api/admin/pipeline-health` returns JSON array of recent health rows; row count matches DB
- Option B: A Discord message is posted to the ops channel after each health check run
- Either path: `check_pipeline_health.js` continues to write to DB without error

## Tests to run

- `npm --prefix web run lint` (Option A only)

- Option B: Confirm Discord message received in ops channel after a manual check_pipeline_health run

**Guard for WI closeout**: confirm via grep that at least one callsite reads from `pipeline_health` (Option A) or that Discord emit is present (Option B).
