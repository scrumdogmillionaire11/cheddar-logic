# WI-1138: Retire Dual-DB Runtime Paths And Align To Single-Writer Contract

**ID**: WI-1138
**Goal**: Remove runtime ambiguity by hard-disabling dual/multi DB initialization paths so production behavior aligns with ADR-0002 single-writer architecture.

**Scope**:

- packages/data/src/db-dual-init.js
- packages/data/src/db-multi.js
- packages/data/index.js
- packages/data/src/db/connection.js
- docs/decisions/ADR-0002-single-writer-db-contract.md

**Out of scope**:

- Changes to SQL schema, migrations, or lockfile semantics
- Changes to web API route behavior
- Refactoring unrelated package exports

**Implementation**:

**Task 1 — `packages/data/src/db-dual-init.js`** (active import; fail-closed via function-body throws)

`db-dual-init.js` is `require()`d by `index.js` line 17 so the module must remain loadable. Do NOT throw at module load time. Instead, replace the body of every exported function (`initDualDb`, `closeDualDb`, `getDb`, `isDualModeActive`) with:

```js
throw new Error(
  '[WI-1138] Dual-DB mode is disabled. Use packages/data/src/db.js (single-writer worker) ' +
  'or closeDatabaseReadOnly() (read-only web). See ADR-0002.'
);
```

Remove the top-of-file comment block that says "retained for future dual-DB deployment" and replace it with:

```
/* DISABLED (WI-1138): Dual-DB mode is no longer supported.
 * All exported functions throw. See ADR-0002.
 */
```

Keep `RECORD_TABLES` and `LOCAL_TABLES` Set exports as-is (they are data, not init paths).

---

**Task 2 — `packages/data/src/db-multi.js`** (zero callers; dead code; tombstone only)

`db-multi.js` has no callers anywhere in the repo. Replace the top-of-file JSDoc block with a tombstone:

```
/* DISABLED (WI-1138): Dead code — no callers. Dual-DB mode is not supported.
 * See ADR-0002. Do not re-enable without a new ADR.
 */
```

Then replace the body of every exported function with the same throw as Task 1.

**Task 3 — `packages/data/index.js`** (two changes: header comment + dual-init JSDoc)

Change 1 — Remove the "recommended for prod" mode header (lines 7-12):

```
 * DUAL-DATABASE MODE (recommended for prod):
 *   - Record DB: shared reference data (read-only)
 *   - Local DB: environment-specific state (writable)
 *   Usage: await require('.').initDualDb({ recordDbPath, localDbPath })
 *
 * SINGLE-DATABASE MODE (legacy, default):
 *   - Single DB for all data
```

Replace with:

```
 * Single-writer DB contract: worker is sole writer; web is read-only.
 * See ADR-0002. Dual-DB mode is disabled — its exports throw on call.
```

Change 2 — Update the JSDoc block above the dual-init exports (~line 227):

```js
  /**
   * Initialize dual-database mode.
   * Currently unused in production entry points.
   * Single-DB path via resolveDatabasePath is the active runtime path.
   */
```

Replace with:

```js
  /**
   * DISABLED (WI-1138) — these functions throw on call.
   * Dual-DB mode is not supported. See ADR-0002.
   * Retained as exports so call sites fail loudly rather than silently missing a symbol.
   */
```

Do not remove the 6 exports (`initDualDb`, `closeDualDb`, `isDualModeActive`, `getDualDb`, `RECORD_TABLES`, `LOCAL_TABLES`). Keeping them ensures callers get an explicit throw rather than a silent `TypeError: X is not a function`.
