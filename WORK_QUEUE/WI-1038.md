# WI-1038: Feature Timestamp Guard Phase 2 - enforce hard block on future-leakage violations

**ID**: WI-1038
**Goal**: Complete WI-0827 by enforcing a hard block whenever any tracked high-risk feature has `available_at > bet_placed_at`, while preserving diagnostic visibility for operators.

**Depends on**: WI-0827

**Dependency baseline (WI-0827 delivered)**:

- Phase 1 audit path exists: violations are detected and attached as `feature_timeliness` metadata.
- Current behavior is fail-open (warn + attach), not enforcement.
- This WI converts that audit signal into deterministic enforcement.

**Acceptance**:

- Feature timeliness violations are enforcement-level for tracked high-risk fields across MLB, NHL, and NBA: cards impacted by `available_at > bet_placed_at` cannot remain executable.
- Deterministic blocked payload contract for enforcement-triggered cards:
  - `execution_status = BLOCKED`
  - `status = PASS`
  - `action = PASS`
  - `classification = PASS`
  - `pass_reason_code = PASS_FEATURE_TIMESTAMP_LEAK` (operator-facing pass reason; non-canonical token allowed here)
  - `decision_v2.official_status = PASS`
  - `decision_v2.primary_reason_code` must remain an existing canonical registered reason code (do not introduce a new taxonomy token in this WI)
- Existing diagnostics are preserved (violation list remains attached for auditability).
- Null/untracked timestamp policy is explicit and enforced by tests:
  - null/missing `available_at` for tracked fields remains non-blocking (fail-open), but must still be represented in `feature_timeliness` diagnostics;
  - parseable future-dated `available_at` on tracked fields is always blocking.
- MLB, NHL, and NBA flows are all covered by automated tests proving:
  - violation case blocks actionable output;
  - non-violation case preserves current behavior;
  - blocked payload fields match the deterministic contract above.
- `.planning/STATE.md` no longer implies WI-0827 is pending once this follow-up lands.
