# WI-0305: Atomic Snapshot Publishing for Stable Cards API

CLAIM: github-copilot 2026-03-05T16:35:00Z

**ID**: WI-0305

**Goal**: Stop games from disappearing/reappearing on `/cards` during model runs by implementing atomic publish via `run_id` + `current_run_id` pointer.

**Scope**:

- `packages/data/db/migrations/` (new migration: 018_create_run_state.sql)
- `packages/data/src/db-multi.js` (getCurrentRunId, setCurrentRunId, insertRun, getRun helpers)
- `apps/worker/src/utils/` (run tagging helpers in job runners)
- `web/src/app/api/cards/route.ts`
- `web/src/app/api/games/route.ts`
- `web/src/app/api/cards/[gameId]/route.ts`

**Out of scope**:

- Props API (`/api/props-games`, future WI)
- Results page filtering (depends on WI-0306)
- Worker model logic changes (cards built same way, just tagged)
- Response caching layer
- Welcome Home API gating (status quo)

**Acceptance**:

- `/api/cards` and `/api/games` return only plays from `SELECT card_payloads WHERE run_id = (SELECT current_run_id FROM run_state LIMIT 1)`.
- Response JSON includes meta object: `{ success, data, meta: { current_run_id, generated_at, run_status, items_count } }`.
- A failed model run (status=FAILED) does NOT update `current_run_id`; display snapshot stays stable.
- Cards page never shows 0 games due to mid-run writes.
- Latest successful run always visible until replaced by newer successful run.
- Run state persists across restarts (SQLite durability).
