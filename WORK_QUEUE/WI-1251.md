# WI-1251: Enforce Bettable Main-Surface Eligibility In `/api/games`

**ID**: WI-1251
**Goal**: Stop `/api/games` from surfacing games whose only surviving plays are `INFO`-only or non-main-surface rows, so the API reflects actual main-board coverage instead of forcing the frontend to drop them later.

**Context**:

The current `/api/games` service layer keeps a pregame row alive when it has any plays at all. That overstates coverage for two known failure modes: MLB games whose surviving rows are F5-only, and NBA games whose surviving rows are `INFO`/evidence only. The frontend already suppresses those rows on the main board, so the API should fail closed before they inflate coverage counts.

**Scope**:
- `web/src/lib/games/service-layer.ts`
- `web/src/lib/games/route-handler.ts`
- `web/src/__tests__/games-service-layer.test.ts`
- `web/src/__tests__/api-games-market-smoke.test.js`
- `WORK_QUEUE/WI-1251.md`

**Out of scope**:
- Reworking frontend filter UX
- Changing `/api/cards` or prop-only rendering contracts unless required by this exact eligibility rule
- Fixing underlying worker/model generation gaps

**Acceptance**:
- Pregame `/api/games` eligibility distinguishes main-surface bettable plays from `INFO`/evidence-only rows.
- A game is not kept alive solely by `INFO`-only rows or MLB F5-only rows when it lacks any bettable main-surface market the frontend can actually render.
- Existing ingest-failure visibility remains intact so operational diagnostics are not hidden.
- Regression coverage proves the endpoint still keeps genuinely actionable games while dropping the false-coverage cases.
