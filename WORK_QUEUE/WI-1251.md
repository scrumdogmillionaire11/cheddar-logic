# WI-1251: Prevent False Market Coverage in `/api/games`

## ID
WI-1251

## Goal
Ensure `/api/games` only treats a game as having market/card coverage when it has at least one **main-surface bettable** card row. Diagnostic, INFO, evidence-only, or unsupported market rows must not make a game look covered.

## Problem
A game can appear "covered" when the only related rows are non-bettable diagnostics or unsupported market artifacts. That hides real ingest/model/card failures and makes the surface look healthier than it is.

## Depends on
WI-1250

Dependency reason:
WI-1250 adds scheduled-game card coverage watchdog behavior. WI-1251 aligns `/api/games` surface filtering with that same concept so web coverage does not disagree with worker coverage.

## Main-Surface Bettable Contract

A card row counts as `/api/games` market coverage only if all are true:

| Rule | Required |
|---|---|
| Has game_id matching scheduled game | yes |
| Market family is supported on main `/api/games` surface | yes |
| Row is not diagnostic/info/evidence-only | yes |
| Row can produce a visible game-card recommendation or explicit pass reason | yes |

### Allowed coverage rows
- Full-game moneyline
- Full-game spread / puckline
- Full-game total
- Supported main-surface player prop rows only if the current `/api/games` surface already renders them

### Disallowed coverage rows
- `INFO`
- diagnostic-only rows
- evidence-only rows
- shadow/near-miss rows
- unsupported market rows
- MLB F5-only rows if the main `/api/games` surface does not render F5 as primary coverage
- rows excluded by `isRenderableGameSurfacePlay()` or equivalent transform authority

If a row would not be visible or explainable on the main game card, it must not satisfy market coverage.

## Required Behavior

### Case 1 — False coverage removed
Given a scheduled game with only INFO/diagnostic/evidence rows:
- `/api/games` must not mark the game as market-covered
- the game must remain visible only through the existing ingest-failure / no-card diagnostic path, if that path is expected for the route
- no playable card should be fabricated

### Case 2 — Real coverage preserved
Given a scheduled game with at least one supported bettable row:
- `/api/games` marks the game as covered
- the visible card uses the supported row
- diagnostic rows do not override the supported row

### Case 3 — Diagnostics preserved
When false coverage is removed:
- ingest-failure visibility must remain intact
- diagnostic/visibility metadata must remain available through the existing route contract, or the test must assert the documented fallback behavior
- the game must not silently disappear unless current route contract explicitly says uncovered games are excluded

## Implementation plan

1. Identify the current coverage predicate in:
   - `web/src/lib/games/service-layer.ts`
   - `web/src/lib/games/route-handler.ts`

2. Replace broad "has any card row" logic with a main-surface bettable predicate.

3. Reuse the existing game-card renderability authority where possible:
   - `isRenderableGameSurfacePlay()`
   - or an equivalent shared predicate if the service layer cannot import transform code safely

4. Add tests proving:
   - INFO-only rows do not count as coverage
   - diagnostic-only rows do not count as coverage
   - unsupported/F5-only rows do not count unless main surface supports them
   - valid full-game row still counts
   - mixed diagnostic + valid row counts as covered by the valid row only
   - ingest-failure visibility remains intact

## Acceptance
- Coverage logic is based on main-surface bettable rows, not arbitrary card row existence.
- INFO/diagnostic/evidence-only rows cannot satisfy market coverage.
- Unsupported market rows cannot satisfy market coverage.
- Mixed valid + diagnostic rows preserve the valid coverage path.
- False-coverage games keep explicit ingest-failure/no-card visibility behavior.
- Tests include deterministic fixtures for false coverage and real coverage.
- No schema changes.
- No worker changes.
