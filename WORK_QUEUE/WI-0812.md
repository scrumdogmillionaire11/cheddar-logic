# WI-0812: Fix market-call card duplication — deterministic IDs + upsert + UNIQUE constraint

| Field | Value |
|---|---|
| **ID** | WI-0812 |
| **Goal** | Eliminate the ~10x market-call card duplication caused by a random UUID suffix in `buildMarketCallCard`. Fix at all three enforcement layers: (1) make card IDs deterministic so the same game+market always produces the same ID, (2) change `insertCardPayload` to upsert so re-runs update the existing card rather than inserting a new one, (3) add a partial UNIQUE INDEX on `card_payloads(game_id, card_type)` for call-type cards as a DB-level guard, using migration 062 to first collapse the ~7,000 existing duplicate rows. |
| **Priority** | High |
| **Depends on** | None — no other WI in flight touches card-factory or insertCardPayload |
| **Coordination flag** | `needs-sync` for `packages/data/db/migrations/062_deduplicate_card_payloads.sql` |

## Background

Every 30 minutes the worker model runs and calls `buildMarketCallCard(...)` for each in-progress game. That function generates:

```js
// packages/models/src/card-factory.js line 126
const cardId = `card-${cardType}-${gameId}-${uuidV4().slice(0, 8)}`;
```

Every run produces a fresh UUID suffix → a unique `id` → a new row inserted (bare `INSERT`, no conflict handling) into `card_payloads`. For a 25-hour NBA game window running every 30 min, a single game/market accumulates ~50 duplicate card_payload rows, all settling with the same outcome.

**Production evidence (as of 2025-03-07):**

| card_type | unique games | total rows | avg dup ratio |
|---|---|---|---|
| `nba-totals-call` | 122 | 3,359 | 27.5× |
| `nba-spread-call` | 68 | 1,950 | 28.7× |
| `nhl-totals-call` | 52 | 1,256 | 24.2× |
| `nhl-moneyline-call` | 15 | 614 | 40.9× |

Dr. Claire health report surfaced this as ⚠️ NBA 10.2× dup / NHL 6.2× dup warnings.

**Fix strategy — three layers:**
1. **Application — card-factory.js**: drop UUID suffix from `buildMarketCallCard` → ID becomes `card-${cardType}-${gameId}` (stable across runs)
2. **Application — cards.js**: change `insertCardPayload` bare INSERT to an `ON CONFLICT(game_id, card_type) DO UPDATE` upsert that refreshes payload_data unless the card is already settled
3. **DB — migration 062**: collapse existing duplicate rows (keep one per `game_id+card_type`, delete orphaned `card_results`), then add a partial UNIQUE INDEX on `card_payloads(game_id, card_type) WHERE card_type LIKE '%-call'`

### Task 2 — Upsert in `insertCardPayload`

**File:** `packages/data/src/db/cards.js` (around line 674)

**Change the `db.prepare(...)` SQL** from a bare `INSERT` to:

```sql
INSERT INTO card_payloads (
  id, game_id, sport, card_type, card_title, created_at,
  expires_at, payload_data, model_output_ids, metadata, run_id
)
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
ON CONFLICT(game_id, card_type) DO UPDATE SET
  payload_data  = excluded.payload_data,
  run_id        = excluded.run_id,
  created_at    = excluded.created_at,
  expires_at    = excluded.expires_at
WHERE NOT EXISTS (
  SELECT 1 FROM card_results
  WHERE card_id = card_payloads.id
    AND status = 'settled'
)
```

**Semantics:**
- **New game/market (no conflict):** plain insert, just like today.
- **Re-run for in-progress game (conflict on `game_id+card_type`):** update `payload_data`, `run_id`, `created_at`, and `expires_at` with latest odds/model output. The existing `id` and its `card_results` FK row are preserved (no PK mutation, safe with `foreign_keys = ON`).
- **Already-settled card (conflict, but settled):** the `WHERE NOT EXISTS` clause prevents the update; the re-run insert is silently ignored.

**Note:** The `ON CONFLICT(game_id, card_type)` clause in the prepared statement is a silent no-op without the UNIQUE INDEX from migration 062. **Migration 062 must be applied before this code is deployed.** Verify the index exists before testing: `SELECT name FROM sqlite_master WHERE type='index' AND name='uq_card_payloads_call_per_game';`

**Done criteria:**
- Unit test: insert same call card twice → second insert updates `payload_data`, does not create second row
- Unit test: insert same call card after it has a `settled` card_result → second insert is ignored, `payload_data` unchanged
- Unit test: insert a driver card (non-call type, no UNIQUE INDEX) → both inserts succeed as separate rows (existing behavior unchanged until driver-card WI)
- Unit test: re-insert of same call card produces exactly one `card_results` row (`INSERT OR IGNORE` in `results.js` absorbs the duplicate)

### Task 3 — Migration 062: deduplicate + UNIQUE INDEX

**File:** `packages/data/db/migrations/062_deduplicate_card_payloads.sql`

```sql
-- Migration 062: Deduplicate market-call card_payloads and enforce uniqueness
-- Root cause: buildMarketCallCard used a random UUID suffix, causing every
-- 30-min model run to insert a fresh card for the same game+market.
--
-- Strategy:
--   1. For each (game_id, card_type) group among %-call cards:
--      - Identify the canonical row = earliest created_at (most historical context)
--      - Delete all card_results rows that reference the non-canonical duplicates
--      - Delete the non-canonical card_payload rows
--   2. Add a partial UNIQUE INDEX so this can never re-accumulate.
--
-- Safe to run multiple times (idempotent via IF NOT EXISTS on index).

-- Step 1a: Delete card_results rows for non-canonical call-card duplicates.
-- "Non-canonical" = not the earliest-created row for that (game_id, card_type) pair.
DELETE FROM card_results
WHERE card_id IN (
  SELECT cp.id
  FROM card_payloads cp
  WHERE cp.card_type LIKE '%-call'
    AND cp.id != (
      SELECT cp2.id
      FROM card_payloads cp2
      WHERE cp2.game_id   = cp.game_id
        AND cp2.card_type = cp.card_type
        AND cp2.card_type LIKE '%-call'
      ORDER BY cp2.created_at ASC
      LIMIT 1
    )
);

-- Step 1b: Delete the non-canonical card_payload rows themselves.
DELETE FROM card_payloads
WHERE card_type LIKE '%-call'
  AND id != (
    SELECT cp2.id
    FROM card_payloads cp2
    WHERE cp2.game_id   = card_payloads.game_id
      AND cp2.card_type = card_payloads.card_type
      AND cp2.card_type LIKE '%-call'
    ORDER BY cp2.created_at ASC
    LIMIT 1
  );

-- Step 2: Add partial UNIQUE INDEX — enforces one call card per game+market
-- going forward. Partial (WHERE LIKE '%-call') so driver cards are unaffected.
CREATE UNIQUE INDEX IF NOT EXISTS uq_card_payloads_call_per_game
  ON card_payloads(game_id, card_type)
  WHERE card_type LIKE '%-call';
```

**Verification queries to run on staging/Pi before/after:**

```sql
-- Before: count per group (should be ~7,100 total for call cards)
SELECT card_type, COUNT(DISTINCT game_id) as games, COUNT(*) as rows,
       ROUND(COUNT(*)*1.0/COUNT(DISTINCT game_id),1) as avg_dup
FROM card_payloads WHERE card_type LIKE '%-call' GROUP BY card_type;

-- After: every avg_dup should be 1.0
-- Also confirm card_results not over-deleted:
SELECT COUNT(*) FROM card_results WHERE status='settled';
```

**Done criteria:**
- Migration runs without error on production DB
- Post-migration: all four call card types show avg_dup = 1.0
- Post-migration: settled card_results count matches pre-migration (one result per surviving canonical card)
- Dr. Claire report shows no ⚠️ dup warnings for next model run cycle
