# WI-0998: Replace EDGE_VERIFICATION_REQUIRED with decision-grade blocker contract

**ID**: WI-0998

**Goal**: Replace `EDGE_VERIFICATION_REQUIRED` with explicit, bettor-legible blocker reasons and enforce user-facing no-play/watch messaging that states cause, transition trigger, and current call.

## Background

Current user-facing output can include:

- `Main blockers: SUPPORT_BELOW_LEAN_THRESHOLD, EDGE_VERIFICATION_REQUIRED`

`SUPPORT_BELOW_LEAN_THRESHOLD` is concrete. `EDGE_VERIFICATION_REQUIRED` is internal process language and does not explain what is actually wrong in betting terms.

This work item implements a decision-grade blocker contract so no-play/hold/watch states report concrete reasons only:

- `LINE_NOT_CONFIRMED`
- `EDGE_RECHECK_PENDING`
- `EDGE_NO_LONGER_CONFIRMED`
- `MARKET_DATA_STALE`
- `PRICE_SYNC_PENDING`

And maps all blocker codes to short user-facing text for Discord/card output.

## Decision Contract

### Action state (separate from blocker reasons)

- `PLAY`
- `LEAN`
- `WATCH`
- `PASS`

### Blocker reasons

- `SUPPORT_BELOW_LEAN_THRESHOLD`
- `LINE_NOT_CONFIRMED`
- `EDGE_RECHECK_PENDING`
- `EDGE_NO_LONGER_CONFIRMED`
- `MARKET_DATA_STALE`
- `PRICE_SYNC_PENDING`

### Invariants

- Internal verification steps may remain in decision logic.
- User-facing output must never expose generic `EDGE_VERIFICATION_REQUIRED`.
- Every watch/pass/no-play explanation must answer:
  - what is wrong,
  - what would change the status,
  - what the current call is.
- Do not manufacture action: if thresholds are not met, `official_plays` stays empty.

## Acceptance

- Decision pipeline no longer emits `EDGE_VERIFICATION_REQUIRED` as a user-facing blocker reason.
- Verification-related blockers are emitted as concrete reason codes:
  - `LINE_NOT_CONFIRMED`
  - `EDGE_RECHECK_PENDING`
  - `EDGE_NO_LONGER_CONFIRMED`
  - `MARKET_DATA_STALE`
  - `PRICE_SYNC_PENDING`
- Formatter layer maps blocker codes to plain user language (no raw internal enum leakage in normal mode).
- WATCH/PASS messaging includes:
  - reason(s),
  - deterministic transition trigger (`Would become ...` or `Waiting on ...`),
  - explicit current call (`No bet yet` / `No action`).
- Example behavior enforced:
  - stale or unverified line -> WATCH/PASS with `Line not confirmed` or `Market data stale`
  - failed recheck -> PASS with `Edge no longer clears threshold`
  - verified edge + support meets floor -> LEAN/PLAY with no verification blocker shown
- Documentation reflects the new blocker contract and messaging guidance.

## Implementation notes

1. Replace generic verification blocker emission in pipeline logic with explicit cause-specific blocker codes.
2. Add/centralize a strict blocker-to-user-text mapper in formatter output path.
3. Enforce separation between action state (`PLAY/LEAN/WATCH/PASS`) and blocker reasons.
4. Add anti-leak test coverage: fail tests if `EDGE_VERIFICATION_REQUIRED` or other raw internal codes leak to user-facing text (except explicit debug mode).
5. Reuse NHL-style `Waiting on` phrasing pattern for WATCH when there is a concrete upgrade path.
6. WATCH vs PASS rule:
   - use `WATCH` only when a concrete, near-term upgrade trigger exists,
   - otherwise use `PASS` with explicit reasons.
