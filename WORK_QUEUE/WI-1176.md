# WI-1176: MLB Pitcher Game Log Field Audit — Use Command/Context Features, Cut Dead Run-Allowed Fields

**ID**: WI-1176
**Goal**: Convert useful `mlb_pitcher_game_logs` fields into explicit pitcher-K model features and remove truly dead fields from the recurring fetch/write path.

**Depends on**: none
**Release classification**: model-input cleanup / quota reduction
**Owner agent**: unassigned
**Time window**: TBD
**Coordination flag**: solo

---

## Summary

`mlb_pitcher_game_logs` currently stores fields that are either unused or only partially wired. Keep and model the useful ones:

* `walks`
* `batters_faced`
* `home_away`

Cut or stop refreshing:

* `hits`
* `earned_runs`

Rationale: `walks` + `batters_faced` create a real command/leash signal; `home_away` can support small context adjustments. `hits` and `earned_runs` are noisy outcome stats and should not stay in the live API path unless a model feature explicitly consumes them.

## Current Problem

The game log fetch writes these fields:

```text
walks
hits
earned_runs
home_away
batters_faced
```

But the live model path does not consistently consume them as first-class features.

This creates two bad outcomes:

1. API/data pipeline waste
2. hidden feature debt where useful data exists but does not influence decisions

---

## Decision

### Keep and use

#### `walks`

Used to compute recent command profile.

#### `batters_faced`

Used as denominator for recent walk rate.

#### `home_away`

Used as small context/split adjustment or confidence flag.

### Stop fetching/writing or mark deprecated

#### `hits`

Do not use for pitcher-K confidence in this WI.

#### `earned_runs`

Do not use for pitcher-K confidence in this WI.

If schema removal is risky, leave columns in DB but stop updating them and mark deprecated in docs.

## New Feature Contract

Add derived fields to pitcher-K projection input:

```ts
type MlbPitcherRecentCommandContext = {
  recent_walks: number | null;
  recent_batters_faced: number | null;
  recent_bb_pct: number | null;
  recent_bb_pct_status: "OK" | "MISSING" | "SMALL_SAMPLE";
  home_away_context: "HOME" | "AWAY" | "MIXED" | "UNKNOWN";
  command_risk_flag: boolean;
};
```

---

## Feature Rules

### 1. Recent BB%

Compute from recent game logs:

```text
recent_bb_pct = sum(walks) / sum(batters_faced)
```

Use the same lookback window already used for pitcher-K recent logs, likely last 10 starts.

### 2. Small sample guard

```text
if recent_batters_faced < 40:
  recent_bb_pct_status = SMALL_SAMPLE
```

Small sample should not heavily affect projection. It may affect confidence only.

### 3. Missing guard

```text
if walks is null OR batters_faced is null:
  recent_bb_pct_status = MISSING
```

Missing data should not crash the model. It should add a degradation reason.

### 4. Command risk flag

Initial thresholds:

```text
recent_bb_pct >= 0.095 -> command_risk_flag = true
recent_bb_pct < 0.095 -> command_risk_flag = false
```

Suggested effect:

```text
if command_risk_flag:
  reduce pitcher-K confidence
  apply small projection penalty
```

Initial penalty:

```text
k_projection_adjustment = -0.15 Ks
confidence_penalty = -5 points
```

Keep this small. Walk rate is useful, but not a hammer.

### 5. Home/away context

From recent logs:

```text
if >= 70% recent logs are HOME -> HOME
if >= 70% recent logs are AWAY -> AWAY
else MIXED
```

Suggested use:

```text
if current game context conflicts with dominant recent context:
  confidence_penalty = -2 points
```

Do not move the projection heavily from `home_away` alone.

## Implementation Plan

### Step 1 — Audit fetch/write source

Find the job that pulls MLB pitcher game logs.

Confirm where these fields are fetched:

```text
walks
hits
earned_runs
home_away
batters_faced
```

Update fetch/write behavior:

* keep `walks`
* keep `batters_faced`
* keep `home_away`
* stop requesting/writing `hits`
* stop requesting/writing `earned_runs`

If the API response naturally includes all fields and there is no quota impact per field, then leave response parsing alone but stop persisting `hits` and `earned_runs`.

### Step 2 — Update `buildPitcherStrikeoutLookback`

Ensure it returns:

```js
walks
batters_faced
home_away
```

Do not return:

```js
hits
earned_runs
```

Add derived aggregate helper:

```js
buildPitcherRecentCommandContext(logs)
```

### Step 3 — Wire command context into pitcher-K projection

Add `recent_command_context` to pitcher object or projection input.

Example:

```js
pitcher.recent_command_context = buildPitcherRecentCommandContext(
  pitcher.strikeout_history
);
```

### Step 4 — Apply model adjustment

In pitcher-K projection logic:

```js
if (recent_command_context.command_risk_flag) {
  projection -= 0.15;
  confidence -= 5;
  reason_codes.push("COMMAND_RISK_RECENT_BB_RATE");
}
```

For small sample:

```js
if (recent_bb_pct_status === "SMALL_SAMPLE") {
  confidence -= 2;
  reason_codes.push("COMMAND_CONTEXT_SMALL_SAMPLE");
}
```

For missing:

```js
if (recent_bb_pct_status === "MISSING") {
  reason_codes.push("COMMAND_CONTEXT_MISSING");
}
```

Do not auto-pass on missing command context.

### Step 5 — Add docs

Update pitcher-K input docs with:

* retained fields
* deprecated fields
* formulas
* thresholds
* penalties
* reason codes

### Step 6 — Add tests

Add unit tests for:

* BB% computation
* small sample behavior
* missing data behavior
* command risk penalty
* home/away context penalty
* deprecated fields not required

## Acceptance Criteria

* `walks`, `batters_faced`, and `home_away` are explicitly consumed by pitcher-K feature generation.
* `recent_bb_pct` is computed from `sum(walks) / sum(batters_faced)`.
* High recent BB% applies a small K projection or confidence penalty.
* Missing command context does not crash or auto-pass.
* `hits` and `earned_runs` are removed from the active model input contract.
* Fetch/write path no longer treats `hits` and `earned_runs` as required fields.
* Docs explain why `hits` and `earned_runs` are excluded.
* Tests prove the command features affect model output.
* Tests prove `hits` and `earned_runs` are not required.

## Failure Modes & Guards

| Failure mode                                     | Guard                                                                   |
| ------------------------------------------------ | ----------------------------------------------------------------------- |
| Walk rate over-penalizes good strikeout pitchers | Small initial penalty only: `-0.15 Ks`, `-5 confidence`                 |
| Small sample creates false command risk          | `recent_batters_faced < 40` → `SMALL_SAMPLE`, confidence-only penalty   |
| Missing logs crash model                         | Missing context adds reason code, no crash                              |
| Hits/earned runs still fetched forever           | Fetch/write test or code audit confirms they are not persisted/required |
| Home/away overfits                               | Tiny confidence penalty only, no direct projection swing                |
| Feature silently unused again                    | Unit test asserts high BB% changes output                               |
