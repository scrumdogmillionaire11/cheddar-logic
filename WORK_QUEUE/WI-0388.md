# WI-0388: Derived Lifecycle Mode on `/api/games` + Cards Toggle

**ID**: WI-0388
**Goal**: Reduce `/cards` noise with a derived board lifecycle layer (pre-game vs active) on `/api/games`, while keeping settlement and `/results` unchanged.

**Scope**:

- `WORK_QUEUE/WI-0388.md`
- `web/src/app/api/games/route.ts`
- `web/src/components/cards-page-client.tsx`
- `web/src/__tests__/api-games-filter.test.js`
- `web/src/__tests__/api-games-repair-budget.test.js`
- `web/src/lib/api-security/validation.ts`

**Out of scope**:

- `/api/cards` and `/api/cards/[gameId]` behavior changes
- Worker model/job logic changes
- DB schema or migration changes
- `/api/results` grading/settlement behavior changes
- Introducing `game_results` as a new dependency for board lifecycle filtering

**Acceptance**:

- `/api/games` defaults to `lifecycle=pregame` (`game_time_utc > now`) so started games are omitted.
- `/api/games?lifecycle=active` returns started games (`game_time_utc <= now`) while preserving settled-game exclusion.
- Active mode excludes clearly non-live statuses when present (`postponed`, `cancelled`/`canceled`, `final`, `closed`, `complete`).
- `/api/games` response includes derived lifecycle fields (`lifecycle_mode`, `display_status`) so UI does not depend on raw backend status strings.
- Cards page exposes a lifecycle toggle (Pre-Game / Active), with URL param as source-of-truth and session storage fallback.
- Settlement remains unchanged: `/results` continues as the settled outcomes surface.
- Non-prod diagnostics include applied lifecycle mode and timestamp used for filtering.
