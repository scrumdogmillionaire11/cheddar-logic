# WI-1249: Add Behavior Regression for INFO Play Main-Surface Exclusion

## ID
WI-1249

## Goal
Guarantee the main /api/games game-card transform never promotes INFO plays into the primary game surface and never renders INFO-only games as playable game cards.

## Problem
The existing guard exists in transform code, but the proposed verification relies on source-string assertions in games-pipeline-contract.test.ts. That can pass while behavior is still broken.

This work item must prove runtime behavior by executing the transform against concrete fixtures.

## Scope
- web/src/lib/game-card/transform/index.ts
- web/src/__tests__/game-card-decision-authority.test.ts
- web/src/__tests__/integration/games-pipeline-contract.test.ts only if keeping a lightweight static contract
- WORK_QUEUE/WI-1249.md

## Out of Scope
- Changing decision pipeline classifications
- Changing worker output
- Changing /api/results
- Changing Discord/POTD behavior
- Broad refactor of game-card transform

## Required Behavior Contract

### 1. Mixed game behavior
Given a game containing:
- one INFO/diagnostic/non-renderable play
- one valid renderable play

Then:
- transformGames() must return the game
- primary/surface play must be the renderable play
- no INFO play may become primaryPlay, driver, top pick, or visible main recommendation

### 2. INFO-only game behavior
Given a game containing only:
- INFO/diagnostic/non-renderable plays

Then:
- transformGames() must exclude the game from the main surface
- exclusion reason must be explicit
- no empty playable card may be emitted

### 3. Guard invariant
isRenderableGameSurfacePlay() remains the authority for deciding whether a play can appear on the main game surface.

## Implementation Plan

### 1. Add behavior-level regression tests
Use web/src/__tests__/game-card-decision-authority.test.ts or create a nearby transform behavior test.

Tests must import and execute:
- transformGames() preferred
- transformToGameCard() only if needed for narrower proof

Do not rely only on fs.readFileSync() or string includes.

### 2. Add mixed fixture
Create a fixture where:
- decision_status = "INFO" or equivalent INFO marker exists
- another play has a real market type such as TOTAL, MONEYLINE, SPREAD, etc.
- assert selected surface play is not INFO

### 3. Add INFO-only fixture
Create a fixture where all plays are INFO/non-renderable.
Assert:
- transformed game list is empty, or
- game is excluded by explicit exclusion result if that is the existing API shape

### 4. Keep static test optional
games-pipeline-contract.test.ts may retain a source-level invariant, but it cannot be the only proof.

## Acceptance
- Runtime test proves mixed INFO + renderable game surfaces only the renderable play.
- Runtime test proves INFO-only game is excluded from main /api/games transform output.
- Tests execute successfully using repo-valid commands.
- No test relies solely on source-string assertions.
- Existing behavior for valid non-INFO games remains unchanged.
- Exclusion/pass reason is explicit, not silent.

### Completion Conditions
This WI may be closed under either condition:

1. Regression Found + Fixed
   - New tests fail against current behavior
   - Code is patched
   - Tests pass

2. No Regression (Guard Already Correct)
   - New tests pass without code changes
   - Behavior is proven correct
   - WI closes as coverage added
