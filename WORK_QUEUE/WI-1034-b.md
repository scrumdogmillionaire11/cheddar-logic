# WI-1034-b: Implement NHL verification contract and blocker registration

**ID**: WI-1034-b
**Goal**: Implement the NHL worker-layer side of the verification contract defined in `WI-1034.md` and documented by `WI-1034-a`. Register required blocker codes, attach structured `CandidateVerification` state to NHL candidates, enforce gate semantics, and ensure `CLEARED` is re-evaluation eligibility (never auto-`PLAY`).

**Depends on**: WI-1034-a

**Scope**:

- `WORK_QUEUE/WI-1034-b.md`
- `packages/models/src/decision-pipeline-v2.js` (register new blocker/action codes)
- `apps/worker/src/jobs/run_nhl_model.js` (NHL candidate verification object, gate checks, resolver wiring)
- `apps/worker/src/jobs/__tests__/run_nhl_model.*` (or nearest existing NHL worker tests)
- `.planning/STATE.md` (claim/update only)

**Out of scope**:

- MLB implementation (`run_mlb_model.js`) - covered in WI-1034-c
- Vegas-agent docs - covered in WI-1034-a
- Web/API transforms and UI rendering
- DB schema migrations
- Cross-sport watchdog rollout beyond NHL path

**Implementation requirements**:

- Register codes in `decision-pipeline-v2.js` (no duplicates):
  - `WATCHDOG_REASONS`: `STARTER_UNCONFIRMED`, `STARTER_MISMATCH`
  - `PRICE_REASONS`: `PRICE_STALE`
  - soft blockers if used in NHL path: `BEST_LINE_UNCONFIRMED`, `WEATHER_STATUS_PENDING`, `MARKET_SOURCE_UNCONFIRMED`
- NHL candidate output includes structured verification object with:
  - `verification_state = NOT_REQUIRED | PENDING | CLEARED | FAILED | EXPIRED`
  - `requirements[]` entries carrying `blocker_code`, `severity`, `status`, `unblock_condition`, `action_type`, `retry_policy`
  - `next_action_summary` when state is `PENDING`
- Gate behavior:
  - User-facing output remains first-fail (`PASS - [REASON_CODE]: ...`) for brevity
  - Internal state captures all cheap-to-detect blockers in the same pass
- Verdict semantics:
  - Keep verdict vocabulary as `PLAY/LEAN/PASS/FADE`
  - Pending verification uses `LEAN` + `verification_state=PENDING`
  - True weak lean remains `LEAN` + `verification_state=CLEARED|NOT_REQUIRED`
- Resolver semantics:
  - `CLEARED` means eligible for re-evaluation only
  - `CLEARED` must not directly emit or force `PLAY`
  - Re-run edge/threshold checks before `PLAY`
- Threshold policy:
  - Price/adverse-move defaults may start at 5 cents
  - Threshold values must be configurable by sport/market (not hard-coded doctrine)

**Acceptance**:

- New code registrations compile and `_assertPipelineCodesRegistered()` passes at startup.
- NHL verification candidates carry non-empty `requirements` whenever `verification_state=PENDING`.
- Pending verification with empty `next_action_summary` is rejected/flagged as contract violation.
- First-fail user output behavior is preserved, while internal requirement list can contain >1 blocker from same evaluation pass.
- `CLEARED` path triggers re-evaluation and does not auto-promote directly to `PLAY`.
- Existing NHL tests pass; add/adjust tests to cover:
  - `LEAN + verification_state=PENDING`
  - `LEAN + verification_state=CLEARED|NOT_REQUIRED`
  - `CLEARED != PLAY`
  - configurable threshold behavior (default + override)
