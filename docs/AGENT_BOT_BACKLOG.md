# Agent Bot Backlog

Conversational AI interface for cheddar-logic — inspired by OpenClaw's model of a persistent, chat-accessible agent running on your own infrastructure.

---

## Context

OpenClaw demonstrated that a Claude agent accessible via messaging apps (Telegram/Discord), with persistent memory and the ability to trigger real actions, is genuinely useful. This document captures how those ideas apply to cheddar-logic specifically.

**What we already have that applies:**
- Scheduler with hourly jobs (`scripts/start-scheduler.sh`)
- Discord webhook notifications (snapshots at 09:00/12:00/18:00 ET)
- Worker that executes model runs and ingestion jobs
- Clean web API (`/api/cards`, `/api/games`)
- SQLite DB with structured card/game data

**What we're missing:** a bidirectional interface — something that can receive questions and commands, not just push outputs.

---

## Tier 1 — Bidirectional Discord/Telegram bot

**Value:** High | **Effort:** Low | **Recommended first step**

Turn the existing one-way Discord notification channel into a two-way command interface. Add Telegram as a mobile-first alternative.

### Capabilities

| Command pattern | What it does |
|---|---|
| "what are tonight's best NHL plays?" | Queries `card_payloads`, returns top edges formatted by Claude |
| "run the NBA model now" | Triggers worker job, replies when done |
| "show games where our total is >3 pts off market" | Translates NL to SQL/API query, returns results |
| "what's our record on NHL totals this month?" | Queries settlement history, formats summary |

### Implementation notes

- New `apps/bot/` process (Node.js, runs alongside worker)
- **Reads:** calls existing web API endpoints — no direct DB access from bot layer
- **Actions:** shells out to worker scripts or hits an internal job-trigger endpoint on the worker
- Claude API handles natural language → structured query/action translation (tool use pattern)
- Telegram: use `telegraf` library
- Discord: extend existing webhook setup with `discord.js` slash commands

### Decisions needed

- Single channel (Discord only, since we already have it) or also Telegram?
- Which actions should be allowed remotely vs. local-only? (e.g., allow "run model" but not "delete records")

## Tier 2 — Proactive market alerts + morning brief

**Value:** High | **Effort:** Medium

Extend the scheduler with smarter, event-driven notifications — not just time-triggered snapshots.

### Capabilities

**Morning brief (replace or augment existing snapshot)**
- Daily summary: today's play count, confidence distribution, anything flagged for watch
- Generated by Claude from card_payloads data, not a raw dump

**Line move alerts**
- When a card's edge crosses a threshold mid-day (pass → lean, lean → official)
- When market moves >1.5 pts against a current play
- Configurable thresholds per sport

**Edge decay warnings**
- Notify when a line has moved enough that a previous official play is now borderline

### Implementation notes

- New scheduler job types: `morning-brief`, `line-move-monitor`
- Line move monitor polls odds ingest diff at configurable interval (e.g., every 30 min)
- Alert content generated by Claude with access to current card state + prior snapshot
- Delivers via same Telegram/Discord channel as Tier 1

## Tier 3 — Remote job control

**Value:** Medium | **Effort:** Low (given Tier 1 bot exists)

On-demand triggering of worker jobs from your phone. Several OpenClaw users run autonomous Claude Code loops from Telegram — the equivalent here is kicking off a model run and getting a summary when it finishes.

### Capabilities

| Command | Maps to |
|---|---|
| "run NHL model" | Triggers NHL worker job |
| "ingest latest odds" | Triggers odds ingestion pipeline |
| "run all models" | Full scheduler pass, not waiting for cron |
| "what jobs ran today?" | Returns scheduler job log summary |

### Implementation notes

- Bot layer sends job request to worker via internal HTTP endpoint (e.g., `POST /internal/jobs/trigger`)
- Worker emits status events back to bot layer on completion
- Simple job queue: array in memory or a `bot_jobs` table in SQLite — no need for Redis/BullMQ at this scale
- Auth: restrict trigger endpoint to localhost only (bot and worker colocate on same machine)

---

## Tier 4 — Persistent memory

**Value:** Medium | **Effort:** Low

The bot remembers context across sessions — preferences, past queries, things you've told it.

### Capabilities

- Remembers preferred sports, time zones, alert thresholds
- Tracks conversation context so follow-up queries work ("what about the under on that game?")
- Stores user-provided context ("I only care about plays with >3% edge")

### Implementation notes

- Add `bot_memory` table to existing SQLite DB (key/value + embedding if needed later)
- Single writer contract: bot process writes to its own table, does not touch worker tables
- Context window management: summarize older memories, keep recent N turns in active context

## What to skip (for now)

| OpenClaw feature | Why it doesn't fit here |
|---|---|
| Browser control | Data is already in structured DB via APIs — no need to scrape |
| File system access | Not a use case for a sports analytics tool |
| WhatsApp / iMessage | Discord + Telegram covers the need; additional integrations add surface area |
| Self-modifying skills | Useful for a personal assistant, unnecessary complexity here |

---

## Recommended build order

1. **Telegram bot + basic card query** (Tier 1, read-only) — ~200 lines, proves the concept
2. **Discord slash commands** (Tier 1, extends existing infra)
3. **Remote job trigger** (Tier 3, natural extension once bot exists)
4. **Morning brief via Claude** (Tier 2, replace raw snapshot with generated summary)
5. **Line move alerts** (Tier 2, event-driven)
6. **Persistent memory** (Tier 4, polish layer)

## Stack

| Layer | Library | Notes |
|---|---|---|
| Telegram bot | `telegraf` | Simple, Node-native |
| Discord commands | `discord.js` | Extends existing webhook infra |

| Memory | Existing SQLite DB (`bot_memory` table) | Respects single-writer contract |
| Job scheduling | Existing scheduler | New job types, same runner |
